# 领域建模写作 Skill

本 skill 规定本仓库**领域建模（Domain Model）类文档**的统一骨架、符号语义、与事件风暴的对齐规则和质量门槛，保证不同限界上下文的领域模型在词汇、粒度、章节结构上**互相对齐、可直接对话**，并与各自的事件风暴文档**一一可追溯**。

> **前置纪律（来自项目 CLAUDE.md）**：当用户指定要做某个模块时，**只读该模块自身文档与其直接对接的领域（domain），不把其它模块杂糅进来**。领域建模只承载本限界上下文的聚合与行为，不代写相邻限界上下文的模型。

> **术语约定（全文强制，与 [event-storming](../event-storming/SKILL.md) skill 一致）**：
> - **限界上下文（Bounded Context）**：领域建模的建模单元与模型边界，是解决方案空间的概念。为行文简洁，下文以「**本上下文**」指代当前限界上下文，以「**相邻上下文**」指代其它限界上下文，以「**上下文外**」指代当前限界上下文之外。正文中具体限界上下文统一写作「XX限界上下文」，**文件名**按仓库既有约定写作 `XX上下文.md`。
> - **领域（Domain）/ 子域（Subdomain）**：问题空间的业务范围及其细分，不与限界上下文混用。

---

## 1. 何时使用本 skill

- 用户说"做 / 写 / 补 / 改 XX 限界上下文的领域建模 / 领域模型"。
- 用户给出一个限界上下文名称，需要把已落地的事件风暴**翻译**为聚合根 / 实体 / 值对象 / 领域服务 / 应用服务 / 领域事件 / 不变规则。
- 用户要把一段事件链（命令→事件→策略→读模型）落实为可指导编码的面向对象模型。

**强前置**：本 skill **必须以已存在的事件风暴文档为建模依据**。若该限界上下文的事件风暴尚不存在，**先停止**，提示用户先走 [event-storming](../event-storming/SKILL.md) skill 落地事件风暴，再回来做领域建模——事件风暴是源，领域模型是派生，跳过事件风暴直接建模会丢失事件/策略/上下文外消费的覆盖度。

不适用：纯接口契约、纯数据库表结构、纯消息实现细节——那些走对应的实现说明文档。

## 2. 文档落位规则

领域建模文档统一落到 [领域模型/](../../../领域模型/) 下，与事件风暴**同服务不同子目录**平行存放：

```
领域模型/<服务>/事件风暴/<限界上下文>.md      ← 源（建模依据）
领域模型/<服务>/领域建模/<限界上下文>.md      ← 本 skill 产物
```

完整目录结构（与 [领域总览](../../../领域模型/领域总览.md) 一致，仅三个服务）：

```
领域模型/
├── 制造资源服务/领域建模/<限界上下文>.md
├── 生产执行服务/领域建模/<限界上下文>.md
└── 设备管理服务/领域建模/<限界上下文>.md
```

> 制造资源服务下若需按子域细分，可在 `领域建模/` 下再加一层 `<子域>/` 子目录；其它服务直接以限界上下文为文件名，不再嵌套。

**新建文件前必做三件事**：
1. `ls` 对应服务的 `领域建模/` 目录确认有没有同名旧文件。**若旧文件存在但归属变了，先与用户确认是改写还是新建**。
2. `ls` 对应服务的 `事件风暴/` 目录确认事件风暴源文件存在——不存在则按 §1 强前置处理。
3. 文档头部"建模依据"链接必须指向该事件风暴源文件（相对路径 `../事件风暴/XX上下文.md`）。

> **路径提示**：本 skill 文件位于 `.claude/skills/domain-modeling/`，指向 `领域模型/` 的相对路径需写作 `../../../领域模型/`（三层 `../` 回到仓库根）。

## 3. 从事件风暴到领域模型的映射规则（核心）

领域模型不是凭空设计，而是把事件风暴里已经识别出的贴纸**逐一对位**到面向对象结构。下表是**强制对齐表**，建模时逐行落实，不得遗漏：

| 事件风暴贴纸（源） | 领域建模对应（产物） | 落实要点 |
|--------------------|--------------------|----------|
| **【聚合:Xxx】** | 聚合根（§1）或其内实体 | 一个事件风暴聚合 → 至少一个聚合根；若聚合内有独立生命周期的部分，拆为实体 |
| **【事件:XxxHappened】** | 领域事件（§6）+ 聚合根行为发出 | 事件名双语照搬事件风暴，不改名；每个事件必须有"发出者"列 |
| **【命令:DoXxx】** | 聚合根行为方法（§1）/ 应用服务入口（§5） | 单聚合命令→聚合根方法；跨聚合命令→领域服务编排；CRUD 命令按事件风暴规范改业务动词 |
| **【策略:当 X 发生时自动 Y】** | 领域服务（§4）或应用服务内编排 | 跨聚合策略→领域服务；单聚合内部策略→聚合根方法或应用服务编排；**不得挂空** |
| **【参与者:XXX】** | 应用服务的调用方（§5 说明） | 参与者不进模型，只在应用服务注释里标注触发来源 |
| **【外部系统】** | 事件消费应用服务（§5）+ ACL | 上下文外协作方通过事件订阅/REST 查询接入，外部模型经防腐层翻译为本上下文语义 |
| **【读模型】** | 查询应用服务（§5）+ 读模型投影 | 读模型不进聚合，单列查询服务方法 |
| **【热点】🔴** | 不变规则（§7）里的待确认项 或 暂缓说明 | 热点不得静默消失；要么固化为 INV，要么在"暂缓模块说明"留痕 |
| 事件风暴 §聚合与不变式 | 不变规则 INV-XX（§7） | 事件风暴的不变式逐条迁入 INV 编号；跨聚合不变式注明"由谁保证" |
| 事件风暴 §对外契约 | 对外契约发布主题（§8） | 主题名、`<svc>` 前缀、主消费方照搬，保持契约面稳定 |
| 事件风暴 §状态流转 | 聚合根状态机（§1 内 + 值对象枚举） | 状态机的每条边→聚合根方法 + 前置条件；终态→INV |

> **可追溯性硬约束**：领域建模文档 §10"与事件风暴的映射"必须给出一张**贴纸→模型元素**的对位表，让读者能从任何一个事件风暴贴纸查到它落到了哪个类/方法/INV。这是本 skill 的验收门槛之一。

> **反向一致性**：若建模过程中发现事件风暴有遗漏（如某命令缺前置条件、某事件无发出者、某策略无归属），**先回去补事件风暴**，再回来改模型——不要在领域模型里私自补事件，那会让两份文档失同步。

## 5. 写作规范（关键）

1. **面向对象，不是面向过程**（CLAUDE.md 硬约束）：聚合根行为是**方法**，不是 `update/set/save` 流水账。方法名用业务动词（`commission` / `suspend` / `reinstate` / `retire`），与事件风暴命令一致。
2. **单一职责（SRP）**：每个聚合根/服务只解决一个变化原因。把跨聚合逻辑拎到领域服务，别堆在聚合根里。
3. **接口隔离（ISP）**：三类资产（设备/工装/计量器具）共享主链但不强迫实现对方特有方法——特有行为各放各的聚合根，共享行为通过共享值对象/事件表达。
4. **行为必须发出事件**：每个改动状态的方法标注 `→ 发出 XxxHappened`；只读方法不发事件。事件名与事件风暴逐字一致。
5. **前置条件显式写**：`（前置：当前为 IN_SERVICE）` 之类，写在方法后。前置条件对应 INV，可双向查。
6. **值对象不可变**：值对象一旦创建不改字段；可变状态走聚合根 + 事件。状态枚举值对象附状态机。
7. **不变式可验证**：写得能让人写出一个失败用例。"必须先有 X 事件才能发生 Y" 合格；"数据要正确" 不合格。
8. **跨聚合不变式注明"由谁保证"**：领域服务 / 唯一索引 / Saga / 事件去重表四选一，**禁止挂在聚合根上**（与事件风暴 §5.5 一致）。
9. **领域服务必须回答"为什么不放在聚合根内"**：答不上来就不是领域服务，要么下沉到聚合根，要么上浮到应用服务。
10. **统一语言一致**：术语在事件风暴、领域建模、领域总览三处保持同形；新增术语需回填事件风暴的"统一语言"行。
11. **不要伪装确定性**：枚举不全时写 `(reason: 维修 / 检定不合格 / ...)` 并留省略号，不硬凑。
12. **ASCII 图保持单一方向**（从上往下或从左往右，不混用），框 + 缩进对齐。
13. **yaml 用注释分组**：`# ─── 标识 ───` / `# ─── 状态 ───` / `# ─── 累积度量 ───`，字段类型写在行尾注释。

## 6. 常见反模式（拒绝写法）

| 反模式 | 为什么不行 | 改怎么写 |
|--------|-----------|---------|
| 没有事件风暴就直接建模 | 丢失事件/策略/上下文外消费的覆盖度 | 先走 event-storming skill，再回来 |
| 聚合根行为用 `updateXxx/saveXxx` | 面向过程流水账，丢失领域语义（CLAUDE.md 硬约束） | 用业务动词，与事件风暴命令对齐 |
| 跨聚合不变式挂在某个聚合根上 | 该聚合根无权校验，会被误以为聚合内规则 | 单列 INV-XX + "由 XxxService + 唯一索引保证" |
| 领域服务不回答"为什么不在聚合根内" | 滥用领域服务，把本该属于聚合根的逻辑外提 | 每个领域服务必带 blockquote 回答此问题 |
| 事件名与事件风暴不一致 | 两份文档失同步，读者分不清哪个为准 | 事件名逐字照搬；要改先改事件风暴 |
| 把读模型塞进聚合根 | 读模型是投影，不该污染写模型 | 单列查询应用服务 + 读模型投影 |
| 把上下文外模型直接引用进本上下文 | 边界塌陷，模型耦合 | 经防腐层（ACL）翻译为本上下文语义值对象 |
| §10 映射表只列聚合不列策略/读模型/热点 | 读者无法从事件风暴贴纸查到落点 | 覆盖所有贴纸类别，尤其策略与热点 |
| 热点在建模中静默消失 | 不确定性被藏起来 | 固化为 INV 或写入"暂缓模块说明" |
| 一篇领域建模覆盖两个限界上下文 | 边界塌陷，无法独立演进 | 拆成两篇，用"对外契约"章节互联 |
| 应用服务里写业务逻辑 | 应用服务只编排，业务逻辑归领域层（CLAUDE.md 分层） | 业务规则下沉到领域服务/聚合根 |

## 7. 撰写流程（操作步骤）

### 7.A 新建文档

1. **确认前置**：`ls` 事件风暴目录确认源文件存在；不存在则停止，提示用户先做事件风暴。
2. **通读事件风暴**：完整读一遍源事件风暴，抽出"贴纸清单"——所有【聚合】【事件】【命令】【策略】【读模型】【热点】逐条列在草稿上。这张清单就是 §10 映射表的左列。
3. **定聚合划分**：把【聚合】贴纸落为聚合根，画 §0 的聚合划分框图。聚合内有独立生命周期的部分拆为实体（若有）。
4. **填聚合根**：每个聚合根按 §1 子结构写——标识/属性/状态/关联/累积度量，行为方法逐个对应事件风暴命令，每个方法标 `→ 发出 XxxHappened` + 前置条件。
5. **抽值对象**：把聚合根属性里的枚举、复合类型提炼为值对象（§2）。状态型值对象附状态机。
6. **落策略**：把【策略】贴纸逐条落位——跨聚合→领域服务（§3）；单聚合内部→聚合根方法或应用服务编排。每个领域服务必答"为什么不在聚合根内"。
7. **落应用服务**：分命令/事件消费/审批/查询四类（§4）。事件消费应用服务把事件风暴 §2.N 的"上下文外事件→反应"表逐条迁入。
8. **迁事件**：把事件风暴的事件逐字迁入 §5 领域事件表，补"发出者"列。
9. **迁不变式**：把事件风暴 §聚合与不变式逐条迁入 §6 INV-XX，跨聚合的注明"由谁保证"。热点能固化的固化，不能固化的进暂缓说明。
10. **迁契约**：把事件风暴 §对外契约迁入 §8，主题名/前缀/消费方照搬。
11. **画关系图**：§8 聚合间关系图 + §9 分层关系图。

## 1. 聚合根

### 1.1 SparePartLot（备件批次）

**一致性边界**：单一批次的库存量与状态。
**职责**：批次入库、扣减、状态流转、消耗累积。

```yaml
SparePartLot:
  # ─── 标识 ───
  lot_id:     LotId                # 批次唯一标识（聚合根ID）
  part_no:    String               # 备件号
  # ─── 基本属性 ───
  vendor:     String
  # ─── 状态 ───
  state:      LotState             # 待检/在库/耗尽
  # ─── 累积度量 ───
  qty:        Integer              # 当前余量
```

**行为**：
- `receive(part_no, qty, vendor)` → 发出 `SparePartLotReceived`（前置：无，新建）
- `accept()` → 发出 `LotAccepted`（前置：当前为 `INCOMING`）
- `deduct(qty)` → 发出 `SparePartConsumed` + `InventoryChanged`（前置：qty>0；扣减后=0 则转 `DEPLETED`）

---

## 2. 值对象

### 2.1 LotState（批次状态）

```yaml
LotState:
  - INCOMING   # 待检
  - IN_STOCK   # 在库
  - DEPLETED   # 耗尽（终态）
```

**状态机**：
```
INCOMING --Accept--> IN_STOCK --Deduct(qty=0)--> DEPLETED (终态)
```

---

## 3. 领域服务

### 3.1 InventoryReorderService（库存再订货服务）

**职责**：库存变更后判定是否触发再订货预警。
```yaml
InventoryReorderService:
  方法:
    onInventoryChanged(lot_id, new_qty):
      # 1. 加载 ReorderRule
      # 2. if new_qty < rule.threshold → 发出 InventoryLowWarned
```
> **为什么不放在聚合根内**：再订货判定需读取 ReorderRule 聚合（跨聚合），SparePartLot 无权持有规则。

## 6. 不变规则

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-01 | `qty ≥ 0` | 由聚合根 `deduct` 前置条件保证 |
| INV-02 | 同一 (part_no, lot_no) 全局唯一 | 跨实例不变式，由唯一索引保证（非聚合根可校验） |
