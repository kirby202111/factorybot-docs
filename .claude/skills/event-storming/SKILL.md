# 事件风暴写作 Skill

本 skill 规定本仓库**事件风暴（Event Storming）类文档**的统一骨架、符号语义与质量门槛，保证不同域之间的事件风暴在词汇、粒度、章节结构上**互相对齐、可直接对话**。

> **前置纪律（来自项目 CLAUDE.md）**：当用户指定要做某个模块时，**只读该模块自身文档与其直接对接的域（domain），不把其它模块杂糅进来**。事件风暴只应承载本域职责，不应代写邻居域的事件链。

---

## 1. 何时使用本 skill

- 用户说"做 / 写 / 补 / 改 XX 域的事件风暴"。
- 用户给出一个限界上下文名称（如"备件管理域"、"排产域"），需要从零落地一篇结构化文档。
- 用户要把一段需求描述（散文）转成"命令 → 事件 → 策略 → 读模型"的形式。

不适用：纯需求澄清、纯接口契约、纯数据库表结构——那些走对应的实现说明文档。

---

## 2. 文档落位规则

事件风暴文档统一落到 [MES领域模型/](../../MES领域模型/) 下，按归属服务放入子目录的 `事件风暴/` 文件夹：

```
MES领域模型/
├── 制造资源服务/事件风暴/<子域>/<XX域>.md
├── 设备管理服务/事件风暴/<XX域>.md
├── 生产执行服务/事件风暴/<XX域>.md   （文件可直接置于服务根，无 `事件风暴/` 目录时按现状归属）
└── 跨服务/事件风暴/<XX域>.md
```

新建文件前，先 `ls` 对应服务目录确认有没有同名旧文件。**若旧文件存在但归属变了，先与用户确认是改写还是新建**，避免双份不一致。

## 3. 符号语义（强制统一）

文中以 `【】` 标注事件风暴贴纸的"颜色语义"。**所有事件风暴文档必须复用这一套，禁止自造**：

- **【事件】** 已发生的事实（过去时，动宾名词化，如 `资产已登记 AssetRegistered`）
- **【命令】** 触发事件的意图（动词原形 + 宾语，如 `登记资产 RegisterAsset`）
- **【参与者】** 触发命令的人或系统（如 `设备工程师`、`Agent`、`接入域`）
- **【聚合】** 一致性边界（聚合根名，如 `Asset` / `RepairOrder`）
- **【策略】** "当 X 发生时自动触发 Y" 的反应式规则
- **【外部系统】** 域外协作方（含其它限界上下文）
- **【读模型】** 投影出的视图 / 查询模型
- **【热点】** 待对齐 / 有争议 / 需 SME 确认的点，**正文用 🔴 前缀**

事件统一双语：`中文已过去式名称 EnglishPascalCase`。命令同理。

## 2. 事件风暴 — 主链

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 域外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐/有争议点

## 5. 写作规范（关键）

1. **事件用过去时**：`AssetRegistered` ✅，`RegisterAsset` 是命令 ❌。中文同理用"已…"。
2. **命令是意图，不是 CRUD**：避免 `UpdateXxx` / `SaveXxx` / `CreateXxx`——找出业务意图的真实名字（`Commission`、`Reinstate`、`Retire`）。
3. **载荷只列关键字段**，不写完整 schema——字段级 schema 是实现说明的事，不属于事件风暴。
4. **聚合不变式必须可验证**：写得能让人写出一个失败用例。"必须先有 X 事件才能发生 Y" 是合格的；"数据要正确" 不是。
5. **跨聚合不变式必须标"由谁保证"**：领域服务 / 唯一索引 / Saga / 事件去重表四选一，**禁止挂在聚合根上**。
6. **状态机的每条边都要可触发**：列出命令 + 前置条件 + 谁来发命令。终态明确写"终态—之后任何事件均拒收"。
7. **域外事件只消费、不重发**：本域不能把订阅来的事件原样再发一次，必须包装成本域语义的事件，否则下游分不清来源。
8. **热点标 🔴**：任何"我建议 X，但需要 SME 确认"的地方都打热点。**热点不是缺陷，是协作信号**——不写热点等于把不确定性藏起来。
9. **不要伪装确定性**：枚举值不全时写 `(reason: 维修 / 检定不合格 / 寿命超限 / ...)` 并留省略号，不要硬凑五个。
10. **ASCII 流程图保持单一方向**（从上往下或从左往右，不混用），多分支用 `+----+` 框 + 缩进对齐。

---

## 6. 常见反模式（拒绝写法）

| 反模式 | 为什么不行 | 改怎么写 |
|--------|-----------|---------|
| 把"接口契约"写进事件风暴 | 实现细节挤掉了领域语义 | 接口走实现说明文档，事件风暴只到"主题 + 关键载荷" |
| 一篇事件风暴覆盖两个限界上下文 | 边界塌陷，无法独立演进 | 拆成两篇，用"对外契约"章节互联 |
| 事件用现在时 / 命令式 | 时序语义错乱，读者分不清"将要发生"还是"已经发生" | 事件统一过去时，命令统一动词原形 |
| 把 `EquipmentOnline` 直接当作 `AssetCommissioned` 用 | 视角混淆——通信视角 ≠ 资产视角 | 在"与 XX 域协作"章节单列对齐表 |
| 所有策略都堆在一节，没有"域外 vs 本域"分组 | 读者无法判定"修改我会不会影响别人" | 按 N.1 / N.2 分组，每条策略指明来源 |
| 跨聚合不变式写在某个聚合的"不变式"列里 | 该聚合根本无权校验它，会被误以为聚合内规则 | 单列一行 INV-04 并注明"由 XxxService + 唯一索引保证" |
| 写 "TODO: 待补充" 占位 | 读者无法判断这是"暂缓"还是"漏掉" | 用 🔴 热点写明"建议 X / 等待 Y 确认"，或写入"暂缓模块说明"章节 |

## 7. 撰写流程（操作步骤）

1. **确认归属**：根据 [领域总览](../../MES领域模型/领域总览.md) 找到本域归哪个服务，定文件路径。
2. **抽出"主语清单"**：列出本域所有聚合根、典型对象、域外触发源——填到第 1 章的表里。
3. **画主链状态机**：每个聚合根的核心生命周期（New → InService → Suspended → Retired 之类），先画 ASCII 图，再补 §2.2 表格。
4. **补足事件载荷**：每个事件列"何时发生 / 关键载荷 / 后续策略"。载荷字段≤6 个，关键即可。
5. **梳理域外消费**：把所有"我订阅的事件"列成 §2.N 的 `事件源 → 反应` 表，明确"是否变更状态 / 是否影响可用性"。
6. **聚合不变式**：每个聚合根至少 2 条不变式，跨聚合的额外列。
7. **策略汇总**：分"域外事件→响应"和"本域内部策略"两张表。
8. **对外契约**：主题命名 + 主消费方，**只列本域发布的**，订阅的不要混进来。
9. **协作边界表**：对每个紧密邻居域出一张 `视角 / 存储 / 判定 / 事件性质 / 流程所有权` 五列表。
10. **热点收尾**：通读一遍，把"我也不确定"的地方全部 🔴 标出，附建议立场。

> **限界上下文**：备件管理域
> **统一语言**：SparePart（备件）, SparePartLot（备件批次）, ConsumptionRecord（消耗记录）, ReorderRule（再订货规则）

## 0. 设计原则与边界澄清

**域内职责**：
- 备件主数据、批次、库存、消耗台账
- 再订货阈值判定与库存预警

**域外职责**：
- 采购订单 → 供应链域
- 维修工单 / 备件预约占用 → 维修域
- 财务出入库凭证 → 系统集成域

[参与者:仓管员] --【命令:入库 ReceiveSparePart】--> 【聚合:SparePartLot】
                                                          |
                                                          v
                          【事件:备件批次已入库 SparePartLotReceived】
                          (part_no, lot_no, qty, vendor, received_at)
                                                          |
                          【策略:入库后重算可用库存】
                                                          v
                          【事件:库存已变更 InventoryChanged】
\`\`\`

### 2.2 域外消费 — 维修域出库

\`\`\`
【外部系统:维修域】发布 PartsConsumed(repair_order_id, part_no, qty)
                            |
                            v
   【聚合:SparePartLot】--【命令:扣减库存 DeductInventory】
                            |
                            v
   【事件:备件已消耗 SparePartConsumed】 + 【事件:库存已变更 InventoryChanged】
                            |
                            v
   【策略:库存低于 ReorderRule.threshold 时】
                            |
                            v
   【事件:库存预警已触发 InventoryLowWarned】 → 发布到 spare.inventory.alert
\`\`\`
