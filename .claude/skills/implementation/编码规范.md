# 编码规范（implementation skill 参考）

> 本文件为 implementation skill 的编码规范参考附录，聚焦“撰写实现设计文档时需遵循的分层、命名与方法约定”，与 [SKILL.md](SKILL.md) 的八层分层、映射规则配合使用。
>
> 强制级别沿用原文：【强制】为硬约束，【建议】为推荐。

---

## 1. 应用模块分层

以领域层为核心，依赖倒置（DIP）：`domain` 定义接口，由 `repository` / `infrastructure` 实现。

调用流向：

```text
testsuites / main  ->  controller / facade  ->  facade-impl  ->  application  ->  domain
                                                                            ^            |
                                                                            |____________|
                                                                          repository / infrastructure（实现 domain 接口）
```

各层职责：

| 层 | 职责 | 关键约束 |
|----|------|----------|
| `facade` | 对外发布的 API 包：SOA/RPC 接口、DTO、Request。一个应用可按用途设计多个 facade 模块 | 不写业务实现 |
| `facade-impl` | 实现 facade 接口；输入解析/校验/转换、输出序列化。多个 facade 可由一个或多个 facade-impl 实现 | 不承载 INV 业务判断 |
| `controller` | HTTP 接口入口（如 Spring Boot Controller）。微服务一般由网关提供，此层可不需要 | 依赖 facade-impl 但**只能调 facade 声明的方法**，防业务泄露 |
| `application` | 定义并协调业务功能：事务控制、只读查询仓储、领域事件触发与监听、操作日志、安全认证 | **只能查询仓储，不能写仓储**；使用中间件须经 domain 声明的接口，不得直接用 Mapper |
| `domain` | 表达业务概念/状态/规则；一个聚合一个 package。含实体、值对象、领域服务、领域事件、仓储接口（读写）、外部服务防腐接口、工厂 | 不依赖 Spring / MyBatis / MQ / HTTP |
| `repository` | 数据查询与持久化。DO 只存在于本层，经 Converter 转为领域对象供上层 | 从 infrastructure 独立出来；DO 不上抛领域层 |
| `infrastructure` | 倒置依赖 domain，负责 RPC 服务与中间件调用实现；防腐层（ACL）在此 | 不承载业务规则 |
| `utility` | 与业务无关的工具方法、全局 Exception | 被各层依赖；必须无业务语义 |

> 实体模式：本 skill 约定**聚合根充血、子实体可贫血**（实体与领域服务共同构成领域模型）。

## 2. POJO 规范【强制】

POJO 区别于 Spring Bean，对象后缀约定如下：

| 后缀 | 含义 | 所在层 |
|------|------|--------|
| `DO` | 数据库模型，表字段平铺 | repository |
| `Model` | 领域模型，逻辑核心 | domain |
| `DTO` | 对外数据传输对象 | facade |
| `Info` | 内部传输对象（application↔domain），与 DTO 功能类似但 DTO 对外、Info 对内 | application |
| `VO` | 值对象（**注意非 View Object**；前后端分离，无视图对象） | domain |
| `Query` | 仓储查询参数（≥3 参数封装），与仓储接口同在 domain | domain |
| `Request` | facade 查询参数（≥3 参数封装） | facade |
| `Converter` | 转换器 `XXAA2BBConverter`，逐字段手写 | 对应分层 |

**Converter 写法【强制】**：逐字段手写，**禁止 `BeanUtils.copyProperties` / `MapStruct`**。理由：使用此类工具后字段变更无法在编译期感知，易致生产事故。

**Converter 落位**：
- `Model2DOConverter` / `DO2ModelConverter` → repository 层
- `Model2DTOConverter` / `DTO2ModelConverter` → facade-impl 层
- `Model2InfoConverter` / `Info2ModelConverter` → application 层

## 3. Bean 规范【强制】

多层结构下为避免命名冲突，Spring Bean 命名约定如下：

| Bean | 说明 | 所在层 |
|------|------|--------|
| `Controller` | HTTP 控制器 | controller |
| `Facade` | 对外服务接口，可细分 `XXWriteFacade` / `XXReadFacade`；实现为 `XXFacadeImpl` | facade（接口）/ facade-impl（实现） |
| `FacadeImpl` | Facade 实现，一般发布为 RPC 服务 | facade-impl |
| `AppService` | 应用层服务，可细分 `XXWriteAppService` / `XXReadAppService` | application |
| `DomainService` | 领域服务 | domain |
| `Repository` | 仓储接口（domain 定义），实现为 `XXRepositoryImpl` | domain（接口） |
| `RepositoryImpl` | 仓储实现，倒置依赖的体现 | repository |
| `Mapper` | MyBatis 查询接口，继承 `XXBaseMapper`（基类定义基本增删改查）；只在 repository 调用 | repository |
| `Client` | 中间件服务依赖接口（domain 定义），实现为 `XXClientImpl`（infrastructure），防腐设计 | domain（接口）/ infrastructure（实现） |
| `FacadeClient` | 外部 RPC 服务依赖接口（domain 定义），实现为 `XXFacadeClientImpl`（infrastructure），防腐层 | domain（接口）/ infrastructure（实现） |
| `Configuration` | 配置类，各层各自配置（如 `DataSourceConfiguration`→repository，`WebConfiguration`→controller） | 对应层 |

## 4. 方法命名【建议】

`Service` / `Repository` / `Client` / `DAO`(`Mapper`) 方法命名前缀：

| 前缀 | 语义 | 示例 |
|------|------|------|
| `query` | 获取单个对象 | `queryById` / `queryByPartNo` |
| `list` | 获取多个对象列表（复数结尾） | `listObjects` |
| `page` | 获取多个对象分页（复数结尾） | `pageObjects` |
| `count` | 获取统计值 | `countByState` |
| `insert` | 插入 | `insertDO` |
| `delete` | 删除 | `deleteById` |
| `update` | 修改 | `updateState` |

> **例外**：聚合根行为方法不套用上述前缀，使用业务动词（如 `accept` / `deduct` / `commission`）。这是 OOD 硬约束——CRUD 前缀只用于数据访问层，不用于表达业务行为。
