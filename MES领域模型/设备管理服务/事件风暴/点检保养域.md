# 点检保养域（Preventive Maintenance Bounded Context）

> **限界上下文**：点检保养域
> **统一语言**：InspectionRule（点检规则）, MaintenanceRule（保养规则）, InspectionItem（点检项）, MaintenanceItem（保养项）, TriggerCondition（触发条件）, ItemTemplate（项目模板）, InspectionPlan（点检计划）, MaintenancePlan（保养计划）, InspectionTask（点检任务）, MaintenanceTask（保养任务）, InspectionResult（点检结果）, MaintenanceResult（保养结果）, TargetAsset（目标资产·抽象引用）, SOPReference（SOP引用）, OverdueState（超期状态）, FixtureLifeJudgement（工装寿命判定）
> **核心关注**：以"事件风暴"方式刻画**预防性运维**从规则定义、计划调度、任务执行到结果签核的完整流程。本域负责"**何时该做、谁来做、怎么做**"；**不负责**资产的身份数据与可用性状态——那归台账域。

---

## 0. 设计原则与边界澄清

点检保养域是设备管理服务下的**运维调度子域**，是 PCBA + 整机组装混合型车间所有"预防性运维活动"的调度权威源。

**域内职责**（本域负责）：
- 点检/保养**规则定义**——触发条件（周期/阈值）、检查/维护项模板、SOP 绑定、责任人/角色指派
- 点检/保养**计划生成**——根据规则与触发条件，自动或手动生成执行计划
- 点检/保养**任务调度与下发**——计划拆解为可执行任务，分配到人/班次
- 点检/保养**执行登记与签核**——逐项记录结果、照片、签名，判定 PASS/FAIL
- **超期检测与预警**——未按时执行的任务标记超期，发布 Overdue 事件
- **工装寿命判定**——消费台账域累积度量，与寿命上限规则比较，发布 FixtureLifeExceeded

**域外职责**（本域**不**负责）：
- 资产身份、归属、状态 → 台账域
- 设备通信与数据采集 → 接入域
- 故障报修、维修执行、备件消耗 → 维修域
- 计量器具检定/校准/证书管理 → 计量检定域
- 资产报废审批流程 → 台账域
- 移动端 UI 交互、消息推送机制 → 应用层
- SOP 文档版本管理 → 文档管理（本域仅引用 SOP ID）

**为什么不拆"设备点检保养域 + 工装点检保养域"**：
1. **流程骨架完全同构**——规则→计划→任务→执行→签核，拆开等于双份代码。
2. **规则模型可统一抽象**——差异仅在触发条件（运行时长 vs 使用次数）和检查项内容，通过 `TargetAsset.asset_kind` + 模板库即可区分。
3. **调度引擎共用**——计划生成、任务分配、超期检测逻辑无差别。
4. **组织一致**——设备和工装的点检保养由同一工程团队执行，无组织鸿沟。

**为什么不拆"点检域 + 保养域"**：
1. **触发机制同构**——都是周期/阈值驱动的预防性活动。
2. **调度模型一致**——Plan → Task → Execute → Sign-off 骨架完全相同。
3. **执行经常联动**——点检发现异常时可能触发即时保养（如润滑），分域增加协调成本。
4. **区别仅在内容**——点检项是"检查 pass/fail"，保养项是"执行动作+可能重置累积值"，结构差异不足以支撑独立限界上下文。

## 1. 规则模型与资产适配

### 1.1 规则统一模型

点检规则和保养规则共享同一规则骨架，差异通过字段语义区分：

```
┌─────────────────────────────────────────────────────────────┐
│ Rule（规则骨架）                                              │
│                                                              │
│  rule_id          : 规则唯一标识                               │
│  rule_kind        : INSPECTION | MAINTENANCE                 │
│  target_asset_kind: EQUIPMENT | FIXTURE | *（通配）           │
│  target_classification: 资产分类（如"回流焊""钢网"）           │
│  trigger          : TriggerCondition                         │
│  item_templates[] : ItemTemplate[]                           │
│  responsible_role : 执行角色                                   │
│  sop_references[] : SOPReference[]                           │
│  plan_horizon     : 计划周期（如每周/每月）                     │
│  task_deadline    : 任务执行时限（从计划生成起算）              │
│  enabled          : 是否启用                                  │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 触发条件

| 触发类型 | 适用资产 | 说明 | 来源 |
|----------|---------|------|------|
| **CALENDAR_PERIOD** | 设备 + 工装 | 日历周期：每班/每天/每周/每月 | 本域内计算 |
| **RUN_HOUR_THRESHOLD** | 设备 | 运行时长累积达阈值：如每 500 小时 | 消费台账域 `EquipmentRunHourAccrued` |
| **CYCLE_THRESHOLD** | 工装 | 使用次数累积达阈值：如每 5000 次 | 消费台账域 `FixtureCycleAccrued` |
| **LIFE_UPPER_LIMIT** | 工装 | 使用次数达到寿命上限 | 消费台账域 `FixtureCycleAccrued` + 本域规则中的 `life_limit` |

> 🔴 **热点：双重触发**——一台设备可能同时配置"日历周期点检"和"运行时长保养"，两者独立触发、独立生成计划。规则引擎必须支持同一资产绑定多条不同触发条件的规则。

### 1.3 项目模板

```
ItemTemplate
├── item_id          : 模板项唯一标识
├── item_kind        : CHECK（检查项，输出 pass/fail + 实测值）
│                     | ACTION（执行项，输出 done/skipped + 可能重置度量）
├── name             : 项目名称
├── description      : 描述/标准
├── standard_value   : 标准值（如"温度 250±5℃"）
├── measurement_unit : 单位
├── check_method     : 检查方式（目视/仪器/软件自检）
├── required_evidence: 是否必须拍照
├── metric_reset     : 保养项专有——执行后是否重置累积度量
│                     （如保养后 run_hours 基准清零）
└── sort_order       : 排序
```

**设备 vs 工装的项目模板差异**：

| 维度 | 设备点检项示例 | 工装点检项示例 |
|------|--------------|--------------|
| 检查项 | 温度校准值、气压值、传送带张力、光源亮度 | 张力值、开口磨损度、定位销偏移、清洁度 |
| 维度 | 设备保养项示例 | 工装保养项示例 |
| 执行项 | 清洁传动系统、更换滤芯、润滑导轨、校准传感器 | 清洗、翻修、张力恢复、补胶 |
| 度量重置 | 保养后可能重置 `run_hours` 基准 | 翻修后可能重置 `accumulated_cycles`（由 `metric_reset` 字段控制） |

## 2. 事件风暴 — 规则定义与触发

### 2.1 规则生命周期

```
[参与者:设备工程师/工艺工程师]
        --【命令:定义点检规则 DefineInspectionRule】--> 【聚合:InspectionRule】
        --【命令:定义保养规则 DefineMaintenanceRule】--> 【聚合:MaintenanceRule】
                                                    |
                                                    v
                            【事件:点检规则已定义 InspectionRuleDefined】
                            【事件:保养规则已定义 MaintenanceRuleDefined】
                                                    |
                                            【策略:规则生效后立即开始监听触发条件】
                                                    |
                                                    v
                            【命令:激活规则 ActivateRule】
                            【事件:规则已激活 RuleActivated】
```

**规则变更事件**：

| 事件 | 触发场景 | 载荷要点 |
|------|---------|---------|
| InspectionRuleDefined | 首次定义点检规则 | rule_id, target_asset_kind, trigger, item_templates, responsible_role |
| MaintenanceRuleDefined | 首次定义保养规则 | rule_id, target_asset_kind, trigger, item_templates, metric_reset_info |
| RuleUpdated | 规则内容变更 | changed_fields, effective_from（立即/下个周期生效） |
| RuleActivated / RuleDeactivated | 启用/停用规则 | — |
| RuleObsolete | 规则废弃（不再使用） | replacement_rule_id（可选） |

> 🔴 **热点：规则变更的生效策略**——变更已有规则时，已生成的计划和任务是否受影响？建议：**已生成的计划/任务不受影响**（快照语义），变更从下个周期生效。避免执行中途规则变更导致混乱。

---

### 2.2 触发与 Due 事件生成

```
【外部系统:台账域】
        -- 发布 eam.asset.metric (RunHourAccrued / CycleAccrued)
                |
                v
        【聚合:InspectionRule / MaintenanceRule】
        --【策略:度量值与阈值比较】
                |
                ├── 达到点检周期阈值 ──► 【命令:标记点检到期 MarkInspectionDue】
                │                           |
                │                           v
                │                    【事件:点检已到期 InspectionDue】(asset_id, rule_id, due_reason)
                │
                ├── 达到保养周期阈值 ──► 【命令:标记保养到期 MarkMaintenanceDue】
                │                           |
                │                           v
                │                    【事件:保养已到期 MaintenanceDue】(asset_id, rule_id, due_reason)
                │
                └── 工装达到寿命上限 ──► 【命令:判定工装寿命超限 JudgeFixtureLife】
                                            |
                                            v
                                     【事件:工装寿命已超限 FixtureLifeExceeded】(fixture_id, accumulated_cycles, life_limit)
```

**触发判定逻辑**：

| 触发类型 | 判定逻辑 | 事件 |
|----------|---------|------|
| CALENDAR_PERIOD | 本域定时器扫描规则 + 资产列表，距上次执行达到周期 | InspectionDue / MaintenanceDue |
| RUN_HOUR_THRESHOLD | 消费 `EquipmentRunHourAccrued`，比较 `accumulated_run_hours - last_reset_baseline ≥ threshold` | MaintenanceDue |
| CYCLE_THRESHOLD | 消费 `FixtureCycleAccrued`，比较 `accumulated_cycles - last_reset_baseline ≥ threshold` | MaintenanceDue |
| LIFE_UPPER_LIMIT | 消费 `FixtureCycleAccrued`，比较 `accumulated_cycles ≥ life_limit` | FixtureLifeExceeded |

> **重要**：`last_reset_baseline`（上次重置基准值）由本域维护——当保养项包含 `metric_reset=true` 时，保养完成后更新基准值。台账域存的是绝对累积值，本域存的是"相对于上次重置的偏移量"。

## 3. 事件风暴 — 计划生成与任务调度

### 3.1 Due → Plan

```
InspectionDue / MaintenanceDue
        |
        v
【策略:到期自动生成计划】
        |
        v
【命令:生成点检计划 GenerateInspectionPlan】
【命令:生成保养计划 GenerateMaintenancePlan】
        |
        v
【聚合:InspectionPlan / MaintenancePlan】
        |
        v
【事件:点检计划已生成 InspectionPlanGenerated】(plan_id, plan_items[], deadline)
【事件:保养计划已生成 MaintenancePlanGenerated】(plan_id, plan_items[], deadline)
```

**计划结构**：

```
InspectionPlan / MaintenancePlan
├── plan_id          : 计划唯一标识
├── plan_kind        : INSPECTION | MAINTENANCE
├── plan_items[]     : PlanItem[]（每条对应一台资产的计划明细）
│   ├── asset_id     : 目标资产
│   ├── rule_id      : 来源规则
│   ├── due_reason   : 到期原因
│   └── deadline     : 该条明细的执行截止时间
├── plan_horizon     : 计划周期（如 2026-W24）
├── status           : DRAFT → ACTIVE → PARTIALLY_COMPLETED → COMPLETED / OVERDUE
└── generated_at     : 生成时间
```

### 3.2 Plan → Task

```
InspectionPlanGenerated / MaintenancePlanGenerated
        |
        v
【策略:计划激活后拆解为任务】
        |
        v
【命令:创建点检任务 CreateInspectionTask】（每条 plan_item → 一个 task）
【命令:创建保养任务 CreateMaintenanceTask】
        |
        v
【聚合:InspectionTask / MaintenanceTask】
        |
        v
【事件:点检任务已创建 InspectionTaskCreated】(task_id, asset_id, items[], assignee, deadline)
【事件:保养任务已创建 MaintenanceTaskCreated】(task_id, asset_id, items[], assignee, deadline)
```

**任务结构**：

```
InspectionTask / MaintenanceTask
├── task_id          : 任务唯一标识
├── task_kind        : INSPECTION | MAINTENANCE
├── plan_id          : 来源计划
├── asset_id         : 目标资产
├── asset_kind       : 设备/工装
├── items[]          : TaskItem[]（从规则模板实例化）
│   ├── item_id      : 项目标识
│   ├── name         : 项目名称
│   ├── standard     : 标准/目标值
│   ├── result       : 执行结果（待执行）
│   ├── evidence_url : 证据（照片/签名）
│   └── metric_reset : 是否重置度量（保养项专有）
├── assignee         : 执行人/角色
├── status           : PENDING → IN_PROGRESS → COMPLETED / SKIPPED / OVERDUE
├── deadline         : 执行截止时间
└── completed_at     : 完成时间
```

> 🔴 **热点：任务分配策略**——自动分配还是手动派工？建议：**规则中指定角色 → 计划生成时自动匹配当班人员 → 支持手动改派**。工装点检/保养可能需先借出到指定区域，任务分配需考虑位置。

## 4. 事件风暴 — 点检执行

```
[参与者:点检员/设备工程师]
        --【命令:开始点检 StartInspection】--> 【聚合:InspectionTask】
                                                    |
                                                    v
                                            【事件:点检已开始 InspectionStarted】
                                                    |
        --【命令:逐项登记点检结果 RecordInspectionItem】-->
                                                    |
                                                    v
                                            【事件:点检项已登记 InspectionItemRecorded】
                                              (item_id, verdict: PASS/FAIL, measured_value, evidence)
                                                    |
                                        ┌──── 所有项登记完成 ────┐
                                        v                        v
                                全部 PASS                 存在 FAIL
                                        |                        |
                                        v                        v
                        【命令:签核点检 ConfirmInspection】  【命令:签核异常点检 ConfirmInspectionWithAnomaly】
                                        |                        |
                                        v                        v
                        【事件:点检已完成                   【事件:点检已完成
                         InspectionCompleted                 InspectionCompleted
                         verdict=PASS】                      verdict=FAIL】
                                                                    |
                                                            【策略:点检异常建议报修】
                                                                    |
                                                                    v
                                                            【事件:建议报修 RepairSuggested】
                                                                    (asset_id, failed_items[])
```

**关键事件**：

| 事件 | 何时发生 | 关键载荷 | 下游消费 |
|------|---------|---------|----------|
| InspectionStarted | 点检员开始执行任务 | task_id, started_at, started_by | — |
| InspectionItemRecorded | 逐项登记 | item_id, verdict, measured_value, evidence_url | — |
| InspectionCompleted(verdict=PASS) | 全部通过签核 | task_id, asset_id, verdict, completed_by | 台账域：无状态变更 |
| InspectionCompleted(verdict=FAIL) | 存在异常签核 | task_id, asset_id, verdict, failed_items[], completed_by | 台账域：SuspendAsset；维修域：建议报修 |
| InspectionSkipped | 跳过（资产停用/外转等原因） | task_id, skip_reason | — |

> 🔴 **热点：部分 FAIL 处理**——点检 10 项中 1 项 FAIL，整次点检判定为 FAIL，但其他 9 项结果仍需保留。签核时标记整体 verdict=FAIL + 列出 failed_items。

## 5. 事件风暴 — 保养执行

```
[参与者:保养员/设备工程师]
        --【命令:开始保养 StartMaintenance】--> 【聚合:MaintenanceTask】
                                                    |
                                                    v
                                            【事件:保养已开始 MaintenanceStarted】
                                                    |
        --【命令:逐项执行保养 PerformMaintenanceItem】-->
                                                    |
                                                    v
                                            【事件:保养项已执行 MaintenanceItemPerformed】
                                              (item_id, action_taken, metric_reset: bool)
                                                    |
                                        ┌──── 所有项执行完成 ────┐
                                        v                        v
                                无需重置度量                需要重置度量
                                        |                        |
                                        v                        v
                        【命令:签核保养 ConfirmMaintenance】  【命令:签核保养并通知度量重置
                                                                    ConfirmMaintenanceWithMetricReset】
                                        |                        |
                                        v                        v
                        【事件:保养已完成                   【事件:保养已完成
                         MaintenanceCompleted               MaintenanceCompleted
                         metric_reset=false】                metric_reset=true,
                                                            reset_asset_id, reset_metric_kind,
                                                            reset_baseline】
```

**关键事件**：

| 事件 | 何时发生 | 关键载荷 | 下游消费 |
|------|---------|---------|----------|
| MaintenanceStarted | 保养员开始执行 | task_id, started_at, started_by | — |
| MaintenanceItemPerformed | 逐项执行 | item_id, action_taken, evidence_url, metric_reset | — |
| MaintenanceCompleted(metric_reset=false) | 执行完成，无度量重置 | task_id, asset_id, completed_by | — |
| MaintenanceCompleted(metric_reset=true) | 执行完成，含度量重置 | task_id, asset_id, reset_metric_kind, reset_baseline, completed_by | 台账域：更新累积值基准 |

**度量重置机制**：

保养项如果配置了 `metric_reset=true`，签核时本域发布 `MaintenanceCompleted` 携带重置信息。台账域消费后，**不修改绝对累积值**，而是记录新的重置基准点。下次阈值比较时，本域使用 `accumulated_value - reset_baseline` 计算偏移量。

| 资产类型 | 重置场景 | 重置度量 | 说明 |
|---------|---------|---------|------|
| 设备 | 大保养后 | run_hours 基准清零 | 如每 2000 小时大保养后 |
| 工装 | 翻修后 | accumulated_cycles 清零 | 如治具大修后（由工程标准判定） |
| 工装 | 钢网清洁后 | **不清零** | 钢网清洁是常规保养，不重置寿命 |

> 🔴 **热点：谁决定是否重置**——`metric_reset` 字段来自规则模板定义，不是执行时临时决定。但执行人可以在签核时标注"本次跳过重置"（如发现翻修不合格）。策略归工程标准，本域记录执行事实。

## 6. 事件风暴 — 超期与预警

### 6.1 超期检测

```
【策略:定时扫描超期任务】
        |
        ├── 点检任务超过 deadline 未完成
        │       |
        │       v
        │   【命令:标记点检超期 MarkInspectionOverdue】
        │       |
        │       v
        │   【事件:点检已超期 InspectionOverdue】(task_id, asset_id, overdue_since)
        │       |
        │       v
        │   发布到 pm.inspection.overdue → 台账域消费：MarkAssetUnavailable
        │
        └── 保养任务超过 deadline 未完成
                |
                v
            【命令:标记保养超期 MarkMaintenanceOverdue】
                |
                v
            【事件:保养已超期 MaintenanceOverdue】(task_id, asset_id, overdue_since)
                |
                v
            发布到 pm.maintenance.overdue → 台账域消费：MarkAssetUnavailable
```

### 6.2 工装寿命预警

```
【外部系统:台账域】
        -- 发布 FixtureCycleAccrued
                |
                v
        【策略:使用次数与寿命上限比较】
                |
                ├── accumulated_cycles ≥ life_limit
                │       |
                │       v
                │   【事件:工装寿命已超限 FixtureLifeExceeded】(fixture_id, accumulated_cycles, life_limit)
                │       |
                │       v
                │   发布到 pm.fixture.life → 台账域消费：SuspendAsset(reason=LifeExceeded)
                │
                └── accumulated_cycles ≥ life_limit × 0.8（预警线）
                        |
                        v
                    【事件:工装寿命预警 FixtureLifeWarning】(fixture_id, accumulated_cycles, life_limit, remaining_ratio)
                        |
                        v
                    发布到 pm.fixture.warning → 通知工程团队（不触发状态变更）
```

> 🔴 **热点：寿命上限存哪里**——`life_limit` 存在本域的 MaintenanceRule 中（它是运维规则，不是资产属性）。台账域只存 `accumulated_cycles`，不做阈值判定。但如果同一类工装可能有不同寿命上限（如不同供应商钢网），需在规则中按资产分类细分。

## 7. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **InspectionRule** | 单条点检规则 | 同一 `target_asset_kind + classification + trigger` 组合不重复；启用状态下至少 1 个 item_template |
| **MaintenanceRule** | 单条保养规则 | 同上；含 `metric_reset=true` 的 item 必须指定 `reset_metric_kind` |
| **InspectionPlan** | 单个周期点检计划 | 每条 plan_item 必须关联一条已激活的 InspectionRule；plan 内 asset 不重复 |
| **MaintenancePlan** | 单个周期保养计划 | 同上 |
| **InspectionTask** | 单次点检任务 | 必须关联一个 InspectionPlan 的 plan_item；所有 item 必须登记结果后才能签核 |
| **MaintenanceTask** | 单次保养任务 | 同上；含 metric_reset 的任务签核时必须携带 reset_baseline |

## 8. 关键策略（Policy）汇总

### 8.1 触发策略

| 触发源 | 策略 | 触发命令 |
|--------|------|----------|
| 台账域 `EquipmentRunHourAccrued` | 运行时长与保养阈值比较 | MarkMaintenanceDue（达到阈值时） |
| 台账域 `FixtureCycleAccrued` | 使用次数与保养阈值/寿命上限比较 | MarkMaintenanceDue / JudgeFixtureLife |
| 日历定时器 | 周期扫描点检规则 | MarkInspectionDue（达到周期时） |

### 8.2 调度策略

| 触发事件 | 策略 | 触发命令 |
|----------|------|----------|
| InspectionDue | 自动生成点检计划 | GenerateInspectionPlan |
| MaintenanceDue | 自动生成保养计划 | GenerateMaintenancePlan |
| InspectionPlanGenerated | 拆解为点检任务 | CreateInspectionTask（per plan_item） |
| MaintenancePlanGenerated | 拆解为保养任务 | CreateMaintenanceTask（per plan_item） |

### 8.3 超期策略

| 触发条件 | 策略 | 触发命令 |
|----------|------|----------|
| InspectionTask 超过 deadline | 标记点检超期 | MarkInspectionOverdue |
| MaintenanceTask 超过 deadline | 标记保养超期 | MarkMaintenanceOverdue |

### 8.4 执行结果策略

| 触发事件 | 策略 | 触发命令 |
|----------|------|----------|
| InspectionCompleted(verdict=FAIL) | 建议报修 | RepairSuggested（发布到维修域） |
| MaintenanceCompleted(metric_reset=true) | 通知台账域度量重置 | （通过事件契约传递，台账域消费 MaintenanceCompleted 的 reset 信息） |
| FixtureLifeExceeded | 通知台账域寿命超限 | （台账域消费后 SuspendAsset） |

### 8.5 资产生命周期联动

| 触发源（台账域事件） | 策略 |
|---------------------|------|
| AssetCommissioned | 激活该资产匹配的所有规则 |
| AssetSuspended | 标记该资产待执行任务为 SKIPPED（如因停用/外转） |
| AssetRetired | 停用该资产所有规则，取消未执行任务 |

## 10. 与台账域的协作边界

| 维度 | 本域（点检保养域） | 台账域 |
|------|-------------------|--------|
| 关心什么 | 运维活动**怎么管** | 资产**能不能用** |
| 存储什么 | 阈值规则：保养周期 / 寿命上限 / 点检周期 | 累积值：run_hours / accumulated_cycles |
| 谁判定触发 | **判定并触发**——订阅度量值，与阈值比较 | **不判定**——只发布度量值 |
| 事件性质 | 调度信号：MaintenanceDue / InspectionDue | 度量事实：EquipmentRunHourAccrued |

**事件流向**：

```
台账域 ──→ eam.asset.metric ──→ 本域（阈值比较 → 触发调度）
本域  ──→ pm.inspection.completed ──→ 台账域（FAIL → SuspendAsset）
本域  ──→ pm.inspection.overdue  ──→ 台账域（MarkAssetUnavailable）
本域  ──→ pm.maintenance.completed ──→ 台账域（metric_reset → 更新基准）
本域  ──→ pm.maintenance.overdue ──→ 台账域（MarkAssetUnavailable）
本域  ──→ pm.fixture.life ──→ 台账域（SuspendAsset）
台账域 ──→ eam.asset.lifecycle ──→ 本域（Commissioned → 激活规则 / Retired → 停用规则）
```

## 11. 与维修域的协作边界

点检保养域与维修域是"预防"与"纠正"的关系：

| 协作场景 | 方向 | 事件 | 说明 |
|---------|------|------|------|
| 点检异常建议报修 | 本域 → 维修域 | `RepairSuggested` | 点检 FAIL 后建议进入维修流程 |
| 维修完成后恢复 | 维修域 → 本域 | `RepairCompleted` | 维修域发布后台账域 Reinstate，本域无需额外动作 |
| 维修域发现需保养 | 维修域 → 本域 | `MaintenanceSuggested` | 维修过程中发现潜在保养需求（如润滑不足） |

> 本域**不直接创建维修工单**——仅发布 `RepairSuggested` 建议，由维修域决定是否创建 RepairOrder。避免预防域直接指挥纠正域，保持各域自治。
