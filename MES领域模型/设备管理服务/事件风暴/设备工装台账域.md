# 设备工装台账域（Asset Registry Bounded Context）

> **限界上下文**：设备工装台账域
> **统一语言**：Asset（资产·抽象）, Equipment（设备）, Fixture（工装夹具）, MeasuringInstrument（计量器具）, AssetIdentifier（资产编码）, AssetSpecificationProfile（能力规格画像）, FixtureMatchingRule（工装匹配规则）, AvailabilityState（可用性状态）, RunHourAccrual（运行时长累积）, CycleAccrual（使用次数累积）, CalibrationCertificate（检定证书·只读引用）
> **核心关注**：以"事件风暴"方式刻画**实物资产从入账到报废**全生命周期事件链。本域只对资产的**身份、归属、状态、生命周期**主数据负责；**不负责**点检/保养/检定/维修的调度与执行流程——那些归「点检保养域」「计量检定域」等独立域建模，本域仅**消费其结果事件**以驱动资产状态变更与可用性重算。

---

## 0. 设计原则与边界澄清

设备工装台账域是设备管理服务下的**主数据子域**，是 PCBA + 整机组装混合型车间所有"实物资产"运行态的权威源。

**域内职责**（本域负责）：
- 设备、工装夹具、计量器具的**入账、赋码、建档、变更、报废**
- 资产**位置、归属、责任人、状态**的全生命周期记录
- 资产**可用性聚合视图**的维护与发布——消费域外运维事件，重算可用性，对外发布
- 资产**能力规格画像**与**工装匹配规则**的发布（供制造资源服务工艺设计引用）
- 资产**累积度量**的维护（运行时长、使用次数、检定证书有效期）——仅存储当前值，阈值与触发规则归点检保养域

**域外职责**（本域**不**负责）：
- 设备数据采集行为本身 → 设备数据接入域
- 工艺路线 / BOM / 工序定义 → 制造资源服务（仅引用本域规格读模型）
- 派工时的资产匹配与占用 → 生产执行服务（订阅本域可用性视图）
- OEE / 设备效率 / 故障率分析 → 数据应用域
- **点检/保养/检定/维修的完整流程**——规则定义、计划生成、任务下发、执行登记、签核、SOP → **点检保养域 / 计量检定域**（本域仅消费其结果事件）
- 备件库存、备件 BOM → 后续「备件管理域」
- 资产折旧、财务接口 → 系统集成域

**为什么不拆"设备台账域 + 工装台账域"两个并列域**：
1. 两类资产**生命周期同构**——入账→投用→停用→报废，事件骨架完全可复用，拆开等于写两份。
2. **可用性引擎共用**——无论哪类资产，可用性重算逻辑同构；拆开会逼出双份。
3. **对外契约一致**——制造资源服务工艺路线绑定的是"资源能力规格"抽象，不区分设备还是工装；本域应以**统一资产规格读模型**对外暴露。
4. **内部聚合区分即可**——同一域内并存 Equipment / Fixture / MeasuringInstrument 三个聚合根，共享主链 + 各自特定事件。
5. **组织一致**——本车间形态下，设备和工装由同一工程团队管理，没有拆分所必需的组织鸿沟。

**为什么点检/保养/检定/维修不放本域**：
- `InspectionPlan` / `MaintenancePlan` / `CalibrationPlan` 是**调度聚合**——关心"何时该做、谁来做、怎么做"，不是资产属性。
- 台账域关注"**资产能不能用**"，运维域关注"**运维活动怎么管**"——生命周期、责任人、变更频率完全不同。
- 混在一起，台账域就变成"台账 + 运维调度"混合体，边界模糊，修改点检规则就要动台账代码。

## 1. 资产分类与台账形态

按"管理特征"分类，决定了事件链与累积度量的差异：

| 聚合根 | 典型对象 | 关键管理特征 | 累积度量 | 域外触发源 |
|--------|---------|-------------|----------|-----------|
| **Equipment**（设备） | 锡膏印刷机、AOI、波峰焊、镭雕机、烧录电脑、测试电脑、智能电批控制器、线边仓库、老化房 | 位置相对固定；绑定采集通道 | run_hours（运行时长） | 接入域时长聚合 |
| **Fixture**（工装夹具） | 钢网、SMT 治具、FCT 测试治具、烧录治具、装配/对位夹具、工装托盘 | 配套使用（绑产品/工序/设备）；有借出归还 | accumulated_cycles（使用次数） | 生产执行/接入域上报 |
| **MeasuringInstrument**（计量器具） | 卡尺、千分尺、扭矩扳手、张力计、温度计、湿度计、绝缘电阻仪 | 强制周期检定；有证书有效期；超期即冻结 | cert_valid_until（证书有效期） | 计量检定域签发 |

> 说明：智能电批的"枪头/批头"按工装管理；老化房的"载具/治具"按工装管理；老化房温控按设备管理。三类聚合在域内并存，不互相继承，仅共享主链事件名。
>
> **阈值（保养周期、点检周期、寿命上限、检定周期）不在本域存储**——它们是运维规则，归点检保养域 / 计量检定域。本域只存"当前累积值"，域外规则引擎根据累积值判定是否触发。

## 2. 事件风暴 — 共享主链（横切，三类聚合共用）

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 域外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐/有争议点

### 2.1 入账与建档

```
[参与者:设备工程师/工装管理员/计量员]
        --【命令:登记资产 RegisterAsset】--> 【聚合:Asset】
                                                  |
                                                  v
                          【事件:资产已登记 AssetRegistered】(asset_kind: Equipment/Fixture/Instrument)
                                                  |
                          【策略:首次登记自动赋码】
                                                  |
                                                  v
                          【命令:分配资产编码 AssignAssetIdentifier】
                          【事件:资产编码已分配 AssetIdentifierAssigned】
                                                  |
                                                  v
                                      --【事件:能力规格已建立 AssetSpecificationProfileEstablished】
                                                  |
                                            (供制造资源服务读)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| AssetRegistered | 工程师登记新资产 | asset_id, asset_kind, model, vendor, custodian_org | 触发赋码 |
| AssetIdentifierAssigned | 系统分配/导入外部码 | asset_code, code_scheme（自编/厂商/资产标签） | 进入"未投用"状态 |
| AssetAttributeChanged | 属性变更（型号/规格/责任部门等） | changed_fields, before, after | 触发规格画像更新 |
| AssetClassificationUpdated | 资产分类变更 | new_classification, reason | — |
| AssetSpecificationProfileEstablished / Updated | 能力规格画像首次建立或更新 | spec_keys, spec_values（如：回流焊·峰值≥260℃） | 发布到 `eam.asset.specification` |

> 🔴 **热点：编码方案**——是用统一的"资产标签"（如 `EQ-202606-00123`）还是按类型前缀分段（`EQ-` / `FX-` / `MI-`）？建议**单一标签前缀 + 类型字段**，避免重码冲突；与 ERP/财务资产编码的对齐留给系统集成域。

### 2.2 状态流转

```
AssetRegistered
        |
        v
        [NEW] --【命令:投用 Commission】--> 【事件:AssetCommissioned】--> [IN_SERVICE]
                                                                              |
                                                                              | --【命令:停用 Suspend】--> 【事件:AssetSuspended】--> [SUSPENDED]
                                                                              |       (reason: 维修/检定超期/工艺停产/借出/寿命超限)      |
                                                                              |                                                            |
                                                                              | <-- 【事件:AssetReinstated】 <-- 【命令:复用 Reinstate】 <-- |
                                                                              |
                                                                              v
                                                                        【命令:报废 Retire】
                                                                              |
                                                                              v
                                                                        【事件:AssetRetired】--> [RETIRED] (终态)
```

**关键事件**

| 事件 | 触发场景 | 不变式 |
|------|---------|--------|
| AssetCommissioned | 验收完成、首次投入产线 | 必须先有 `AssetSpecificationProfileEstablished` |
| AssetSuspended | 故障/超期/借出/工艺暂停/寿命超限 | 载荷必带 `reason` |
| AssetReinstated | 维修结束/检定通过/归还 | 必须有对应的 Suspended 才能复用 |
| AssetRetired | 报废、转出、丢失 | 终态——**之后任何事件均拒收** |

> **Suspend / Reinstate 的命令来源**：本域自身可发起（外转、借出归还），也可由**域外运维事件消费后自动触发**（详见 §2.5）。

### 2.4 累积度量维护

台账域维护三类资产的"当前累积值"，供域外规则引擎判定触发条件：

```
【外部系统:设备数据接入域】
        -- 发布 EquipmentOnline / Offline 区间聚合
                |
                v
        【聚合:Equipment】--【命令:累计运行时长 AccrueRunHour】
                |
                v
        【事件:运行时长已累计 EquipmentRunHourAccrued】(run_hours_delta, accumulated_run_hours, source)
                |
                v
        发布到 eam.asset.metric（供点检保养域订阅，用于判定"是否达到保养阈值"）

【外部系统:生产执行服务 / 接入域】
        -- 上报工装使用次数
                |
                v
        【聚合:Fixture】--【命令:累计使用次数 AccrueCycle】
                |
                v
        【事件:使用次数已累计 FixtureCycleAccrued】(cycles_delta, accumulated_cycles, source)
                |
                v
        发布到 eam.asset.metric（供点检保养域订阅，用于判定"是否达到寿命预警"）

【外部系统:计量检定域】
        -- 发布检定证书签发事件
                |
                v
        【聚合:MeasuringInstrument】--【命令:更新证书有效期 UpdateCertValidity】
                |
                v
        【事件:检定证书有效期已更新 InstrumentCertValidityUpdated】(cert_no, valid_until)
```

> 🔴 **热点：运行时长来源**——`EquipmentRunHourAccrued` 由谁计算？建议由**接入域**根据 `EquipmentOnline/Offline` 区间聚合后**周期性发布**到本域订阅；本域不直接采集状态流。
> 🔴 **热点：cycles 来源**——`FixtureCycleAccrued` 谁来报？建议**优先由生产执行服务在派工/过站时上报**（最权威）；当工装与设备绑定且通过设备采集事件可推断时，由接入域通过策略派生上报。
> 🔴 **热点：阈值判定归谁**——累积值在本域，但"是否达到阈值"的判定与触发归**点检保养域**。本域只发布 `EquipmentRunHourAccrued` / `FixtureCycleAccrued`，不发布 `MaintenanceDue` / `FixtureLifeThresholdReached`——后两者是调度事件，不是资产度量事件。

### 2.5 域外运维事件消费与可用性投影

本域**不定义**点检/保养/检定/维修的事件链（Due → Plan → Performed → Overdue），**只消费**这些域发布的结果事件，驱动资产状态变更与可用性重算：

```
┌──────────────────────────────────────────────────────────────┐
│ 域外事件源（点检保养域 / 计量检定域 / 生产执行服务 / 接入域） │
└──────────────────────────┬───────────────────────────────────┘
                           │ 发布结果事件
                           v
┌──────────────────────────────────────────────────────────────┐
│ 本域：事件消费 → 状态变更 → 可用性重算                       │
│                                                              │
│  [点检保养域事件]                                            │
│    InspectionCompleted(verdict=FAIL) ──► SuspendAsset        │
│    InspectionCompleted(verdict=PASS) ──► (无状态变更)        │
│    InspectionOverdue ──────────────────► MarkAssetUnavailable│
│    MaintenanceCompleted(run_hour_reset) ──► (无状态变更，    │
│                                           但点检保养域会通知  │
│                                           本域重置累积值)     │
│    MaintenanceOverdue ──────────────────► MarkAssetUnavailable│
│                                                              │
│  [计量检定域事件]                                            │
│    CalibrationCertificateIssued(cert_no, valid_until)        │
│                        ──► UpdateCertValidity + Reinstate    │
│    CalibrationFailed ──► SuspendAsset(reason=CalibrationFailed)│
│    CalibrationExpired ──► SuspendAsset(reason=CalibrationExpired)│
│    CalibrationDispatched ──► SuspendAsset(reason=Calibrating) │
│                                                              │
│  [维修域事件]                                                 │
│    RepairCompleted ──────► ReinstateAsset                    │
│    ScrapRecommendation ──► 触发报废审批子流程（§2.6）        │
│                                                              │
│  [生产执行服务/接入域事件]                                   │
│    MalfunctionReported ──► SuspendAsset(reason=UnderRepair)  │
│    RepairEscalated ──────► (无状态变更，仅记录)              │
│                                                              │
│  [接入域事件]                                                │
│    ChannelEscalated ─────► 派生 MalfunctionReported          │
│                           (通信故障→报修)                     │
│                                                              │
└──────────────────────────┬───────────────────────────────────┘
                           │ 状态变更后
                           v
        【聚合:AvailabilityProjection】
        --【命令:重算可用性 RecomputeAvailability】
        --【事件:可用性已变更 AssetAvailabilityChanged】(asset_id, available: true/false, blocking_reasons[])
                |
                v
        发布到主题: eam.asset.availability
                |
                v
        【外部系统:生产执行服务】订阅，用于派工准入校验
```

**可用性不变式**：`AssetAvailabilityChanged.available = true` 当且仅当：
- 状态为 IN_SERVICE
- 且无未解除的 InspectionOverdue / MaintenanceOverdue
- 且（若为计量器具）cert_valid_until > now()
- 且无 ACTIVE 的 MalfunctionReported（未 RepairCompleted）
- 且（若为工装）accumulated_cycles 未超过点检保养域判定的寿命上限

> 🔴 **热点：FixtureLifeExceeded 由谁判定**？——建议由**点检保养域**根据 `FixtureCycleAccrued` 累积值 + 寿命上限规则判定后发布 `FixtureLifeExceeded` 事件到本域；本域消费后执行 SuspendAsset。本域自身不存"寿命上限"阈值，不做判定。

### 2.6 报废审批子流程

报废不是独立域，而是**资产生命周期的终态变更**，归属本域。但报废可由多域发起建议，本域统一审批执行：

```
┌──────────────────────────────────────────────────────────────┐
│ 报废发起来源（多域）                                          │
│                                                              │
│  【来源1:点检保养域】                                         │
│    FixtureLifeExceeded（工装寿命超限）→ 建议报废              │
│                                                              │
│  【来源2:维修域】                                             │
│    ScrapRecommendation（维修不经济/无法修复）→ 建议报废       │
│                                                              │
│  【来源3:计量检定域】                                         │
│    CalibrationFailed（检定不合格且无法修复）→ 建议报废        │
│                                                              │
│  【来源4:管理决策】                                           │
│    设备工程师/管理层直接发起报废申请                          │
│                                                              │
└──────────────────────────┬───────────────────────────────────┘
                           │
                           v
        【聚合:ScrapApplication】
        --【命令:提交报废申请 SubmitScrapApplication】
                |
                v
        【事件:报废申请已提交 ScrapApplicationSubmitted】
         (application_id, asset_id, source, source_event_id, reason, applicant)
                |
                v
        【策略:根据来源自动填充审批流】
                |
                v
        ┌──── 审批决策 ────┐
        v                   v
    批准                驳回
        |                   |
        v                   v
【命令:批准报废申请        【命令:驳回报废申请
 ApproveScrapApplication】 RejectScrapApplication】
        |                   |
        v                   v
【事件:报废申请已批准       【事件:报废申请已驳回
 ScrapApplicationApproved】 ScrapApplicationRejected】
 (application_id, asset_id,  (application_id, reason)
  approved_by, approved_at)       |
        |                         v
        v                   通知来源域（如维修域工单退回 DIAGNOSED 重新评估）
【命令:报废资产 RetireAsset】
        |
        v
【事件:资产已报废 AssetRetired】--> [RETIRED] (终态)
```

**报废申请模型**：

```
ScrapApplication
├── application_id  : 申请唯一标识
├── asset_id        : 报废资产
├── asset_kind      : 设备/工装/计量器具
├── source          : INSPECTION | REPAIR | CALIBRATION | ADMINISTRATIVE
├── source_event_id : 来源事件 ID（如 RepairOrder 的 order_id）
├── reason          : 报废原因
├── residual_value  : 资产残值（辅助审批决策）
├── applicant       : 申请人
├── approval_flow   : 审批流（按来源自动匹配）
│   ├── approvers[] : 审批人列表（工程主管→财务→厂级）
│   └── current_step: 当前审批步骤
├── status          : SUBMITTED → APPROVED / REJECTED
└── attachments[]   : 附件（照片、诊断报告等）
```

**审批流匹配规则**：

| 来源 | 审批流 | 说明 |
|------|--------|------|
| INSPECTION（寿命超限） | 工程主管 → 厂级 | 工装寿命到期，审批较简 |
| REPAIR（维修不经济） | 工程主管 → 财务 → 厂级 | 涉及维修成本 vs 残值比较，需财务确认 |
| CALIBRATION（检定不合格） | 计量管理员 → 工程主管 | 法规相关，计量管理员先确认 |
| ADMINISTRATIVE（管理决策） | 工程主管 → 财务 → 厂级 | 完整审批链 |

> 🔴 **热点：报废审批是否阻塞维修工单**——维修域发布 `ScrapRecommendation` 后，维修工单进入 `PENDING_SCRAP_DECISION` 状态。本域驳回报废申请后，需通知维修域（发布 `ScrapApplicationRejected`），维修域将工单退回 `DIAGNOSED` 重新评估方案。

**关键事件**：

| 事件 | 何时发生 | 关键载荷 | 下游消费 |
|------|---------|---------|----------|
| ScrapApplicationSubmitted | 多域发起报废建议 | asset_id, source, reason | 审批引擎 |
| ScrapApplicationApproved | 审批通过 | asset_id, approved_by | 执行 RetireAsset；通知来源域 |
| ScrapApplicationRejected | 审批驳回 | asset_id, reason | 通知来源域（维修域工单退回） |

## 6. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **Asset**（抽象） | 单件资产从入账到报废 | 同一资产编码全局唯一；`AssetRetired` 后任何事件均拒收（终态） |
| **AssetSpecificationProfile** | 单件资产的能力规格画像 | 必须先有 `AssetCommissioned` 之前的 Establish；规格 key 集合受工程标准约束 |
| **AvailabilityProjection** | 单件资产的当前可用性 | 任一影响可用性的事件必须重算并发布 `AssetAvailabilityChanged`，幂等 |
| **FixtureMatchingRule** | 工装 × 适用范围（产品/工序/设备型号） | 规则变更必须发布 SpecificationUpdated 通知制造资源服务 |
| **ScrapApplication** | 单次报废申请 | 同一资产不可重复提交未决报废申请；审批通过后必须执行 RetireAsset |

> 注：`InspectionPlan` / `MaintenancePlan` / `CalibrationPlan` **不在本域**——它们是调度聚合，归点检保养域 / 计量检定域。

## 7. 关键策略（Policy）汇总

本域的策略只有两类：**域外事件消费策略**和**可用性重算策略**。

### 7.1 域外事件 → 资产状态变更

| 触发事件（域外） | 来源域 | 策略 | 触发命令 |
|-----------------|--------|------|----------|
| InspectionCompleted(verdict=FAIL) | 点检保养域 | 点检异常停用 | SuspendAsset(reason=InspectionAnomaly) |
| InspectionOverdue | 点检保养域 | 点检超期停用 | MarkAssetUnavailable |
| MaintenanceOverdue | 点检保养域 | 保养超期停用 | MarkAssetUnavailable |
| CalibrationFailed | 计量检定域 | 检定不合格冻结 | SuspendAsset(reason=CalibrationFailed) |
| CalibrationExpired | 计量检定域 | 证书过期冻结 | SuspendAsset(reason=CalibrationExpired) |
| CalibrationDispatched | 计量检定域 | 送检中冻结 | SuspendAsset(reason=Calibrating) |
| CalibrationCertificateIssued | 计量检定域 | 检定通过复用 | ReinstateAsset + UpdateCertValidity |
| MalfunctionReported | 生产执行/接入域 | 报修即停用 | SuspendAsset(reason=UnderRepair) |
| RepairCompleted | 维修域 | 维修完成复用 | ReinstateAsset |
| ScrapRecommendation | 维修域 | 维修不经济建议报废 | 触发报废审批子流程（§2.6） |
| FixtureLifeExceeded | 点检保养域 | 寿命超限冻结 | SuspendAsset(reason=LifeExceeded) |

### 7.2 本域内部策略

| 触发事件 | 策略 | 触发命令 |
|----------|------|----------|
| AssetTransferredOut | 外转后自动停用 | SuspendAsset(reason=TransferredOut) |
| 任意可用性影响事件（含域外消费后的状态变更） | 重算可用性 | RecomputeAvailability → 发布 AssetAvailabilityChanged |
| 接入域 ChannelEscalated（重连超阈值） | 派生故障上报 | ReportMalfunction(symptom=CommunicationLost) |
| ScrapApplicationApproved | 审批通过执行报废 | RetireAsset → 发布 AssetRetired |
| ScrapApplicationRejected | 审批驳回通知来源域 | 通知来源域（如维修域工单退回） |

> 注：`EquipmentRunHourAccrued 达阈值 → GenerateMaintenancePlan` 和 `FixtureCycleAccrued 达阈值 → RaiseFixtureLifeWarning` 这类**调度策略**不在本域——由点检保养域订阅本域发布的度量事件后自行判定与触发。

## 8. 对外契约（读模型 / 主题）

下游消费方只看以下投影，不直接读本域聚合：

| 主题（Topic 命名约定） | 内容 | 主消费方 |
|------------------------|------|----------|
| `eam.asset.lifecycle` | Registered / Commissioned / Suspended / Reinstated / Retired / Transferred | 制造资源服务、生产执行服务、追溯域、财务集成 |
| `eam.asset.availability` | AssetAvailabilityChanged + 当前 blocking_reasons | **生产执行服务（派工准入）** |
| `eam.asset.specification` | 能力规格画像、工装匹配规则 | **制造资源服务（工艺路线设计）** |
| `eam.asset.metric` | EquipmentRunHourAccrued / FixtureCycleAccrued / InstrumentCertValidityUpdated | **点检保养域 / 计量检定域**（用于阈值判定与触发） |

## 9. 与「设备数据接入域」的对齐点

本域与 [设备数据接入域.md](设备数据接入域.md) 共享 `equipment_id` 但**视角不同**，必须对齐避免事件混淆：

| 维度 | 本域（台账域） | 接入域（数据接入域） |
|------|---------------|---------------------|
| 视角 | 资产管理视角（"它是谁、归谁、能不能用"） | 通信视角（"它的连接通不通、报文进没进来"） |
| 主链事件 | AssetCommissioned / Suspended / Retired | EquipmentOnline / Offline / Recovered |
| `EquipmentRegistered` | **本域为权威源**（资产建档） | 接入域订阅，用于绑定网关与协议适配器 |
| `equipment_id` | 由本域 `AssetIdentifierAssigned` 颁发 | 接入域使用同一 ID，不另行分配 |

**事件流向**：
- **本域 → 接入域**：`AssetRegistered` / `AssetSpecificationProfileUpdated` / `AssetRetired` （触发通道挂载与卸载）
- **接入域 → 本域**：`ChannelEscalated`（派生 MalfunctionReported）、`EquipmentRunHourAccrued` 计算源（基于 Online/Offline 区间聚合）

> 🔴 **热点：EquipmentOnline ≠ AssetCommissioned**——一台资产可能 Commissioned 但暂时 Offline（断网/停机），也可能 Online 但 Suspended（在维修自检）。**两者必须独立维护**，不可互为充要条件。

## 10. 与点检保养域、维修域、计量检定域的协作边界

这是本域与运维域的职责分界线——**本域管"能不能用"，运维域管"怎么管"**：

| 维度 | 本域（台账域） | 点检保养域 | 维修域 | 计量检定域 |
|------|---------------|-----------|--------|-----------|
| 关心什么 | 资产**能不能用** | 预防性运维**怎么管** | 纠正性运维**怎么修** | 检定活动**怎么管** |
| 存储什么 | 累积值：run_hours / accumulated_cycles / cert_valid_until | 阈值规则：保养周期 / 寿命上限 / 点检周期 | 维修工单 / 诊断记录 / 备件消耗 | 检定规则 / 证书全量 / 检定周期 |
| 谁判定触发 | **不判定**——只发布度量值 | **判定并触发**——订阅度量值，与阈值比较 | **故障事件驱动**——不比较阈值 | **判定**——证书有效期与周期比较 |
| 事件性质 | 度量事实 | 调度信号：Due / Overdue / Completed | 执行结果：Completed / ScrapRecommendation | 证书事实：Issued / Failed / Expired |
| 流程所有权 | 报废审批子流程 | 规则→计划→任务→执行→签核 | 报修→派工→诊断→维修→验证→关单 | 规则→计划→送检→检定→证书→收回 |

**事件流向**：
- **本域 → 运维域**：`eam.asset.metric` 主题（累积度量变更，供阈值判定）；`eam.asset.lifecycle`（资产生命周期变更）
- **点检保养域 → 本域**：点检/保养的**结果事件**（§2.5 消费表），本域消费后驱动状态变更
- **维修域 → 本域**：`RepairCompleted`（恢复资产）、`ScrapRecommendation`（触发报废审批）
- **计量检定域 → 本域**：`CalibrationCertificateIssued`/`CalibrationFailed`/`CalibrationExpired`/`CalibrationDispatched`
- **不互相调用的操作**：本域不调用运维域的"生成计划"接口；运维域不调用本域的"停用资产"接口——全部通过事件异步解耦

---

## 11. 与制造资源服务、生产执行服务的协作边界

- **制造资源服务**仅消费 `eam.asset.specification`：工艺路线引用"能力规格 key/value"和"工装匹配规则"，**不复制**台账实体；本域资产的 ID/编码可作为只读防腐层快照保留，但累积度量、运维记录一律回查本域或运维域。
- **生产执行服务**消费 `eam.asset.availability` + `eam.asset.location`：派工时检查"当前可用 + 在目标工位"；**不直接修改**资产状态——派工占用通过自身的"工单×资产占用"模型记录，资产状态变更必须发命令回本域。
- **资产报废、外转**等强生命周期变更只能从本域发起，避免下游服务越权。
