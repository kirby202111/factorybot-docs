> **限界上下文**：设备工装台账域

## 0. 建模总览

本域是设备管理服务下的**主数据子域**，是所有实物资产（设备、工装夹具、计量器具）运行态的权威源。核心职责是管理资产的**身份、归属、状态、生命周期**主数据，以及维护**可用性聚合视图**供下游消费。

## 1. 聚合根

### 1.1 Equipment（设备）

**一致性边界**：单台设备从入账到报废的全生命周期状态与度量。

**职责**：管理设备的身份建档、状态流转、位置归属、采集通道绑定、运行时长累积。

```yaml
Equipment:
  # ─── 标识 ───
  asset_id:              AssetId                    # 资产唯一标识（聚合根ID）
  identifier:            AssetIdentifier            # 资产编码（赋码后生成）

  # ─── 基本属性 ───
  model:                 String                     # 设备型号
  vendor:                String                     # 制造商
  classification:        AssetClassification        # 资产分类
  specification_profile: EquipmentSpecificationProfile  # 能力规格画像

  # ─── 状态 ───
  availability_state:    AvailabilityState          # 可用性状态枚举

  # ─── 位置与归属 ───
  location:              AssetLocation              # 当前位置
  custodian_org:         String                     # 责任部门
  custodian_person:      String                     # 责任人

  # ─── 设备特有 ───
  station_id:            StationReference?          # 部署工位（可为空）
  data_channel_id:       ChannelReference?          # 关联采集通道（可为空）

  # ─── 累积度量 ───
  run_hour_accrual:      RunHourAccrual             # 运行时长累积

  # ─── 可用性阻塞原因 ───
  blocking_reasons:      List<BlockingReason>       # 当前阻塞原因列表
```

**行为**：
- `register(asset_kind, model, vendor, custodian_org)` → 发出 `AssetRegistered`
- `assignIdentifier(code, code_scheme)` → 发出 `AssetIdentifierAssigned`
- `commission()` → 发出 `AssetCommissioned`（前置：规格画像已建立）
- `suspend(reason)` → 发出 `AssetSuspended`
- `reinstate()` → 发出 `AssetReinstated`（前置：当前为 SUSPENDED）
- `retire()` → 发出 `AssetRetired`（终态，之后拒收任何命令）
- `relocate(to_location)` → 发出 `AssetRelocated`
- `changeCustodian(new_custodian)` → 发出 `AssetCustodianChanged`
- `transferOut(target_org)` → 发出 `AssetTransferredOut` + 策略触发 `suspend(TransferredOut)`
- `deployToStation(station_id)` → 发出 `EquipmentDeployedToStation`
- `undeploy(reason)` → 发出 `EquipmentUndeployed`
- `linkDataChannel(channel_id)` → 发出 `EquipmentDataChannelLinked`
- `unlinkDataChannel(reason)` → 发出 `EquipmentDataChannelUnlinked`
- `accrueRunHours(delta, source)` → 发出 `EquipmentRunHourAccrued`
- `updateMetricBaseline(new_baseline, source_task_id)` → 发出 `MetricBaselineReset`
- `updateSpecificationProfile(spec_keys, spec_values)` → 发出 `AssetSpecificationProfileUpdated`

### 1.4 AvailabilityProjection（可用性投影）

**一致性边界**：单件资产的当前可用性判定结果——综合所有阻塞因素后得出"可用/不可用"结论。

**职责**：聚合所有影响可用性的因素，计算并发布可用性变更事件。这是一个**投影聚合**——它不拥有资产本身的数据，而是消费各类事件后重算可用性结论。

```yaml
AvailabilityProjection:
  # ─── 标识 ───
  asset_id:              AssetId                    # 关联的资产ID（聚合根ID）

  # ─── 投影结果 ───
  available:             Boolean                    # 当前是否可用
  blocking_reasons:      List<BlockingReason>       # 当前阻塞原因列表

  # ─── 各维度阻塞状态 ───
  lifecycle_blocked:     Boolean                    # 生命周期状态阻塞（非IN_SERVICE）
  inspection_overdue:    Boolean                    # 点检超期阻塞
  maintenance_overdue:   Boolean                    # 保养超期阻塞
  calibration_blocked:   Boolean                    # 计量阻塞（超期/不合格/送检中）
  malfunction_active:    Boolean                    # 活跃故障未修复
  life_exceeded:         Boolean                    # 寿命超限（工装）
  comm_lost:             Boolean                    # 通信中断（设备）
  critical_alarm:        Boolean                    # 关键报警（设备）
```

**行为**：
- `recompute(triggering_event)` → 综合所有维度重算 available 值，发出 `AssetAvailabilityChanged`
- `markUnavailable(reason)` → 添加阻塞原因，重算可用性
- `clearBlocking(reason_type)` → 移除指定类型的阻塞原因，重算可用性

> **设计说明**：`AvailabilityProjection` 与三类资产聚合根是**1:1 关系**，但作为独立聚合根，因为：
> 1. 可用性重算需要消费**多个域外事件**，一致性边界不同于资产主数据；
> 2. 可用性变更的发布节奏和下游消费方与资产生命周期事件不同；
> 3. 未来可用性判定逻辑可能独立演进（如增加新的阻塞维度），不污染资产聚合根。

### 1.5 ScrapApplication（报废申请）

**一致性边界**：单次报废申请从提交到审批完结的全流程。

**职责**：管理报废申请的提交、审批流推进、审批决策。

```yaml
ScrapApplication:
  # ─── 标识 ───
  application_id:        ApplicationId              # 申请唯一标识（聚合根ID）

  # ─── 关联资产 ───
  asset_id:              AssetId                    # 报废资产
  asset_kind:            AssetKind                  # 资产类型（设备/工装/计量器具）

  # ─── 来源 ───
  source:                ScrapSource                # 发起来源枚举
  source_event_id:       String?                    # 来源事件ID（如维修工单号）
  reason:                String                     # 报废原因

  # ─── 辅助信息 ───
  residual_value:        Decimal?                   # 资产残值
  applicant:             String                     # 申请人
  attachments:           List<Attachment>           # 附件

  # ─── 审批流 ───
  approval_flow:         ApprovalFlow               # 审批流实例
  status:                ScrapApplicationStatus     # 申请状态

  # ─── 审计 ───
  submitted_at:          DateTime
  decided_at:            DateTime?
  decided_by:            String?
```

**行为**：
- `submit(asset_id, source, reason, applicant)` → 发出 `ScrapApplicationSubmitted`（前置：该资产无未决报废申请）
- `approve(approver)` → 推进审批流步骤；若最终步骤通过，发出 `ScrapApplicationApproved`
- `reject(approver, reason)` → 发出 `ScrapApplicationRejected`

### 1.6 FixtureMatchingRule（工装匹配规则）

**一致性边界**：单条工装匹配规则的定义——规定某工装适用于哪些产品/工序/设备型号。

**职责**：发布工装与产品/工序/设备的匹配关系，供制造资源服务工艺设计引用。

```yaml
FixtureMatchingRule:
  # ─── 标识 ───
  rule_id:               RuleId                     # 规则唯一标识（聚合根ID）

  # ─── 关联工装 ───
  fixture_asset_id:      AssetId                    # 关联的工装资产ID

  # ─── 适用范围 ───
  applies_to:            MatchingScope              # 适用范围（产品族/工序/设备型号）

  # ─── 匹配键 ───
  rule_keys:             Map<String, String>        # 匹配键值对（开口规格、Pin数、张力范围等）
```

**行为**：
- `define(fixture_asset_id, applies_to, rule_keys)` → 发出 `FixtureMatchingRuleDefined`
- `update(rule_keys)` → 发出规格更新通知

## 2. 值对象

### 2.1 AssetId（资产唯一标识）

```yaml
AssetId:
  value: String    # UUID 或数据库序列生成的唯一标识
```

> 不可变。由系统在 `AssetRegistered` 事件时生成。

### 2.2 AssetIdentifier（资产编码）

```yaml
AssetIdentifier:
  code:        String       # 资产编码，如 EQ-202606-00123
  code_scheme: CodeScheme   # 编码方案枚举
```

```yaml
CodeScheme:  # 编码方案
  - SELF_ASSIGNED    # 系统自编
  - VENDOR           # 厂商编码
  - ASSET_TAG        # 资产标签
```

> 不可变。一旦赋码不可变更。编码方案为单一标签前缀 + 类型字段，避免重码冲突。

### 2.3 AssetKind（资产类型）

```yaml
AssetKind:
  - EQUIPMENT              # 设备
  - FIXTURE                # 工装夹具
  - MEASURING_INSTRUMENT   # 计量器具
```

> 不可变枚举。在 `AssetRegistered` 时确定，之后不变。

### 2.4 AssetClassification（资产分类）

```yaml
AssetClassification:
  primary_category:   String    # 一级分类
  secondary_category: String    # 二级分类
```

> 可变——通过 `AssetClassificationUpdated` 事件变更。

### 2.5 AvailabilityState（可用性状态）

```yaml
AvailabilityState:
  - NEW        # 已登记，未投用
  - IN_SERVICE # 投用中
  - SUSPENDED  # 停用
  - RETIRED    # 报废（终态）
```

> 状态机驱动，仅通过领域事件变更。

**状态机**：

```
NEW ──Commission──▶ IN_SERVICE
                       │    ▲
                  Suspend  Reinstate
                       │    │
                       ▼    │
                    SUSPENDED
                       │
                    Retire
                       │
                       ▼
                    RETIRED (终态)
```

**不变式**：
- `NEW → IN_SERVICE`：前置条件为 `AssetSpecificationProfileEstablished`
- `IN_SERVICE → SUSPENDED`：载荷必带 `reason`
- `SUSPENDED → IN_SERVICE`：必须有对应的 SUSPENDED 状态
- `→ RETIRED`：终态，之后任何事件均拒收

### 2.6 AssetLocation（资产位置）

```yaml
AssetLocation:
  factory:    String    # 工厂
  workshop:   String    # 车间
  line:       String?   # 产线
  station:    String?   # 工位
  remark:     String?   # 自由文本备注
```

> 四级位置 + 自由备注。产线/工位的引用对接生产执行服务的工厂建模域。

### 2.7 RunHourAccrual（运行时长累积）

```yaml
RunHourAccrual:
  accumulated_run_hours: Decimal    # 累积运行时长（单调递增，永不减小）
  reset_baseline:        Decimal    # 度量基准值（上次重置时的累积值快照）
```

> **关键语义**：`accumulated_run_hours` 单调递增，永不修改绝对累积值。`reset_baseline` 是快照标记，由点检保养域在保养完成时提供。偏移量 = `accumulated_run_hours - reset_baseline`。

### 2.8 CycleAccrual（使用次数累积）

```yaml
CycleAccrual:
  accumulated_cycles: Integer       # 累积使用次数（单调递增，翻修清零除外）
  reset_baseline:     Integer       # 度量基准值
```

> 与 `RunHourAccrual` 同构。`FixtureRefurbished(life_reset=true)` 时允许重置 `accumulated_cycles`。

### 2.9 CertValidity（检定证书有效期）

```yaml
CertValidity:
  cert_no:     String     # 证书编号
  valid_until: Date       # 有效期截止日
  issued_by:   String     # 签发机构
```

> 仅由消费计量检定域事件后更新。`valid_until > now()` 是计量器具可用性的必要条件。

### 2.10 BlockingReason（阻塞原因）

```yaml
BlockingReason:
  reason_type:  BlockingReasonType   # 阻塞类型枚举
  source:       String?              # 来源标识（如工单号、证书号）
  occurred_at:  DateTime             # 阻塞发生时间
```

```yaml
BlockingReasonType:
  - UNDER_REPAIR           # 维修中
  - INSPECTION_ANOMALY     # 点检异常
  - INSPECTION_OVERDUE     # 点检超期
  - MAINTENANCE_OVERDUE    # 保养超期
  - CALIBRATION_FAILED     # 检定不合格
  - CALIBRATION_EXPIRED    # 证书过期
  - CALIBRATING            # 送检中
  - LIFE_EXCEEDED          # 寿命超限
  - COMMUNICATION_LOST     # 通信中断
  - CRITICAL_ALARM         # 关键报警
  - TRANSFERRED_OUT        # 外转
  - MALFUNCTION_ACTIVE     # 活跃故障
```

### 2.11 IssuanceRecord（借出记录）

```yaml
IssuanceRecord:
  station_id:       String      # 借往工位
  issued_to:        String      # 借用人
  issued_at:        DateTime    # 借出时间
  expected_return:  DateTime?   # 预计归还时间
```

> 工装特有值对象。归还后置空 `issued_to`。

### 2.12 ScrapSource（报废来源）

```yaml
ScrapSource:
  - INSPECTION      # 点检保养域：寿命超限
  - REPAIR          # 维修域：维修不经济/无法修复
  - CALIBRATION     # 计量检定域：检定不合格且无法修复
  - ADMINISTRATIVE  # 管理决策：直接发起
```

### 2.13 ScrapApplicationStatus（报废申请状态）

```yaml
ScrapApplicationStatus:
  - SUBMITTED   # 已提交
  - APPROVED    # 已批准
  - REJECTED    # 已驳回
```

### 2.14 ApprovalFlow（审批流）

```yaml
ApprovalFlow:
  approvers:     List<ApproverStep>    # 审批人步骤列表
  current_step:  Int                   # 当前审批步骤（0=未开始）
```

```yaml
ApproverStep:
  step_order:   Int         # 步骤顺序
  role:         String      # 审批角色（如 工程主管 / 财务 / 厂级）
  approver_id:  String?     # 实际审批人ID（审批时填入）
  decided:      Boolean     # 是否已决策
```

> 审批流根据 `ScrapSource` 自动匹配：
> - INSPECTION → 工程主管 → 厂级
> - REPAIR → 工程主管 → 财务 → 厂级
> - CALIBRATION → 计量管理员 → 工程主管
> - ADMINISTRATIVE → 工程主管 → 财务 → 厂级

### 2.15 MatchingScope（工装匹配适用范围）

```yaml
MatchingScope:
  product_family:  String?    # 适用产品族
  process_step:    String?    # 适用工序
  equipment_model: String?    # 适用设备型号
```

### 2.16 SpecificationProfile（能力规格画像）

```yaml
SpecificationProfile:
  spec_keys:    List<String>             # 规格键名列表
  spec_values:  Map<String, String>      # 规格键值对
```

> 三类资产各自持有 `SpecificationProfile` 的特化子类型：
> - `EquipmentSpecificationProfile`：温区数、传送宽度、最大板尺寸等
> - `FixtureSpecificationProfile`：开口规格、Pin数、张力范围等
> - `InstrumentSpecificationProfile`：量程、精度等级、分度值等

## 3. 领域服务

领域服务封装**跨聚合**的业务逻辑或**不自然归属**单一聚合根的操作。

### 3.1 AssetRegistrationService（资产登记服务）

**职责**：编排资产入账与赋码的跨步骤流程。

```yaml
AssetRegistrationService:
  方法:
    registerAsset(kind, model, vendor, custodian_org):
      # 1. 创建资产聚合根，发出 AssetRegistered
      # 2. 自动赋码策略：分配 AssetIdentifier，发出 AssetIdentifierAssigned
      # 3. 建立初始能力规格画像，发出 AssetSpecificationProfileEstablished
      # 4. 创建 AvailabilityProjection 初始实例
```

> **为什么不放在聚合根内**：登记涉及创建多个聚合根（Asset + AvailabilityProjection），跨聚合的一致性编排由领域服务承担。

### 3.2 AvailabilityRecomputeService（可用性重算服务）

**职责**：消费域外运维事件后，驱动资产状态变更与可用性重算。这是本域最核心的领域服务。

```yaml
AvailabilityRecomputeService:
  方法:
    # ─── 点检保养域事件消费 ───
    onInspectionCompleted(asset_id, verdict):
      if verdict == FAIL → asset.suspend(InspectionAnomaly)
                           projection.markUnavailable(INSPECTION_ANOMALY)
      if verdict == PASS → (无状态变更)

    onInspectionOverdue(asset_id):
      projection.markUnavailable(INSPECTION_OVERDUE)

    onMaintenanceOverdue(asset_id):
      projection.markUnavailable(MAINTENANCE_OVERDUE)

    onMaintenanceCompleted(asset_id, metric_reset, reset_metric_kind, reset_baseline):
      projection.clearBlocking(MAINTENANCE_OVERDUE)
      if metric_reset → asset.updateMetricBaseline(reset_baseline, source_task_id)

    # ─── 计量检定域事件消费 ───
    onCalibrationCertificateIssued(asset_id, cert_no, valid_until, issued_by):
      asset.updateCertValidity(cert_no, valid_until, issued_by)
      asset.reinstate()  # 如果之前因计量原因被 suspend
      projection.clearBlocking(CALIBRATING/CALIBRATION_FAILED/CALIBRATION_EXPIRED)

    onCalibrationFailed(asset_id):
      asset.suspend(CalibrationFailed)
      projection.markUnavailable(CALIBRATION_FAILED)

    onCalibrationExpired(asset_id):
      asset.suspend(CalibrationExpired)
      projection.markUnavailable(CALIBRATION_EXPIRED)

    onCalibrationDispatched(asset_id):
      asset.suspend(Calibrating)
      projection.markUnavailable(CALIBRATING)

    # ─── 维修域事件消费 ───
    onRepairCompleted(asset_id):
      projection.clearBlocking(UNDER_REPAIR / MALFUNCTION_ACTIVE)
      # 综合所有阻塞原因判定是否 ReinstateAsset
      recomputeAndDecideReinstate(asset_id)

    onScrapRecommendation(asset_id, source_event_id):
      # 不直接停用资产，而是触发报废审批子流程
      initiateScrapApplication(asset_id, REPAIR, source_event_id)

    # ─── 生产执行服务/接入域事件消费 ───
    onMalfunctionReported(asset_id):
      asset.suspend(UnderRepair)
      projection.markUnavailable(MALFUNCTION_ACTIVE)

    # ─── 接入域事件消费 ───
    onChannelEscalated(asset_id):
      asset.suspend(CommunicationLost)
      projection.markUnavailable(COMMUNICATION_LOST)

    onCriticalAlarmRaised(asset_id):
      asset.suspend(CriticalAlarm)
      projection.markUnavailable(CRITICAL_ALARM)

    # ─── 点检保养域寿命事件消费 ───
    onFixtureLifeExceeded(asset_id):
      asset.suspend(LifeExceeded)
      projection.markUnavailable(LIFE_EXCEEDED)

    # ─── 内部辅助 ───
    recomputeAndDecideReinstate(asset_id):
      # 重算可用性，若无任何阻塞维度 → ReinstateAsset
      projection.recompute(triggering_event)
      if projection.available → asset.reinstate()

    initiateScrapApplication(asset_id, source, source_event_id):
      # 委托 ScrapApplication 聚合根创建申请
```

> **设计说明**：此服务是本域的**事件消费中枢**，所有域外运维事件都经过此服务翻译为资产状态变更命令。该服务**不包含阈值判定逻辑**——阈值判定归点检保养域/计量检定域。

### 3.3 ScrapApprovalService（报废审批服务）

**职责**：编排报废审批子流程，包括审批流匹配、审批推进、审批结果执行。

```yaml
ScrapApprovalService:
  方法:
    submitScrapApplication(asset_id, source, source_event_id, reason, applicant):
      # 1. 校验：同一资产不可有未决报废申请
      # 2. 根据 source 匹配审批流
      # 3. 创建 ScrapApplication 聚合根
      # 发出 ScrapApplicationSubmitted

    approveApplication(application_id, approver):
      # 推进审批流步骤
      # 若最终步骤通过 → ScrapApplicationApproved → 执行 RetireAsset
      # 发出 ScrapApplicationApproved

    rejectApplication(application_id, approver, reason):
      # 发出 ScrapApplicationRejected → 通知来源域
```

## 4. 应用服务

应用服务是**外部调用入口**，负责：
1. 接收外部请求（REST / 消息消费）
2. 事务边界管理
3. 调用领域服务编排业务流程
4. 发布集成事件到对外主题

### 4.1 AssetCommandService（资产命令应用服务）

**职责**：处理资产生命周期的写操作命令。

```yaml
AssetCommandService:
  方法:
    registerAsset(cmd: RegisterAssetCommand):
      # 委托 AssetRegistrationService.registerAsset()
      # 事务内完成

    commissionAsset(asset_id):
      # 加载资产聚合根 → asset.commission()
      # 发布事件到 eam.asset.lifecycle

    suspendAsset(asset_id, reason):
      # 加载资产聚合根 → asset.suspend(reason)
      # 触发可用性重算
      # 发布事件到 eam.asset.lifecycle + eam.asset.availability

    reinstateAsset(asset_id):
      # 加载资产聚合根 → asset.reinstate()
      # 触发可用性重算
      # 发布事件到 eam.asset.lifecycle + eam.asset.availability

    retireAsset(asset_id):
      # 加载资产聚合根 → asset.retire()
      # 触发可用性重算
      # 发布事件到 eam.asset.lifecycle + eam.asset.availability

    relocateAsset(asset_id, to_location):
      # 加载资产聚合根 → asset.relocate(to_location)
      # 发布事件到 eam.asset.location

    changeCustodian(asset_id, new_custodian):
      # 加载资产聚合根 → asset.changeCustodian(new_custodian)

    transferOutAsset(asset_id, target_org):
      # 加载资产聚合根 → asset.transferOut(target_org)
      # 策略自动触发 suspend(TransferredOut)

    # ─── 设备特有 ───
    deployEquipmentToStation(asset_id, station_id): ...
    undeployEquipment(asset_id, reason): ...
    linkDataChannel(asset_id, channel_id): ...
    unlinkDataChannel(asset_id, reason): ...
    accrueRunHours(asset_id, delta, source): ...

    # ─── 工装特有 ───
    issueFixtureToStation(asset_id, station_id, issued_to, expected_return): ...
    returnFixtureToStorage(asset_id, condition): ...
    accrueCycles(asset_id, delta, source): ...
    cleanFixture(asset_id, method): ...
    refurbishFixture(asset_id, log, life_reset): ...

    # ─── 计量器具特有 ───
    updateCertValidity(asset_id, cert_no, valid_until, issued_by): ...

    # ─── 度量基准 ───
    updateMetricBaseline(asset_id, metric_kind, new_baseline, source_task_id): ...

    # ─── 规格画像 ───
    updateSpecificationProfile(asset_id, spec_keys, spec_values): ...
```

## 6. 不变规则（Invariants）

不变规则是领域模型中**必须始终为真**的业务约束，无论经过何种操作序列。

### 6.1 资产身份不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-01 | 同一资产编码全局唯一 | `AssetIdentifier.code` 在全局范围内不可重复 |
| INV-02 | 资产类型一旦确定不可变更 | `AssetKind` 在 `AssetRegistered` 时确定，之后不可修改 |
| INV-03 | `AssetRetired` 后任何事件均拒收 | 报废是终态，不可逆 |
| INV-04 | 同一资产不可重复提交未决报废申请 | 防止并行报废申请冲突 |

### 6.2 状态流转不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-05 | `AssetCommissioned` 前必须已建立能力规格画像 | `AssetSpecificationProfileEstablished` 是投用的前置条件 |
| INV-06 | `AssetSuspended` 载荷必带 `reason` | 停用必须有明确原因 |
| INV-07 | `AssetReinstated` 必须有对应的 `AssetSuspended` | 只能从 SUSPENDED 状态恢复 |
| INV-08 | 状态流转必须遵循状态机定义 | NEW→IN_SERVICE→SUSPENDED⇄IN_SERVICE→RETIRED |

### 6.3 累积度量不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-09 | `accumulated_run_hours` 单调递增 | 运行时长绝对累积值永不减小 |
| INV-10 | `accumulated_cycles` 单调递增（翻修清零除外） | 工装使用次数永不减小，除非 `FixtureRefurbished(life_reset=true)` |
| INV-11 | `reset_baseline` 仅由点检保养域通过 `MaintenanceCompleted(metric_reset=true)` 驱动更新 | 本域不主动修改基准值 |
| INV-12 | 偏移量 = 累积值 - 基准值 ≥ 0 | 由单调递增和基准值语义保证 |

### 6.4 可用性不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-13 | `AssetAvailabilityChanged.available = true` 当且仅当所有阻塞维度均不成立 | 可用性是所有维度的合取 |
| INV-14 | 任意影响可用性的事件必须触发重算并发布 `AssetAvailabilityChanged` | 可用性变更必须幂等发布 |
| INV-15 | 计量器具可用必须 `cert_valid_until > now()` | 证书过期则不可用 |
| INV-16 | `SuspendAsset` 导致状态变为 SUSPENDED + available=false | 硬阻塞：双重不可用 |
| INV-17 | `MarkAssetUnavailable` 仅导致 available=false，状态保持 IN_SERVICE | 条件阻塞：合规性不满足但资产本身在线 |

### 6.5 报废审批不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-18 | `ScrapApplicationApproved` 后必须执行 `RetireAsset` | 审批通过 → 资产退役，强一致 |
| INV-19 | 审批流步骤必须按顺序推进 | 不可跳步审批 |
| INV-20 | 审批驳回后需通知来源域 | 如维修域工单退回 DIAGNOSED 重新评估 |

### 6.6 跨域协作不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-21 | `EquipmentOnline ≠ AssetCommissioned` | 通信在线 ≠ 资产投用，两者独立维护 |
| INV-22 | 本域不存储阈值规则 | 保养周期、点检周期、寿命上限、检定周期归点检保养域/计量检定域 |
| INV-23 | 本域不判定"是否达到阈值" | 阈值判定归点检保养域，本域只发布度量值 |
| INV-24 | 资产报废/外转等强生命周期变更只能从本域发起 | 避免下游服务越权 |

## 9. 领域服务与应用服务的关系

```
┌──────────────────────────────────────────────────────────────────┐
│                        外部调用入口                               │
│                                                                  │
│  REST API ──▶ AssetCommandService (应用服务)                     │
│               ├─ 单聚合操作：直接委托聚合根方法                    │
│               └─ 跨聚合编排：委托领域服务                         │
│                                                                  │
│  消息消费 ──▶ AssetEventConsumer (应用服务)                      │
│               └─ 委托 AvailabilityRecomputeService (领域服务)     │
│                                                                  │
│  审批操作 ──▶ ScrapApplicationService (应用服务)                 │
│               └─ 委托 ScrapApprovalService (领域服务)             │
│                                                                  │
│  查询请求 ──▶ AssetQueryService (应用服务，读模型)               │
│               └─ 直接查询读模型投影                               │
└──────────────────────────────────────────────────────────────────┘
```

**原则**：
- 应用服务不包含业务逻辑，只做编排和事务边界
- 领域服务封装跨聚合的业务规则
- 聚合根内的业务逻辑由聚合根自身保证
- 不变式由聚合根 + 领域服务共同守护

| 事件风暴概念 | 领域建模对应 |
|-------------|-------------|
| 【聚合:Asset】 | Equipment / Fixture / MeasuringInstrument 三个聚合根，共享主链事件 |
