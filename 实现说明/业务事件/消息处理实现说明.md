# 业务事件消息处理实现说明

> 适用范围：本文用于说明业务事件采用 Transactional Outbox + Kafka 的整体实现方式和代码职责划分。

---

## 1. 设计目标

领域服务在处理业务命令时，经常需要同时完成两件事：

1. 修改本服务数据库中的业务状态。
2. 发布业务事件，通知其它服务或后续处理流程。

如果业务代码在数据库事务提交后直接调用 Kafka Producer，则存在崩溃窗口：

```text
DB commit 成功
应用进程在 send Kafka 前宕机
Kafka 永远收不到该业务事件
```

Transactional Outbox 的目标是将“业务状态变更”和“待发布事件记录”放在同一个本地数据库事务中提交：

```text
同一个本地事务：
  1. 更新业务表
  2. 插入 outbox_event(PENDING)

事务提交后：
  3. Outbox Publisher 异步读取 outbox_event
  4. 发布 Kafka
  5. 更新 outbox_event 状态
```

这样即使应用在任意时刻崩溃，未发布成功的事件仍然保留在 outbox 表中，可以由后台 Publisher 继续投递。

## 3. Outbox 状态模型

Outbox 记录建议保留以下状态：

| 状态 | 是否可被 Publisher 拉取 | 是否终态 | 说明 |
|---|---:|---:|---|
| `PENDING` | 是 | 否 | 初始状态，等待发布。 |
| `PUBLISHING` | 否 | 否 | 已被 Publisher 领取，正在发送。 |
| `SENT` | 否 | 是 | 已成功发送到 Kafka。 |
| `RETRYABLE` | 是，但需到达 `next_retry_at` | 否 | 上次发送失败，等待重试。 |
| `DEAD_LETTER` | 否 | 是 | 重试耗尽或不可恢复错误，等待人工处理或补偿。 |
| `DISCARDED` | 否 | 是 | 人工确认不再发布。 |

状态流转：

```text
               ┌────────────────────────┐
               │                        ▼
PENDING ──→ PUBLISHING ──send ok──→ SENT
   ▲            │
   │            ├─send failed, retryable──→ RETRYABLE
   │            │                            │
   │            │                            └─next_retry_at reached──┘
   │            │
   │            ├─retry exhausted────────→ DEAD_LETTER
   │            │
   │            └─lock timeout───────────→ RETRYABLE
   │
   └─manual replay from DEAD_LETTER, if fixed

DEAD_LETTER ──manual discard──→ DISCARDED
```

Outbox 表结构、索引、领取 SQL、恢复 SQL 统一维护在《[MySQL配置说明](MySQL配置说明.md)》。

## 4. 生产侧实现

### 4.1 应用服务事务

应用服务必须在同一个 `@Transactional` 中完成业务状态变更和 outbox 写入。

```java
@Transactional(rollbackFor = Exception.class)
public void handleCommand(BusinessCommand command) {
    Aggregate aggregate = repository.getById(command.aggregateId());

    DomainEvent event = aggregate.handle(command);

    repository.save(aggregate);

    OutboxEvent outboxEvent = outboxEventFactory.create(
        event,
        topicResolver.resolve(event),
        partitionKeyResolver.resolve(event)
    );

    outboxEventRepository.save(outboxEvent);
}
```

要求：

- 业务状态变更和 outbox 写入必须在同一个本地事务中提交。
- 事务内只写本地数据库，不直接发送 Kafka。
- 如果事务回滚，业务状态和 outbox 记录一起回滚。

### 4.2 聚合根职责

聚合根只表达业务事实，不依赖消息基础设施。

```java
public class Aggregate {

    public DomainEvent handle(BusinessCommand command) {
        // 校验业务规则
        // 修改聚合状态
        // 返回领域事件
    }
}
```

禁止聚合根直接依赖：

- `KafkaTemplate`
- `ApplicationEventPublisher`
- `OutboxRepository`
- 任何消息发送器

### 4.3 事件 Envelope

业务事件建议统一使用 Envelope。Envelope 字段、Kafka Header 建议和版本演进规则见《[Kafka配置说明](Kafka配置说明.md)》。

统一 Envelope 的目的：

- 消费方可以用 `event_id` 做幂等。
- 监控系统可以按 `event_type`、`source_service` 聚合。
- 后续 schema 演进有统一入口。
- 追踪链路可以通过 `trace_id` 串联。

## 5. Outbox Publisher 实现

Publisher 负责把 outbox 表中的待发布记录投递到 Kafka，并更新状态。

### 5.1 定时扫描

建议使用 Spring Scheduler 或独立 Worker。

```java
@Component
public class OutboxPublisherJob {

    private final OutboxPublisher outboxPublisher;

    @Scheduled(fixedDelayString = "${app.outbox.publisher.fixed-delay:1000}")
    public void publish() {
        outboxPublisher.publishBatch();
    }
}
```

### 5.2 领取事件

多实例部署时，必须避免多个 Publisher 同时处理同一行。

领取要求：

- 只领取 `PENDING` / `RETRYABLE` 且到达 `next_retry_at` 的记录。
- 领取和置为 `PUBLISHING` 必须在同一个短事务中完成。
- Kafka 发送不能放在领取事务内。
- `PUBLISHING` 超时后必须可恢复为 `RETRYABLE`。

具体 MySQL SQL 见《[MySQL配置说明](MySQL配置说明.md)》。

### 5.3 发布 Kafka

Publisher 领取到事件后，逐条或批量发送 Kafka。

```java
public void publishOne(OutboxEvent event) {
    kafkaOutboxMessageSender.send(event);
    outboxEventRepository.markSent(event.getId(), Instant.now());
}
```

实现建议：

- 低吞吐场景可以同步等待 Kafka ack，逻辑简单。
- 中高吞吐场景可以使用异步 future + 批量状态更新。
- 发送失败后，根据异常类型和重试次数更新为 `RETRYABLE` 或 `DEAD_LETTER`。
- Producer 可靠性参数见《[Kafka配置说明](Kafka配置说明.md)》。

## 6. 消费侧实现

Outbox + Kafka 提供的是至少一次投递，因此重复消息是正常情况。消费方必须使用 `event_id` 幂等去重。

消费处理顺序：

```text
Kafka poll 收到消息
  → 开启本地 DB 事务
    → 插入 consumed_event 幂等记录
    → 执行业务变更
  → DB commit
  → 手动 ack Kafka offset
```

要求：

- 消费幂等记录和业务处理必须在同一个本地事务中提交。
- 如果幂等记录唯一键冲突，说明已经处理过，直接 ack。
- 如果 DB commit 前抛异常，不 ack，Kafka 后续重投。
- 如果 DB commit 成功但 ack 前应用崩溃，Kafka 会重复投递，消费方通过幂等表跳过。

消费幂等表见《[MySQL配置说明](MySQL配置说明.md)》。Consumer 参数、ack 顺序、DLT 配置见《[Kafka配置说明](Kafka配置说明.md)》。

## 7. 事务边界

### 7.1 生产侧事务边界

```text
一个业务命令 = 一个本地数据库事务
事务内：
  - 加载聚合根
  - 执行业务方法
  - 保存业务状态
  - 插入 outbox_event
事务外：
  - 不保证立即发布 Kafka
```

禁止：

```text
@Transactional 方法中直接调用 kafkaTemplate.send()
```

允许但不作为可靠性来源：

```text
事务提交后 afterCommit() 立即唤醒 Publisher
```

### 7.2 消费侧事务边界

```text
Kafka poll 收到消息
  → 开启本地 DB 事务
    → 幂等检查/插入 consumed_event
    → 执行业务变更
  → DB commit
  → 手动 ack Kafka offset
```

## 8. 快速路径：afterCommit 唤醒 Publisher

为了降低延迟，可以在事务提交后触发一次本地发布信号，但仍保留后台轮询兜底。

```java
@Transactional(rollbackFor = Exception.class)
public void handleCommand(BusinessCommand command) {
    // 保存业务状态和 outbox_event

    TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            outboxPublishSignal.tryWakeup();
        }
    });
}
```

注意：

- `afterCommit` 只负责唤醒 Publisher，不直接替代 Publisher。
- 即使 `afterCommit` 未执行，定时任务仍会扫描 `PENDING`。
- 如果需要极低延迟，可使用内存队列传递 outbox id，但发布前仍应从 DB 读取并校验状态。

## 9. 可观测性与告警

必须暴露以下指标：

| 指标 | 含义 |
|---|---|
| `outbox_pending_count` | PENDING + RETRYABLE 待发布数量。 |
| `outbox_publishing_count` | PUBLISHING 数量。 |
| `outbox_dead_letter_count` | DEAD_LETTER 数量。 |
| `outbox_publish_latency_seconds` | `published_at - created_at`。 |
| `outbox_publish_success_total` | 成功发布总数。 |
| `outbox_publish_failure_total` | 发布失败总数。 |
| `outbox_retry_total` | 重试总数。 |
| `consumer_duplicate_event_total` | 消费方幂等去重次数。 |
| `consumer_dead_letter_total` | 消费方进入 DLT 的消息数量。 |

告警建议：

| 告警条件 | 严重性 |
|---|---|
| `DEAD_LETTER > 0` 持续 5 分钟 | 高 |
| 最老 PENDING 超过 1 分钟 | 中/高，取决于业务 SLA |
| PENDING 数量持续增长 | 高 |
| PUBLISHING 锁超时数量增长 | 中 |
| Kafka send failure rate 超过阈值 | 高 |
| 消费 DLT 有新增 | 高 |

## 10. 数据清理与归档

Outbox 表不建议无限增长。

清理和归档规则见《[MySQL配置说明](MySQL配置说明.md)》。原则上：

- `SENT` 只保留短期运维窗口。
- `DEAD_LETTER` 保留到人工处理完成。
- 需要长期审计的事件应归档到对象存储、数据湖或事件审计库。

```text

    MyBatisOutboxEventMapper.java
```

业务域使用方式：

```text
<business-module>/application
  ApplicationService

```

不要让业务域直接依赖 Kafka 细节，只依赖 outbox 抽象。

## 12. 实现检查清单

### 12.1 生产侧

- [ ] 业务状态变更和 outbox 写入在同一个 `@Transactional` 中。
- [ ] 聚合根不直接依赖 Kafka。
- [ ] 每条事件都有全局唯一 `event_id`。
- [ ] 每条事件都有 `event_type`、`event_version`、`occurred_at`。
- [ ] Kafka `partition_key` 能保证同一聚合根有序。
- [ ] Publisher 支持多实例并发领取。
- [ ] Publisher 支持失败重试和死信。
- [ ] `PUBLISHING` 锁超时可恢复。
- [ ] `SENT` 数据有清理策略。

### 12.2 消费侧

- [ ] `enable-auto-commit=false`。
- [ ] 业务处理成功后手动 ack。
- [ ] 使用 `event_id + consumer_group` 幂等去重。
- [ ] 消费失败有重试和 DLT。
- [ ] DLT 有告警和人工处理流程。

### 12.3 可观测性

- [ ] 监控 PENDING 积压。
- [ ] 监控发布延迟。
- [ ] 监控 DEAD_LETTER。
- [ ] 监控 Kafka producer 错误率。
- [ ] 监控消费方 DLT。

## 13. 推荐实施顺序

1. 建立业务事件 MySQL 表结构，见《[MySQL配置说明](MySQL配置说明.md)》。
2. 建立业务事件 Kafka Topic 与 DLT，见《[Kafka配置说明](Kafka配置说明.md)》。
3. 实现 `OutboxEventFactory`，统一生成 Envelope、payload、headers。
4. 改造应用服务：业务状态变更时同步写入 outbox。
5. 实现 `OutboxPublisher`：扫描、领取、发送 Kafka、更新状态。
6. 增加 `OutboxRecoveryJob`：恢复超时的 `PUBLISHING`。
7. 增加 `OutboxCleanupJob`：清理过期 `SENT`。
8. 为消费者建立幂等表 `consumed_event`。
9. 实现消费者手动 ack、失败重试、DLT。
10. 增加监控指标和告警。
11. 先选择低频、影响范围可控的业务事件试点。
12. 验证稳定后推广到其它业务事件。

---

## 14. 关键原则总结

1. **业务事务只负责写业务表和 outbox 表，不直接承担可靠 Kafka 投递。**
2. **Outbox 表是可靠性的来源，Publisher 是投递执行器。**
3. **Kafka 投递按至少一次处理，消费方必须幂等。**
4. **同一聚合根使用相同 partition key，保证单聚合根事件有序。**
5. **低频业务事件适合走 Outbox，高频原始采集数据不要走 Outbox。**
6. **即使当前只有服务内部消费者，只要是业务契约事件，也统一走 Kafka + Outbox。**
