# Java + Spring + Outbox + Kafka 消息处理通用实现说明

> 适用范围：本文用于指导当前 MES 项目中各领域服务实现可靠领域事件发布与消费。重点面向 Java、Spring Boot、关系型数据库、Transactional Outbox、Kafka 的组合方案。
>
> 目标：保证领域状态变更与领域事件发布之间具备可靠的最终一致性，避免“数据库事务已提交但 Kafka 消息丢失”的双写问题。

---

## 1. 总体设计目标

领域服务在处理业务命令时，经常需要同时完成两件事：

1. 修改本服务数据库中的业务状态。
2. 向 Kafka 发布领域事件，通知其它域或服务。

例如台账域执行资产退役：

```text
RetireAssetCommand
  → 修改资产状态为 RETIRED
  → 发布 AssetRetired 到 eam.asset.lifecycle
```

如果业务代码在数据库提交后直接调用 Kafka Producer，则存在崩溃窗口：

```text
DB commit 成功
应用进程在 send Kafka 前宕机
Kafka 永远收不到 AssetRetired
```

Outbox 模式的目标是将“业务状态变更”和“待发布事件记录”放进同一个数据库事务中提交：

```text
同一个本地事务：
  1. 更新业务表
  2. 插入 outbox_event(PENDING)

事务提交后：
  3. Outbox Publisher 异步读取 outbox_event
  4. 发布 Kafka
  5. 标记 outbox_event 为 SENT
```

这样即使应用在任意时刻崩溃，未发布成功的事件仍然保留在 outbox 表中，可以被后台发布器继续投递。

## 3. Outbox 表设计

### 3.1 表名

建议每个服务拥有自己的 outbox 表：

```text
outbox_event
```

如果服务内有多个库或多个 bounded context，也可以按库拆分：

```text
asset_outbox_event
pm_outbox_event
repair_outbox_event
```

但优先推荐服务内统一一张 `outbox_event`，通过 `source_context`、`topic`、`aggregate_type` 区分。

### 3.2 字段定义

```sql
CREATE TABLE outbox_event (
    id                  VARCHAR(64)   NOT NULL PRIMARY KEY,
    event_id            VARCHAR(64)   NOT NULL,
    topic               VARCHAR(128)  NOT NULL,
    event_type          VARCHAR(128)  NOT NULL,
    event_version       INT           NOT NULL DEFAULT 1,

    aggregate_type      VARCHAR(64)   NOT NULL,
    aggregate_id        VARCHAR(128)  NOT NULL,
    aggregate_version   BIGINT        NULL,

    partition_key       VARCHAR(128)  NOT NULL,
    payload             JSON          NOT NULL,
    headers             JSON          NULL,

    source_service      VARCHAR(64)   NOT NULL,
    source_context      VARCHAR(64)   NOT NULL,
    trace_id            VARCHAR(128)  NULL,
    correlation_id      VARCHAR(128)  NULL,
    causation_id        VARCHAR(128)  NULL,

    status              VARCHAR(32)   NOT NULL,
    retry_count         INT           NOT NULL DEFAULT 0,
    max_retry_count     INT           NOT NULL DEFAULT 20,
    next_retry_at       DATETIME      NOT NULL,

    locked_by           VARCHAR(128)  NULL,
    locked_at           DATETIME      NULL,

    last_error_code     VARCHAR(128)  NULL,
    last_error_message  TEXT          NULL,

    occurred_at         DATETIME      NOT NULL,
    created_at          DATETIME      NOT NULL,
    published_at        DATETIME      NULL,
    dead_lettered_at    DATETIME      NULL,

    UNIQUE KEY uk_outbox_event_id (event_id),
    KEY idx_outbox_publishable (status, next_retry_at, created_at),
    KEY idx_outbox_aggregate (aggregate_type, aggregate_id, created_at),
    KEY idx_outbox_topic_status (topic, status, created_at)
);
```

> 数据库为 PostgreSQL 时，`JSON` 可替换为 `JSONB`；MySQL 5.7+ 可使用 `JSON`。如果使用 JPA 不便映射 JSON，也可以先使用 `TEXT` 存储 JSON 字符串。

### 3.3 字段说明

| 字段 | 说明 |
|---|---|
| `id` | outbox 表主键，可与 `event_id` 相同，也可独立生成。 |
| `event_id` | 全局事件唯一 ID，消费方幂等去重的核心字段。 |
| `topic` | Kafka topic，例如 `eam.asset.lifecycle`。 |
| `event_type` | 领域事件类型，例如 `AssetRetired`。 |
| `event_version` | 事件契约版本，用于 schema 演进。 |
| `aggregate_type` | 聚合根类型，例如 `Asset`、`ScrapApplication`。 |
| `aggregate_id` | 聚合根 ID，例如资产 ID。 |
| `aggregate_version` | 聚合根版本号，可用于消费方判断顺序或并发冲突。 |
| `partition_key` | Kafka 分区键。台账域建议使用 `asset_id`，保证单资产事件有序。 |
| `payload` | 领域事件正文。 |
| `headers` | Kafka headers 的业务扩展。 |
| `source_service` | 来源服务，例如 `equipment-management-service`。 |
| `source_context` | 来源限界上下文，例如 `asset-ledger`。 |
| `trace_id` | 链路追踪 ID。 |
| `correlation_id` | 业务流程关联 ID，同一业务流程中的多个事件可共用。 |
| `causation_id` | 导致当前事件产生的上游事件或命令 ID。 |
| `status` | 消息状态，见下文。 |
| `retry_count` | 已重试次数。 |
| `max_retry_count` | 最大重试次数。 |
| `next_retry_at` | 下次可投递时间。 |
| `locked_by` | 被哪个 publisher 实例锁定。 |
| `locked_at` | 锁定时间。 |
| `last_error_*` | 最近一次失败原因。 |
| `occurred_at` | 领域事件发生时间。 |
| `created_at` | outbox 行创建时间。 |
| `published_at` | 成功发布到 Kafka 的时间。 |
| `dead_lettered_at` | 进入死信状态的时间。 |

## 4. 消息状态类型定义

### 4.1 状态枚举

Java 枚举建议如下：

```java
public enum OutboxStatus {
    /** 已写入 outbox，等待发布 */
    PENDING,

    /** 已被某个 publisher 实例领取，正在发送 */
    PUBLISHING,

    /** 已成功发送到 Kafka */
    SENT,

    /** 发送失败，但仍可重试 */
    RETRYABLE,

    /** 超过最大重试次数，或被判定为不可恢复错误 */
    DEAD_LETTER,

    /** 人工处理后决定忽略，不再发布 */
    DISCARDED
}
```

### 4.2 状态语义

| 状态 | 是否可被 Publisher 拉取 | 是否终态 | 说明 |
|---|---:|---:|---|
| `PENDING` | 是 | 否 | 初始状态。事务提交后等待发布。 |
| `PUBLISHING` | 否 | 否 | Publisher 已领取，正在发送。防止多实例重复领取。 |
| `SENT` | 否 | 是 | Kafka 已确认成功。 |
| `RETRYABLE` | 是，但需到达 `next_retry_at` | 否 | 上次发送失败，等待重试。 |
| `DEAD_LETTER` | 否 | 是 | 重试耗尽或不可恢复错误，等待人工处理或补偿。 |
| `DISCARDED` | 否 | 是 | 人工确认不再发布。 |

> 最小实现可以只有 `PENDING`、`SENT`、`DEAD_LETTER` 三个状态。但为了可观测、可并发、可恢复，建议保留上面的完整状态集。

## 5. 消息状态流转

### 5.1 正常流转

```text
PENDING
  └─ Publisher 领取成功
      → PUBLISHING
          └─ Kafka send ack 成功
              → SENT
```

触发条件：

1. 业务事务中插入 outbox 行，初始状态为 `PENDING`。
2. Publisher 定时扫描 `PENDING` 且 `next_retry_at <= now()` 的记录。
3. Publisher 通过数据库锁或状态更新将记录置为 `PUBLISHING`。
4. Kafka Producer 收到 broker ack 后，将状态置为 `SENT`。

### 5.2 可重试失败流转

```text
PUBLISHING
  └─ Kafka 发送失败，且 retry_count < max_retry_count
      → RETRYABLE
          └─ 到达 next_retry_at
              → PUBLISHING
```

典型可重试错误：

- Kafka broker 短暂不可用。
- 网络超时。
- Leader 选举中。
- Producer 发送超时。
- 可恢复的认证/连接抖动。

处理策略：

```text
retry_count += 1
next_retry_at = now + backoff(retry_count)
status = RETRYABLE
last_error_message = 当前异常摘要
```

推荐退避策略：

```text
第 1 次：5 秒
第 2 次：15 秒
第 3 次：30 秒
第 4 次：1 分钟
第 5 次：5 分钟
之后：最多 30 分钟一次
```

可使用指数退避 + 上限：

```java
Duration delay = min(Duration.ofMinutes(30), Duration.ofSeconds((long) Math.pow(2, retryCount) * 5));
```

### 5.3 重试耗尽流转

```text
PUBLISHING / RETRYABLE
  └─ retry_count >= max_retry_count
      → DEAD_LETTER
```

触发条件：

- 重试次数达到上限。
- 明确识别为不可恢复错误。

不可恢复错误示例：

- topic 不存在且禁止自动创建。
- 序列化失败。
- payload 不符合事件 schema。
- partition key 缺失。
- 事件版本不受支持。

进入 `DEAD_LETTER` 后，必须有监控告警。运维或开发人员处理后可以：

1. 修复数据，将状态改回 `RETRYABLE`。
2. 确认该事件不应再发，将状态改为 `DISCARDED`。
3. 人工重新生成一条补偿事件。

### 5.4 Publisher 崩溃恢复流转

可能出现：记录已被置为 `PUBLISHING`，但 Publisher 在发送 Kafka 前后宕机。

恢复规则：

```text
PUBLISHING 且 locked_at 超过超时时间
  → RETRYABLE
```

建议超时时间：

```text
publisher.lock-timeout = 5 分钟
```

恢复任务定期执行：

```sql
UPDATE outbox_event
SET status = 'RETRYABLE',
    locked_by = NULL,
    locked_at = NULL,
    next_retry_at = CURRENT_TIMESTAMP,
    last_error_code = 'PUBLISHING_LOCK_TIMEOUT',
    last_error_message = 'Publisher lock timeout, reset to retryable'
WHERE status = 'PUBLISHING'
  AND locked_at < CURRENT_TIMESTAMP - INTERVAL '5' MINUTE;
```

注意：如果崩溃发生在 Kafka 已发送成功但尚未更新 `SENT` 之前，恢复后会重复发送同一 `event_id`。因此消费方必须实现幂等。

### 5.5 状态流转总图

```text
               ┌────────────────────────┐
               │                        ▼
PENDING ──→ PUBLISHING ──send ok──→ SENT
   ▲            │
   │            ├─send failed, retryable──→ RETRYABLE
   │            │                            │
   │            │                            └─next_retry_at reached──┘
   │            │
   │            ├─retry exhausted────────→ DEAD_LETTER
   │            │
   │            └─lock timeout───────────→ RETRYABLE
   │
   └─manual replay from DEAD_LETTER, if fixed

DEAD_LETTER ──manual discard──→ DISCARDED
```

## 6. 业务侧写入 Outbox 的步骤

### 6.1 应用服务处理命令

应用服务必须在同一个 `@Transactional` 中完成业务状态变更和 outbox 写入。

示例：资产退役。

```java
@Service
public class AssetApplicationService {

    private final AssetRepository assetRepository;
    private final OutboxEventRepository outboxEventRepository;
    private final DomainEventMapper domainEventMapper;

    @Transactional
    public void retireAsset(RetireAssetCommand command) {
        Asset asset = assetRepository.getById(command.assetId());

        AssetRetired event = asset.retire(command.reason(), command.operatorId());

        assetRepository.save(asset);

        OutboxEvent outboxEvent = domainEventMapper.toOutboxEvent(
            event,
            "eam.asset.lifecycle",
            asset.getId().value()
        );

        outboxEventRepository.save(outboxEvent);
    }
}
```

### 6.2 聚合根只产生领域事件，不直接发 Kafka

聚合根负责业务规则和状态变化：

```java
public class Asset {

    public AssetRetired retire(String reason, String operatorId) {
        if (this.status == AssetStatus.RETIRED) {
            throw new DomainException("资产已退役，不能重复退役");
        }

        this.status = AssetStatus.RETIRED;
        this.retiredAt = Instant.now();

        return new AssetRetired(
            EventId.newId(),
            this.id.value(),
            reason,
            operatorId,
            Instant.now()
        );
    }
}
```

> 聚合根不要依赖 KafkaTemplate、ApplicationEventPublisher 或 OutboxRepository。聚合根只表达领域事实。

### 6.3 事件 Envelope

Kafka 中建议统一使用事件信封结构：

```json
{
  "event_id": "01J...",
  "event_type": "AssetRetired",
  "event_version": 1,
  "source_service": "equipment-management-service",
  "source_context": "asset-ledger",
  "aggregate_type": "Asset",
  "aggregate_id": "EQ-2026-0001",
  "aggregate_version": 12,
  "occurred_at": "2026-06-17T10:15:30Z",
  "trace_id": "...",
  "correlation_id": "...",
  "causation_id": "...",
  "payload": {
    "asset_id": "EQ-2026-0001",
    "reason": "SCRAP_APPROVED",
    "operator_id": "u1001",
    "retired_at": "2026-06-17T10:15:30Z"
  }
}
```

统一 Envelope 的好处：

- 消费方可以用 `event_id` 做幂等。
- 监控系统可以按 `event_type`、`source_service` 聚合。
- 后续 schema 演进有统一入口。
- 追踪链路可以通过 `trace_id` 串联。

## 7. Outbox Publisher 实现步骤

### 7.1 定时扫描

建议使用 Spring Scheduler 或独立 Worker。

```java
@Component
public class OutboxPublisherJob {

    private final OutboxPublisher outboxPublisher;

    @Scheduled(fixedDelayString = "${app.outbox.publisher.fixed-delay:1000}")
    public void publish() {
        outboxPublisher.publishBatch();
    }
}
```

### 7.2 批量领取事件

多实例部署时，必须避免多个 Publisher 同时处理同一行。

推荐 SQL：

```sql
SELECT *
FROM outbox_event
WHERE status IN ('PENDING', 'RETRYABLE')
  AND next_retry_at <= CURRENT_TIMESTAMP
ORDER BY created_at
LIMIT :batchSize
FOR UPDATE SKIP LOCKED;
```

领取后置为 `PUBLISHING`：

```sql
UPDATE outbox_event
SET status = 'PUBLISHING',
    locked_by = :instanceId,
    locked_at = CURRENT_TIMESTAMP
WHERE id IN (:ids);
```

该过程建议放在一个短事务中完成。

### 7.3 发布 Kafka

Publisher 领取到事件后，逐条或批量发送 Kafka。

```java
public void publishOne(OutboxEvent event) {
    ProducerRecord<String, String> record = new ProducerRecord<>(
        event.getTopic(),
        event.getPartitionKey(),
        event.getPayload()
    );

    record.headers().add("event_id", event.getEventId().getBytes(StandardCharsets.UTF_8));
    record.headers().add("event_type", event.getEventType().getBytes(StandardCharsets.UTF_8));
    record.headers().add("event_version", String.valueOf(event.getEventVersion()).getBytes(StandardCharsets.UTF_8));
    record.headers().add("source_service", event.getSourceService().getBytes(StandardCharsets.UTF_8));
    record.headers().add("trace_id", nullToEmpty(event.getTraceId()).getBytes(StandardCharsets.UTF_8));

    kafkaTemplate.send(record).get();

    outboxEventRepository.markSent(event.getId(), Instant.now());
}
```

> `send().get()` 会阻塞等待 Kafka ack，逻辑简单但吞吐较低。中高吞吐场景可以使用异步 future + 批量状态更新。

### 7.4 成功后标记 SENT

```sql
UPDATE outbox_event
SET status = 'SENT',
    published_at = CURRENT_TIMESTAMP,
    locked_by = NULL,
    locked_at = NULL,
    last_error_code = NULL,
    last_error_message = NULL
WHERE id = :id
  AND status = 'PUBLISHING';
```

### 7.5 失败后标记 RETRYABLE 或 DEAD_LETTER

```java
catch (Exception ex) {
    if (isRetryable(ex) && event.getRetryCount() + 1 < event.getMaxRetryCount()) {
        outboxEventRepository.markRetryable(
            event.getId(),
            event.getRetryCount() + 1,
            nextRetryAt(event.getRetryCount() + 1),
            errorCode(ex),
            abbreviate(ex.getMessage())
        );
    } else {
        outboxEventRepository.markDeadLetter(
            event.getId(),
            event.getRetryCount() + 1,
            errorCode(ex),
            abbreviate(ex.getMessage())
        );
    }
}
```

## 8. Kafka Producer 配置

### 8.1 Spring Boot 配置示例

```yaml
spring:
  kafka:
    bootstrap-servers: ${KAFKA_BOOTSTRAP_SERVERS:localhost:9092}
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: all
      retries: 10
      properties:
        enable.idempotence: true
        max.in.flight.requests.per.connection: 5
        delivery.timeout.ms: 120000
        request.timeout.ms: 30000
        retry.backoff.ms: 1000
        linger.ms: 10
        batch.size: 32768
        compression.type: zstd
```

### 8.2 关键配置说明

| 配置 | 推荐值 | 说明 |
|---|---:|---|
| `acks` | `all` | 等待 leader 和 ISR 确认，避免只写 leader 后丢失。 |
| `enable.idempotence` | `true` | 开启 Kafka Producer 幂等，降低 producer 重试导致的重复写入。 |
| `retries` | `10` 或更高 | Kafka 客户端内部重试次数。Outbox 还会做业务层重试。 |
| `delivery.timeout.ms` | `120000` | Producer 完成发送的总超时时间。 |
| `request.timeout.ms` | `30000` | 单次请求超时。 |
| `retry.backoff.ms` | `1000` | Kafka 客户端内部重试间隔。 |
| `linger.ms` | `5~20` | 等待更多消息组成批次，提高吞吐。低延迟要求可设小。 |
| `batch.size` | `32768` 或更高 | 批量大小。 |
| `compression.type` | `zstd` / `lz4` | 推荐压缩，减少网络和 broker 存储压力。 |
| `max.in.flight.requests.per.connection` | `5` | 开启幂等时 Kafka 推荐不超过 5。 |

### 8.3 是否需要 Kafka 事务 Producer

Outbox 模式下通常不需要 Kafka 事务 Producer。

原因：

- 业务状态与事件记录的一致性由数据库事务保证。
- Kafka 发布失败可由 outbox 重试保证。
- 消费方通过 `event_id` 幂等处理重复消息。

Kafka 事务 Producer 适合解决“Kafka 内部多 topic / 多 partition 的原子写入”，不能直接解决“数据库事务 + Kafka 发送”的原子性问题。因此本方案优先使用普通 Producer + 幂等 Producer + Outbox。

## 9. Kafka Topic 设计

### 9.1 命名规范

建议统一：

```text
<domain>.<aggregate>.<dimension>
```

台账域示例：

```text
eam.asset.lifecycle
eam.asset.availability
eam.asset.specification
eam.asset.metric
eam.asset.location
eam.asset.scrap
```

### 9.2 分区键

原则：同一聚合根的事件必须有序。

台账域建议：

| Topic | partition key |
|---|---|
| `eam.asset.lifecycle` | `asset_id` |
| `eam.asset.availability` | `asset_id` |
| `eam.asset.specification` | `asset_id` 或 `specification_profile_id` |
| `eam.asset.metric` | `asset_id` |
| `eam.asset.location` | `asset_id` |
| `eam.asset.scrap` | `asset_id` 或 `scrap_application_id`；若关心资产状态联动，优先 `asset_id` |

### 9.3 Topic 参数建议

```bash
kafka-topics.sh --create \
  --topic eam.asset.lifecycle \
  --bootstrap-server localhost:9092 \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --config retention.ms=604800000 \
  --config cleanup.policy=delete
```

建议配置：

| 参数 | 建议 | 说明 |
|---|---:|---|
| `partitions` | 6、12、24 起步 | 根据服务吞吐和消费并行度调整。分区数后续可增加，但会影响 key 的分布位置。 |
| `replication.factor` | 3 | 生产环境建议 3 副本。 |
| `min.insync.replicas` | 2 | 配合 producer `acks=all` 使用。 |
| `retention.ms` | 7~30 天 | 领域事件建议保留足够长，支持故障恢复和新消费者追赶。 |
| `cleanup.policy` | `delete` | 领域事件通常保留日志，不建议 compact 替代完整事件流。 |

> 如果某些主题需要长期可重放，不要只依赖 Kafka retention，应将事件归档到对象存储、数据湖或事件审计库。

## 10. 消费方实现

### 10.1 消费方必须幂等

Outbox 模式提供的是至少一次投递：

```text
At-least-once delivery
```

因此重复消息是正常情况。消费方必须使用 `event_id` 去重。

建议每个消费服务建立消费记录表：

```sql
CREATE TABLE consumed_event (
    event_id          VARCHAR(64)  NOT NULL,
    consumer_group    VARCHAR(128) NOT NULL,
    topic             VARCHAR(128) NOT NULL,
    event_type        VARCHAR(128) NOT NULL,
    processed_at      DATETIME     NOT NULL,
    PRIMARY KEY (event_id, consumer_group)
);
```

消费逻辑：

```java
@Transactional
public void handle(EventEnvelope<AssetRetiredPayload> event) {
    if (consumedEventRepository.exists(event.eventId(), consumerGroup)) {
        return;
    }

    // 执行业务处理，例如关闭该资产未完成维修工单
    repairOrderService.closeOpenOrdersForRetiredAsset(event.payload().assetId());

    consumedEventRepository.save(new ConsumedEvent(event.eventId(), consumerGroup));
}
```

更推荐利用数据库唯一键避免并发重复：

```text
先 insert consumed_event
若唯一键冲突，说明已处理，直接 ack
若插入成功，继续执行业务处理
```

但如果先插入幂等表再处理业务，业务处理失败时要跟幂等表处于同一事务中回滚。

### 10.2 Spring Kafka Consumer 配置

```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: ${KAFKA_BOOTSTRAP_SERVERS:localhost:9092}
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      enable-auto-commit: false
      auto-offset-reset: earliest
      properties:
        isolation.level: read_committed
        max.poll.records: 100
        session.timeout.ms: 45000
        heartbeat.interval.ms: 15000
    listener:
      ack-mode: manual
      concurrency: 3
```

关键点：

| 配置 | 推荐 | 说明 |
|---|---:|---|
| `enable-auto-commit` | `false` | 禁止自动提交 offset，避免业务处理失败但 offset 已提交。 |
| `ack-mode` | `manual` | 业务事务成功后手动 ack。 |
| `auto-offset-reset` | `earliest` | 新消费组从最早开始消费，适合领域事件补投影。 |
| `max.poll.records` | `100` | 控制单批处理量。 |
| `concurrency` | 与分区数匹配 | 并发数不应超过分区数太多。 |
| `isolation.level` | `read_committed` | 如果上游使用 Kafka 事务则只读已提交消息；本方案不强依赖，但可保留。 |

### 10.3 Listener 示例

```java
@KafkaListener(
    topics = "eam.asset.lifecycle",
    groupId = "repair-service.asset-lifecycle-consumer"
)
public void onMessage(
    ConsumerRecord<String, String> record,
    Acknowledgment acknowledgment
) {
    try {
        EventEnvelope envelope = objectMapper.readValue(record.value(), EventEnvelope.class);
        assetLifecycleEventHandler.handle(envelope);
        acknowledgment.acknowledge();
    } catch (Exception ex) {
        throw ex;
    }
}
```

### 10.4 消费失败处理

推荐使用 Spring Kafka 的 `DefaultErrorHandler`：

```java
@Bean
public DefaultErrorHandler defaultErrorHandler(KafkaTemplate<String, String> kafkaTemplate) {
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(
        kafkaTemplate,
        (record, ex) -> new TopicPartition(record.topic() + ".DLT", record.partition())
    );

    FixedBackOff backOff = new FixedBackOff(5000L, 5L);
    DefaultErrorHandler errorHandler = new DefaultErrorHandler(recoverer, backOff);

    errorHandler.addNotRetryableExceptions(
        IllegalArgumentException.class,
        EventSchemaException.class
    );

    return errorHandler;
}
```

消费侧 DLQ topic 命名：

```text
eam.asset.lifecycle.DLT
eam.asset.availability.DLT
```

## 11. 事务边界

### 11.1 生产侧事务边界

生产侧必须遵守：

```text
一个业务命令 = 一个本地数据库事务
事务内：
  - 加载聚合根
  - 执行业务方法
  - 保存业务状态
  - 插入 outbox_event
事务外：
  - 不保证立即发布 Kafka
```

禁止：

```text
@Transactional 方法中直接调用 kafkaTemplate.send()
```

允许但不作为可靠性来源：

```text
事务提交后 afterCommit() 立即唤醒 publisher 或尝试发送
```

### 11.2 消费侧事务边界

消费侧必须遵守：

```text
Kafka poll 收到消息
  → 开启本地 DB 事务
    → 幂等检查/插入 consumed_event
    → 执行业务变更
  → DB commit
  → 手动 ack Kafka offset
```

如果 DB commit 之前抛异常，不 ack，Kafka 后续重投。

如果 DB commit 成功但 ack 之前应用崩溃，Kafka 会重复投递，消费方通过 `event_id` 幂等表跳过。

## 12. 快速路径：afterCommit 唤醒 Publisher

为了降低延迟，可以在事务提交后触发一次本地发布信号，但仍保留后台轮询兜底。

```java
@Transactional
public void retireAsset(RetireAssetCommand command) {
    // 保存业务状态和 outbox_event

    TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            outboxPublishSignal.tryWakeup();
        }
    });
}
```

注意：

- `afterCommit` 只负责唤醒 Publisher，不直接替代 Publisher。
- 即使 `afterCommit` 未执行，定时任务仍会扫描 `PENDING`。
- 如果需要极低延迟，可使用内存队列传递 outbox id，但发布前仍应从 DB 读取并校验状态。

## 13. 可观测性与告警

### 13.1 指标

必须暴露以下指标：

| 指标 | 含义 |
|---|---|
| `outbox_pending_count` | PENDING + RETRYABLE 待发布数量。 |
| `outbox_publishing_count` | PUBLISHING 数量。 |
| `outbox_dead_letter_count` | DEAD_LETTER 数量。 |
| `outbox_publish_latency_seconds` | `published_at - created_at`。 |
| `outbox_publish_success_total` | 成功发布总数。 |
| `outbox_publish_failure_total` | 发布失败总数。 |
| `outbox_retry_total` | 重试总数。 |
| `kafka_producer_send_latency` | Kafka 发送耗时。 |
| `consumer_duplicate_event_total` | 消费方幂等去重次数。 |
| `consumer_dead_letter_total` | 消费方进入 DLT 的消息数量。 |

### 13.2 告警建议

| 告警条件 | 严重性 |
|---|---|
| `DEAD_LETTER > 0` 持续 5 分钟 | 高 |
| 最老 PENDING 超过 1 分钟 | 中/高，取决于业务 SLA |
| PENDING 数量持续增长 | 高 |
| PUBLISHING 锁超时数量增长 | 中 |
| Kafka send failure rate 超过阈值 | 高 |
| 消费 DLT 有新增 | 高 |

## 14. 数据清理与归档

### 14.1 SENT 数据清理

Outbox 表不建议无限增长。

策略：

```text
SENT 保留 3~7 天
DEAD_LETTER 长期保留直到人工处理
DISCARDED 保留 30 天或按审计要求
```

清理 SQL 示例：

```sql
DELETE FROM outbox_event
WHERE status = 'SENT'
  AND published_at < CURRENT_TIMESTAMP - INTERVAL '7' DAY
LIMIT 1000;
```

大表建议使用按日期分区，直接 drop 旧分区。

### 14.2 审计要求

如果领域事件需要长期审计，不建议依赖 outbox 表长期保留。建议：

1. Kafka 保留 7~30 天，用于故障恢复。
2. 成功事件异步归档到对象存储/数据湖/审计库。
3. outbox 表只保留短期运维窗口。

## 15. Topic 与事件版本演进

### 15.1 兼容性原则

事件 schema 演进必须遵守：

- 新增字段必须可选或有默认值。
- 不删除已有字段。
- 不改变已有字段语义。
- 不改变字段类型。
- 重大不兼容变更使用新的 `event_version` 或新的 `event_type`。

### 15.2 版本字段

每条事件必须带：

```text
event_type
event_version
```

消费方根据类型和版本分发：

```java
switch (envelope.eventType()) {
    case "AssetRetired" -> handleAssetRetired(envelope.toPayload(AssetRetiredPayload.class));
    case "AssetSuspended" -> handleAssetSuspended(envelope.toPayload(AssetSuspendedPayload.class));
    default -> ignoreOrLog(envelope);
}
```

## 19. 实现检查清单

### 19.1 生产侧

- [ ] 业务状态变更和 outbox 写入在同一个 `@Transactional` 中。
- [ ] 聚合根不直接依赖 Kafka。
- [ ] 每条事件都有全局唯一 `event_id`。
- [ ] 每条事件都有 `event_type`、`event_version`、`occurred_at`。
- [ ] Kafka `partition_key` 能保证同一聚合根有序。
- [ ] Publisher 支持多实例并发领取。
- [ ] Publisher 支持失败重试和死信。
- [ ] `PUBLISHING` 锁超时可恢复。
- [ ] `SENT` 数据有清理策略。

### 19.2 Kafka

- [ ] Producer `acks=all`。
- [ ] Producer `enable.idempotence=true`。
- [ ] Topic 副本数生产环境为 3。
- [ ] Topic `min.insync.replicas=2`。
- [ ] Topic 分区数满足消费并发需求。
- [ ] 禁止生产环境依赖 topic 自动创建。

### 19.3 消费侧

- [ ] `enable-auto-commit=false`。
- [ ] 业务处理成功后手动 ack。
- [ ] 使用 `event_id + consumer_group` 幂等去重。
- [ ] 消费失败有重试和 DLT。
- [ ] DLT 有告警和人工处理流程。

### 19.4 可观测性

- [ ] 监控 PENDING 积压。
- [ ] 监控发布延迟。
- [ ] 监控 DEAD_LETTER。
- [ ] 监控 Kafka producer 错误率。
- [ ] 监控消费方 DLT。

## 20. 推荐实施顺序

1. 新增 `outbox_event` 表和 `OutboxStatus` 枚举。
2. 实现 `OutboxEventFactory`，统一生成 Envelope、payload、headers。
3. 改造应用服务：业务状态变更时同步写入 outbox。
4. 实现 `OutboxPublisher`：扫描、领取、发送 Kafka、更新状态。
5. 增加 `OutboxRecoveryJob`：恢复超时的 `PUBLISHING`。
6. 增加 `OutboxCleanupJob`：清理过期 `SENT`。
7. 配置 Kafka Producer：`acks=all`、`enable.idempotence=true`。
8. 为每个 Topic 建立消费者幂等表 `consumed_event`。
9. 实现消费者手动 ack、失败重试、DLT。
10. 增加监控指标和告警。
11. 先在台账域低频主题试点，例如 `eam.asset.lifecycle`、`eam.asset.scrap`。
12. 验证稳定后推广到其它领域事件主题。

## 21. 关键原则总结

1. **业务事务只负责写业务表和 outbox 表，不直接承担可靠 Kafka 投递。**
2. **Outbox 表是可靠性的来源，Publisher 是投递执行器。**
3. **Kafka 投递按至少一次处理，消费方必须幂等。**
4. **同一聚合根使用相同 partition key，保证单聚合根事件有序。**
5. **低频领域事件走 Outbox，高频设备原始数据不要走 Outbox。**
6. **即使当前只有服务内部消费者，只要是领域契约事件，也统一走 Kafka + Outbox。**
