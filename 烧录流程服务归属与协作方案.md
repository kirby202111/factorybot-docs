# 烧录流程服务归属与协作方案

## 一、功能归属判定

### 1.1 归属结论

**烧录执行、烧录结果记录、烧录校验归属生产执行服务**，作为工位数据采集的一种，与 AOI/SPI 遵循相同的归属逻辑。

| 数据/功能 | 归属服务 | 理由 |
|----------|---------|------|
| 烧录工序定义、烧录参数模板 | 制造资源服务 | 属于工艺路线中的工序配置 |
| 烧录设备状态、设备运行参数 | 设备管理服务 | 属于设备维度数据 |
| **烧录执行、烧录结果记录、烧录校验** | **生产执行服务** | 与过点是同一业务事务，遵循 AOI/SPI 同样的归属逻辑 |

### 1.2 归属依据

烧录的本质是：操作工在烧录工位上，使用烧录设备将固件/配置数据写入产品，并记录烧录结果。套用工位数据采集归属的三个考量维度：

| 考量 | 烧录场景 | 结论 |
|------|---------|------|
| **事务一致性** | 烧录结果（成功/失败/校验和）与过点记录是同一业务事务——烧录失败则产品不可放行，拆开则需分布式事务 | 必须同服务 |
| **时序对齐** | 烧录结果需实时判定放行/拦截，跨服务延迟不可接受 | 必须同服务 |
| **业务语义** | 操作工扫码过点 + 烧录设备执行写入 + 结果自动关联 = 一个完整用例 | 同服务更自然 |

核心原则：**谁决定了过点的放行/拦截，数据就归属谁**。烧录结果直接决定产品能否通过该工位，因此归属生产执行服务。

### 1.3 需要避免的误区

❌ **设备管理服务承担版本合规校验**：版本合规校验的依据是工艺路线中定义的烧录工序参数，属于过点校验逻辑的一部分，不应泄漏到设备管理服务。

❌ **设备管理服务编排过点流程**：设备管理服务的核心特征是"数据流与生产流解耦，不感知过点行为"，承担流程编排直接违反这一定位。

❌ **烧录结果与过点记录分属不同事务**：烧录成功但过点信息传递失败，会导致产品已烧录但系统无过点记录，造成账实不符。

## 二、跨服务协作方式

烧录执行在生产执行服务，但其上下游涉及其他服务的数据，协作关系如下：

```
┌─ 制造资源服务 ──────────────────────────────────────┐
│  工艺路线中定义"烧录"工序:                             │
│  - 烧录文件版本 / 校验算法 / 必选标记                   │
│  - requireDeviceData = true                           │
│  通过 ProcessRouteActivated 事件同步到生产执行服务缓存    │
└───────────────────────────────────────────────────────┘

┌─ 设备管理服务 ──────────────────────────────────────┐
│  烧录设备的:                                          │
│  - 设备状态（运行/故障/保养中）→ 通过事件同步到生产执行服务 │
│  - 设备级运行数据（烧录器温度、电压等）→ 过点时按需查询    │
└───────────────────────────────────────────────────────┘

┌─ 生产执行服务 ──────────────────────────────────────┐
│  烧录工位的完整流程:                                   │
│  1. 扫码过点 → 读取工艺缓存，校验当前工位含烧录工序       │
│  2. 校验烧录设备状态可用                                │
│  3. 触发烧录操作 / 接收烧录设备回传结果                  │
│  4. 校验烧录结果（成功/失败 + 固件版本 + 校验和）         │
│  5. 与过点记录写入同一事务                              │
│  6. 烧录失败 → 拦截放行，触发质量异常                    │
└───────────────────────────────────────────────────────┘
```

## 三、流程方案

根据烧录软件的可对接程度，分为两种流程方案。

### 3.1 方案一：烧录软件可对接（推荐）

**适用条件**：烧录软件为自研，或第三方厂商开放接口/提供集成 SDK。

**核心思路**：烧录软件直接对接生产执行服务，由生产执行服务作为流程编排者。

```
烧录软件                          生产执行服务              设备管理服务       制造资源服务
  │                                  │                        │                 │
  │ ①操作工扫码                       │                        │                 │
  │                                  │                        │                 │
  │ ②调用"烧录过点初始化"接口          │                        │                 │
  │ ─────────────────────────────→   │                        │                 │
  │                                  │                        │                 │
  │                             ③工艺校验(本地缓存)            │                 │
  │                             设备状态校验(本地缓存)          │                 │
  │                                  │                        │                 │
  │ ④返回 { allow: true,             │                        │                 │
  │    firmwareVersion: "V2.3.1",    │                        │                 │
  │    checksum: "0xA3F1" }          │                        │                 │
  │ ←─────────────────────────────   │                        │                 │
  │                                  │                        │                 │
  │ ⑤执行烧录（烧录软件自主控制）       │                        │                 │
  │                                  │                        │                 │
  │ ⑥烧录完成，调用"烧录结果回传"接口   │                        │                 │
  │ ─────────────────────────────→   │                        │                 │
  │    { barcode, result,            │                        │                 │
  │      actualVersion, checksum }   │                        │                 │
  │                                  │                        │                 │
  │                             ⑦校验烧录结果                  │                 │
  │                              版本匹配? 校验和一致?          │                 │
  │                             写入过点记录(同一事务) ✓        │                 │
  │                                  │                        │                 │
  │ ⑧返回过点结果                     │                        │                 │
  │ ←─────────────────────────────   │                        │                 │
```

**关键设计要点**：

1. **烧录专用 API**：生产执行服务对外暴露烧录专用接口（非通用过点接口），烧录软件只需对接两个接口：
   - `POST /api/exec/burn/init` — 烧录过点初始化
   - `POST /api/exec/burn/result` — 烧录结果回传

2. **初始化阶段完成校验**：烧录软件在步骤④拿到明确指令（烧哪个版本、校验和是什么），避免盲目烧录后才发现版本不对。

3. **事务一致性**：步骤⑥⑦保证烧录结果与过点记录在同一事务中，强一致性。

4. **幂等处理**：同一 barcode 的初始化和结果回传均需幂等，防止烧录软件重试导致重复过点。

### 3.2 方案二：烧录软件封闭，无法直接对接生产执行服务

**适用条件**：烧录软件为第三方厂商封闭系统，只能将数据推送到设备数据采集接口。

**核心思路**：设备管理服务作为数据通道，通过领域事件将烧录数据传递给生产执行服务，由生产执行服务完成业务决策。

```
烧录软件                   设备管理服务                  生产执行服务
  │                           │                            │
  │ ①烧录软件将烧录数据        │                            │
  │   推送到设备数据采集接口     │                            │
  │ ─────────────────────→    │                            │
  │                           │                            │
  │                      ②仅做数据暂存                      │
  │                      不校验、不决策                      │
  │                      发布领域事件:                      │
  │                      BurnDataReceived                  │
  │                      { deviceId, barcode,              │
  │                        result, version,                │
  │                        checksum, timestamp }           │
  │                           │ ──────────────────────→    │
  │                           │                            │
  │                           │                       ③消费事件，执行过点校验
  │                           │                       版本校验(工艺缓存)
  │                           │                       写入过点记录
  │                           │                            │
  │                           │                  ④过点结果事件(可选回传)
  │                           │ ←──────────────────────    │
```

**关键设计要点**：

1. **设备管理服务仅做数据通道**：接收烧录数据、暂存、发布事件，不执行任何业务校验或流程编排。

2. **领域事件 BurnDataReceived**：设备管理服务在收到烧录数据后发布，生产执行服务订阅消费。

3. **此方案为最终一致性**：烧录结果与过点记录不在同一事务中，需补偿机制保障。

#### 事务一致性缺口与补偿机制

**风险场景**：烧录成功，但过点信息传递失败 → 产品已烧录但系统无过点记录 → 账实不符。

**补偿方案：待确认队列 + 超时告警**

```
生产执行服务
├── 烧录过点记录
│   ├── 状态: CONFIRMED（正常闭环）
│   └── 状态: PENDING（收到烧录数据，过点校验未通过或写入失败）
│
├── 定时扫描 PENDING 记录（≤ 30s）
│   ├── 重试校验与写入
│   └── 重试失败 → 告警，人工介入
│
└── 防重复：同一 barcode 的烧录数据幂等处理
```

**操作工侧体验**：

```
烧录软件上：烧录成功 ✓
MES终端上：  该产品状态显示"待确认"（黄色）
            → 自动确认成功 → 变为"已过点"（绿色）
            → 确认失败     → 变为"异常"（红色），需人工处理
```

## 四、两种方案对比

| 维度 | 方案一：可对接 | 方案二：封闭系统 |
|------|--------------|---------------|
| 数据通道 | 烧录软件 → 生产执行服务 | 烧录软件 → 设备管理服务 → 生产执行服务 |
| 业务决策者 | 生产执行服务 | 生产执行服务 |
| 事务一致性 | **强一致**（同一事务） | **最终一致**（事件 + 补偿） |
| 操作工体验 | 烧录即过点，一步完成 | 烧录后需确认，两步完成 |
| 设备管理服务职责 | 不参与 | **仅数据通道**，不校验不决策 |
| 实现复杂度 | 低 | 中（需补偿机制） |

---

## 五、实施建议

1. **优先推动方案一**：即使烧录软件是第三方的，也要求厂商提供回调接口或集成 SDK，让烧录软件能直接调用生产执行服务的 API。这是架构最干净、一致性最强的方案。

2. **短期过渡用方案二**：如果短期内确实无法对接，使用事件驱动 + 补偿机制过渡，但需在操作工侧明确展示"待确认"状态，避免误判。

3. **长期约束**：设备部/采购部在后续设备选型时，将"支持 MES 接口对接"作为采购条件，逐步将方案二迁移至方案一。
