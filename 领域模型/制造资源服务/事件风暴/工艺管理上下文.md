# 工艺管理上下文（Process Management Bounded Context）

> **限界上下文**：工艺管理上下文
> **统一语言**：Operation（工序）, ParameterTemplate（参数模板）, ProcessRoute（工艺路线·逻辑实体，由 route_id 标识）, RouteVersion（工艺路线版本·聚合根，含步骤序列与生命周期）, RouteStep（路线步骤·RouteVersion 内实体）, AntiErrorItem（防错项·步骤配置）, QualityGateBinding（质量门禁绑定·步骤对质量规则的引用）, EquipmentSpecBinding（设备规格绑定·步骤对台账能力规格的引用）, FixtureMatchingBinding（工装匹配绑定·步骤对台账工装匹配规则的引用）, ReentryPoint（再入点·可返修/返工再入的步序）, RealtimeDataCheckConfig（实时数据校验配置·关键工序）, RouteVersionState（版本状态：DRAFT/SUBMITTED/ACTIVATED/DEPRECATED/ARCHIVED）, SpecificationKey（规格键·引用台账上下文能力规格画像）, RouteCache（工艺路线缓存·过点执行上下文本地投影）
> **核心关注**：以"事件风暴"方式刻画**生产执行的数据基座**——工序定义、参数模板、工艺路线编排、版本生命周期、步骤配置与生效发布。本上下文只对"**工艺怎么定义、版本怎么生效发布、步骤绑定哪些防错/质量/设备/工装规则**"负责；**不负责**物料主数据与产品 BOM（归物料上下文）、检验标准与质量门禁规则定义（归质量上下文）、设备/工装能力规格与匹配规则的事实源（归设备工装台账上下文）、过点判定与路线推进执行（归过点执行上下文）、在制品位置投影（归在制品追踪上下文）。本上下文是**配置型主数据上下文**，写少读多，对外以"版本生效事件 + 只读规格读模型"为协作契约。

## 0. 设计原则与边界澄清

工艺管理上下文是制造资源服务下的**配置子域**，是 PCBA + 整机组装混合型车间所有"工艺路线与版本"的权威源。它解决的核心问题是：**一件产品按什么工序顺序、在什么工位、用什么设备/工装、过什么防错与质量门禁来制造**——并以**不可变版本**形式发布给生产执行服务消费。

**上下文内职责**（本上下文负责）：
- **工序定义**——SMT 印刷、SPI、贴片、回流焊、AOI、波峰焊、FCT、烧录、锁螺丝、装配、老化、包装等工序类型的定义与维护
- **参数模板**——工序级参数模板（如回流焊温区曲线、波峰焊链速、电批扭矩），供步骤配置引用
- **工艺路线编排**——为产品 × 产线编排工序步骤序列（PCBA 段 / 整机组装段）
- **版本生命周期**——草稿 → 提交审核 → 生效 → 弃用 → 归档的完整状态机
- **步骤配置与绑定**——每步绑定防错项、质量门禁规则、设备规格、工装匹配规则、参数模板、再入点
- **版本生效发布**——发布 `ProcessRouteActivated`，供过点执行上下文刷新工艺缓存
- **再入点配置**——标记哪些步序可作为返修/返工再入点（供过点执行上下文做再入点校验）

**上下文外职责**（本上下文**不**负责）：
- 物料主数据、产品 BOM、供应商、替代料 → 物料上下文（本上下文仅以 `product_id` 引用产品，以 `bom_item_ref` 引用 BOM 项做防错）
- 检验标准、缺陷分类、质量门禁规则定义 → 质量上下文（本上下文仅以 `quality_gate_rule_id` 引用规则）
- 设备/工装/计量器具身份、能力规格画像、工装匹配规则的事实源 → 设备工装台账上下文（本上下文仅以 `spec_keys` / `fixture_matching_rule_id` 引用）
- 过点判定、防错校验执行、路线推进、跳站校验 → 过点执行上下文（本上下文只发布版本生效事件 + 提供只读读模型，不执行过点）
- 在制品身份、位置投影、齐套状态 → 在制品追踪上下文
- 首件检验流程编排、首件放行门禁判定 → 首件处理上下文
- 单件返修任务、批量返工工单的执行 → 返修上下文 / 返工上下文（本上下文仅提供再入点配置供其判定）

**为什么不把"工艺管理"和"质量"并入一个限界上下文**：
1. **关注点不同**——工艺管理关注"工序怎么排、版本怎么发布"（流程编排视角），质量关注"检验标准是什么、缺陷怎么分类、门禁规则怎么判定"（质量标准视角）。
2. **生命周期与变化频率不同**——工艺路线随产品/工程变更发布（低频，版本化），质量门禁规则随检验标准修订（独立低频）。混在一起会让"改一条门禁规则"动到"路线版本"代码。
3. **干系人不同**——工艺管理由工艺工程师/工艺主管负责，质量规则由质量工程师负责，审核流与发布节奏不同。
4. **解耦即可协作**——本上下文以 `quality_gate_rule_id` 引用质量规则，规则变更不影响已生效路线版本（版本不可变，INV-04）；过点执行时由质量门禁缓存反映最新规则。

**为什么不把"工艺管理"和"物料"并入一个限界上下文**：
1. 物料主数据（产品/BOM/供应商/替代料）是**跨工艺复用**的基础数据，与工艺路线是"被引用 vs 引用方"关系。
2. 物料由采购/物控团队维护，工艺由工艺团队维护，组织边界清晰。
3. 本上下文仅以 `product_id` / `bom_item_ref` 引用物料上下文，不复制 BOM 实体。

**为什么不把"过点执行"并入本上下文**：
1. **写少读多 vs 高频写入**——本上下文是配置型主数据（写入主要发生在版本发布时），过点是高频实时事件流（每板毫秒级）。混在一起会让配置发布阻塞过点主链。
2. **事务边界不同**——过点记录与测试结果同事务（领域总览 §5.3），工艺版本生效是独立发布事务。合并会逼出分布式事务。
3. **缓存解耦已足够**——过点执行上下文订阅 `ProcessRouteActivated` 维护本地缓存，缓存未命中降级 REST 查询本上下文（领域总览 §5.1）。

## 1. 主语清单与对象形态

工艺管理上下文处理的对象分三类：**可复用定义**（工序、参数模板）、**版本化路线**（核心聚合）、**步骤绑定**（版本内实体）。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 聚合根

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **Operation**（工序） | SMT 印刷、SPI、贴片、回流焊、AOI、波峰焊、FCT、烧录、电批锁螺丝、装配、老化、包装 | 可复用；跨路线共享；有 ACTIVE / DEPRECATED 状态 | 工艺工程师定义 |
| **ParameterTemplate**（参数模板） | 回流焊温区曲线模板、波峰焊链速模板、电批扭矩模板 | 绑定到 Operation；可复用；有 ACTIVE / DEPRECATED 状态 | 工艺工程师定义 |
| **RouteVersion**（工艺路线版本） | 某 PCBA 产品的回流焊段路线 v3、某整机的装配段路线 v2 | 版本化；生效后不可变；含步骤序列；唯一 ACTIVE 版本约束 | 工艺工程师编排 + 工艺主管审核 |

### 1.2 RouteVersion 内实体（非独立聚合）

| 实体 | 归属 | 关键管理特征 | 引用的上下文外对象 |
|------|------|-------------|-------------------|
| **RouteStep**（路线步骤） | RouteVersion | 步序号、操作工序、工位类型、再入点标记 | Operation（本上下文） |
| **AntiErrorItem**（防错项） | RouteStep | 防错维度（物料/工装/设备准入/首件门禁/前置工序/实时数据）及参数 | 物料上下文 BOM 项、台账上下文规格 |
| **QualityGateBinding**（质量门禁绑定） | RouteStep | 对质量门禁规则的引用 | 质量上下文 `quality_gate_rule_id` |
| **EquipmentSpecBinding**（设备规格绑定） | RouteStep | 对设备能力规格的引用（spec_keys） | 设备工装台账上下文 `eam.asset.specification` |
| **FixtureMatchingBinding**（工装匹配绑定） | RouteStep | 对工装匹配规则的引用 | 设备工装台账上下文 `fixture_matching_rule_id` |
| **RealtimeDataCheckConfig**（实时数据校验配置） | RouteStep | 关键工序的实时数据校验项与阈值 | 设备数据接入上下文（过点时查询） |

> **关键边界**：本上下文只存**引用**（`rule_id` / `spec_keys` / `bom_item_ref`），**不复制**上下文外实体内容。过点执行时由过点执行上下文综合各缓存执行校验。这保证上下文外规则变更不污染已生效版本（INV-04），同时过点读到的是最新规则（最终一致）。

> **双段生产架构**：路线按 PCBA 段与整机组装段分别编排（同一产品可有 PCBA 路线与整机装配路线，由 `route_type` 区分）；两段在过点执行上下文以单件 `routeVersion` 推进衔接，本上下文不感知段间拼接。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 工序与参数模板定义（可复用基础）

工序与参数模板是路线编排的"积木"，独立维护、跨路线复用。

```
[参与者:工艺工程师]
        --【命令:定义工序 DefineOperation】--> 【聚合:Operation】
                                                  |
                                                  v
                          【事件:工序已定义 OperationDefined】
                           (operation_id, name, category: PCBA/ASSEMBLY, station_type, capability_keys[])
                                                  |
                          【策略:工序定义后可挂参数模板】
                                                  v
        --【命令:定义参数模板 DefineParameterTemplate】--> 【聚合:ParameterTemplate】
                                                          |
                                                          v
                          【事件:参数模板已定义 ParameterTemplateDefined】
                           (template_id, operation_id, param_keys[], unit_scheme)
                                                  |
                                                  v
                          发布到 process.operation.lifecycle / process.parameter.template
                          （供路线编排时引用）
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| OperationDefined | 工艺工程师新建工序 | operation_id, name, category, station_type, capability_keys[] | 可被路线步骤引用 |
| OperationUpdated | 工序能力/属性变更 | changed_fields, before, after | 触发引用影响检查（策略：FlagAffectedDrafts，标记 DRAFT 版本 stale_reference） |
| OperationDeprecated | 工序弃用 | operation_id, reason, deprecated_at | 限制新草稿引用；已生效版本不受影响（INV-04） |
| ParameterTemplateDefined | 新建参数模板 | template_id, operation_id, param_keys[] | 可被步骤引用 |
| ParameterTemplateUpdated | 参数模板修订 | template_id, changed_fields | 影响引用该模板的草稿（已生效版本不受影响） |

> 🔴 **热点：工序与产品/产线的耦合度**——工序是"通用类型"还是按产品族细分（如"产品A的回流焊"vs"通用回流焊"）？建议**通用工序 + 步骤配置承载产品差异**，避免工序爆炸。待与工艺工程团队确认工序粒度。

> 🔴 **热点：参数模板与台账能力规格的对齐**——回流焊温区曲线（参数模板）与台账上下文 `AssetSpecificationProfile`（设备能力规格，如"温区数、最大板尺寸"）是两个视角。本上下文参数模板是"工艺要求值"，台账规格是"设备能力上限"，过点时由过点执行上下文比较两者。两者 key 命名空间需对齐。待与设备工装台账上下文确认 spec_key 命名规范。

### 2.2 工艺路线版本生命周期（状态机）

RouteVersion 是本上下文的核心聚合根，承载步骤序列与完整生命周期。

```
                         [DRAFT]
                           |
            ┌──────────────┼────────────── 直接归档 AbandonDraft ──────────────┐
            │              │                                                    │
            │      提交审核 SubmitReview                                          │
            │       (前置: 至少 1 步且每步必填绑定满足, INV-06)                    │
            │              │                                                    │
            │              v                                                    │
            │        [SUBMITTED]                                                │
            │              │                                                    │
            │     ┌────────┴────────┐                                          │
            │     v                 v                                          │
            │  审核通过 Activate   审核驳回 Reject                                │
            │  (前置: 引用校验全通过,  (回到 DRAFT,                              │
            │   INV-07/INV-08)       载荷带 rejection_notes)                    │
            │     │                 │                                          │
            │     v                 v                                          │
            │  [ACTIVATED]      [DRAFT]                                         │
            │     │                                                            │
            │     │ 弃用 Deprecate                                              │
            │     │ (被新版本生效时自动触发 / 手动弃用)                            │
            │     v                                                            │
            │  [DEPRECATED]                                                     │
            │     │                                                            │
            │     │ 归档 Archive                                                │
            │     v                                                            │
            └─▶ [ARCHIVED] ◀───────────────────────────────────────────────────┘
                  (终态—之后任何事件均拒收, INV-03)
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| → DRAFT（新建） | CreateRouteVersion | 产品存在（查物料上下文）；同 route 无 ACTIVATED 冲突或为显式升级 | 工艺工程师 |
| DRAFT → SUBMITTED | SubmitRouteReview | 至少 1 步；每步必填绑定满足（operation_id + 工位类型 + 必需防错项，INV-06） | 工艺工程师 |
| SUBMITTED → ACTIVATED | ActivateRouteVersion | 审核通过；引用校验全通过（operation 非 DEPRECATED、quality_gate_rule 存在、spec_keys 在台账可解析，INV-07/INV-08） | 工艺主管 |
| SUBMITTED → DRAFT | RejectRouteReview | 审核驳回，带 rejection_notes | 工艺主管 |
| ACTIVATED → DEPRECATED | DeprecateRouteVersion | 当前 ACTIVATED；新版本生效自动触发或手动弃用 | `RouteActivationService`（新版本生效时）/ 工艺主管 |
| DEPRECATED → ARCHIVED | ArchiveRouteVersion | 当前 DEPRECATED；无在制品引用（可选，🔴 热点） | 工艺主管 / 归档策略 |
| DRAFT → ARCHIVED | AbandonDraft | 当前 DRAFT | 工艺工程师 |
| → ARCHIVED 之后 | （任何） | 终态，拒收（INV-03） | — |

**关键事件**

| 事件 | 触发场景 | 不变式 |
|------|---------|--------|
| RouteVersionCreated | 工程师新建草稿 | 初始状态 DRAFT；绑定产品/路线类型 |
| RouteReviewSubmitted | 提交审核 | 前置 DRAFT；步骤与必填绑定满足（INV-06） |
| ProcessRouteActivated | 审核通过、版本生效（**本上下文核心输出事件**） | 前置 SUBMITTED；引用校验通过（INV-07/INV-08）；生效后内容不可变（INV-04）；同 route 唯一 ACTIVE（INV-02） |
| RouteReviewRejected | 审核驳回 | 回到 DRAFT；带 rejection_notes |
| ProcessRouteDeprecated | 被新版本取代或手动弃用 | 前置 ACTIVATED；弃用不删除内容（历史过点记录仍可回溯，INV-04） |
| RouteVersionArchived | 归档 | 终态（INV-03）；允许从 DEPRECATED / DRAFT 转入 |

> 🔴 **热点：是否允许同 route 并存多个 ACTIVATED 版本**——例如 PCBA 段 v3 与整机段 v2 同时 ACTIVE（不同 route_type）。建议**按 (route_id, route_type) 维度唯一 ACTIVE**：同一产品同一类型的路线同时只能有一个生效版本；不同类型（PCBA/装配）可并存。待与工艺工程团队确认 route 粒度。

> 🔴 **热点：DEPRECATED → ARCHIVED 是否要求无在制品引用**——归档是否需要等待所有引用该版本的在制品完成？建议**归档不依赖在制品状态**（版本不可变，归档只是标记不再用于新过点；已过点记录与在制品按其锁定的 routeVersion 继续，INV-04/INV-09）。归档后读模型仍可查询。待与在制品追踪上下文确认归档后查询需求。

### 2.3 步骤配置与多维绑定（DRAFT 态专属）

步骤配置只能在 DRAFT 态进行；一旦 ACTIVATED，步骤内容连同所有绑定冻结（INV-04）。这是"防错项/质量门禁/设备/工装/再入点"如何挂到路线上的核心链路。

```
[参与者:工艺工程师] 在 DRAFT 版本中编排步骤
        |
        v
【命令:配置步骤 ConfigureStep】--> 【聚合:RouteVersion(DRAFT)】
        |
        v
【事件:步骤已配置 RouteStepConfigured】
 (route_id, route_version, step_no, operation_id, station_type, is_reentry_point)
        |
        +---- 多维绑定（DRAFT 态可并行追加，均作用于该 RouteStep） ----+
        |
        v
【命令:绑定防错项 BindAntiErrorItem】
        |
        v
【事件:防错项已绑定 AntiErrorItemBound】
 (step_no, dimension: MATERIAL/FIXTURE/EQUIPMENT_ADM/FA_GATE/PRECEDENCE/REALTIME_DATA, params{})
        |
        v
【命令:绑定质量门禁 BindQualityGate】--> 【事件:质量门禁已绑定 QualityGateBound】(step_no, quality_gate_rule_id)
【命令:绑定设备规格 BindEquipmentSpec】--> 【事件:设备规格已绑定 EquipmentSpecBound】(step_no, spec_keys[])
【命令:绑定工装匹配规则 BindFixtureMatchingRule】--> 【事件:工装匹配规则已绑定 FixtureMatchingRuleBound】(step_no, fixture_matching_rule_id)
【命令:绑定参数模板 BindParameterTemplate】--> 【事件:参数模板已绑定 ParameterTemplateBound】(step_no, template_id)
【命令:配置再入点 ConfigureReentryPoint】--> 【事件:再入点已配置 ReentryPointConfigured】(step_no, reentry_rules: 缺陷类型→合法再入步序)
        |
        v
（所有绑定完成后可 SubmitRouteReview, §2.2）
```

**防错项维度对照**（与过点执行上下文 §2.1 防错校验维度一一对应，本上下文只定义"哪些维度在该步生效 + 参数"，不执行校验）：

| 防错维度 | 本上下文配置内容 | 过点执行上下文执行内容 |
|---------|-----------------|---------------------|

> 这些查询不引入分布式事务——校验失败则绑定/生效命令被拒绝（INV-07/INV-08），不产生半成品状态。

> 🔴 **热点：是否需要缓存上下文外引用以加速设计时校验**——工艺工程师在设计路线时频繁校验引用，是否维护物料/质量/台账的本地只读投影？建议**当前阶段不维护**（写少读多场景下设计频次低，同步查询足够；引入投影会增加一致性复杂度）。若后续设计体验差再引入。待与平台确认设计时延要求。

## 3. 事件风暴 — 版本审核子流程

版本审核是生效的前置门禁，确保工艺路线发布前经过工艺主管确认。

```
[参与者:工艺工程师] 完成 DRAFT 配置
        |
        v
【命令:提交审核 SubmitRouteReview】--> 【聚合:RouteVersion(DRAFT)】
        |
        v
【事件:审核已提交 RouteReviewSubmitted】(route_version, submitted_by, submitted_at)
        |
        v
【策略:自动预校验引用完整性】（RouteValidationService, INV-06）
        |
        v
预校验（同步查上下文外引用）
   ├── 步骤数 ≥ 1？
   ├── 每步 operation_id 非 DEPRECATED？
   ├── 必填防错项齐全？
   └── 引用对象存在性（质量规则/设备规格/工装规则）
        |
        ├── 预校验失败 --> 【事件:审核提交已拒绝 ReviewSubmissionRejected】(failed_checks[]) --> 回 DRAFT
        |
        v (预校验通过)
[参与者:工艺主管] 审核决策
        |
        +-------------------+-------------------+
        v                   v                   v
    批准生效              驳回               要求修改
    ActivateRouteVersion  RejectRouteReview   (退回 DRAFT 修改, 同驳回流)
        |                   |
        v                   v
【事件:工艺路线版本已生效    【事件:审核已驳回 RouteReviewRejected】
 ProcessRouteActivated】     (route_version, rejection_notes, rejected_by)
 (见 §2.4)                   |
        |                   v
        v              回到 DRAFT（可修改后重新提交）
   (进入 §2.4 生效发布主链)
```

> **审核驳回 vs 生效拒绝的语义区分**：
> - **审核驳回（RouteReviewRejected）**：工艺主管在审核环节否决（业务决策），回到 DRAFT 可修改重提。
> - **生效拒绝（ActivationRejected）**：主管已批准但生效时引用校验失败（技术校验，如引用对象在审核期间被弃用），版本停留 SUBMITTED 等待修复。
>
> 两者均不进入 ACTIVATED，但状态流向不同：驳回回 DRAFT，生效拒绝留 SUBMITTED（🔴 热点：是否回 DRAFT 待确认）。

> 🔴 **热点：生效拒绝后的状态流向**——主管批准后生效校验失败（引用对象在审核窗口期被弃用），版本应回 DRAFT 还是留 SUBMITTED？建议**回 DRAFT 并带 `activation_blocked_reason`**，工程师修复引用后重新提交。待与工艺工程团队确认审核流是否支持"批准但待修复"中间态。

## 4. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **Operation** | 单个工序定义 | INV-01 operation_id 全局唯一；capability_keys 受工程标准约束；DEPRECATED 后新草稿不可引用，已生效版本引用不受影响（INV-04） |
| **ParameterTemplate** | 单个参数模板 | template_id 全局唯一；绑定到唯一 operation_id；DEPRECATED 后新草稿不可引用 |
| **RouteVersion** | 单个路线版本（含步骤与绑定） | INV-02 同 (route_id, route_type) 同时仅一个 ACTIVATED（**跨实例**，由 `RouteActivationService` + 唯一索引保证）；INV-03 ARCHIVED 为终态，之后任何事件均拒收；INV-04 ACTIVATED 后步骤内容与所有绑定不可变（保证过点记录 routeVersion 快照完整性）；INV-06 提交审核前置：至少 1 步且每步必填绑定满足；INV-07 生效前置：所有 operation 非 DEPRECATED；INV-08 生效前置：所有上下文外引用（quality_gate_rule_id / spec_keys / fixture_matching_rule_id）可解析存在 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同 route 唯一 ACTIVE 版本）→ `RouteActivationService` 领域服务在生效时查询读模型 + 数据库唯一索引 (route_id, route_type, status=ACTIVATED) 共同保证；新版本生效自动 Deprecate 旧版本（同事务 outbox）
> - **INV-04**（生效后不可变）→ RouteVersion 聚合根在 ACTIVATED 态拒收任何 ConfigureStep / Bind* 命令；保证过点执行上下文 `CheckpointRecord.route_version` 快照可永久回溯
> - **INV-07 / INV-08**（生效引用校验）→ `RouteValidationService` 领域服务在生效时同步查询本上下文 + 质量上下文 + 台账上下文（REST），校验失败拒绝 ActivateRouteVersion
> - **INV-09**（版本一致性，变更仅影响变更后首次过点，领域总览 §5.1）→ 本上下文发布 `ProcessRouteActivated` + 过点执行上下文 `ProcessRouteCache` 投影 + `CheckpointRecord.route_version` 快照 + `RoutingProgress` 锁定版本，由过点引擎在首次过点时固化版本保证（跨上下文，主责在过点执行上下文）。该不变式与物料上下文 INV-09、质量上下文 INV-09 跨上下文语义同构、统一编号
> - **INV-10**（事务发件箱至少一次投递，领域总览 §5.4）→ outbox 模式 + 消费端幂等（source_event_id 去重表，过点执行上下文侧保证）。该不变式与物料上下文 INV-10、质量上下文 INV-10 跨上下文语义同构、统一编号
>
> 注：`RouteStep` / `AntiErrorItem` / `QualityGateBinding` / `EquipmentSpecBinding` / `FixtureMatchingBinding` 是 **RouteVersion 聚合内实体**，不独立成聚合——它们的生命周期完全依附于所属 RouteVersion，且仅在 DRAFT 态可变更（INV-04）。

## 7. 与过点执行上下文的协作边界

本上下文是工艺路线的**权威源**，过点执行上下文是**消费方**，通过"本地缓存 + 降级查询"解耦（领域总览 §5.1）：

| 维度 | 本上下文（工艺管理上下文） | 过点执行上下文 |
|------|------------------------|--------------|
| 视角 | 配置视角（"工艺怎么定义、版本怎么发布"） | 执行视角（"过点瞬间怎么判定、怎么记录"） |
| 存储 | 工序定义、参数模板、路线版本、步骤与绑定（主数据） | 过点记录、路线推进、工单进度、ProcessRouteCache（投影） |
| 谁判定 | 规则与版本定义（不执行过点判定） | 执行判定（综合多维度放行/拦截） |
| 事件性质 | 配置生效事实：ProcessRouteActivated / Deprecated | 过点执行事实：CheckpointReleased / Blocked |
| 流程所有权 | 工序定义 → 路线编排 → 审核 → 生效发布 → 弃用/归档 | 扫码 → 防错 → 决策 → 记录 → 推进 → 累计 |
| 版本一致性 | 发布生效事件为权威源 | 消费生效事件刷新缓存；过点记录锁定 routeVersion（INV-09） |
| 写入频率 | 写少读多（版本发布时写入） | 高频实时（每板毫秒级） |

**事件流向**：
- **本上下文 → 过点执行上下文**：`process.route.lifecycle`（`ProcessRouteActivated` 刷新 ProcessRouteCache，`ProcessRouteDeprecated` 标记旧版本不可用于新过点）
- **过点执行上下文 → 本上下文**：（无事件反向发布；仅缓存未命中时 REST 降级查询 RouteVersionReadModel，≤500ms）
- **不互相调用的操作**：本上下文不调用过点执行上下文的"过点/刷新缓存"接口——全部通过事件异步解耦；过点执行上下文不调用本上下文的"创建/生效版本"接口——配置变更归本上下文

> **版本快照原则**（INV-04 + INV-09）：过点记录 `CheckpointRecord.route_version` 是不可变快照。本上下文后续发布新版本或弃用旧版本，**不影响已过点记录与在制品**——在制品按首次过点锁定的 routeVersion 继续推进；新版本仅对变更后首次过点的在制品生效。

## 8. 与质量上下文、物料上下文的协作边界

三者同属制造资源服务，是配置型主数据的姊妹上下文，以"引用"协作、不互相复制实体：

| 维度 | 本上下文（工艺管理） | 质量上下文 | 物料上下文 |
|------|------------------|----------|----------|
| 视角 | 流程编排视角（"工序怎么排、版本怎么发"） | 质量标准视角（"检验标准是什么、门禁怎么判"） | 物料主数据视角（"产品/BOM/供应商是什么"） |
| 存储 | 工序、参数模板、路线版本、步骤绑定 | 检验标准、缺陷分类、质量门禁规则 | 物料主数据、产品 BOM、供应商、替代料 |
| 干系人 | 工艺工程师/工艺主管 | 质量工程师 | 采购/物控 |
| 被引用方式 | — | 本上下文以 `quality_gate_rule_id` 引用其规则 | 本上下文以 `product_id` / `bom_item_ref` 引用其产品/BOM |
| 变更影响 | 路线版本生效影响过点 | 规则变更影响过点质量门禁缓存（不回写已生效版本, INV-04） | BOM 变更影响物料防错（不回写已生效版本） |

**事件流向**：
- **质量上下文 → 本上下文**：`QualityGateDeprecated`（标记引用该规则的 DRAFT 版本 stale_reference，INV-04 保证已生效版本不变）
- **物料上下文 → 本上下文**：（当前无事件订阅；设计时同步查询产品/BOM 存在性）
- **本上下文 → 质量/物料上下文**：（无反向事件；质量上下文通过过点执行上下文 `mes.testresult.structured` 获取测试结果做业务判定，不经过本上下文）
- **不互相调用的操作**：本上下文不定义检验标准/BOM；质量/物料上下文不定义工艺路线——各自自治，仅通过引用 + 设计时校验协作

> 🔴 **热点：质量门禁规则变更是否需通知本上下文重校验已生效版本**——质量上下文修订某 quality_gate_rule 的判定逻辑（非弃用），已生效版本引用的 rule_id 不变，过点时读质量门禁缓存自动反映最新逻辑。建议**规则修订不通知本上下文**（rule_id 不变即引用有效），仅规则**弃用**才通知标记 DRAFT。待与质量上下文确认规则"修订 vs 弃用"语义。

## 9. 与设备工装台账上下文的协作边界

本上下文是设备能力规格与工装匹配规则的**引用方**，台账上下文是**事实源**：

| 维度 | 本上下文（工艺管理） | 设备工装台账上下文 |
|------|------------------|------------------|
| 视角 | 工艺要求视角（"这步需要什么能力的设备/工装"） | 资产管理视角（"资产有什么能力、能不能用"） |
| 存储 | spec_keys（引用）、fixture_matching_rule_id（引用） | AssetSpecificationProfile（能力规格画像·事实源）、FixtureMatchingRule（工装匹配规则·事实源） |
| 谁定义规格 | 仅引用 spec_keys，不定义设备能力 | 定义并维护能力规格画像与工装匹配规则（`eam.asset.specification`） |
| 变更影响 | 台账规格变更不回写已生效版本（INV-04）；过点时由过点执行上下文综合两缓存校验 | 规则变更发布 `FixtureMatchingRuleUpdated` 通知制造资源服务重新加载（台账上下文 §4 已声明） |

**事件流向**：
- **台账上下文 → 本上下文**：`eam.asset.specification`（能力规格画像/工装匹配规则变更，本上下文设计时校验引用可解析性；当前不订阅运行时事件，仅设计时同步查询）
- **本上下文 → 台账上下文**：（无反向事件）
- **过点时的接力**：本上下文步骤配置定义"需要 spec_keys[] 的设备能力"，过点执行上下文过点时综合 ProcessRouteCache（来自本上下文）+ EquipmentAvailabilityCache（来自台账上下文）校验"设备可用且能力匹配"

> 🔴 **热点：spec_key 命名空间对齐**——本上下文 `EquipmentSpecBinding.spec_keys` 与台账上下文 `AssetSpecificationProfile.spec_keys` 必须共享同一命名空间（如"回流焊.温区数"、"贴片.最大板尺寸"），否则过点时能力匹配校验无法比较。待与设备工装台账上下文确认 spec_key 字典与命名规范。

> 🔴 **热点：台账 FixtureMatchingRuleUpdated 的消费**——台账上下文 §4 声明"FixtureMatchingRuleUpdated 通知制造资源服务重新加载"。本上下文是否需订阅此事件刷新已生效版本的引用？建议**不刷新已生效版本**（INV-04 不可变）；过点时由过点执行上下文读台账缓存反映最新规则。本上下文仅在 DRAFT 设计时校验 rule_id 存在性。待与台账上下文确认"通知重新加载"的具体语义。

## 10. 与返修上下文、返工上下文的协作边界

本上下文提供**再入点配置**，返修/返工上下文与过点执行上下文消费它做再入判定：

| 维度 | 本上下文（工艺管理） | 返修上下文（单件） | 返工上下文（批量） |
|------|------------------|----------------|----------------|
| 视角 | 路线配置视角（"哪些步序可再入、缺陷→再入映射"） | 单件离线返修执行 | 批量返工工单执行 |
| 存储 | RouteStep.is_reentry_point、reentry_rules（缺陷类型→合法再入步序） | 返修任务、诊断、再入点判定 | 返工工单、返工路线 |
| 谁判定再入点 | 定义合法再入点与映射规则 | 读取本上下文再入点配置 + 诊断结果判定单件再入点 | 读取本上下文再入点配置判定批量再入点 |
| 协作方式 | 提供只读再入点配置（REST 查询 / 读模型） | 过点执行上下文 §5.2 消费返修完成事件后校验 reentry_point ∈ routeVersion 合法步序 | 同返修，批量维度 |

**事件流向**：
- **本上下文 → 返修/返工上下文**：（无直接事件；返修/返工上下文通过过点执行上下文或 REST 查询本上下文读模型获取再入点配置）
- **返修/返工上下文 → 本上下文**：（无反向事件；再入点校验由过点执行上下文执行，结果 `ReentryRejected` 回返修/返工上下文）
- **不互相调用的操作**：本上下文不创建返修/返工任务；返修/返工上下文不修改路线版本——各自自治

> 🔴 **热点：再入点配置的查询入口**——返修上下文"读取返修规则与再入点配置"（领域总览 §4.2）是直接查本上下文，还是经过点执行上下文缓存？建议**过点执行上下文 ProcessRouteCache 已含 steps[] 与 is_reentry_point/reentry_rules**（若 §2.4 载荷含完整步骤），返修/返工上下文可复用过点执行上下文缓存或直接 REST 查本上下文读模型。待与返修/返工上下文确认查询路径与载荷需求。
