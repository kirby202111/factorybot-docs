# 质量上下文（Quality Bounded Context）

> **限界上下文**：质量上下文
> **统一语言**：InspectionStandard（检验标准·聚合根，含判定项与接收准则）, FirstArticleInspectionStandard（首件检验标准·InspectionStandard 子类型）, IqcStandard（来料检验标准·InspectionStandard 子类型）, InspectionItem（检验项·InspectionStandard 内实体）, AcceptanceCriterion（接收准则·检验项的合格判定阈值/规则）, DefectCatalog（缺陷分类字典·聚合根）, DefectCategory（缺陷分类项·DefectCatalog 内实体，含严重度）, DefectSeverity（缺陷严重度：CRITICAL / MAJOR / MINOR）, QualityGateRule（质量门禁规则·聚合根，版本化，含判定逻辑与引用的检验标准/缺陷分类）, GateVerdict（门禁判定结果：PASS / HOLD / BLOCK）, QualityGateRuleState（规则状态：DRAFT / SUBMITTED / ACTIVATED / DEPRECATED / ARCHIVED）, QualityVerdict（质量业务判定·消费 TestResult 翻译为业务语义 + 缺陷分类）, DefectRecord（缺陷记录·QualityVerdict 内实体）, BatchQualityAnomaly（批量质量异常·缺陷影响范围分析结果）, AnomalyScope（异常影响范围：SINGLE_INCIDENT / BATCH_SYSTEMIC）, IqcRequest（来料质检请求）, IqcResult（来料质检结果）, OverrideAuthorizationPolicy（放行授权策略·HOLD 人工放行权限配置）, QualityGateCache（质量门禁缓存·过点执行上下文本地投影）, InspectionStandardCache（检验标准缓存·首件处理上下文本地投影）
> **核心关注**：以"事件风暴"方式刻画**生产执行的数据基座**——检验标准、缺陷分类字典、质量门禁规则的配置与发布，以及**对过点执行上下文 `TestResult` 的业务判定（翻译原始判定为业务语义 + 缺陷分类）**与**批量质量异常分析**。本上下文只对"**检验标准是什么、缺陷怎么分类、门禁规则怎么配、TestResult 的业务判定结果是什么、批量异常怎么识别**"负责；**不负责**工序定义与工艺路线编排（归工艺管理上下文）、物料主数据与库存事实源（归物料上下文）、过点判定与防错校验执行（归过点执行上下文，本上下文仅发布规则供其缓存消费 + 发布业务判定结果供首件/统计消费）、在制品位置投影（归在制品追踪上下文）、首件检验流程编排（归首件处理上下文，本上下文仅发布首件检验标准与首件单元业务判定结果）、单件返修任务执行（归返修上下文，本上下文仅提供缺陷分类字典辅助诊断）、批量返工工单执行（归返工上下文，本上下文仅发布批量异常信号供其评估生成返工工单）。本上下文是**配置型主数据 + 业务判定**混合上下文，写少读多（主数据）与中频写入（业务判定/异常分析）并存，对外以"规则生效/弃用事件 + 检验标准变更事件 + 质量业务判定事件 + 批量异常信号事件"为协作契约。

## 0. 设计原则与边界澄清

质量上下文是制造资源服务下的**质量标准与业务判定子域**，是 PCBA + 整机组装混合型车间所有"质量怎么判、缺陷怎么分、门禁怎么配、TestResult 的业务语义是什么、批量异常怎么识别"的权威源。它解决两类核心问题：**(1) 配置问题**——检验标准、缺陷分类字典、质量门禁规则的定义与发布；**(2) 判定问题**——消费过点执行上下文的 `TestResult` 翻译为业务判定（`QualityVerdict` + 缺陷分类），并基于缺陷累积分析识别批量质量异常。

**上下文内职责**（本上下文负责）：
- **检验标准管理**——首件检验标准、来料检验标准（IQC）、常规过点检验标准的定义、变更、弃用；含检验项与接收准则
- **缺陷分类字典**——缺陷分类项的定义与维护（如焊点不良、缺件、错件、桥接、冷焊等），含严重度分级（CRITICAL / MAJOR / MINOR）
- **质量门禁规则生命周期**——草稿 → 提交审核 → 生效 → 弃用 → 归档的完整状态机；规则含判定逻辑（引用检验标准 + 缺陷分类）
- **规则生效/弃用发布**——发布 `QualityGateRuleActivated`（供过点执行上下文刷新 `QualityGateCache`）、`QualityGateDeprecated`（供工艺管理上下文标记 stale_reference + 过点执行移除缓存）
- **检验标准变更发布**——发布首件检验标准变更（供首件处理上下文刷新 `InspectionStandardCache`）
- **测试结果业务判定**——消费过点执行上下文 `mes.testresult.structured`（`TestResult`，含 `raw_verdict` 原始判定），结合缺陷分类字典与门禁规则，翻译为业务判定 `QualityVerdict`（含缺陷分类 + 业务判定结果），发布 `quality.inspection.verdict` 供首件处理上下文消费与缺陷统计/SPC
- **批量质量异常分析**——基于缺陷累积（`QualityVerdict` + 返修诊断回流）分析缺陷影响范围（单件偶发 vs 批次性系统性），识别批量异常并发布信号供返工上下文评估生成返工工单
- **来料质检（IQC）执行**——消费采购入库前的来料质检请求，执行检验并发布结果（供物料上下文接收入库）
- **放行授权策略配置**——HOLD 状态的人工放行权限规则配置（供过点执行上下文 `OverrideCheckpoint` 与首件处理上下文 `OverrideGate` 引用）

**上下文外职责**（本上下文**不**负责）：
- 工序定义、参数模板、工艺路线、版本生命周期 → 工艺管理上下文（本上下文仅以 `quality_gate_rule_id` 被其步骤绑定引用，提供规则存在性校验）
- 物料主数据、产品 BOM、库存事实源 → 物料上下文（本上下文仅消费其入库事件驱动 IQC，或由采购/仓管系统触发 IQC）
- 过点判定、防错校验执行、放行/拦截决策、过点记录 → 过点执行上下文（本上下文仅发布规则供其 `QualityGateCache` 缓存消费；过点级同步门禁判定 `QualityGateEvaluated` 由过点执行上下文基于缓存规则执行，本上下文不参与过点主事务）
- 测试数据接入适配层、`TestResult` 结构化 → 过点执行上下文（本上下文仅消费其产出的 `TestResult` 做业务判定，不重复采集）
- 在制品身份、位置投影、齐套状态 → 在制品追踪上下文
- 首件标识、首件检验流程编排、首件放行门禁判定 → 首件处理上下文（本上下文仅发布首件检验标准 + 首件单元业务判定结果供其消费）
- 单件返修任务、故障诊断、再入点判定 → 返修上下文（本上下文仅提供缺陷分类字典辅助诊断 REST 查询，并消费其诊断结论做缺陷分析）
- 批量返工工单、返工工艺路线 → 返工上下文（本上下文仅发布批量异常信号供其评估生成返工工单）
- 设备/工装能力规格与匹配规则 → 设备工装台账上下文
- OEE / 良率分析 / SPC 可视化 → 数据应用上下文（本上下文仅发布 `QualityVerdict` 供其统计消费，不建设分析看板）

**为什么不把"质量"和"工艺管理"并入一个限界上下文**：
1. **关注点不同**——工艺管理关注"工序怎么排、版本怎么发"（流程编排视角），质量关注"检验标准是什么、缺陷怎么分类、门禁规则怎么判、TestResult 业务语义是什么"（质量标准与判定视角）。
2. **生命周期与变化频率不同**——工艺路线随产品/工程变更发布（低频，版本化），质量门禁规则随检验标准修订（独立低频），业务判定是中频（每次过点 TestResult 触发）。混在一起会让"改一条门禁规则"或"判一次 TestResult"动到"路线版本"代码。
3. **干系人不同**——工艺管理由工艺工程师/工艺主管负责，质量规则由质量工程师负责，审核流与发布节奏不同。
4. **解耦即可协作**——工艺管理上下文以 `quality_gate_rule_id` 引用本上下文规则，规则变更不影响已生效路线版本（版本不可变，工艺管理 INV-04）；过点执行时由 `QualityGateCache` 反映最新规则。与工艺管理上下文 §0"为什么不把工艺管理和质量并入一个限界上下文"立场一致。

**为什么不把"质量"和"物料"并入一个限界上下文**：
1. **关注点不同**——物料关注"物料是什么、BOM 怎么配、库存多少"（物料与库存视角），质量关注"检验标准、缺陷分类、门禁规则、业务判定"（质量视角）。
2. **来料质检（IQC）是协作而非合并理由**——IQC 执行归本上下文，入库事实归物料上下文。两者通过"IQC 完成 → 物料入库"事件接力，无需合并。
3. **干系人不同**——物料由物控/采购团队维护，质量由质量工程师维护，组织边界清晰。

**为什么测试结果业务判定归本上下文而非过点执行上下文**：
1. **语义解释职责**——过点执行上下文的测试数据接入适配层只做"结构化"（解析 + 字段映射 + 单位归一化 + 原始判定 `raw_verdict` 透传），**不翻译** PASS/FAIL 业务语义（过点执行上下文 §2.5 已声明）。业务语义解释（"这个 `raw_verdict` 对应什么缺陷、是否需 HOLD"）归本上下文。
2. **过点同步门禁 vs 异步业务判定**——过点执行上下文基于 `QualityGateCache` 做**同步门禁判定**（`QualityGateEvaluated`，PASS/HOLD/BLOCK，驱动过点放行/拦截，在过点主事务内）；本上下文消费 `TestResult` 做**异步业务判定**（`QualityVerdict`，含缺陷分类 + 业务判定，供首件检验/缺陷统计/SPC 消费）。两者服务不同目的：门禁=流量控制（快），判定=质量评估与记录（深）。
3. **首件检验需要缺陷级判定**——首件处理上下文需要的不是简单 PASS/BLOCK，而是缺陷分类与判定细节（首件处理上下文 §2.3 热点明确偏好消费本上下文的 `quality.inspection.verdict`，而非过点执行上下文的 `QualityGateEvaluated`）。
4. **缺陷统计/SPC 需要缺陷分类**——数据应用上下文的良率/SPC 分析需要缺陷分类维度，过点执行上下文不产出缺陷分类。

> 🔴 **热点：同步门禁与异步判定的一致性**——过点执行上下文的同步门禁 `QualityGateEvaluated`（基于缓存规则）与本上下文的异步业务判定 `QualityVerdict`（基于规则 + 缺陷字典）可能不一致（如门禁 PASS 但判定发现轻微缺陷仍记入统计）。建议立场：**门禁结果驱动过点放行/拦截（权威）**，业务判定结果用于首件检验汇总与缺陷统计（记录性），两者语义不同不要求强一致。门禁用缓存规则做快速判定，判定用完整规则 + 缺陷字典做深度分类。待与过点执行上下文确认两者一致性预期。

> 🔴 **热点：业务判定的时效性**——`QualityVerdict` 异步消费 `TestResult`，存在秒级延迟。首件处理上下文在末步检验工位放行后汇总检验结果时，若 `QualityVerdict` 尚未到达，是否等待？建议立场：**首件处理上下文开启等待窗口**（参考过点执行上下文 TestResult 等待窗口模式），窗口内 `quality.inspection.verdict` 到达则汇总，超时则降级用过点执行的 `QualityGateEvaluated` 作为 fallback。待与首件处理上下文确认等待窗口与降级策略。

## 1. 主语清单与对象形态

质量上下文处理的对象分三类：**配置型聚合**（检验标准、缺陷分类字典、质量门禁规则）、**判定型聚合**（质量业务判定、批量质量异常）、**执行型聚合**（来料质检请求/结果）。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 聚合根

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **InspectionStandard**（检验标准） | 首件检验标准、来料检验标准（IQC）、常规过点检验标准 | 可复用；跨规则共享；有 ACTIVE / DEPRECATED 状态；按类型区分（FIRST_ARTICLE / IQC / ROUTINE） | 质量工程师定义 |
| **DefectCatalog**（缺陷分类字典） | SMT 焊点缺陷字典、组装缺陷字典 | 可复用；跨规则共享；有 ACTIVE / DEPRECATED 状态；含缺陷分类项与严重度 | 质量工程师定义 |
| **QualityGateRule**（质量门禁规则·版本化） | 某 PCBA 产品的 AOI 工位门禁规则 v2、某整机的 FCT 工位门禁规则 v1 | 版本化；生效后不可变；含判定逻辑（引用检验标准 + 缺陷分类）；唯一 ACTIVE 版本约束 | 质量工程师编排 + 质量主管审核 |
| **QualityVerdict**（质量业务判定） | 某 SN 在 AOI 工位的业务判定（含缺陷分类） | 消费 TestResult 产出；含缺陷记录；不可变；中频写入 | 过点执行上下文 `mes.testresult.structured` |
| **BatchQualityAnomaly**（批量质量异常） | 某工单某工序的批次性缺陷异常 | 基于缺陷累积分析产出；含影响范围（SINGLE_INCIDENT / BATCH_SYSTEMIC）+ 受影响 SN 清单 | 由缺陷累积分析策略触发 |
| **IqcRequest**（来料质检请求） | 某批次来料的 IQC 请求 | 关联物料批次 + 检验标准；PENDING / PASSED / FAILED 状态 | 采购/仓管系统入库前触发 |

### 1.2 QualityGateRule 内实体（非独立聚合）

| 实体 | 归属 | 关键管理特征 | 引用的上下文外对象 |
|------|------|-------------|-------------------|
| **GateLogic**（门禁判定逻辑） | QualityGateRule | 判定规则（阈值/逻辑表达式/引用检验标准）；产出 PASS/HOLD/BLOCK | InspectionStandard（本上下文） |
| **DefectMapping**（缺陷映射） | QualityGateRule | 原始判定/测量项 → 缺陷分类项的映射 | DefectCatalog（本上下文） |
| **HoldAuthorizationBinding**（HOLD 授权绑定） | QualityGateRule | 该规则 HOLD 时允许的放行授权策略引用 | OverrideAuthorizationPolicy（本上下文） |

> **关键边界**：本上下文只存**质量规则与判定**，**不复制**上下文外实体内容。工艺管理上下文以 `quality_gate_rule_id` 引用本上下文规则（不复制规则实体），过点执行上下文读 `QualityGateCache`（投影）执行同步门禁判定。这保证规则生效后过点记录的规则快照完整（与工艺管理上下文 INV-04 同构的版本不可变原则）。

> **双段生产架构**：PCBA 段（SMT 焊接/检测类缺陷为主）与整机组装段（装配/功能类缺陷为主）的检验标准与缺陷分类分别维护；质量门禁规则按工位 × 产品段配置。两段在过点执行上下文以单件推进衔接，本上下文不感知段间拼接。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 检验标准与缺陷分类字典定义（可复用基础）

检验标准与缺陷分类字典是质量门禁规则编排的"积木"，独立维护、跨规则复用。

```
[参与者:质量工程师]
        --【命令:定义检验标准 DefineInspectionStandard】--> 【聚合:InspectionStandard】
                                                  |
                                                  v
                          【事件:检验标准已定义 InspectionStandardDefined】
                           (standard_id, name, type: FIRST_ARTICLE/IQC/ROUTINE,
                            inspection_items[], acceptance_criteria[])
                                                  |
                          【策略:检验标准定义后可被门禁规则引用】
                                                  v
        --【命令:定义缺陷分类字典 DefineDefectCatalog】--> 【聚合:DefectCatalog】
                                                          |
                                                          v
                          【事件:缺陷分类字典已定义 DefectCatalogDefined】
                           (catalog_id, name, categories[]: [{code, name, severity: CRITICAL/MAJOR/MINOR}])
                                                  |
                                                  v
                          发布到 quality.standard.lifecycle / quality.defect.catalog
                          （供门禁规则编排引用、返修上下文 REST 查询辅助诊断）
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| InspectionStandardDefined | 质量工程师新建检验标准 | standard_id, name, type, inspection_items[], acceptance_criteria[] | 可被门禁规则引用；首件检验标准变更通知首件处理上下文刷新缓存 |
| InspectionStandardUpdated | 检验标准修订（检验项/接收准则变更） | standard_id, type, changed_fields | 影响引用该标准的草稿规则；首件检验标准变更通知首件处理上下文刷新缓存 |
| InspectionStandardDeprecated | 检验标准弃用 | standard_id, reason, deprecated_at | 限制新草稿规则引用；已生效规则不受影响（INV-04） |
| DefectCatalogDefined | 质量工程师新建缺陷分类字典 | catalog_id, name, categories[] | 可被门禁规则引用、返修上下文 REST 查询 |
| DefectCatalogUpdated | 缺陷分类字典修订（新增/调整分类项） | catalog_id, changed_categories[] | 影响引用该字典的草稿规则；业务判定使用最新字典 |
| DefectCatalogDeprecated | 缺陷分类字典弃用 | catalog_id, reason | 限制新草稿规则引用；已生效规则不受影响 |

> 🔴 **热点：检验标准的粒度**——检验标准是按"工序类型"（如 AOI 检验标准、FCT 检验标准）还是按"产品 × 工序"（如产品A的 AOI 检验标准）定义？建议**通用工序级标准 + 门禁规则承载产品差异**（与工艺管理上下文工序粒度热点同构），避免标准爆炸。待与质量工程团队确认标准粒度。

> 🔴 **热点：缺陷分类字典的统一性**——PCBA 段缺陷（焊点不良/桥接/立碑等）与整机组装段缺陷（螺丝漏打/功能不良/外观瑕疵等）是统一字典还是分段字典？建议**统一字典 + 段标签**（`segment: PCBA/ASSEMBLY`），便于跨段缺陷统计，同时支持按段过滤。待与质量工程团队确认字典组织方式。

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| → DRAFT（新建） | CreateQualityGateRule | 规则类型/工位/产品范围已明确；同 rule 无 ACTIVATED 冲突或为显式升级 | 质量工程师 |
| DRAFT → SUBMITTED | SubmitRuleReview | 判定逻辑完整（GateLogic 非空）；引用校验满足（InspectionStandard 非 DEPRECATED、DefectCatalog 非 DEPRECATED，INV-06） | 质量工程师 |

| SUBMITTED → DRAFT | RejectRuleReview | 审核驳回，带 rejection_notes | 质量主管 |
| ACTIVATED → DEPRECATED | DeprecateQualityGateRule | 当前 ACTIVATED；新版本生效自动触发或手动弃用 | `RuleActivationService`（新版本生效时）/ 质量主管 |
| DEPRECATED → ARCHIVED | ArchiveQualityGateRule | 当前 DEPRECATED | 质量主管 / 归档策略 |
| DRAFT → ARCHIVED | AbandonDraft | 当前 DRAFT | 质量工程师 |
| → ARCHIVED 之后 | （任何） | 终态，拒收（INV-03） | — |

**关键事件**

| 事件 | 触发场景 | 不变式 |
|------|---------|--------|
| QualityGateRuleCreated | 工程师新建草稿 | 初始状态 DRAFT；绑定规则类型/工位/产品范围 |
| RuleReviewSubmitted | 提交审核 | 前置 DRAFT；判定逻辑与引用校验满足（INV-06） |
| **QualityGateRuleActivated**（规则生效·本上下文核心输出事件之一） | 审核通过、版本生效 | 前置 SUBMITTED；引用校验通过（INV-07/INV-08）；生效后内容不可变（INV-04）；同 rule 唯一 ACTIVE（INV-02） |
| RuleReviewRejected | 审核驳回 | 回到 DRAFT；带 rejection_notes |
| **QualityGateDeprecated**（规则弃用·本上下文核心输出事件之一） | 被新版本取代或手动弃用 | 前置 ACTIVATED；弃用不删除内容（历史过点记录与业务判定仍可回溯，INV-04） |
| QualityGateRuleArchived | 归档 | 终态（INV-03）；允许从 DEPRECATED / DRAFT 转入 |

> 🔴 **热点：是否允许同 rule 并存多个 ACTIVATED 版本**——建议**按 rule_id 维度唯一 ACTIVE**：同一规则同时只能有一个生效版本；新版本生效自动 Deprecate 旧版本。与工艺管理上下文 RouteVersion 的 (route_id, route_type) 维度、物料上下文 Bom 的 (product_id, bom_type) 维度一致。待与质量工程团队确认规则粒度。

> 🔴 **热点：规则"修订"与"弃用"的语义区分**——工艺管理上下文 §8 热点要求"规则修订不通知本上下文（rule_id 不变即引用有效），仅规则弃用才通知标记 DRAFT"。本上下文回填立场：**修订 = 新版本生效（QualityGateRuleActivated，rule_id 不变，version 递增）**，过点执行上下文刷新缓存，工艺管理上下文不标记 stale（rule_id 引用仍有效）；**弃用 = 规则整体退役（QualityGateDeprecated，无后继版本）**，工艺管理上下文标记 stale_reference，过点执行上下文移除缓存。两者事件不同，消费方按需订阅。

### 2.4 测试结果业务判定（消费 TestResult → QualityVerdict）

本上下文消费过点执行上下文 `mes.testresult.structured` 的 `TestResult`（含 `raw_verdict` 原始判定，不翻译业务语义），结合缺陷分类字典与门禁规则，翻译为业务判定 `QualityVerdict`（含缺陷分类 + 业务判定结果），发布 `quality.inspection.verdict` 供首件处理上下文消费与缺陷统计/SPC。这是本上下文**业务判定的核心链路**。

```
【外部系统:过点执行上下文】
        -- 发布 mes.testresult.structured: TestResultStructured
           (test_id, sn, station_id, test_type, raw_verdict, measured_items[],
            source_channel, source_ts, structured_ts)
                |
                v
【策略:消费 TestResult 执行业务判定】（QualityVerdictService 编排, 幂等见 INV-09）
                |
                v
        查询该 station_id × product_scope 的 ACTIVE QualityGateRule
        + DefectCatalog（缺陷分类字典）
                |
                v
        应用 gate_logic + defect_mapping 翻译业务语义
        (raw_verdict + measured_items → business_verdict + defect_records[])
                |
                v
【聚合:QualityVerdict】 --【命令:发布质量业务判定 IssueQualityVerdict】
                |
                v
【事件:质量业务判定已发布 QualityVerdictIssued】
 (verdict_id, test_id, sn, station_id, work_order_id, product_id,
  business_verdict: PASS/HOLD/BLOCK, defect_records[]: [{defect_code, severity, location?, measured_value?}],
  rule_id, rule_version, source_test_result_id, verdict_ts)
                |
                v
发布到 quality.inspection.verdict
                |
                v
【外部系统:首件处理上下文】订阅

【外部系统:数据应用上下文】订阅

```

**业务判定与过点同步门禁的关系**：

| 维度 | 过点执行上下文（同步门禁） | 本上下文（异步业务判定） |
|------|------------------------|----------------------|
| 触发源 | 过点扫码后防错校验阶段 | 消费 `mes.testresult.structured` |
| 时机 | 同步，在过点主事务内 | 异步，秒级延迟 |
| 输入 | `QualityGateCache`（缓存规则）+ `TestResult` | `QualityGateRule`（完整规则）+ `DefectCatalog` + `TestResult` |
| 输出 | `QualityGateEvaluated`（PASS/HOLD/BLOCK，驱动放行/拦截） | `QualityVerdict`（PASS/HOLD/BLOCK + 缺陷分类，供首件/统计） |
| 用途 | 流量控制（过点放行/拦截） | 质量评估与记录（首件检验汇总、缺陷统计、SPC） |
| 一致性 | 权威（驱动过点决策） | 记录性（不覆盖过点决策） |

> 🔴 **热点：业务判定是否对所有过点 TestResult 执行**——每个过点 TestResult 都触发异步业务判定可能产生大量 `QualityVerdict`。建议立场：**对所有 TestResult 执行业务判定**（缺陷统计需要全量数据），但判定逻辑应轻量（规则匹配 + 缺陷映射，无复杂计算）。若性能压力，可按工位/产品配置采样判定。待与质量工程团队确认判定覆盖范围。

> 🔴 **热点：raw_verdict 与 business_verdict 的映射复杂度**——设备的 `raw_verdict`（如 AOI 的 NG 信号）到 `business_verdict` + `defect_code` 的映射可能是简单查表，也可能需结合 `measured_items` 综合判定。建议立场：**简单映射走 `defect_mapping` 查表**（规则内配置），**复杂判定走策略引擎**（本上下文内嵌轻量策略引擎，支持阈值/逻辑表达式）。待与质量工程团队确认映射复杂度预期。

> 🔴 **热点：BLOCK 判定与过点拦截的时序**——本上下文的 `QualityVerdict(business_verdict=BLOCK)` 异步到达时，过点可能已放行（过点同步门禁 PASS）。建议立场：**过点同步门禁为权威**，异步 BLOCK 判定不回溯拦截已放行的过点，而是记入缺陷统计供后续批量异常分析（§2.5）触发返工。待与过点执行上下文确认异步 BLOCK 的处理预期。

### 2.5 批量质量异常分析与信号发布

> 🔴 **热点：异常检测的时效性**——缺陷累积分析是实时流式还是周期批量？建议立场：**准实时流式**（每条 `QualityVerdict` 到达即增量累积 + 阈值判定），保证批量异常及时识别。若流式压力大，退化为分钟级窗口批量分析。待与平台确认流式处理能力。

> 🔴 **热点：affected_sn_list 的准确性**——`BatchQualityAnomalyDetected` 携带的 `affected_sn_list[]` 是已确认缺陷的 SN，还是疑似受影响的 SN（包括尚未过点检测的）？建议立场：**已确认缺陷 SN + 疑似范围 SN**（按工序位置推断），返工上下文消费后决定返工 SN 清单（返工上下文 §1.2 热点：`trigger_source=QUALITY` 时由本上下文提供不良 SN 清单）。待与返工上下文确认 SN 清单粒度。

## 4. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **InspectionStandard** | 单个检验标准 | INV-01 standard_id 全局唯一；type 一经确定不可变更；DEPRECATED 后新草稿规则不可引用，已生效规则不受影响（INV-04） |
| **DefectCatalog** | 单个缺陷分类字典 | catalog_id 全局唯一；DEPRECATED 后新草稿规则不可引用；业务判定使用最新字典（已生效规则引用的字典变更不回写规则, INV-04） |
| **QualityGateRule** | 单个规则版本（含判定逻辑与映射） | INV-02 同 rule_id 同时仅一个 ACTIVATED（**跨实例**，由 `RuleActivationService` + 唯一索引保证）；INV-03 ARCHIVED 为终态，之后任何事件均拒收；INV-04 ACTIVATED 后判定逻辑与映射不可变（保证过点记录规则快照完整性）；INV-06 提交审核前置：判定逻辑完整且引用校验满足；INV-07 生效前置：所有 InspectionStandard 非 DEPRECATED；INV-08 生效前置：所有 DefectCatalog 非 DEPRECATED、OverrideAuthorizationPolicy 存在且有效 |
| **QualityVerdict** | 单次业务判定 | INV-09 同一 TestResult 不重复判定（`source_test_result_id` 去重）；判定不可变（一经发布不可修改）；含缺陷分类与业务判定结果 |
| **BatchQualityAnomaly** | 单次批量异常识别 | INV-12 同一 (work_order_id, station_id, defect_code) 异常不重复识别（去重表）；`anomaly_scope=BATCH_SYSTEMIC` 必带 `affected_sn_list[]` + `defect_frequency` + `threshold_breached` |
| **IqcRequest** | 单次来料质检请求 | INV-10 同一触发信号不重复创建 IqcRequest（`source_event_id` 去重）；PENDING → PASSED/FAILED 单向流转；FAILED 必带 `failed_items[]` + `disposition` |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同 rule_id 唯一 ACTIVE 版本）→ `RuleActivationService` 领域服务在生效时查询读模型 + 数据库唯一索引 (rule_id, status=ACTIVATED) 共同保证；新版本生效自动 Deprecate 旧版本（同事务 outbox）
> - **INV-04**（生效后不可变）→ QualityGateRule 聚合根在 ACTIVATED 态拒收任何修改命令；保证过点执行上下文 `CheckpointRecord` 的规则快照可永久回溯
> - **INV-07 / INV-08**（生效引用校验）→ `RuleValidationService` 领域服务在生效时同步查询本上下文 InspectionStandard / DefectCatalog / OverrideAuthorizationPolicy，校验失败拒绝 ActivateQualityGateRule
> - **INV-09**（业务判定幂等）→ `QualityVerdictService` 领域服务 + `source_test_result_id` 去重表，避免 TestResult 事件重复投递导致重复判定
> - **INV-11**（规则版本一致性，变更仅影响变更后首次过点）→ 本上下文发布 `QualityGateRuleActivated` + 过点执行上下文 `QualityGateCache` 投影 + `CheckpointRecord` 规则快照，由过点引擎在首次过点时固化版本保证（跨上下文，主责在过点执行上下文）
> - **INV-13**（事务发件箱至少一次投递，领域总览 §5.4）→ outbox 模式 + 消费端幂等（source_event_id 去重表，过点执行上下文侧保证）
>
> 注：`GateLogic` / `DefectMapping` / `HoldAuthorizationBinding` 是 **QualityGateRule 聚合内实体**，不独立成聚合——它们的生命周期完全依附于所属 QualityGateRule，且仅在 DRAFT 态可变更（INV-04）。`QualityGateCache`（过点执行上下文）/ `InspectionStandardCache`（首件处理上下文）是**读模型/投影**，不承担事务一致性责任。

## 7. 与过点执行上下文的协作边界

本上下文是质量门禁规则的**权威源**与测试结果业务判定的**判定方**，过点执行上下文是**消费方**（规则缓存）与**发布方**（TestResult），通过"本地缓存 + 降级查询"与"事件异步判定"解耦（领域总览 §5.1）：

| 维度 | 本上下文（质量上下文） | 过点执行上下文 |
|------|------------------|--------------|
| 视角 | 质量标准与判定视角（"规则怎么配、TestResult 业务语义是什么、缺陷怎么分"） | 执行视角（"过点瞬间怎么判定、怎么记录"） |
| 存储 | 检验标准、缺陷分类字典、质量门禁规则、业务判定结果、批量异常 | 过点记录、路线推进、工单进度、QualityGateCache（投影） |
| 谁判定 | 规则与标准定义（不执行过点同步门禁）；异步业务判定（QualityVerdict） | 执行同步门禁判定（EvaluateQualityGate，基于缓存规则）；过点放行/拦截决策 |
| 事件性质 | 配置生效事实：QualityGateRuleActivated / Deprecated；业务判定事实：QualityVerdictIssued | 过点执行事实：CheckpointReleased / Blocked；测试结果事实：TestResultStructured |
| 流程所有权 | 标准定义→规则编排→审核→生效发布→业务判定→异常分析 | 扫码→防错→同步门禁判定→决策→记录→推进→累计 |
| 版本一致性 | 发布生效事件为权威源 | 消费生效事件刷新缓存；过点记录锁定规则版本（INV-11） |
| 写入频率 | 写少读多（规则发布）+ 中频（业务判定） | 高频实时（每板毫秒级） |

**事件流向**：
- **本上下文 → 过点执行上下文**：`quality.gate.lifecycle`（`QualityGateRuleActivated` 刷新 QualityGateCache，`QualityGateDeprecated(superseded_by=null)` 移除缓存）
- **过点执行上下文 → 本上下文**：`mes.testresult.structured`（`TestResultStructured` 驱动本上下文异步业务判定 → `QualityVerdictIssued`）
- **降级查询**：过点执行上下文 `QualityGateCache` 未命中时 REST 查询本上下文 `QualityGateRuleReadModel`（与工艺路线降级查询同 SLA, ≤500ms）
- **不互相调用的操作**：本上下文不调用过点执行上下文的"过点/刷新缓存"接口——全部通过事件异步解耦；过点执行上下文不调用本上下文的"创建/生效规则"接口——配置变更归本上下文

> **同步门禁与异步判定的职责切分**（回填过点执行上下文 §2.6 热点 + 首件处理上下文 §2.3 热点）：
> - **过点执行上下文**做**同步门禁判定**（`EvaluateQualityGate`，基于 `QualityGateCache` 缓存规则 + TestResult → `QualityGateEvaluated` PASS/HOLD/BLOCK，驱动过点放行/拦截，在过点主事务内）。这是"流量控制"功能，要求快。
> - **本上下文**做**异步业务判定**（`IssueQualityVerdict`，消费 TestResult + 完整规则 + 缺陷字典 → `QualityVerdictIssued` PASS/HOLD/BLOCK + 缺陷分类，供首件检验汇总/缺陷统计/SPC）。这是"质量评估与记录"功能，要求深。
> - 两者**不要求强一致**：同步门禁为权威（驱动过点决策），异步判定为记录性（不覆盖过点决策）。

> 🔴 **热点：HOLD 人工放行的权限模型**——过点执行上下文 §2.6 热点"HOLD 与人工放行"要求"待与质量上下文确认放行权限模型"。本上下文回填：**HOLD 放行权限归本上下文配置**（`OverrideAuthorizationPolicy`，定义哪些角色可授权放行 HOLD 状态的过点）。过点执行上下文的 `OverrideCheckpoint` 与首件处理上下文的 `OverrideGate` 引用本上下文的 `OverrideAuthorizationPolicy` 校验授权人权限。权限校验是同步 REST 查询（过点放行需实时校验），建议过点执行上下文缓存权限策略。待与过点执行上下文/首件处理上下文确认权限校验方式（缓存 vs 实时查询）。

| 事件性质 | 缺陷字典（REST 只读）；批量异常：BatchQualityAnomalyDetected | 返修执行结果：UnitReworkCompleted / UnitScrappedAfterRework | 返工工单再入：BatchReworkOrderReleased |
| 协作方式 | 提供缺陷分类字典（REST 只读）；发布批量异常信号 | 返修上下文 REST 查询本上下文缺陷字典辅助诊断 | 返工上下文消费本上下文批量异常信号评估生成返工工单 |

**事件流向**：
- **本上下文 → 返修上下文**：（无直接事件；返修上下文 REST 查询本上下文 `DefectCatalogReadModel` 辅助诊断）
- **返修上下文 → 本上下文**：`rework.diagnosis`（`DiagnosisResultRecorded` 诊断结论回流，供本上下文缺陷累积分析）
- **本上下文 → 返工上下文**：`quality.anomaly.batch`（`BatchQualityAnomalyDetected` 批量异常信号，返工上下文消费后评估生成 `BatchReworkOrder`）
- **返工上下文 → 本上下文**：（无反向事件）
- **不互相调用的操作**：本上下文不创建返修/返工任务；返修/返工上下文不定义缺陷分类/不重判质量——各自自治

> **单件返修 vs 批量返工的边界判定**（回填返修上下文 §10、返工上下文 §"与返修上下文协作边界"热点）：
> - 本上下文做**缺陷影响范围分析**（`BatchAnomalyDetectionService`）：
>   - **单件偶发缺陷**（`SINGLE_INCIDENT`）：不发布批量异常信号；通过过点执行上下文的质量类拦截分流（`mes.unit.routed-to-rework`）触发返修上下文单件返修。
>   - **批次性系统性缺陷**（`BATCH_SYSTEMIC`）：发布 `BatchQualityAnomalyDetected` 信号触发返工上下文批量返工工单。
> - 判定逻辑归本上下文，返修/返工上下文只接收已判定的触发信号。

> 🔴 **热点：缺陷影响范围判定的准确性**——单件偶发 vs 批次性系统性的判定阈值与规则归本上下文配置（§2.5 热点）。误判会导致：单件偶发误判为批量异常 → 不必要的批量返工；批次性系统性误判为单件偶发 → 仅单件返修，批量缺陷持续。建议立场：**保守判定**（倾向于批量异常，宁可多返工不可漏检系统性缺陷），并由质量工程师定期复核异常识别记录调优阈值。待与质量工程团队确认判定阈值与复核机制。

> 🔴 **热点：返修诊断回流的消费**——本上下文消费 `rework.diagnosis`（`DiagnosisResultRecorded`）做缺陷累积分析。返修诊断的 `fault_cause` 与本上下文 `DefectCatalog` 的 `defect_code` 如何映射？建议立场：**返修上下文诊断时引用本上下文缺陷分类字典**（REST 查询），`DiagnosisResultRecorded` 携带 `defect_code`（与本上下文字典对齐），本上下文消费后直接累积。待与返修上下文确认诊断结论是否携带标准 defect_code。
