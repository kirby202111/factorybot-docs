# 工单管理上下文（Work Order Management Bounded Context）

> **限界上下文**：工单管理上下文
> **统一语言**：WorkOrder（工单·正常生产工单）, WorkOrderItem（工单明细·产品+目标数量）, WorkOrderBinding（工单绑定·BOM 版本 + 工艺路线版本引用）, WorkOrderStatus（工单状态）, WorkOrderClosure（工单结案·完工判定与关闭）, SourceWorkOrderRef（源工单引用·被返工上下文引用）
> **核心关注**：以"事件风暴"方式刻画**正常生产工单的生命周期管理**——工单的接收、下达、状态流转（下达/进行中/暂停/完工/关闭/取消）、与 BOM 和工艺路线版本的绑定、完工判定与结案。本上下文只对"**一张工单的身份、目标、绑定、状态生命周期**"负责；**不负责**工单执行进度累计（归过点执行上下文 `WorkOrderProgress`，本上下文消费其进度事件做完工判定）、**不负责**工艺路线定义与版本生命周期（归工艺管理上下文，本上下文只读引用路线版本）、**不负责** BOM 与物料主数据（归物料上下文，本上下文只读引用 BOM 版本）、**不负责**批量返工工单（归返工上下文 `BatchReworkOrder`，本上下文的 `WorkOrder` 是其 `source_work_order_id` 的来源）、**不负责**单件返修任务（归返修上下文）、**不负责**在制品位置投影（归在制品追踪上下文）、**不负责**排产调度算法（🔴 待 SME 确认是否纳入本上下文，见 §0 暂缓）。

---

## 0. 设计原则与边界澄清

工单管理上下文是生产执行服务的**工单生命周期子域**，是 PCBA + 整机组装混合型车间所有"正常生产工单"的身份与状态权威源。它接收上游（ERP / 人工创建）下达的工单，绑定 BOM 与工艺路线版本，驱动状态流转，并在过点执行上下文累计的进度达到目标时完工结案。

**上下文内职责**（本上下文负责）：
- **工单接收与建档**——消费上游 ERP 下达指令或人工创建，生成 `WorkOrder`（含产品、目标数量、计划开工/完工时间、产线）
- **工单绑定**——下达前绑定 BOM 版本（引用物料上下文）与工艺路线版本（引用工艺管理上下文），锁定版本快照
- **工单下达**——工单从 CREATED 转 RELEASED，允许过点执行上下文基于该工单过点
- **工单状态流转**——下达 / 开始 / 暂停 / 恢复 / 完工 / 关闭 / 取消的状态机管理
- **工单变更**——数量调整、计划时间调整、产线调整（🔴 工艺路线/BOM 版本变更对在制工单的影响待对齐，见 §2.3）
- **工单完工判定**——消费过点执行上下文 `WorkOrderProgressAccrued`，判定 `good_qty >= target_qty` 后转 COMPLETED
- **工单结案**——COMPLETED 后人工/系统确认结案，转 CLOSED（终态）

**上下文外职责**（本上下文**不**负责）：
- 工单执行进度累计（good/bad/reworked qty） → 过点执行上下文（本上下文消费 `mes.workorder.progress` 做完工判定，不重复累计）
- 工艺路线定义、版本生命周期、生效发布 → 工艺管理上下文（本上下文只读引用 route_id + route_version）
- BOM 定义、物料主数据、替代料 → 物料上下文（本上下文只读引用 bom_id + bom_version）
- 批量返工工单生成、返工工艺路线 → 返工上下文（`BatchReworkOrder` 引用本上下文的 `WorkOrder` 作为 `source_work_order_id`，返工工单是独立实体，不在本上下文管理）
- 单件返修任务、单件再入点判定 → 返修上下文
- 在制品身份、位置投影、齐套状态 → 在制品追踪上下文（本上下文发布工单下达/状态变更事件供其投影齐套）
- 首件检验流程编排 → 首件处理上下文（本上下文发布工单下达事件供其触发首件标识）
- 排产调度算法（APS / 有限产能排程） → 🔴 待 SME 确认归本上下文还是独立排产上下文（见暂缓模块说明）
- 与 ERP 的工单同步协议 → 本上下文的 ERP 工单接收适配层（ACL，端口：`WorkOrderReceptionPort`，负责归一化 ERP 下达指令后由工单聚合消费；不设独立"系统集成上下文"，集成是防腐层性质归各业务上下文自带 ACL，参考过点执行上下文测试数据接入适配层模式）

**为什么不把工单管理并入过点执行上下文**：
1. **生命周期与节奏不同**——过点是毫秒级实时判定，工单是天到周级生命周期（下达/暂停/完工/结案各有独立节奏），混入过点引擎会使其职责膨胀。
2. **聚合边界清晰**——过点聚合是 `CheckpointRecord` / `RoutingProgress` / `WorkOrderProgress`；工单聚合是 `WorkOrder`（含 `WorkOrderItem` / `WorkOrderBinding`）。无共用聚合，强合等于耦合。
3. **流程所有权不同**——过点管"过站瞬间的判定与推进"，工单管"工单身份与状态生命周期 + 完工结案"。两者通过事件契约解耦即可协作。
4. **进度口径分离**——`WorkOrderProgress`（过点执行累计执行进度）与 `WorkOrder.target_qty`（本上下文的目标数量）是两个独立维度，完工判定由本上下文基于两者比对做出，避免过点引擎承担工单生命周期职责。

**为什么不把工单管理并入返工上下文**：
1. **工单性质根本不同**——本上下文的 `WorkOrder` 是**正常生产工单**（ERP 下达，走正常工艺路线）；返工上下文的 `BatchReworkOrder` 是**批量返工工单**（质量异常/ECN 触发，走独立返工工艺路线）。两者触发源、工艺路线、再入机制、终点完全不同。
2. **聚合不同**——`WorkOrder`（含 BOM 绑定 + 正常工艺路线绑定 + 目标数量 + 完工判定）vs `BatchReworkOrder`（含 SN 清单 + 返工路线 + 批量再入点 + 返工进度）。无共用聚合。
3. **引用关系单向**——返工工单引用源工单（`source_work_order_id`），源工单不引用返工工单。合并会反向耦合正常工单到返工流程。
4. **服务前缀不同**——`wo.*`（work order）vs `brework.*`（batch rework），事件名、主题、消费方严格区分，**严禁混用**。

**为什么工单管理归生产执行服务而非制造资源服务**：
1. **数据性质不同**——制造资源服务管**配置性数据**（BOM、工艺路线、质量规则，写少读多）；工单是**事务性数据**（高频下达/状态流转/结案，与生产执行强绑定）。
2. **协作紧密度**——工单与过点执行（进度累计）、在制品追踪（齐套投影）、首件处理（首件触发）、返工（源工单引用）同属生产执行链条，归生产执行服务减少跨服务事务。
3. **领域总览 §1 定位**——生产执行服务是"现场生产活动执行引擎"，工单是执行引擎的输入与目标载体，归生产执行服务符合定位。

## 1. 主语清单与触发入口

工单管理上下文的入口是上下文外触发（ERP 下达 / 人工创建 / 上游变更），统一建模为 `WorkOrder`。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 上下文外触发源

```
┌──────────────────────────────────────────────────────────────────┐
│ 工单入口                                                          │
│                                                                  │
│  【来源1:ERP / 上游计划系统】                                     │
│    生产计划下达工单（产品、数量、计划开工/完工时间、产线）        │
│    → 经本上下文 ERP 接收适配层(ACL)归一化后，生成 WorkOrder      │
│      （🔴 ERP 同步协议与字段映射归本适配层，热点标注）            │
│                                                                  │
│  【来源2:人工创建】                                               │
│    生产计划员通过终端创建工单（试产/补单/紧急单）                 │
│    → 本上下文接收指令生成 WorkOrder                              │
│                                                                  │
│  【来源3:过点执行上下文 (mes.*)】                                  │
│    WorkOrderProgressAccrued(good_qty >= target_qty)              │
│    → 本上下文消费进度事件做完工判定，转 COMPLETED                │
│                                                                  │
│  【来源4:工艺管理上下文 (process.*，REST 只读)】                  │
│    工艺路线版本（非事件订阅，下达时 REST 查询并锁定版本）         │
│    → 工单绑定时引用 route_id + route_version                     │
│                                                                  │
│  【来源5:物料上下文 (REST 只读)】                                  │
│    BOM 版本（非事件订阅，下达时 REST 查询并锁定版本）             │
│    → 工单绑定时引用 bom_id + bom_version                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 聚合根主语清单

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **WorkOrder**（工单） | ERP 下达工单、人工创建工单、试产工单 | 唯一身份；绑定 BOM + 工艺路线版本；状态单向流转；目标数量驱动完工 | ERP / 人工创建 |
| **WorkOrderItem**（工单明细·可选多产品） | 单产品工单 / 多产品套件工单 | 产品 + 目标数量 + 单位；🔴 多产品工单是否支持待 SME 确认 | 由 WorkOrder 触发 |
| **WorkOrderBinding**（工单绑定·值对象） | BOM 版本 + 工艺路线版本快照 | 下达前锁定版本；变更仅影响未过点在制品（参考过点执行 INV-09） | 由 WorkOrder 触发 |
| **WorkOrderClosure**（工单结案） | 完工判定 + 结案确认 | 消费过点执行进度判定完工；COMPLETED 后人工/系统结案 | 过点执行进度事件 / 人工 |

**工单模型**：

```
WorkOrder
├── work_order_id        : 工单唯一标识
├── work_order_no        : 工单编号（业务可读，🔴 编号规则待对齐）
├── product_id           : 产品 ID（引用物料上下文产品主数据）
├── target_qty           : 目标数量
├── unit                 : 单位（PCS / SET）
├── bom_binding          : WorkOrderBinding.bom（bom_id, bom_version, 锁定快照）
├── route_binding        : WorkOrderBinding.route（route_id, route_version, 锁定快照）
├── planned_start_at     : 计划开工时间
├── planned_finish_at    : 计划完工时间
├── production_line_id   : 产线 ID（🔴 排程归属待确认）
├── source               : ERP / MANUAL
├── source_event_id      : 来源事件 ID（幂等去重，若来自 ERP 事件）
├── status               : CREATED | RELEASED | IN_PROGRESS | PAUSED | COMPLETED | CLOSED | CANCELLED
├── closure              : WorkOrderClosure（完工判定与结案信息）
├── created_at           : 创建时间
└── released_at          : 下达时间（RELEASED 时填）
```

> 🔴 **热点：工单编号规则**——`work_order_no` 是业务可读编号（如 `WO-2026-06-001`），还是直接用 `work_order_id`（UUID）？建议：**双字段**——`work_order_id` 内部 UUID，`work_order_no` 业务可读编号由编号生成器产出。编号规则（是否含产线/产品/日期前缀）待与 SME 确认。

> 🔴 **热点：多产品工单是否支持**——PCBA 通常一单一产品，整机组装可能一单多产品（套件）。建议：**单产品工单为主，多产品作为热点暂缓**——`WorkOrderItem` 预留为值对象集合，但当前阶段一单一产品，多产品待 SME 确认后展开。

> 🔴 **热点：排程归属**——产线分配、开工时间排程是归本上下文、独立排产上下文、还是上游 ERP？建议：**当前阶段本上下文只接收已排程的工单**（产线 + 计划时间由 ERP/排产系统给定），本上下文不做有限产能排程算法。若 SME 要求本上下文做产线分配，后续扩展排程子流程。待 SME 确认排程归属。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 工单接收与下达（主链第一段）

```
【外部系统:ERP / 人工创建】
        -- 下达工单指令（产品、数量、计划时间、产线）
                |
                v
【策略:消费下达指令评估生成工单】（幂等：source_event_id 去重, INV-02）
                |
                v
【命令:生成工单 CreateWorkOrder】--> 【聚合:WorkOrder】
        |
        ┌──── 读取辅助 ────┐
        v                   v
【外部系统:物料上下文】  【外部系统:工艺管理上下文】
（BOM 版本, REST 只读）  （工艺路线版本, REST 只读）
        |                   |
        └───────┬───────────┘
                v
        【事件:工单已创建 WorkOrderCreated】
         (work_order_id, work_order_no, product_id, target_qty, bom_binding, route_binding, planned_start_at, source)
                |
                v
        发布到 wo.order.created
                |
                v
【策略:工单创建后下达】（🔴 是否创建即下达，还是分两步，见热点）
                |
                v
【命令:下达工单 ReleaseWorkOrder】--> 【聚合:WorkOrder】
                |
                v
        【事件:工单已下达 WorkOrderReleased】
         (work_order_id, route_binding.route_version, production_line_id, released_at)
                |
                v
        发布到 wo.order.released
                |
                v
【外部系统:过点执行上下文】订阅
（工单已下达，允许基于该工单过点；缓存工单状态供过点校验）
【外部系统:在制品追踪上下文】订阅
（工单已下达，初始化齐套状态投影）
【外部系统:首件处理上下文】订阅
（工单已下达，触发首件标识判定：换线/换料/ECN）
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| WorkOrderCreated | 消费下达指令生成工单 | work_order_id, work_order_no, product_id, target_qty, bom_binding, route_binding, planned_start_at, source | 自动下达（INV-02 幂等去重） |
| WorkOrderReleased | 工单下达，允许过点 | work_order_id, route_version, production_line_id, released_at | 通知过点执行 / WIP / 首件处理 |

> 🔴 **热点：创建即下达 vs 分两步**——ERP 下达的工单是否创建后立即 RELEASED，还是 CREATED 后需人工确认下达？建议：**ERP 下达即 RELEASED**（source=ERP 自动下达）；**人工创建默认 CREATED**，需人工 `ReleaseWorkOrder` 确认下达（防误建）。待与 SME 确认是否需要"待下达"中间态。

> 🔴 **热点：BOM/工艺路线版本锁定时机**——BOM 与工艺路线版本是在 `CreateWorkOrder` 时锁定（CREATED 即固化），还是在 `ReleaseWorkOrder` 时锁定（下达时固化）？建议：**CreateWorkOrder 时锁定**（工单创建即固化版本快照，避免下达前版本变更导致绑定漂移），与过点执行上下文 INV-09"过点记录锁定 routeVersion"模式一致。待与工艺管理上下文 / 物料上下文确认版本查询接口与锁定机制。

> 🔴 **热点：工单下达与首件触发的时序**——`WorkOrderReleased` 发布后，首件处理上下文触发首件标识判定。首件未放行前，过点执行上下文是否拦截批量过点？建议：**首件门禁由过点执行上下文执行**（消费 `fai.article.blocked` 拦截，消费 `fai.article.released` 放行），本上下文只发布 `WorkOrderReleased` 通知首件处理上下文触发判定，不直接参与门禁。待与首件处理上下文确认触发契约。

### 2.2 状态流转

```
[CREATED] ──ReleaseWorkOrder──▶ [RELEASED] ──StartWorkOrder──▶ [IN_PROGRESS] ──CompleteWorkOrder──▶ [COMPLETED] ──CloseWorkOrder──▶ [CLOSED]
                                    │                                │                                (终态—完工判定通过, INV-04)
                                    │                                │
                                    │                                └──PauseWorkOrder──▶ [PAUSED] ──ResumeWorkOrder──▶ [IN_PROGRESS]
                                    │
                                    └──CancelWorkOrder──▶ [CANCELLED]
                                                            (终态—取消, INV-03)

[RELEASED] ──CancelWorkOrder──▶ [CANCELLED]（下达后无在制可取消）
[IN_PROGRESS] ──CancelWorkOrder──▶ [CANCELLED]（🔴 已有过点在制是否允许取消，见热点）
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| CREATED → RELEASED | ReleaseWorkOrder | 当前 CREATED；BOM + 工艺路线版本已绑定（INV-01） | 系统（ERP 来源自动）/ 计划员（人工来源确认） |
| RELEASED → IN_PROGRESS | StartWorkOrder | 当前 RELEASED；首次过点放行（消费过点执行 `CheckpointReleased`，🔴 是否需要此状态见热点） | 系统（首次过点自动转态） |
| IN_PROGRESS → PAUSED | PauseWorkOrder | 当前 IN_PROGRESS；暂停原因记录 | 生产主管 / 系统（设备故障/质量异常联动） |
| PAUSED → IN_PROGRESS | ResumeWorkOrder | 当前 PAUSED；暂停原因已解除 | 生产主管 |
| IN_PROGRESS → COMPLETED | CompleteWorkOrder | 当前 IN_PROGRESS；`good_qty >= target_qty`（INV-04，消费过点执行进度） | 系统（进度达标自动转态） |
| COMPLETED → CLOSED | CloseWorkOrder | 当前 COMPLETED；结案确认（人工/系统） | 计划员 / 系统（超时自动结案） |
| CREATED/RELEASED → CANCELLED | CancelWorkOrder | 当前 CREATED 或 RELEASED；无在制过点（INV-05） | 计划员 |
| IN_PROGRESS → CANCELLED | CancelWorkOrder | 当前 IN_PROGRESS；🔴 已有过点在制的取消策略见热点 | 计划员（需授权） |
| → CLOSED / CANCELLED 之后 | （任何） | 终态，拒收（INV-03） | — |

> **可逆边说明**：仅 `PauseWorkOrder` / `ResumeWorkOrder`（暂停/恢复）允许回退，其余状态单向流转（INV-06）。

> 🔴 **热点：是否需要 IN_PROGRESS 状态**——工单下达后首次过点放行时是否自动转 IN_PROGRESS？还是 RELEASED 即视为进行中？建议：**保留 IN_PROGRESS**——区分"已下达待生产"与"正在生产"，便于排产看板与在制统计。首次过点放行自动转态，无需人工干预。待与 SME 确认是否需要此中间态。

> 🔴 **热点：IN_PROGRESS 工单的取消策略**——已有过点在制的工单是否允许取消？建议：**允许取消但需授权 + 标记在制 SN 处理方式**（继续完工 / 强制报废 / 转其他工单）。取消后剩余在制 SN 由过点执行上下文标记不再推进。待与 SME 确认取消策略与在制处理。

> 🔴 **热点：暂停后过点执行如何响应**——工单 PAUSED 后，过点执行上下文是否拦截该工单的后续过点？建议：**拦截**——本上下文发布 `WorkOrderPaused` 事件，过点执行上下文订阅后缓存工单状态，过点校验时 `work_order_status != IN_PROGRESS` 则 `CheckpointBlocked(blocking_reason=WorkOrderPaused)`。待与过点执行上下文确认工单状态缓存机制。

### 2.3 工单变更

工单变更分为**可变更字段**（数量、计划时间、产线）与**版本绑定变更**（BOM/工艺路线版本）两类，后者对在制工单的影响需谨慎处理。

```
[RELEASED / IN_PROGRESS] 状态下
        |
        v
【参与者:计划员】 --【命令:变更工单 ChangeWorkOrder】--> 【聚合:WorkOrder】
        |
        ├── 可变更字段（target_qty / planned_start_at / planned_finish_at / production_line_id）
        |       |
        |       v
        |   【事件:工单已变更 WorkOrderChanged】
        |    (work_order_id, changed_fields[], change_reason, changed_at)
        |       |
        |       v
        |   发布到 wo.order.changed
        |       |
        |       v
        |   【外部系统:过点执行上下文】订阅
        |   （若 target_qty 变更，影响完工判定阈值；若产线变更，🔴 在制迁移策略见热点）
        |
        └── 版本绑定变更（bom_binding / route_binding）
                |
                v
            【事件:工单版本绑定已变更 WorkOrderBindingChanged】
             (work_order_id, old_route_version, new_route_version, effective_scope)
                |
                v
            发布到 wo.order.binding-changed
                |
                v
            【外部系统:过点执行上下文】订阅
            （effective_scope=NEW_ONLY: 仅影响未过点在制品，已过点按原 routeVersion 推进, INV-07）
            （effective_scope=ALL: 🔴 强制全部在制切新版本，需授权，见热点）
```

> 🔴 **热点：版本绑定变更的影响范围**——BOM/工艺路线版本变更后，已过点的在制品是否切换到新版本？建议：**默认 NEW_ONLY**（仅影响变更后首次过点的在制品，已过点按 `CheckpointRecord.route_version` 锁定版本继续推进，参考过点执行 INV-09）；**ALL 模式需授权**（如 ECN 强制召回在制切换，需记录授权人 + 原因 + 影响的 SN 清单）。待与工艺管理上下文 / ECN 模块确认版本变更触发机制。

> 🔴 **热点：target_qty 变更的完工判定影响**——工单进行中调整目标数量（增补 / 削减），完工判定阈值是否实时更新？建议：**实时更新**——`WorkOrderChanged(target_qty)` 发布后，过点执行上下文与本上下文的完工判定均用最新 `target_qty`。若削减数量低于已完工 `good_qty`，自动触发 COMPLETED。待与 SME 确认数量变更的审批流。

> 🔴 **热点：产线变更的在制迁移**——工单进行中变更产线，已过点的在制品是否物理迁移？建议：**产线变更仅影响未过点在制品**，已过点在制品按原产线工艺路线推进至完工；新产线需重新齐套校验。待与在制品追踪上下文确认产线变更的齐套重算机制。

### 2.4 工单完工判定与结案

工单完工判定基于过点执行上下文累计的 `WorkOrderProgress.good_qty` 与本上下文的 `WorkOrder.target_qty` 比对。结案是完工后的确认动作。

```
【外部系统:过点执行上下文】
        -- 发布 WorkOrderProgressAccrued(work_order_id, good_qty, bad_qty, reworked_qty)
                |
                v
【策略:消费进度事件做完工判定】（幂等：work_order_id + accrued_at 去重, INV-08）
                |
                v
【命令:判定完工 CompleteWorkOrder】--> 【聚合:WorkOrder】
        |
        v
比较 good_qty >= target_qty ?
        |
        ├── 否 --> 保持 IN_PROGRESS，等待后续进度
        |
        └── 是 --> 【事件:工单已完工 WorkOrderCompleted】
                    (work_order_id, completed_qty, good_qty, target_qty, completed_at)
                    |
                    v
                发布到 wo.order.completed
                    |
                    v
                【策略:完工后自动结案 or 人工结案】（🔴 见热点）
                    |
                    v
                【命令:结案工单 CloseWorkOrder】--> 【聚合:WorkOrder】
                    |
                    v
                【事件:工单已结案 WorkOrderClosed】
                 (work_order_id, closure_reason, closed_at)
                    |
                    v
                发布到 wo.order.closed
                    |
                    v
                【外部系统:数据应用上下文】订阅
                （工单结案统计、良率分析、成本归集）
                【外部系统:在制品追踪上下文】订阅
                （工单结案，清理在制投影）
```

> 🔴 **热点：完工判定口径**——完工判定是 `good_qty >= target_qty`，还是需考虑 `reworked_qty` 与在制 `pending_qty`？建议：**`good_qty >= target_qty`**——`good_qty` 仅由过点放行累计（过点执行 INV-08），返修再入后放行计入 `reworked_qty` 不重复计入 `good_qty`，故完工判定基于 `good_qty` 即可。若允许超产，需 `allow_overcomplete` 标志。待与 SME 确认是否允许超产及超产上限。

> 🔴 **热点：完工后是否自动结案**——COMPLETED 后是否自动转 CLOSED，还是需人工确认结案？建议：**默认人工结案**——COMPLETED 后等待计划员确认结案（核对良率、损耗、成本归集），超时（如 7 天）自动结案。避免自动结案掩盖异常工单。待与 SME 确认结案时效与是否需人工确认。

> 🔴 **热点：未完工强制结案**——工单未达 `target_qty` 但需结案（如客户取消部分数量、剩余数量不生产），是否允许 IN_PROGRESS → CLOSED？建议：**不允许直接 CLOSED**，需先 `CancelWorkOrder`（标记剩余数量取消）或调整 `target_qty` 至当前 `good_qty` 后再 COMPLETED → CLOSED。保证状态机单向流转的严格性。待与 SME 确认未完工结案策略。

### 2.5 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动完工判定与状态联动，**不重发**这些事件——必须包装成本上下文语义的事件（INV-09）。BOM 与工艺路线版本通过 REST 查询，不属于事件订阅范畴。

| WorkOrderProgressAccrued | 过点执行上下文 (`mes.workorder.progress`) | CompleteWorkOrder | IN_PROGRESS → COMPLETED（good_qty >= target_qty, INV-04） | work_order_id + accrued_at 去重（INV-08） |
| CheckpointReleased（首次过点） | 过点执行上下文 (`mes.checkpoint.lifecycle`) | StartWorkOrder | RELEASED → IN_PROGRESS（首次过点自动转态, INV-10） | work_order_id 去重（首次过点转态仅一次） |

| 工单下达指令 | 本上下文 ERP 工单接收适配层（ACL，归一化 ERP 指令，🔴 协议待对齐） | CreateWorkOrder | 新建 WorkOrder（CREATED → RELEASED） | source_event_id 去重表（INV-02） |

```
【外部系统:过点执行上下文】
        -- 发布 WorkOrderProgressAccrued(work_order_id, good_qty, bad_qty, reworked_qty, accrued_at)
                |
                v
【策略:消费进度判定完工】（幂等：work_order_id + accrued_at 去重, INV-08）
                |

                |
                v
        读取 WorkOrder.target_qty 与 WorkOrderProgress.good_qty 比对
                |
                ├── good_qty < target_qty --> 保持 IN_PROGRESS

```

> 🔴 **热点：返工工单引用对源工单状态的影响**——返工上下文创建 `BatchReworkOrder` 引用 `source_work_order_id` 后，源工单是否需标记"有返工在制"？建议：**不标记状态**——源工单状态机不引入"返工中"态，返工是独立工单流；源工单的 `good_qty` 已由过点执行累计（返工再入后放行计入 `reworked_qty` 不重复计入 good），完工判定不受返工影响。本上下文只做只读投影（统计源工单关联的返工工单数）供查询。待与返工上下文确认是否需反向通知。

> 🔴 **热点：进度事件的消费频率**——`WorkOrderProgressAccrued` 每次过点放行都发布，高频消费可能压力。建议：**幂等消费 + 批量判定**——本上下文维护 `WorkOrderProgressProjection` 读模型（最新 good_qty），每次事件刷新投影并判定完工，幂等去重保证重复消费无副作用。待与平台运维确认消费吞吐与投影刷新策略。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **WorkOrder** | 单个工单 | INV-01 下达前必须绑定 BOM 版本 + 工艺路线版本（bom_binding / route_binding 非空）；INV-03 CLOSED / CANCELLED 为终态，之后任何事件均拒收；INV-06 状态单向流转，仅 PauseWorkOrder / ResumeWorkOrder 允许回退 |
| **WorkOrderBinding**（值对象） | 工单版本绑定快照 | INV-07 版本绑定在 CreateWorkOrder 时锁定（bom_version / route_version 快照）；版本变更默认 NEW_ONLY（仅影响未过点在制品，已过点按 CheckpointRecord.route_version 推进） |
| **WorkOrderClosure** | 工单结案判定 | INV-04 完工判定基于 `good_qty >= target_qty`（good_qty 来自过点执行上下文，本上下文不重复累计）；INV-11 COMPLETED 后才能 CLOSED，不允许跳过 COMPLETED 直接 CLOSED |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同一 ERP 下达指令不重复创建工单）→ `WorkOrderService` 领域服务 + source_event_id 事件去重表
> - **INV-05**（CANCELLED 前需无在制过点或授权处理在制）→ `WorkOrderService` 领域服务 + 过点执行上下文查询在制 SN + 授权记录
> - **INV-08**（进度事件幂等消费）→ `WorkOrderProgressProjection` 读模型 + (work_order_id, accrued_at) 去重表
> - **INV-09**（上下文外事件只消费不重发）→ 主题前缀隔离（本上下文只发 `wo.*`，订阅 `mes.*` / `brework.*`）+ 事件命名约定
> - **INV-10**（首次过点自动转态 RELEASED → IN_PROGRESS）→ `WorkOrderStatusProjection` 读模型 + work_order_id 去重（首次过点转态仅一次）
> - **INV-12**（工单状态对过点的约束）→ 过点执行上下文订阅 `wo.order.*` 维护 `WorkOrderStatusCache`，过点校验时 `work_order_status ∈ {RELEASED, IN_PROGRESS}` 才允许过点（PAUSED / COMPLETED / CLOSED / CANCELLED 拦截）

## 与过点执行上下文的协作边界

本上下文与过点执行上下文是"工单生命周期"与"过点执行进度"的分工关系，通过事件契约解耦。这是本上下文最紧密的协作方，必须对齐避免事件混淆。

| 维度 | 本上下文（工单管理上下文） | 过点执行上下文 |
|------|------------------|--------------|
| 视角 | 工单生命周期视角（"工单怎么下、状态怎么流转、何时完工结案"） | 过点执行进度视角（"过点瞬间怎么判定、工单进度怎么累计"） |
| 存储 | 工单 / 工单绑定 / 工单结案 / 工单状态投影 | 过点记录 / 路线推进 / 工单进度（WorkOrderProgress）/ 本地缓存 |
| 谁判定 | 工单状态流转（下达/暂停/完工/结案）、完工判定（good_qty >= target_qty） | 过点放行/拦截决策、工单执行进度累计（good/bad/reworked qty） |
| 事件性质 | 工单生命周期事实：WorkOrderReleased / Paused / Completed / Closed | 过点执行事实：CheckpointReleased / WorkOrderProgressAccrued |
| 流程所有权 | 接收→建档→绑定→下达→状态流转→完工判定→结案 | 扫码→防错→决策→记录→推进→累计进度 |
| 进度口径 | target_qty（目标数量，静态） | good_qty / bad_qty / reworked_qty（执行进度，动态累计） |

**事件流向**：

```
本上下文 ──wo.order.released──▶ 过点执行上下文（缓存工单状态，允许过点, INV-12）
本上下文 ──wo.order.paused──▶ 过点执行上下文（缓存工单状态为 PAUSED，过点拦截）
本上下文 ──wo.order.resumed──▶ 过点执行上下文（恢复过点放行）
本上下文 ──wo.order.changed──▶ 过点执行上下文（target_qty 变更影响完工判定阈值）
本上下文 ──wo.order.binding-changed──▶ 过点执行上下文（版本变更影响未过点在制品, INV-07）
本上下文 ──wo.order.cancelled──▶ 过点执行上下文（缓存工单状态为 CANCELLED，过点拦截）
过点执行上下文 ──mes.workorder.progress──▶ 本上下文（进度事件驱动完工判定, INV-04/INV-08）
过点执行上下文 ──mes.checkpoint.lifecycle──▶ 本上下文（首次过点驱动 RELEASED→IN_PROGRESS, INV-10）
```

**职责切分要点**：
- **本上下文管**：工单身份 + 目标数量 + BOM/工艺路线版本绑定 + 状态生命周期 + 完工判定 + 结案
- **过点执行上下文管**：工单执行进度累计（good/bad/reworked qty） + 过点时工单状态校验（缓存工单状态）
- **不互相调用的操作**：本上下文不直接修改 `WorkOrderProgress`（归过点执行）；过点执行上下文不直接修改 `WorkOrder.status`（归本上下文）。全部通过事件异步解耦。
- **完工判定的所有权**：本上下文基于过点执行累计的 `good_qty` 与本上下文的 `target_qty` 比对做完工判定，过点执行上下文不判定完工（它只累计进度）。

> 🔴 **热点：工单状态缓存的刷新机制**——过点执行上下文订阅 `wo.order.*` 维护 `WorkOrderStatusCache`，缓存未命中时如何降级？建议：**降级 REST 查询本上下文**（≤500ms），与工艺路线缓存降级模式一致（领域总览 §5.1）。待与过点执行上下文确认工单状态缓存的降级策略与一致性容忍度。

## 与返工上下文的协作边界（关键区分表）

本上下文（工单管理 `wo`）与返工上下文（返工 `brework`）**名称都含"工单"但完全不同**，必须严格区分，避免事件名、主题前缀、聚合、消费方混淆。这张表是防止"WorkOrder 与 BatchReworkOrder 混淆"的核心边界。

| 维度 | 本上下文（工单管理上下文 wo） | 返工上下文（brework） |
|------|------------------|--------------|
| **工单性质** | **正常生产工单**（ERP 下达，走正常工艺路线） | **批量返工工单**（质量异常/ECN/人工决策触发，走独立返工工艺路线） |
| **服务前缀** | `wo.*` | `brework.*` |
| **聚合根** | WorkOrder（含 WorkOrderItem / WorkOrderBinding / WorkOrderClosure） | BatchReworkOrder（含 ReworkRoute / ReentryPoint / ReworkProgress） |
| **触发机制** | ERP 下达 / 人工创建 | 批量质量异常 / 工程变更 / 人工决策 |
| **流程起点** | 接收下达指令生成工单 | 消费批量触发信号生成返工工单 |
| **工艺路线** | 正常工艺路线（从工艺管理上下文引用，绑定 route_version） | 返工工艺路线（从工艺管理上下文引用，与正常路线不同，绑定 rework_route_version） |
| **再入机制** | 无（正常工单不涉及再入） | 批量再入点（返工工单定义，整批统一再入） |
| **流程终点** | 完工（good_qty >= target_qty）+ 结案 | 整批从再入点回主线或批量报废 |
| **进度口径** | target_qty（目标数量，静态） + 消费过点执行 good_qty 判完工 | ReworkProgress（返工进度，按 SN 维度累计） |
| **所属服务** | 生产执行服务 | 生产执行服务 |

**关键澄清**：
- 两者**单向引用**——返工上下文的 `BatchReworkOrder.source_work_order_id` 引用本上下文的 `WorkOrder`；本上下文不引用返工工单。
- **事件名严格区分**：`WorkOrderReleased` / `WorkOrderCompleted`（本上下文，正常工单级）vs `BatchReworkOrderReleased` / `BatchReworkOrderCompleted`（返工上下文，批量返工级）。**严禁混用或互相引用**。
- **无直接事件往来**——本上下文不订阅 `brework.*`，返工上下文不订阅 `wo.*`。返工上下文只读引用 `source_work_order_id`（REST 查询本上下文获取源工单信息），本上下文可选投影返工引用（🔴 是否需反向通知见 §2.5 热点）。
- **进度独立性**——本上下文的 `WorkOrder.target_qty` 与返工上下文的 `ReworkProgress` 是两个独立进度，不混淆。源工单的 `good_qty` 已由过点执行累计（返工再入后放行计入 `reworked_qty` 不重复计入 good），完工判定不受返工影响。

## 与工艺管理上下文、物料上下文的协作边界

本上下文是 BOM 与工艺路线版本配置数据的**消费方**，通过"REST 查询 + 版本锁定快照"解耦：

| 维度 | 本上下文（工单管理上下文） | 工艺管理上下文 | 物料上下文 |
|------|------------------------|---------------------------|---|
| 视角 | 工单视角（"工单绑哪个 BOM、哪个工艺路线版本"） | 配置视角（"工艺路线怎么定义、版本怎么生效"） | 配置视角（"BOM 怎么定义、物料主数据怎么管"） |
| 存储 | 工单 + 版本绑定快照（bom_version / route_version 锁定） | 工序定义、工艺路线、版本、参数模板 | BOM、产品主数据、替代料、供应商 |
| 谁判定 | 绑定时锁定版本快照（CreateWorkOrder 时固化） | 发布版本生效事件为权威源 | 维护 BOM 版本为权威源 |
| 事件性质 | 工单生命周期事实 | 配置生效事实：ProcessRouteActivated | BOM 变更事实（🔴 事件待对齐） |
| 流程所有权 | 接收→绑定→下达→流转→完工→结案 | 工序定义→路线编排→版本生效发布 | BOM 定义→版本管理→替代料维护 |
| 版本一致性 | 创建时锁定版本快照；变更默认 NEW_ONLY（INV-07） | 发布生效事件为权威源 | 维护 BOM 版本为权威源 |

**事件流向**：
- **工艺管理/物料上下文 → 本上下文**：REST 查询（创建工单时绑定版本，非事件订阅）
- **本上下文 → 工艺管理/物料上下文**：（无直接事件，本上下文只读引用版本）
- **版本锁定机制**：CreateWorkOrder 时 REST 查询工艺管理上下文（route_id + 当前生效 route_version）与物料上下文（bom_id + 当前生效 bom_version），锁定为 `WorkOrderBinding` 快照。后续工艺路线/BOM 版本变更不影响已创建工单的绑定（INV-07），除非主动 `ChangeWorkOrder` 触发 `WorkOrderBindingChanged`。

> 🔴 **热点：版本查询的时机与缓存**——CreateWorkOrder 时 REST 查询 BOM/工艺路线版本，是否需本地缓存？建议：**不缓存**——工单创建是低频操作（相比过点毫秒级），实时 REST 查询保证版本准确性，避免缓存漂移。若性能压力，可考虑短 TTL 缓存。待与工艺管理上下文 / 物料上下文确认版本查询接口性能。

> 🔴 **热点：BOM/工艺路线版本变更的主动通知**——工艺管理上下文发布 `ProcessRouteActivated` 后，是否需主动通知本上下文"有新版本可用"？建议：**不主动通知**——本上下文只在 CreateWorkOrder 时查询当前生效版本锁定，已创建工单的绑定不受新版本影响（INV-07）。若需 ECN 强制切换在制工单版本，由 ECN 模块主动触发 `ChangeWorkOrder(effective_scope=ALL)`，本上下文不被动响应版本变更。待与 ECN 模块确认触发机制。
