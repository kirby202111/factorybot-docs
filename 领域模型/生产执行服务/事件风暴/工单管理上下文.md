# 工单管理上下文（Work Order Management Bounded Context）

> **限界上下文**：工单管理上下文
> **统一语言**：WorkOrder（工单·正常生产工单）, WorkOrderItem（工单明细·当前单产品，预留多明细扩展）, WorkOrderBinding（工单绑定·BOM 版本 + 工艺路线版本引用）, WorkOrderStatus（工单状态）, WorkOrderClosure（工单结案·完工判定与关闭）, MesSerialNo（MES 工单流水号）, ProductSerialNo（产品流水号·开工前按订单数量生成并在首工序绑定产品）, SourceWorkOrderRef（源工单引用·被返工上下文引用）
> **核心关注**：以"事件风暴"方式刻画**正常生产工单的生命周期管理**——工单的 SAP 接收、计划员审核、MES 工单流水号生成、下达、状态流转（下达/进行中/暂停/完工/关闭/取消）、与 BOM 和工艺路线版本的绑定、完工判定与结案。本上下文只对"**一张工单的身份、目标、绑定、状态生命周期**"负责；**不负责**工单创建源头（全部来自 SAP，MES 不允许创建工单）、**不负责**工单执行进度累计（归过点执行上下文 `WorkOrderProgress`，本上下文消费其进度事件做完工判定）、**不负责**工艺路线定义与版本生命周期（归工艺管理上下文，本上下文只读引用路线版本）、**不负责** BOM 与物料主数据（归物料上下文，本上下文只读引用 BOM 版本）、**不负责**批量返工工单（归返工上下文 `BatchReworkOrder`，本上下文的 `WorkOrder` 是其 `source_work_order_id` 的来源）、**不负责**单件返修任务（归返修上下文）、**不负责**在制品位置投影（归在制品追踪上下文）、**不负责**排产调度算法（🔴 待 SME 确认是否纳入本上下文，见 §0 暂缓）。

## 0. 设计原则与边界澄清

工单管理上下文是生产执行服务的**工单生命周期子域**，是 PCBA + 整机组装混合型车间所有"正常生产工单"的身份与状态权威源。它只接收 SAP 下发的工单并建档，MES 内部**无法也不允许创建工单**；SAP 传入后工单处于 CREATED，由计划员审核，审核通过后生成该工单对应的 MES 工单流水号（当前一张 SAP 工单对应一个工单流水号，模型保留一对多扩展空间）并下达为 RELEASED，之后才允许车间生产。

**上下文内职责**（本上下文负责）：
- **工单接收与建档**——消费 SAP 下发的工单生成 `WorkOrder`（含产品、目标数量、计划开工/完工时间、产线），初始状态为 CREATED；MES 不提供人工创建工单能力
- **工单审核与 MES 工单流水号生成**——计划员审核 SAP 工单，审核通过后调用产品流水号前缀解析工具，根据生产产品的物料类型、工单类型和所属年份确定流水号前缀，并在相同前缀下递增生成 MES 工单流水号，作为下达前置条件
- **产品流水号发放边界**——开工前计划员按订单数量创建产品流水号，工艺路线第一步将产品流水号绑定到对应产品；该机制限定可投产数量，不存在超产场景
- **工单绑定**——下达前绑定 BOM 版本（引用物料上下文）与工艺路线版本（引用工艺管理上下文），锁定版本快照
- **工单下达**——工单从 CREATED 转 RELEASED，允许过点执行上下文基于该工单过点
- **工单状态流转**——下达 / 开始 / 暂停 / 恢复 / 完工 / 关闭 / 取消的状态机管理
- **工单变更**——数量调整（仅 RELEASED 前允许，IN_PROGRESS 后禁止调整 target_qty）、计划时间调整、产线调整；BOM/工艺路线版本在下达时锁定，工单下达后不允许变更，若已在制后需修改产品 BOM 或工艺路线，则走返工流程
- **工单完工判定**——消费过点执行上下文 `WorkOrderProgressAccrued`，判定 `good_qty >= target_qty` 后转 COMPLETED
- **工单结案**——完工产品数量达到 `target_qty` 后自动结案，转 CLOSED（终态）；这里的“完工产品”不是单纯最后一道工序过点完成，而是最后一道工序过点完成且无未关闭质量单、返修单等关联异常的汇总状态

**上下文外职责**（本上下文**不**负责）：
- 工单执行进度累计（good/bad/reworked qty） → 过点执行上下文（本上下文消费 `mes.workorder.progress` 做完工判定，不重复累计）
- 工艺路线定义、版本生命周期、生效发布 → 工艺管理上下文（本上下文只读引用 route_id + route_version）
- BOM 定义、物料主数据、替代料 → 物料上下文（本上下文只读引用 bom_id + bom_version）
- 批量返工工单生成、返工工艺路线 → 返工上下文（`BatchReworkOrder` 引用本上下文的 `WorkOrder` 作为 `source_work_order_id`；本上下文仅保留源工单关联返工的只读投影，不改变源工单状态）
- 单件返修任务、单件再入点判定 → 返修上下文
- 在制品身份、位置投影、齐套状态 → 在制品追踪上下文（本上下文发布工单下达/状态变更事件供其投影齐套）
- 首件检验流程编排 → 首件处理上下文（本上下文发布工单下达事件供其触发首件标识）
- 排产调度算法（APS / 有限产能排程） → 当前阶段不做；SAP 下发的产线、计划开工/完工时间仅作为参考计划信息，MES 只在审核下达时做基础校验，后续保留扩展排程子流程（见暂缓模块说明）
- 与 SAP 的工单同步协议 → SAP / 中间件通过消息推送工单下发事件；本上下文 SAP 工单接收适配层（ACL，端口：`WorkOrderReceptionPort`）消费消息并按 `source_event_id` 幂等归一化后交由工单聚合建档

**为什么不把工单管理并入过点执行上下文**：
1. **生命周期与节奏不同**——过点是毫秒级实时判定，工单是天到周级生命周期（下达/暂停/完工/结案各有独立节奏），混入过点引擎会使其职责膨胀。
2. **聚合边界清晰**——过点聚合是 `CheckpointRecord` / `RoutingProgress` / `WorkOrderProgress`；工单聚合是 `WorkOrder`（含 `WorkOrderItem` / `WorkOrderBinding`）。无共用聚合，强合等于耦合。
3. **流程所有权不同**——过点管"过站瞬间的判定与推进"，工单管"工单身份与状态生命周期 + 完工结案"。两者通过事件契约解耦即可协作。
4. **进度口径分离**——`WorkOrderProgress`（过点执行累计执行进度）与 `WorkOrder.target_qty`（本上下文的目标数量）是两个独立维度，完工判定由本上下文基于两者比对做出，避免过点引擎承担工单生命周期职责。

**为什么不把工单管理并入返工上下文**：
1. **工单性质根本不同**——本上下文的 `WorkOrder` 是**正常生产工单**（SAP 下发，走正常工艺路线）；返工上下文的 `BatchReworkOrder` 是**批量返工工单**（质量异常/ECN 触发，走独立返工工艺路线）。两者触发源、工艺路线、再入机制、终点完全不同。
2. **聚合不同**——`WorkOrder`（含 BOM 绑定 + 正常工艺路线绑定 + 目标数量 + 完工判定）vs `BatchReworkOrder`（含 SN 清单 + 返工路线 + 批量再入点 + 返工进度）。无共用聚合。
3. **引用关系单向**——返工工单引用源工单（`source_work_order_id`），源工单不引用返工工单。合并会反向耦合正常工单到返工流程。
4. **服务前缀不同**——`wo.*`（work order）vs `brework.*`（batch rework），事件名、主题、消费方严格区分，**严禁混用**。

**为什么工单管理归生产执行服务而非制造资源服务**：
1. **数据性质不同**——制造资源服务管**配置性数据**（BOM、工艺路线、质量规则，写少读多）；工单是**事务性数据**（高频下达/状态流转/结案，与生产执行强绑定）。
2. **协作紧密度**——工单与过点执行（进度累计）、在制品追踪（齐套投影）、首件处理（首件触发）、返工（源工单引用）同属生产执行链条，归生产执行服务减少跨服务事务。
3. **领域总览 §1 定位**——生产执行服务是"现场生产活动执行引擎"，工单是执行引擎的输入与目标载体，归生产执行服务符合定位。

## 1. 主语清单与触发入口

工单管理上下文的入口只有 SAP 下发工单，统一建模为 `WorkOrder`。SAP 传入后处于 CREATED；计划员审核通过并生成 MES 工单流水号后，工单下达为 RELEASED，车间才允许生产。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.2 聚合根主语清单

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **WorkOrder**（工单） | SAP 下发工单 | 唯一身份；SAP 来源引用；计划员审核；生成 MES 工单流水号；当前 SAP 工单与工单流水号一对一，预留一对多扩展；绑定 BOM + 工艺路线版本；状态单向流转；目标数量驱动完工 | SAP / 计划员审核 |
| **WorkOrderItem**（工单明细·当前单产品） | 单产品工单明细 | 当前阶段一张工单仅允许一个产品明细；`WorkOrderItem` 作为值对象预留多明细扩展，但事件、完工判定与版本绑定均按单产品口径收敛 | 由 WorkOrder 触发 |
| **WorkOrderBinding**（工单绑定·值对象） | BOM 版本 + 工艺路线版本快照 | 下达前锁定版本；变更仅影响未过点在制品（参考过点执行 INV-09） | 由 WorkOrder 触发 |
| **WorkOrderClosure**（工单结案） | 完工产品数量达成后的自动结案 | 消费过点执行汇总后的完工产品数量判定结案；完工产品 = 最后一道工序过点完成 + 无未关闭质量单/返修单等关联异常；数量达到 `target_qty` 后自动 CLOSED | 过点执行进度事件 / 质量与返修闭环状态投影 |
| ProductSerialNo（产品流水号） | 开工前按订单数量创建的产品流水号 | 数量等于 `target_qty`；工艺路线第一步绑定到产品；限定可投产数量，因此不存在超产场景 | 计划员 / 过点执行上下文首工序绑定 |

**工单模型**：

```
WorkOrder
├── work_order_id        : 工单唯一标识
├── work_order_no        : SAP 工单号（业务可读，来自 SAP）
├── mes_serial_no        : MES 工单流水号（计划员审核通过后生成；当前与 SAP 工单一对一，预留一对多扩展）
├── serial_no_prefix     : 产品流水号前缀（由物料类型 + 工单类型 + 所属年份解析）
├── product_id           : 产品 ID（引用物料上下文产品主数据）
├── target_qty           : 目标数量
├── work_order_type      : 工单类型（来自 SAP 或经 ACL 归一化）
├── production_year      : 所属年份（用于产品流水号前缀解析）
├── material_type        : 生产产品的物料类型（用于产品流水号前缀解析）
├── unit                 : 单位（PCS / SET）
├── bom_binding          : WorkOrderBinding.bom（bom_id, bom_version, 审核下达时锁定快照，CREATED 阶段可为空）
├── route_binding        : WorkOrderBinding.route（route_id, route_version, 审核下达时锁定快照，CREATED 阶段可为空）
├── planned_start_at     : 计划开工时间（SAP 下发，仅参考）
├── planned_finish_at    : 计划完工时间（SAP 下发，仅参考）
├── production_line_id   : 产线 ID（SAP 下发，仅参考；MES 审核下达时校验，不做排程）
├── source               : SAP
├── source_event_id      : 来源事件 ID（幂等去重，若来自 SAP 事件）
├── status               : CREATED | RELEASED | IN_PROGRESS | PAUSED | COMPLETED | CLOSED | CANCELLED
├── closure              : WorkOrderClosure（完工判定与结案信息）
├── created_at           : 创建时间
└── released_at          : 下达时间（RELEASED 时填）
```

> **编号语义已确认**：`work_order_no` 表示 SAP 工单号，来源于 SAP；`mes_serial_no` 表示该工单在 MES 内部的工单流水号，由计划员审核通过时生成，并作为 `ReviewAndReleaseWorkOrder` 的前置条件。当前阶段一张 SAP 工单对应一个 MES 工单流水号，但模型保留一对多扩展空间，未来可在同一 `work_order_no` 下按产线/批次/拆分单生成多个 `mes_serial_no`。

> **MES 工单流水号规则已确认**：流水号生成依赖产品流水号前缀。前缀由 `ProductSerialPrefixResolver`（工具类/领域辅助组件）根据生产产品的 `material_type`、`work_order_type` 与 `production_year` 解析；相同前缀下序号递增增长。流水号一经生成不可回收，取消/作废工单也不复用已发号段；并通过唯一索引保证 `mes_serial_no` 全局唯一。

> **已确认：流水号前缀映射表归属**——`material_type + work_order_type + production_year -> serial_no_prefix` 的映射规则由**本上下文配置字典**维护；`ProductSerialPrefixResolver` 作为本上下文的领域辅助组件读取该配置并解析前缀。物料上下文只提供 `material_type` 等输入枚举/主数据，不承担 MES 工单流水号规则维护职责。

> **已确认：当前阶段仅支持单产品工单**——一张 `WorkOrder` 只包含一个产品 + 一个目标数量，`WorkOrderItem` 作为值对象保留扩展位，但当前事件载荷、完工判定、BOM/工艺路线版本绑定均按单产品口径设计；多产品套件工单暂不展开，避免把整机套件/BOM 展开逻辑混入工单生命周期上下文。

> **已确认：排程归属**——SAP 下发工单时携带 `production_line_id`、`planned_start_at`、`planned_finish_at`；MES 工单管理当前只接收并在计划员审核下达时做基础校验（产线存在、产线支持该产品/工艺路线、计划时间格式与先后关系合法），不做有限产能排程。上述产线与计划时间仅作为参考计划信息，不代表 MES 已完成产能承诺；后续保留扩展排程子流程。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

```
（工单已下达，触发首件标识判定：换线/换料/ECN）
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| SapWorkOrderReceived | SAP 工单经 ACL 归一化后被 MES 接收建档 | work_order_id, work_order_no, product_id, target_qty, planned_start_at, planned_finish_at, production_line_id, source_event_id | 创建 CREATED 工单，等待计划员审核 |
| WorkOrderCreated | SAP 工单接收后生成 CREATED 工单 | work_order_id, work_order_no, product_id, target_qty, planned_start_at, planned_finish_at, production_line_id, source=SAP | 等待计划员审核、生成 MES 工单流水号并锁定版本 |
| WorkOrderBindingLocked | 计划员审核下达时查询并锁定 BOM/工艺路线版本 | work_order_id, bom_binding, route_binding, locked_by, locked_at | 满足工单下达前置条件 |
| MesSerialNoGenerated | 计划员审核通过并生成 MES 工单流水号 | work_order_id, work_order_no, mes_serial_no, serial_no_prefix, generated_by, generated_at | 满足工单下达前置条件 |
| WorkOrderReleased | 工单审核通过并下达，允许过点 | work_order_id, work_order_no, mes_serial_no, serial_no_prefix, route_version, production_line_id, released_at | 通知过点执行 / WIP / 首件处理 |

> **已确认：SAP 传入后需人工审核下达**——MES 不允许创建工单；SAP 工单进入 MES 后为 CREATED，必须由计划员审核，审核通过时生成 MES 工单流水号并下达为 RELEASED，之后才允许车间生产。

> **已确认：BOM/工艺路线版本在计划员审核下达时锁定**——SAP 工单进入 MES 的 CREATED 阶段不固化版本；计划员执行 `ReviewAndReleaseWorkOrder` 时，实时查询物料上下文与工艺管理上下文，锁定 BOM 版本和工艺路线版本快照，并生成 `WorkOrderBindingLocked`。若版本缺失、未发布或不适用于该产品/产线，则拒绝下达。

> 🔴 **热点：工单下达与首件触发的时序**——`WorkOrderReleased` 发布后，首件处理上下文触发首件标识判定。首件未放行前，过点执行上下文是否拦截批量过点？建议：**首件门禁由过点执行上下文执行**（消费 `fai.article.blocked` 拦截，消费 `fai.article.released` 放行），本上下文只发布 `WorkOrderReleased` 通知首件处理上下文触发判定，不直接参与门禁。待与首件处理上下文确认触发契约。

### 2.2 状态流转

```
[CREATED] ──ReviewAndReleaseWorkOrder──▶ [RELEASED] ──StartWorkOrder──▶ [IN_PROGRESS] ──CompleteWorkOrder──▶ [COMPLETED] ──CloseWorkOrder──▶ [CLOSED]
                                    │                                │                                (终态—完工判定通过, INV-04)
                                    │                                │
                                    │                                └──PauseWorkOrder──▶ [PAUSED] ──ResumeWorkOrder──▶ [IN_PROGRESS]
                                    │
                                    └──CancelWorkOrder──▶ [CANCELLED]
                                                            (终态—取消, INV-03)

[RELEASED] ──CancelWorkOrder──▶ [CANCELLED]（下达后无在制可取消）
[IN_PROGRESS] ──CancelWorkOrder──▶ [CANCELLED]（已有过点在制时需授权，并记录在制 SN 处理方式）
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| CREATED → RELEASED | ReviewAndReleaseWorkOrder | 当前 CREATED；来源为 SAP；计划员审核通过；SAP 参考产线/计划时间已通过基础校验；已根据 `material_type + work_order_type + production_year` 解析 `serial_no_prefix` 并在此前缀下递增生成全局唯一 `mes_serial_no`；BOM + 工艺路线版本已绑定（INV-01） | 计划员 |
| RELEASED → IN_PROGRESS | StartWorkOrder | 当前 RELEASED；首次过点放行（消费过点执行 `CheckpointReleased`） | 系统（首次过点自动转态） |
| IN_PROGRESS → PAUSED | PauseWorkOrder | 当前 IN_PROGRESS；暂停原因记录 | 生产主管 / 系统（设备故障/质量异常联动） |
| PAUSED → IN_PROGRESS | ResumeWorkOrder | 当前 PAUSED；暂停原因已解除 | 生产主管 |
| IN_PROGRESS → COMPLETED | CompleteWorkOrder | 当前 IN_PROGRESS；`good_qty >= target_qty`（INV-04，消费过点执行进度） | 系统（进度达标自动转态） |
| COMPLETED → CLOSED | CloseWorkOrder | 当前 COMPLETED；结案确认（人工/系统） | 计划员 / 系统（超时自动结案） |
| CREATED/RELEASED → CANCELLED | CancelWorkOrder | 当前 CREATED 或 RELEASED；无在制过点（INV-05） | 计划员 |
| IN_PROGRESS → CANCELLED | CancelWorkOrder | 当前 IN_PROGRESS；已有过点在制时必须具备授权记录、取消原因，并明确在制 SN 处理方式（继续完工 / 强制报废 / 转其他工单） | 计划员（需授权） |
| → CLOSED / CANCELLED 之后 | （任何） | 终态，拒收（INV-03） | — |

> **可逆边说明**：仅 `PauseWorkOrder` / `ResumeWorkOrder`（暂停/恢复）允许回退，其余状态单向流转（INV-06）。

> **已确认：保留 IN_PROGRESS 状态**——`RELEASED` 表示“已下达、允许生产但尚未实际开工”，`IN_PROGRESS` 表示“已有首次过点放行、正在生产”。转态由过点执行上下文发布的首次 `CheckpointReleased` 自动触发 `StartWorkOrder`，不需要人工开工动作；这能支持排产看板区分“待生产”和“生产中”。

> **已确认：IN_PROGRESS 工单允许授权取消**——已有过点在制的工单可取消，但必须记录授权人、取消原因与在制 SN 处理方式（继续完工 / 强制报废 / 转其他工单）。取消后本上下文发布 `WorkOrderCancelled`；过点执行上下文订阅后缓存工单状态并按在制处理策略拦截或引导剩余 SN，不由本上下文直接修改 SN 状态。

> **已确认：PAUSED 后必须拦截过点**——工单进入 `PAUSED` 后，本上下文发布 `WorkOrderPaused`；过点执行上下文订阅后将该工单状态缓存为 PAUSED，后续过点校验必须拦截并产生 `CheckpointBlocked(blocking_reason=WorkOrderPaused)`。`WorkOrderResumed` 发布后才恢复放行。

### 2.3 工单变更

工单变更仅覆盖**可变更字段**（数量、SAP 参考计划时间、SAP 参考产线），且数量变更仅允许在工单尚未进入 `IN_PROGRESS` 前完成；一旦已有首次过点，`target_qty` 不再调整。BOM/工艺路线版本在 `ReviewAndReleaseWorkOrder` 时锁定，下达后不允许变更；若工单已在制后发现产品 BOM 或工艺路线需要调整，应通过返工流程处理，不在本上下文直接切换版本。当前阶段 MES 不做有限产能排程，`planned_start_at` / `planned_finish_at` / `production_line_id` 仅作为 SAP 下发的参考计划信息；若后续扩展排程子流程，可在本节基础上新增正式排程确认事件。

```
[CREATED / RELEASED] 状态下
        |
        v
【参与者:计划员】 --【命令:变更工单 ChangeWorkOrder】--> 【聚合:WorkOrder】
        |
        v
    可变更字段（target_qty / planned_start_at[参考] / planned_finish_at[参考] / production_line_id[参考]）
        |
        v
    【事件:工单已变更 WorkOrderChanged】
     (work_order_id, changed_fields[], change_reason, changed_at)
        |
        v
    发布到 wo.order.changed
        |
        v
    【外部系统:过点执行上下文】订阅
    （若 target_qty 在 RELEASED 前变更，更新完工判定阈值；若 SAP 参考产线/计划时间变更，仅更新参考计划投影，当前不触发排程重算）

[IN_PROGRESS] 状态下尝试变更 target_qty
        |
        v
【聚合:WorkOrder】拒绝
        |
        v
【事件:工单数量变更已拒绝 WorkOrderQuantityChangeRejected】
 (work_order_id, requested_target_qty, reject_reason=WorkOrderAlreadyInProgress)
        |
        v
提示：若在制后发生数量变化，只能取消剩余量、拆分/新建工单，或按异常流程处理

[RELEASED / IN_PROGRESS] 状态下尝试变更 bom_binding / route_binding
        |
        v
【聚合:WorkOrder】拒绝
        |
        v
【事件:工单版本变更已拒绝 WorkOrderBindingChangeRejected】
 (work_order_id, requested_bom_version, requested_route_version, reject_reason=WorkOrderAlreadyReleasedOrInProgress)
        |
        v
提示：若已在制后需要修改产品 BOM 或工艺路线，需走返工流程
```

> **已确认：IN_PROGRESS 后禁止调整 target_qty**——工单一旦发生首次过点进入 `IN_PROGRESS`，目标数量不再实时调整，完工判定继续使用进入在制时的 `target_qty`。若在制后发生客户取消、数量增补或剩余数量不再生产，应通过取消剩余量、拆分/新建工单或异常流程处理，避免运行中工单的完工口径漂移。

> **已确认：在制后禁止变更 BOM/工艺路线版本**——`WorkOrderBinding` 在审核下达时锁定；`RELEASED` / `IN_PROGRESS` 状态下不允许直接变更 BOM 或工艺路线版本，也不提供 `effective_scope=NEW_ONLY/ALL` 的切换模式。若生产过程中发现产品 BOM 或工艺路线需调整，应由质量/工程变更触发返工流程处理，避免同一正常生产工单内混用多个 BOM/路线口径。

> **已确认：产线/计划时间仅作参考**——SAP 下发的 `production_line_id`、`planned_start_at`、`planned_finish_at` 在当前阶段只作为参考计划信息。MES 工单管理只在审核下达时校验其基础合法性，不因为参考产线/计划时间变更而触发在制迁移或排程重算；后续若扩展排程子流程，再新增正式排程确认/调整事件。

### 2.4 工单完工判定与结案

工单完工与结案基于过点执行上下文汇总后的**完工产品数量**与本上下文的 `WorkOrder.target_qty` 比对。这里的“完工产品”不是指产品最后一道工序刚刚过点完成，而是最后一道工序过点完成，并且没有未关闭的质量单、返修单等关联异常后的汇总状态。因此若仍存在在制品、不良品、返修中产品或未闭环质量单，对应产品不会计入完工产品数量；当完工产品数量达到 `target_qty` 时，本上下文自动完成并结案。

```
【外部系统:过点执行上下文】
        -- 发布 WorkOrderProgressAccrued(work_order_id, completed_product_qty, wip_qty, bad_qty, repair_pending_qty)
                |
                v
【策略:消费完工产品数量做完工与结案判定】（幂等：work_order_id + accrued_at 去重, INV-08）
                |
                v
【命令:自动完成并结案工单 CompleteAndCloseWorkOrder】--> 【聚合:WorkOrder】
        |
        v
比较 completed_product_qty >= target_qty ?
        |
        ├── 否 --> 保持 IN_PROGRESS，等待后续完工产品汇总
        |
        └── 是 --> 【事件:工单已完工 WorkOrderCompleted】
                    (work_order_id, completed_product_qty, target_qty, completed_at)
                    |
                    v
                【策略:完工产品数量达成后自动结案】
                    |
                    v
                【事件:工单已结案 WorkOrderClosed】
                 (work_order_id, closure_reason=CompletedProductQtyReachedTarget, closed_at)
                    |
                    v
                发布到 wo.order.completed / wo.order.closed
                    |
                    v
                【外部系统:数据应用上下文】订阅
                （工单结案统计、良率分析、成本归集）
                【外部系统:在制品追踪上下文】订阅
                （工单结案，清理在制投影）
```

> **已确认：不存在超产场景**——开工前计划员会按 `target_qty` 创建等量产品流水号，工艺路线第一步要求将产品流水号绑定到对应产品；未发放产品流水号的产品无法进入首工序过点。因此工单完工判定仍为 `good_qty >= target_qty`，但正常执行中 `good_qty` 不应超过 `target_qty`；超出投产应由产品流水号发放/首工序绑定门禁拦截，而不是通过 `allow_overcomplete` 放开。


> **已确认：完工产品数量达成后自动结案**——`COMPLETED` 不是“最后一道工序刚过点”的瞬时状态，而是产品最后一道工序过点完成且无未关闭质量单、返修单等关联异常后的汇总状态。若仍存在在制品、不良品、返修中产品或未闭环质量单，对应产品不会计入 `completed_product_qty`。因此当 `completed_product_qty >= target_qty` 时，本上下文可自动产生 `WorkOrderCompleted` 并紧接着自动产生 `WorkOrderClosed`，无需人工结案确认。

> **已确认：不存在未完工强制结案**——若完工产品数量未达到 `target_qty`，工单不能直接 CLOSED；必须通过取消剩余量、拆分/新建工单或异常流程使完工口径闭环后，再由完工产品数量达成规则自动结案。

### 2.5 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动完工判定与状态联动，**不重发**这些事件——必须包装成本上下文语义的事件（INV-09）。BOM 与工艺路线版本通过 REST 查询，不属于事件订阅范畴。

> **已确认：进度事件在完工状态变化时发布**——`WorkOrderProgressAccrued` 不要求每次普通过点都发布；仅当产品完工汇总状态发生变化时发布，例如产品最后一道工序完成、质量单关闭、返修单关闭等导致 `completed_product_qty` / `wip_qty` / `bad_qty` / `repair_pending_qty` 变化。这样既能驱动工单自动完工/结案，又避免每次过点都触发工单管理高频判定。本上下文维护 `WorkOrderProgressProjection` 记录最新汇总，并以 `work_order_id + accrued_at` 幂等去重。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **WorkOrder** | 单个工单 | INV-01 下达前必须绑定 BOM 版本 + 工艺路线版本（bom_binding / route_binding 非空）；INV-03 CLOSED / CANCELLED 为终态，之后任何事件均拒收；INV-06 状态单向流转，仅 PauseWorkOrder / ResumeWorkOrder 允许回退；INV-13 当前阶段一张 SAP 工单仅允许一个有效 MES 工单流水号，模型保留同一 SAP 工单多个 `mes_serial_no` 的扩展空间；INV-15 当前阶段一张工单仅允许一个产品明细，多产品明细暂不启用；INV-16 IN_PROGRESS 后禁止调整 target_qty |
| **WorkOrderBinding**（值对象） | 工单版本绑定快照 | INV-07 版本绑定在 ReviewAndReleaseWorkOrder 时锁定（bom_version / route_version 快照）；RELEASED / IN_PROGRESS 后禁止直接变更 BOM 或工艺路线版本，若已在制后需修改产品 BOM 或工艺路线，则走返工流程 |
| **WorkOrderClosure** | 工单结案判定 | INV-04 完工与结案判定基于 `completed_product_qty >= target_qty`（completed_product_qty 来自过点执行上下文汇总，含最后工序完成且无未关闭质量单/返修单等关联异常，本上下文不重复累计）；INV-11 COMPLETED 后立即自动 CLOSED，不允许跳过完工口径直接 CLOSED；INV-17 开工前按 `target_qty` 创建等量产品流水号，首工序必须绑定有效产品流水号，因此正常场景不存在超产 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同一 SAP 下发指令不重复创建工单）→ `WorkOrderService` 领域服务 + source_event_id 事件去重表
> - **INV-05**（CANCELLED 前需无在制过点或授权处理在制）→ `WorkOrderService` 领域服务 + 过点执行上下文查询在制 SN + 授权记录
> - **INV-08**（进度事件幂等消费）→ `WorkOrderProgressProjection` 读模型 + (work_order_id, accrued_at) 去重表
> - **INV-09**（上下文外事件只消费不重发）→ 主题前缀隔离（本上下文只发 `wo.*`，订阅 `mes.*` / `brework.*`）+ 事件命名约定

## 与过点执行上下文的协作边界

本上下文与过点执行上下文是"工单生命周期"与"过点执行进度"的分工关系，通过事件契约解耦。这是本上下文最紧密的协作方，必须对齐避免事件混淆。

| 维度 | 本上下文（工单管理上下文） | 过点执行上下文 |
|------|------------------|--------------|
| 视角 | 工单生命周期视角（"工单怎么下、状态怎么流转、何时完工结案"） | 过点执行进度视角（"过点瞬间怎么判定、工单进度怎么累计"） |
| 存储 | 工单 / 工单绑定 / 工单结案 / 工单状态投影 | 过点记录 / 路线推进 / 工单进度（WorkOrderProgress）/ 本地缓存 |
| 谁判定 | 工单状态流转（下达/暂停/完工/结案）、完工判定（good_qty >= target_qty） | 过点放行/拦截决策、工单执行进度累计（good/bad/reworked qty） |
| 事件性质 | 工单生命周期事实：WorkOrderReleased / Paused / Completed / Closed | 过点执行事实：CheckpointReleased / WorkOrderProgressAccrued |
| 流程所有权 | 接收→建档→绑定→下达→状态流转→完工判定→结案 | 扫码→防错→决策→记录→推进→累计进度 |
| 进度口径 | target_qty（目标数量，静态） + completed_product_qty 完工判定阈值 | completed_product_qty / wip_qty / bad_qty / repair_pending_qty（执行与异常闭环汇总，动态累计） |

**事件流向**：

```
本上下文 ──wo.order.released──▶ 过点执行上下文（缓存工单状态，允许过点, INV-12）
本上下文 ──wo.order.paused──▶ 过点执行上下文（缓存工单状态为 PAUSED，过点拦截）
本上下文 ──wo.order.resumed──▶ 过点执行上下文（恢复过点放行）
本上下文 ──wo.order.changed──▶ 过点执行上下文（target_qty 变更影响完工判定阈值）
本上下文 ──wo.order.binding-change-rejected──▶ 计划员审核工作台/数据应用上下文（提示已下达/在制工单不可切换版本，需走返工流程）
本上下文 ──wo.order.cancelled──▶ 过点执行上下文（缓存工单状态为 CANCELLED，过点拦截）
本上下文 ──wo.order.completed──▶ 下游上下文（完工产品数量达成，系统随后自动结案）
本上下文 ──wo.order.closed──▶ 下游上下文（工单自动结案，终态）
过点执行上下文 ──mes.workorder.progress──▶ 本上下文（完工产品数量事件驱动自动完工与结案, INV-04/INV-08）
过点执行上下文 ──mes.checkpoint.lifecycle──▶ 本上下文（首次过点驱动 RELEASED→IN_PROGRESS, INV-10）
```

**职责切分要点**：
- **本上下文管**：工单身份 + 目标数量 + BOM/工艺路线版本绑定 + 状态生命周期 + 基于完工产品数量的自动完工与结案
- **过点执行上下文管**：工单执行进度与异常闭环汇总（completed_product_qty / wip_qty / bad_qty / repair_pending_qty） + 过点时工单状态校验（缓存工单状态）
- **不互相调用的操作**：本上下文不直接修改 `WorkOrderProgress`（归过点执行）；过点执行上下文不直接修改 `WorkOrder.status`（归本上下文）。全部通过事件异步解耦。
- **完工与结案判定的所有权**：本上下文基于过点执行汇总的 `completed_product_qty` 与本上下文的 `target_qty` 比对；当完工产品数量达成时自动产生 `WorkOrderCompleted` 与 `WorkOrderClosed`。过点执行上下文负责给出“产品是否已完工”的汇总口径：最后一道工序完成且无未关闭质量单、返修单等关联异常。

> **已确认：工单状态缓存以事件投影为主，REST 查询兜底**——过点执行上下文订阅 `wo.order.*` 维护 `WorkOrderStatusCache`，正常过点链路优先读取本地缓存进行工单状态校验，避免每次过点同步调用工单管理影响产线节拍。缓存未命中、缓存过期或事件版本不可信时，同步 REST 查询本上下文兜底；查询成功后刷新缓存并继续判定。若 REST 查询失败，则保守拦截本次过点，避免在工单 `PAUSED` / `CANCELLED` / `CLOSED` 状态下继续生产。

## 与返工上下文的协作边界（关键区分表）

本上下文（工单管理 `wo`）与返工上下文（返工 `brework`）**名称都含"工单"但完全不同**，必须严格区分，避免事件名、主题前缀、聚合、消费方混淆。这张表是防止"WorkOrder 与 BatchReworkOrder 混淆"的核心边界。

| 维度 | 本上下文（工单管理上下文 wo） | 返工上下文（brework） |
|------|------------------|--------------|
| **工单性质** | **正常生产工单**（SAP 下发，走正常工艺路线） | **批量返工工单**（质量异常/ECN/人工决策触发，走独立返工工艺路线） |
| **服务前缀** | `wo.*` | `brework.*` |
| **聚合根** | WorkOrder（含 WorkOrderItem / WorkOrderBinding / WorkOrderClosure） | BatchReworkOrder（含 ReworkRoute / ReentryPoint / ReworkProgress） |
| **触发机制** | SAP 下发 + 计划员审核下达 | 批量质量异常 / 工程变更 / 人工决策 |
| **流程起点** | 接收下达指令生成工单 | 消费批量触发信号生成返工工单 |
| **工艺路线** | 正常工艺路线（从工艺管理上下文引用，绑定 route_version） | 返工工艺路线（从工艺管理上下文引用，与正常路线不同，绑定 rework_route_version） |
| **再入机制** | 无（正常工单不涉及再入） | 批量再入点（返工工单定义，整批统一再入） |
| **流程终点** | 完工（good_qty >= target_qty）+ 结案 | 整批从再入点回主线或批量报废 |
| **进度口径** | target_qty（目标数量，静态） + 消费过点执行 good_qty 判完工 | ReworkProgress（返工进度，按 SN 维度累计） |
| **所属服务** | 生产执行服务 | 生产执行服务 |

**关键澄清**：
- 两者**单向引用**——返工上下文的 `BatchReworkOrder.source_work_order_id` 引用本上下文的 `WorkOrder`；本上下文不引用返工工单。
- **事件名严格区分**：`WorkOrderReleased` / `WorkOrderCompleted`（本上下文，正常工单级）vs `BatchReworkOrderReleased` / `BatchReworkOrderCompleted`（返工上下文，批量返工级）。**严禁混用或互相引用**。
- **只读投影**：本上下文可消费 `BatchReworkOrderCreated` 维护源工单关联返工投影（关联返工工单数、状态摘要），供追溯与看板查询；该投影不改变 `WorkOrder.status`，也不把返工流程并入本上下文。

- **进度独立性**——本上下文的 `WorkOrder.target_qty` 与返工上下文的 `ReworkProgress` 是两个独立进度，不混淆。源工单的 `good_qty` 已由过点执行累计（返工再入后放行计入 `reworked_qty` 不重复计入 good），完工判定不受返工影响。

## 与工艺管理上下文、物料上下文的协作边界

本上下文是 BOM 与工艺路线版本配置数据的**消费方**，通过"REST 查询 + 版本锁定快照"解耦：

| 维度 | 本上下文（工单管理上下文） | 工艺管理上下文 | 物料上下文 |

| 存储 | 工单 + 版本绑定快照（bom_version / route_version 锁定） | 工序定义、工艺路线、版本、参数模板 | BOM、产品主数据、替代料、供应商 |
| 谁判定 | 审核下达时锁定版本快照（ReviewAndReleaseWorkOrder 时固化） | 发布版本生效事件为权威源 | 维护 BOM 版本为权威源 |

| 版本一致性 | 审核下达时锁定版本快照；变更默认 NEW_ONLY（INV-07） | 发布生效事件为权威源 | 维护 BOM 版本为权威源 |

**事件流向**：
- **工艺管理/物料上下文 → 本上下文**：REST 查询（计划员审核下达时绑定版本，非事件订阅）
- **本上下文 → 工艺管理/物料上下文**：（无直接事件，本上下文只读引用版本）
- **版本锁定机制**：ReviewAndReleaseWorkOrder 时 REST 查询工艺管理上下文（route_id + 当前生效 route_version）与物料上下文（bom_id + 当前生效 bom_version），锁定为 `WorkOrderBinding` 快照。CREATED 阶段不固化版本；`RELEASED` / `IN_PROGRESS` 后禁止直接变更 BOM 或工艺路线版本（INV-07）。若已在制后发现产品 BOM 或工艺路线需要调整，应通过返工流程处理，避免同一正常生产工单内混用多个版本口径。

> **已确认：版本查询与锁定发生在计划员审核下达时**——`ReviewAndReleaseWorkOrder` 实时查询 BOM/工艺路线版本并锁定快照；CREATED 阶段不锁定版本。工单审核下达是低频操作（相比过点毫秒级），建议默认不缓存，避免缓存漂移；若性能压力，可考虑短 TTL 缓存。

> **已确认：不响应 BOM/工艺路线版本主动通知来切换在制工单**——本上下文只在 `ReviewAndReleaseWorkOrder` 时查询并锁定当前生效版本；CREATED 工单尚未锁定版本，已 RELEASED / IN_PROGRESS 工单的绑定不受新版本发布影响且不允许直接切换。若需因 ECN/质量问题处理已在制产品，应由返工流程承接。

- 多产品工单（一单多产品套件）→ 已确认当前阶段不支持；`WorkOrderItem` 仅作为值对象扩展位，事件与完工判定按单产品口径设计
- 与 SAP 的工单同步协议（字段映射、双向同步、状态回传）→ 已确认主链为 SAP / 中间件消息推送工单下发；字段映射由本上下文 SAP 工单接收适配层（ACL，端口：`WorkOrderReceptionPort`）承担；状态回传是否需要仍待 SAP 集成专题确认
- BOM / 工艺路线版本定义与版本生命周期 → 工艺管理上下文 / 物料上下文（制造资源服务）
- 批量返工工单生成、返工工艺路线 → 返工上下文（已落地，`brework.*`，引用本上下文源工单）
- 工单执行进度累计（good/bad/reworked qty）→ 过点执行上下文（已落地，`mes.workorder.progress`）
- 在制品身份、位置投影、齐套状态计算 → 在制品追踪上下文（已落地）
- 首件标识触发、首件检验流程编排 → 首件处理上下文（已落地）
- 工单结案的成本归集、良率分析 → 数据应用上下文（工单达到 CLOSED 后消费事件做统计，不作为结案前置条件）
- 字段级 Schema（工单/绑定/结案的字段精度）→ 实现说明文档
- 工单管理终端 UI 交互、工单看板 → 应用层

这些将通过本上下文已经定义的事件契约衔接，避免回头返工。
