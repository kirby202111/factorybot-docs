# 过点执行上下文 · 测试数据接入适配层（Test Data Ingestion Adapter）

> **所属上下文**：过点执行上下文（本文件为其端口适配层的详细设计）
> **层级**：适配层（非独立限界上下文）—— 唯一消费方为过点引擎，与过点记录同事务，无独立业务价值
> **统一语言**：TestStation（测试工位）, TestDataGateway（测试数据网关）, TestResult（测试结果）, TestAdapter（接入适配器）, FormatMapper（格式映射器）, FtpFileAdapter（FTP文件适配器）, HttpCallbackAdapter（HTTP回调适配器）, ManualEntryAdapter（手工填报适配器）, TestSession（测试会话）, TestItem（测试项）
> **核心关注**：以"事件风暴"方式刻画测试数据从多种异构渠道汇集到生产执行服务的完整事件链；只关心**数据如何接入与结构化**，不负责过点判定逻辑（由过点引擎负责）。

---

## 0. 设计原则与边界澄清

测试数据接入适配层是过点执行上下文的**端口适配层**（非独立限界上下文），解决"测试数据获取渠道多样化"与"过点逻辑统一化"之间的矛盾。核心设计思想：**过点引擎不感知数据从哪来，接入适配器不关心数据怎么判**。

### 0.1 为什么作为过点执行上下文的适配层

| 考量 | 说明 |
|------|------|
| 事务一致性 | 测试结果与过点记录是同一业务事务——"这块板测完了、结果是什么、能不能放行"不可拆分 |
| 时序对齐 | 测试结果需实时回传决定放行/拦截，跨服务延迟不可接受 |
| 业务语义 | 测试是工艺路线中的工序，操作工刷条码过点 + 测试数据关联 = 一个完整用例 |
| 与设备数据接入上下文的分工 | 设备数据接入上下文负责**设备维度的时序数据**（温度曲线、抛料率），不绑定单件，为独立限界上下文；测试数据接入适配层负责**单件维度的测试结果**，与过点事务强关联，为过点执行上下文的适配层 |

### 0.2 域内职责（本域负责）

- 定义统一的 `TestResult` 数据结构，屏蔽接入渠道差异
- FTP 文件适配：定时扫描、文件稳定检测、解析、幂等写入
- HTTP 回调适配：暴露回调接口、验签、格式识别与映射
- 手工填报适配：暴露填报接口、权限校验、审计记录
- 所有适配器写入前执行去重（按 SN + stationCode + testItem 幂等）
- 将结构化后的 `TestResult` 写入，供过点引擎查询

### 0.3 域外职责（本域**不**负责）

- PASS/FAIL 的业务判定 → 过点引擎
- 过站推进、WIP 状态变更 → 过点引擎
- 质量门禁规则执行 → 质量上下文（制造资源服务）
- 测试设备的物理连接、协议握手 → 设备数据接入上下文
- 测试数据的长期存储与查询 → 数据存储策略（暂缓模块）

### 0.4 为什么这样切

在电子制造车间，测试工位的数据接入是最"脏"的环节——同一个车间里，FCT 工装通过 HTTP 回调推送 JSON，老化测试电脑只在本地写 CSV 等 FTP 来取，一台 15 年前的 ICT 设备除了手工填根本没别的办法。如果把每种接入方式的处理逻辑直接写在过点代码里，过点引擎会变成一坨"if FTP then... else if HTTP then... else if manual then..."的意大利面。

本域的做法是：**所有接入渠道汇入同一个 Gateway 接口，过点引擎只调 `gateway.fetchTestResult(sn, stationCode)` 一个方法**。新增一种接入方式 = 新增一个 Adapter，过点引擎零改动。

## 1. 接入渠道分类与形态

针对车间实际存在的测试数据获取方式，按**触发模式**归为三类：

| 类别 | 接入方式 | 触发模式 | 典型场景 | 数据格式 | 适配器 |
|------|---------|---------|---------|---------|--------|
| A. 文件拉取 | FTP / 共享目录 | 主动拉取（定时扫描） | 烧录电脑本地 CSV/XML 日志、老化测试工装输出文件、FCT 测试报告 | CSV、XML、JSON Lines | FtpFileAdapter |
| B. HTTP 推送 | 测试软件回调 MES 接口 | 被动接收 | 新一代 FCT 工装、AOI 检测软件、SPI 检测软件 | JSON（格式因厂商而异） | HttpCallbackAdapter |
| C. 人工录入 | Web 页面手工填报 | 人工触发 | 老旧 ICT 设备（无网口/串口）、简易测试架、临时抽检 | 表单数据 | ManualEntryAdapter |

```
                         ┌─────────────────────────────┐
                         │     TestDataGateway          │
                         │     (统一存取接口)            │
                         │                              │
                         │  fetchTestResult(sn, station)│
                         │  onTestDataReceived(...)      │
                         └──────┬─────────┬────────────┘
                                │         │
                ┌───────────────┘         └───────────────┐
                ▼                                         ▼
    ┌───────────────────────┐             ┌───────────────────────────┐
    │   FtpFileAdapter      │             │   HttpCallbackAdapter      │
    │   (定时扫描 + 解析)    │             │   (被动接收 + 格式映射)     │
    │                       │             │                            │
    │  - 文件稳定检测        │             │  POST /api/exec/test-data/  │
    │  - CSV/XML/JSON 解析   │             │       callback             │
    │  - 按文件名+hash 去重  │             │  - 验签                     │
    └───────────────────────┘             │  - FormatResolver           │
                                          │    ├─ ChromaFormatMapper    │
                ┌───────────────────────┐ │    ├─ FctFormatMapper       │
                │  ManualEntryAdapter   │ │    └─ GenericJsonMapper     │
                │  (手工填报)            │ │                            │
                │                       │ └───────────────────────────┘
                │  POST /api/exec/test-  │
                │       data/manual      │
                │  - 权限校验             │
                │  - 审计日志             │
                └───────────────────────┘
```

## 2. 测试数据接入事件风暴（核心）

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 域外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐/有争议点

### 2.1 测试数据接入总览事件链（三类渠道 → 统一出口）

```
                    ┌──────────────────────────────────────────┐
                    │          三种接入渠道（并行）              │
                    └──────────────────────────────────────────┘
                              │          │          │
                    FTP 拉取  │   HTTP推送 │  手工填报 │
                              ▼          ▼          ▼
                    ┌──────────────────────────────────────────┐
                    │         TestDataGateway                  │
                    │         (统一写入入口)                    │
                    └──────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────────────────────────────┐
                    │         【聚合:TestResult】               │
                    │   (sn + stationCode + testItem 唯一)     │
                    └──────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────────────────────────────┐
                    │         【事件:测试数据已就绪              │
                    │          TestDataReady】                  │
                    └──────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────────────────────────────┐
                    │         过点引擎（消费方）                 │
                    │         fetchTestResult(sn, station)      │
                    │         → 判定 PASS/FAIL → 放行/拦截      │
                    └──────────────────────────────────────────┘
```

**关键设计**：所有渠道写入同一份 `TestResult` 存储，过点引擎只读这份存储，不关心数据来源。即使同一块板同时收到 FTP 文件和 HTTP 回调两份数据（冗余接入），去重机制保证只有一份生效。

### 2.2 FTP 文件适配器事件链（A 类：主动拉取）

```
[参与者:定时调度器(Cron)] --【命令:扫描FTP目录 ScanFtpDirectory】--> 【聚合:FtpFileAdapter】
                                                                        |
                                                                        v
                                                          【事件:目录已扫描 DirectoryScanned】
                                                          (发现新文件列表 new_files[])
                                                                        |
                                                    【策略:逐个处理新文件，跳过已处理文件名】
                                                                        |
                                                                        v
                                                          【命令:检测文件稳定性 WaitFileStable】
                                                          (等待 N 秒内文件大小不变)
                                                                        |
                                                    +-- 稳定 --> 【事件:文件已稳定 FileStabilized】
                                                    |                    |
                                                    |                    v
                                                    |        【命令:解析文件 ParseFile】
                                                    |        (根据 stationCode 选择解析器:
                                                    |         CsvParser / XmlParser / JsonLinesParser)
                                                    |                    |
                                                    |        +-- 解析成功 --> 【事件:文件已解析 FileParsed】
                                                    |        |                    |
                                                    |        |                    v
                                                    |        |        【命令:逐行写入测试结果 BatchIngestTestResults】
                                                    |        |                    |
                                                    |        |                    v
                                                    |        |        【聚合:TestResult】(逐条写入，幂等)
                                                    |        |                    |
                                                    |        |                    v
                                                    |        |        【事件:测试数据已就绪 TestDataReady】
                                                    |        |        (sn, stationCode, testItems[], source=FTP)
                                                    |        |
                                                    |        +-- 解析失败 --> 【事件:文件解析失败 FileParseFailed】
                                                    |                            |
                                                    |                            v
                                                    |                【命令:移入异常目录 + 告警】
                                                    |                【事件:异常文件已隔离 FileQuarantined】
                                                    |
                                                    +-- 超时未稳定 --> 【事件:文件稳定性超时 FileStabilizeTimedOut】
                                                                              |
                                                                              v
                                                                  【命令:跳过本次，下次扫描重试】
```

**FTP 文件解析器映射**

| stationCode 前缀 | 文件格式 | 解析器 | 典型来源 |
|-----------------|---------|--------|---------|
| BURN-* | CSV | CsvParser | 烧录电脑 J-Link/UniFlash 输出日志 |
| AGING-* | CSV / XML | CsvParser / XmlParser | 老化房测试工装 |
| FCT-* | JSON Lines / XML | JsonLinesParser / XmlParser | FCT 测试电脑 |
| ICT-* | CSV | CsvParser | 老旧 ICT 设备（如有文件输出） |

> 🔴 **热点：文件稳定检测时长**——FTP 文件可能还在写入时就被扫描到。建议检测窗口 5s（两次 stat 间隔 5s，文件大小不变视为稳定），可配置。烧录日志文件较大（>10MB）时可能需要更长时间。

> 🔴 **热点：文件清理策略**——解析成功的文件是立即删除还是保留？建议保留 7 天到归档目录，便于问题追溯；7 天后自动清理，与设备数据接入上下文的死信保留策略一致。

### 2.3 HTTP 回调适配器事件链（B 类：被动接收）

```
[外部系统:测试软件/FCT工装] --【命令:回调测试结果 CallbackTestResult】
        │
        ▼
[聚合:HttpCallbackAdapter] --【命令:校验请求签名 VerifySignature】
        │
        +-- 验签失败 --> 【事件:验签失败 SignatureVerificationFailed】
        │                    |
        │                    v
        │              【命令:拒绝请求 + 记录安全告警】
        │              【事件:未授权回调已拒绝 UnauthorizedCallbackRejected】
        │
        +-- 验签通过 --> 【事件:回调已接收 CallbackReceived】
                            |
                            v
                【命令:识别数据格式 ResolveFormat】
                (根据 Content-Type / stationCode / payload 特征)
                            |
                +-- 格式识别成功 --> 【事件:格式已识别 FormatResolved】
                |                        |
                |                        v
                |            【命令:映射为标准TestResult MapToTestResult】
                |            (调用对应 FormatMapper)
                |                        |
                |            +-- 映射成功 --> 【事件:测试数据已就绪 TestDataReady】
                |            |                   (sn, stationCode, testItems[], source=HTTP)
                |            |
                |            +-- 映射失败 --> 【事件:格式映射失败 FormatMappingFailed】
                |                                |
                |                                v
                |                    【命令:落入人工审核队列 + 告警】
                |                    【事件:待审核数据已入队 ManualReviewEnqueued】
                |
                +-- 格式识别失败 --> 【事件:未知格式 UnknownFormatDetected】
                                        |
                                        v
                            【命令:保留原始 payload + 告警】
                            【事件:原始载荷已保留 RawPayloadPreserved】
```

**FormatMapper 映射表**

| Mapper | 适用厂商/设备 | 识别特征 | 映射规则 |
|--------|-------------|---------|---------|
| ChromaFormatMapper | Chroma FCT 测试仪 | `Content-Type: application/json` + `$.vendor == "Chroma"` | 字段路径映射表 |
| FctFormatMapper | 自研/通用 FCT 工装 | `$.testType == "FCT"` | 通用 JSON → TestItem[] |
| IctFormatMapper | 特定 ICT 厂商 | `$.fixture_id` 存在且 `$.pins` 存在 | 引脚级结果展开 |
| GenericJsonMapper | 其他 JSON 回调 | 兜底 | 尝试自动推断字段，失败则入人工审核 |

> 🔴 **热点：FormatMapper 的扩展机制**——建议通过配置文件（YAML/JSON）定义映射规则，新增厂商时只需新增配置项而非改代码。但首次接入新厂商时通常需要联调，配置化只能降低后续变更成本。

> 🔴 **热点：测试软件重复回调**——某些测试软件在超时后会重发回调，必须在写入 TestResult 时做幂等（SN + stationCode + testItem 唯一约束），防止同一份测试结果被重复处理。

### 2.4 手工填报适配器事件链（C 类：人工触发）

```
[参与者:生产操作工] --【命令:打开填报页面 OpenManualEntryPage】--> 【外部系统:MES Web UI】
        │
        ▼
[聚合:ManualEntryAdapter] --【命令:加载工位测试模板 LoadTestTemplate】
        │ (根据 stationCode 读取工艺路线中定义的测试项清单)
        │
        ▼
【事件:测试模板已加载 TestTemplateLoaded】
(显示该工位需要填报的测试项: item_name, unit, 输入框类型)
        │
        ▼
[参与者:操作工] --【命令:提交测试数据 SubmitManualTestData】--> 【聚合:ManualEntryAdapter】
        │
        ▼
【命令:校验操作工权限 VerifyOperatorPermission】
        │ (操作工工位绑定 + 班次有效性)
        │
        +-- 权限不足 --> 【事件:填报已拒绝 EntryRejected】(reason=UNAUTHORIZED)
        │
        +-- 权限通过 --> 【命令:校验数据完整性 ValidateDataCompleteness】
                            │
                            +-- 必填项缺失 --> 【事件:填报已拒绝 EntryRejected】(reason=INCOMPLETE)
                            │
                            +-- 校验通过 --> 【事件:手工数据已提交 ManualDataSubmitted】
                                                |
                                                v
                                    【命令:写入TestResult + 审计日志 WriteWithAudit】
                                                |
                                                v
                                    【聚合:TestResult + AuditLog】
                                                |
                                                v
                                    【事件:测试数据已就绪 TestDataReady】
                                    (sn, stationCode, testItems[], source=MANUAL, operator_id)
                                                |
                                                v
                                    【事件:手工填报已审计 ManualEntryAudited】
                                    (operator_id, timestamp, client_ip, 变更前/后值)
```

> 🔴 **热点：手工填报的可信度**——手工填报数据天然不如自动采集可信，是否需要在过点判定时区别对待？建议在 `TestResult.source` 字段标记 `MANUAL`，质量上下文可根据 source 类型差异化处理（如手工填报的 PASS 需要班组长复核）。

> 🔴 **热点：老旧设备的替换节奏**——如果某 ICT 设备半年后会替换为新设备（支持 HTTP 回调），则手工填报只是过渡方案。建议在工位主数据中标记 `testDataAcquisitionMode`（MANUAL / FTP / HTTP），替换设备后只需改配置，无需改代码。

### 2.5 TestDataGateway 统一写入事件链

无论哪种渠道，最终都通过 Gateway 写入，保证一致性：

```
[任一适配器] --【命令:写入测试结果 IngestTestResult】--> 【聚合:TestDataGateway】
                                                              |
                                                              v
                                                【命令:幂等检查 CheckIdempotency】
                                                (唯一键: sn + stationCode + testItem)
                                                              |
                                        +-- 已存在(重复) --> 【事件:重复数据已忽略 DuplicateTestDataIgnored】
                                        |                        (记录 dedup_key, source)
                                        |
                                        +-- 新数据 --> 【命令:写入TestResult】
                                                            |
                                                            v
                                                【聚合:TestResult】
                                                (sn, stationCode, testItem, rawValue,
                                                 unit, source, ingestedAt, operator_id)
                                                            |
                                                            v
                                                【事件:测试数据已就绪 TestDataReady】
                                                (sn, stationCode, testItems[], source)
                                                            |
                                              【策略:通知过点引擎(若板正在等待)】
                                                            |
                                                            v
                                                【命令:唤醒等待中的过点事务 WakePendingPassThrough】
```

### 2.6 过点引擎消费事件链（本域边界）

这是过点引擎如何消费 TestDataReady 的视角，属于过点上下文但在此画出以展示协作：

```
【事件:操作工扫码过点 StationScanned】 (过点上下文事件)
        │
        ▼
【聚合:过点引擎】 --【命令:查询测试数据 FetchTestData(sn, stationCode)】
        │
        │   调用 TestDataGateway.fetchTestResult(sn, stationCode)
        │
        v
┌───────────────────────────────────────────────┐
│  情况1: 测试数据已就绪                         │
│  → 获取 TestResult → 执行判定 → 放行/拦截     │
│                                               │
│  情况2: 测试数据尚未到达                       │
│  → 工艺配置 requireTestData = true?            │
│    ├─ true:  拦截，提示"等待测试数据..."        │
│    │         订阅 TestDataReady 事件，           │
│    │         超时 N 秒后升级告警                 │

│                                               │
│  情况3: 测试数据源 = MANUAL                    │
│  → 正常判定，但在过点记录中标记                 │
│    data_source=MANUAL，供质量上下文后续复核         │
└───────────────────────────────────────────────┘
```

## 3. 跨渠道共享聚合

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **TestResult** | 单件单工位的单测试项结果 | (sn, stationCode, testItem) 唯一；rawValue 不可被后续写入覆盖（首次写入胜出）；source 标记不可变 |
| **FtpFileAdapter** | 单个 FTP 目录的扫描状态 | 同一文件名 + hash 不重复处理；扫描游标单调递增 |
| **HttpCallbackAdapter** | 单次 HTTP 回调请求 | 同一 requestId 不重复处理；验签失败不计入业务数据 |
| **ManualEntryAdapter** | 单次手工填报 | operator_id 不可为空；每次提交记录完整审计日志 |
| **FormatMapper** | 单次格式映射 | 输入 payload 不变 → 输出 TestResult 不变（纯函数语义） |
| **AuditLog** | 手工填报的审计轨迹 | 不可删除、不可修改；保留操作前/后快照 |

## 4. 关键策略（Policy）汇总

| 触发事件 | 策略 | 触发命令 |
|----------|------|----------|
| DirectoryScanned（发现新文件） | 逐个处理，跳过已处理文件名 | WaitFileStable |
| FileStabilized | 按 stationCode 选择解析器 | ParseFile |
| FileParseFailed | 移入异常目录，保留原文件 | QuarantineFile + NotifyOps |
| FileStabilizeTimedOut | 跳过，下次扫描重试 | — |
| CallbackReceived | 验签 → 格式识别 → 映射 | ResolveFormat → MapToTestResult |
| SignatureVerificationFailed | 拒绝 + 安全告警 | RejectUnauthorized |
| FormatMappingFailed / UnknownFormatDetected | 保留原始 payload，入人工审核 | PreserveRawPayload + EnqueueManualReview |
| ManualDataSubmitted | 权限校验 → 完整性校验 → 写入 | WriteWithAudit |
| DuplicateTestDataIgnored | 记录去重日志 | — |
| TestDataReady | 若有过点事务在等待，唤醒之 | WakePendingPassThrough |
| TestDataPending（过点侧，超时未到达） | 超时升级 | NotifySupervisor |

## 5. 对外契约（供过点引擎消费）

过点引擎只通过 `TestDataGateway` 访问测试数据，不直接接触任何 Adapter：

| 接口 | 方向 | 说明 |
|------|------|------|
| `TestDataGateway.fetchTestResult(sn, stationCode)` | 过点引擎 → Gateway | 同步查询，返回 `TestResult[]` 或 `null`（数据未到达） |
| `TestDataGateway.subscribeTestDataReady(sn, stationCode, timeout)` | 过点引擎 → Gateway | 异步等待，数据到达或超时后回调 |
| `TestDataGateway.ingest(sn, stationCode, rawData, source)` | Adapter → Gateway | 所有适配器的统一写入入口 |

过点引擎**不**直接访问的接口（Adapter 层内部）：
- `POST /api/exec/test-data/callback` → HttpCallbackAdapter 专用
- `POST /api/exec/test-data/manual` → ManualEntryAdapter 专用
- FTP 目录 → FtpFileAdapter 专用（不是 API，是文件系统）
