# 过点执行上下文（Checkpoint Execution Bounded Context）

> **限界上下文**：过点执行上下文
> **统一语言**：WorkOrder（工单）, RouteVersion（工艺路线版本·只读引用）, CheckpointRecord（过点记录·不可变）, CheckpointDecision（放行/拦截决策）, AntiErrorCheck（防错校验）, StationSequenceCheck（跳站校验）, RoutingProgress（工艺路线推进·单件当前步序）, WorkOrderProgress（工单执行进度）, TestResult（测试结果·测试数据接入适配层产出）, ProcessRouteCache（工艺路线缓存·投影）, EquipmentAvailabilityCache（设备可用性缓存·投影）, QualityGateCache（质量门禁缓存·投影）, KitStatusCache（齐套状态缓存·投影·消费 wip.kit.status）, FixtureUsageReport（工装使用上报）, EquipmentMalfunctionReport（设备故障上报）, ReworkReentry（返修再入）, BatchReworkReentry（返工再入）
> **核心关注**：以"事件风暴"方式刻画**现场生产活动执行引擎**——扫码过点、防错校验、放行/拦截决策、过点记录、工艺路线推进与跳站校验、工单执行进度累计；测试数据接入适配层从多种异构渠道（FTP 文件、HTTP 回调、手工填报）汇集数据并结构化为 `TestResult` 供过点引擎查询。本上下文只对"**一次过点的判定与记录、单件在路线上的推进、工单维度的进度累计**"负责；**不负责**工艺路线定义与版本生命周期（归工艺管理上下文）、质量门禁规则配置（归质量上下文）、设备资产身份与可用性事实源（归设备工装台账上下文）、设备原始信号采集（归设备数据接入上下文）、在制品位置投影与齐套状态（归在制品追踪上下文）、首件检验流程编排（归首件处理上下文）、单件返修任务（归返修上下文）、批量返工工单（归返工上下文）。

## 0. 设计原则与边界澄清

过点执行上下文是生产执行服务的**核心子域**，是 PCBA + 整机组装混合型车间所有"过站活动"的执行权威源。

**上下文内职责**（本上下文负责）：
- **扫码过点**——操作员在工位扫码触发过点，绑定 SN × 工单 × 工位 × 设备
- **防错校验**——过点前校验物料齐套、工装、设备、工艺版本、首件门禁、前置工序完成性等防错条件
- **放行/拦截决策**——综合防错校验、测试结果、质量门禁、设备可用性，判定 PASS / BLOCK
- **过点记录**——不可变地持久化过点事实（携带 `routeVersion`，保证历史追溯不受后续工艺变更影响）
- **工艺路线推进与跳站校验**——推进单件在路线上的当前步序，校验是否跳站/漏站
- **工单执行进度累计**——按过点放行结果累计工单完成量、良品/不良品量
- **测试数据接入适配层**——从 FTP 文件 / HTTP 回调 / 手工填报等异构渠道汇集数据，结构化为 `TestResult`，供过点引擎同事务查询
- **本地缓存维护**——工艺路线缓存、设备可用性缓存、质量门禁缓存（订阅上下文外事件刷新，缓存未命中降级远程查询）
- **工装使用上报**——过点时上报工装使用次数到台账上下文（工装累积度量的权威来源之一）
- **设备故障上报**——过点时发现设备异常，上报到维修上下文

**上下文外职责**（本上下文**不**负责）：
- 工序定义、工艺路线、参数模板、版本生效发布 → 工艺管理上下文（本上下文仅订阅 `ProcessRouteActivated` 刷新缓存，缓存未命中降级 REST 查询）
- 检验标准、缺陷分类、质量门禁规则配置 → 质量上下文（本上下文仅读缓存执行判定）
- 设备/工装/计量器具身份、归属、状态、可用性事实源 → 设备工装台账上下文（本上下文仅订阅 `eam.asset.availability` 刷新缓存，降级查询）
- 设备原始信号采集、协议适配、补传 → 设备数据接入上下文（本上下文过点时按需 REST 查询设备实时数据）
- 在制品身份、位置投影、流转历史、齐套状态 → 在制品追踪上下文（本上下文发布过点位置变更供其投影）
- 首件标识、首件检验流程编排、首件放行门禁判定 → 首件处理上下文（本上下文仅消费其放行/拦截门禁）
- 单件离线返修任务、故障诊断、再入点判定 → 返修上下文（本上下文发布分流事件、消费返修完成再入）
- 订单/批次级返工工单、返工工艺路线 → 返工上下文（本上下文消费返工工单再入）
- OEE / 良率分析 / SPC → 数据应用上下文

**为什么测试数据接入不独立成限界上下文**：
1. **无独立业务价值**——测试数据接入适配层的唯一消费方是过点引擎，其产出的 `TestResult` 与过点判定强绑定，脱离过点无独立语义。
2. **同事务边界**——`TestResult` 写入与过点判定属于同一业务事务（领域总览 §5.3），拆开会逼出分布式事务或最终一致缺口。
3. **职责是"适配"而非"领域"**——它解决的是"异构渠道数据如何变成结构化 `TestResult`"的端口适配问题，是防腐层（ACL）性质，不是领域子域。

**为什么在制品追踪 / 首件 / 返修 / 返工 不并入本上下文**：
1. **生命周期与变化频率不同**——过点是高频实时事件流（每板毫秒级），WIP 位置投影是读模型累积，首件是低频门禁，返修/返工是离线任务流。混在一起会使过点引擎膨胀为"执行 + 追踪 + 检验 + 返修"混合体。
2. **聚合边界清晰**——过点的聚合是 `CheckpointRecord` / `RoutingProgress`；WIP 的聚合是 `WipUnit` / `KitStatus`；首件的聚合是 `FirstArticle`；返修的聚合是 `ReworkTask`；返工的聚合是 `BatchReworkOrder`。无共用聚合，强合等于耦合。
3. **流程所有权不同**——过点管"过站瞬间"，返修/返工管"离线再处理"，首件管"批量放行门禁"，各自有独立状态机与 SLA。
4. **通过事件契约解耦即可协作**——本上下文发布过点位置变更供 WIP 投影、发布分流事件供返修接单、消费首件放行/返修返工再入事件，无需合并。

## 1. 主语清单与缓存形态

过点执行上下文处理的对象分两类：**写侧聚合**（过点判定与推进的事实）与**读侧缓存/投影**（订阅上下文外事件维护的本地视图）。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 写侧聚合根

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **CheckpointRecord**（过点记录） | 每板/每件过点事实 | 不可变；携带 `routeVersion`；与 `TestResult` 同事务 | 操作员扫码触发 |
| **CheckpointDecision**（放行/拦截决策） | PASS / BLOCK 决策 | 综合多维度判定；BLOCK 必带 `blocking_reason` | 由防错校验/测试结果驱动 |
| **RoutingProgress**（工艺路线推进） | 单件在路线上的当前步序 | 跟随 `routeVersion`；跳站校验 | 过点放行驱动 |
| **WorkOrderProgress**（工单执行进度） | 工单维度的完成量累计 | 良品/不良品/完成量；按过点放行累计 | 过点放行驱动 |
| **TestResult**（测试结果） | AOI/SPI/FCT/烧录/电批结构化结果 | 测试数据接入适配层产出；与过点判定同事务 | FTP/HTTP/手工填报异构渠道 |

### 1.2 读侧缓存/投影（订阅上下文外事件维护）

| 缓存/投影 | 数据来源 | 刷新方式 | 降级策略 | 用途 |
|-----------|---------|---------|---------|------|
| **ProcessRouteCache** | 工艺管理上下文 | 订阅 `ProcessRouteActivated`（秒级） | 缓存未命中 REST 查询制造资源服务（≤500ms） | 过点时读取步序、防错项、跳站规则 |
| **EquipmentAvailabilityCache** | 设备工装台账上下文 | 订阅 `eam.asset.availability`（秒级） | 缓存未命中降级查询台账 | 过点设备准入校验 |
| **QualityGateCache** | 质量上下文 | 订阅质量门禁规则变更 / 启动加载 | REST 查询质量上下文 | 过点质量门禁判定 |
| **KitStatusCache** | 在制品追踪上下文 | 订阅 `wip.kit.status`（秒级） | 缓存未命中降级查询在制品追踪上下文 | 首次过点齐套校验（kit_ready=false 则拦截） |
| **设备实时数据**（按需查询，非缓存） | 设备数据接入上下文 | 过点时按需 REST 查询（≤200ms） | 超时降级为拦截 + 告警 | 过点设备数据校验（如当前温度、当前程序） |

> **缓存一致性原则**（领域总览 §5.1/§5.2）：工艺/设备数据写少读多，本上下文维护本地缓存优先读；缓存未命中允许实时查询作降级路径；工艺版本变更仅影响变更后**首次过点**的在制品，历史过点记录因携带 `routeVersion` 不受影响。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 扫码过点与防错校验（主链）

```
[参与者:操作员] 在工位扫码
        |
        v
【命令:扫码过点 ScanCheckpoint】--> 【聚合:CheckpointRecord】
                                          |
                                          v
                          【事件:过点已扫码 CheckpointScanned】(sn, work_order_id, station_id, equipment_id, scanned_at)
                                          |
                          【策略:扫码后自动执行防错校验】（AntiErrorCheckService 编排）
                                          |
                                          v
                          【命令:执行防错校验 RunAntiErrorChecks】
                                          |
            ┌──── 防错校验维度（并行）────┐
            v                              v
   工艺版本校验                          物料/工装校验
   (ProcessRouteCache 命中?              (BOM 替代料、工装匹配)
    routeVersion 有效?)                            |
            |                              v
            |                       设备准入校验
            |                       (EquipmentAvailabilityCache:
            |                        available=true 且在目标工位?)
            |                              |
            v                              v
   首件门禁校验                          前置工序完成性
   (消费首件处理上下文门禁)              (RoutingProgress: 前序步序已完成?)
            |                              |
            └──────────────┬───────────────┘
                           v
                  【事件:防错已校验 AntiErrorChecked】
                   (sn, station_id, passed: bool, failed_checks[])
                           |
                ┌──────────┴──────────┐
                v                     v
            passed=true            passed=false
                |                     |
                v                     v
   【命令:判定放行 DecideCheckpoint(release)】   【命令:判定拦截 DecideCheckpoint(block)】
                |                     |
                v                     v
   【事件:过点已放行 CheckpointReleased】     【事件:过点已拦截 CheckpointBlocked】
    (sn, route_version, decision=PASS)         (sn, blocking_reason, failed_checks)
                |                     |
                |                     v
                |            【策略:拦截后分流】
                |                ┌────┴────┐
                |                v         v
                |            通知安灯     分流至返修
                |                |   (发布 UnitRoutedToRework, §5)
                |                v
                |            【读模型:拦截看板】
                v
   (进入 §2.1 续: 记录 + 推进 + 累计, 同事务)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| CheckpointScanned | 操作员工位扫码 | sn, work_order_id, station_id, equipment_id, scanned_at | 触发防错校验（INV-01） |
| AntiErrorChecked | 防错校验完成 | sn, station_id, passed, failed_checks[] | passed→判定放行；!passed→判定拦截（INV-02） |
| CheckpointReleased | 综合判定 PASS | sn, route_version, decision=PASS | 同事务记录 + 推进路线 + 累计进度（INV-03） |
| CheckpointBlocked | 综合判定 BLOCK | sn, blocking_reason, failed_checks[] | 分流（安灯/返修）；不计入良品（INV-04） |

> 🔴 **热点：防错校验的同步 vs 异步**——防错校验含设备实时数据 REST 查询（≤200ms）与缓存命中判定，整体需在过点 SLA 内完成。建议防错校验**同步执行**（过点是同步交互），设备实时数据查询超时则计为该校验项 FAIL（保守拦截），避免过点长时间阻塞。待与 SME 确认过点 SLA（建议 ≤1s）。

> 🔴 **热点：拦截后是否自动分流返修**——`CheckpointBlocked` 后是否自动发布 `UnitRoutedToRework`？建议：**仅当 `blocking_reason ∈ {TEST_FAIL, AOI_NG, ...}`（质量类）时自动分流**；设备/物料类拦截走安灯人工处理，不自动进返修。分流判定逻辑由策略引擎执行，具体质量类原因枚举待与质量上下文 / 返修上下文对齐。

### 2.2 状态流转

#### 2.2.1 RoutingProgress（单件工艺路线推进）状态机

```
                 [NOT_STARTED]
                       |
              首次过点放行 AdvanceRouting
                       |
                       v
            ┌──▶ [IN_PROGRESS] ◀── 返修/返工再入 ReenterFromRework/BatchRework ───┐
            │            │                                                         │
            │     过点放行 AdvanceRouting                                          │
            │            │                                                         │
            │            v                                                         │
            │   ┌── 步序校验 ──┐                                                   │
            │   v              v                                                   │
            │ 下一站正确     跳站/漏站                                              │
            │   |              |                                                   │
            │   v              v                                                   │
            │ 推进成功     StationSkipViolation                                     │
            │   |              | (拦截, 不推进)                                     │
            │   v              v                                                   │
            │ [IN_PROGRESS]  [BLOCKED_SKIP]                                        │
            │   |              |                                                   │
            │   |       人工纠正后 ReenterFromSkip                                 │
            │   |              |                                                   │
            │   v              v                                                   │
            │ 最终步序放行                                                        │
            │   |                                                                │
            │   v                                                                │
            └─[COMPLETED] ◀──────────────────────────────────────────────────────┘
                  (终态—单件完成全部步序, INV-05)
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| NOT_STARTED → IN_PROGRESS | AdvanceRouting | 首次过点放行（CheckpointReleased） | 过点引擎（同事务） |
| IN_PROGRESS → IN_PROGRESS（推进） | AdvanceRouting | 过点放行且下一站步序正确（INV-06） | 过点引擎 |
| IN_PROGRESS → BLOCKED_SKIP | CheckStationSequence | 跳站/漏站检出（当前步序 ≠ 预期） | 跳站校验策略 |
| BLOCKED_SKIP → IN_PROGRESS | ReenterFromSkip | 人工纠正（退回正确工位或授权跳站） | 操作员/工程师 |
| IN_PROGRESS → COMPLETED | AdvanceRouting | 最终步序放行（INV-05） | 过点引擎 |
| 任意非终态 → IN_PROGRESS | ReenterFromRework / ReenterFromBatchRework | 消费返修/返工完成再入事件，再入点校验通过（INV-07） | 再入投影消费上下文外事件 |
| → COMPLETED 之后 | （任何推进） | 终态，拒收推进（INV-05） | — |

#### 2.2.2 WorkOrderProgress（工单执行进度）

工单进度不是状态机而是**累积值**，由过点放行事件驱动累计：

```
CheckpointReleased(decision=PASS)
        |
        v
【命令:累计工单进度 AccrueWorkOrderProgress】--> 【聚合:WorkOrderProgress】
        |
        v
【事件:工单进度已累计 WorkOrderProgressAccrued】
 (work_order_id, completed_qty, good_qty, bad_qty, route_version, accrued_at)
        |
        v
发布到 mes.workorder.progress
```

> **进度累计口径**（INV-08）：`good_qty` 仅由 `CheckpointReleased` 累计；`CheckpointBlocked` 计入 `bad_qty`；返修/返工再入后再次过点放行的，按业务规则计入 `reworked_qty`（单独计数，不重复计入 `good_qty`），具体口径待与 SME 确认。

### 2.3 工艺路线缓存与版本一致性

本上下文**不定义**工艺路线与版本（归工艺管理上下文），仅**订阅**其生效事件维护本地缓存，缓存未命中降级 REST 查询。

```
【外部系统:工艺管理上下文】
        -- 发布 ProcessRouteActivated(route_id, route_version, effective_at)
                |
                v
【读模型:ProcessRouteCache】 --【命令:刷新工艺缓存 RefreshProcessRouteCache】
                |
                v
【事件:工艺缓存已刷新 ProcessRouteCacheRefreshed】(route_id, route_version, source_event_id)
                |
                v
后续过点使用新 routeVersion（仅影响变更后首次过点的在制品, INV-09）
```

**降级查询路径**：

```
过点时 ProcessRouteCache 未命中 (route_id, route_version)
        |
        v
【命令:降级查询工艺路线 QueryRouteFallback】--> 【外部系统:制造资源服务(REST, ≤500ms)】
        |
        v
回填 ProcessRouteCache + 继续过点判定
        |
        v
若 REST 超时/失败 --> CheckpointBlocked(blocking_reason=RouteCacheMiss)
```

> **版本一致性不变式**（INV-09，领域总览 §5.1）：工艺版本变更仅影响**变更后首次过点**的在制品；已过点的在制品按其 `CheckpointRecord.route_version` 继续推进，不受新版本影响。`RoutingProgress` 跟随单件首次过点时锁定的 `routeVersion`，后续推进沿用同一版本。

> 🔴 **热点：缓存未命中降级查询的频率**——工艺数据写少读多，正常情况下缓存命中率应 >99%。降级查询是兜底，不应成为常态。建议监控降级查询率，超阈值告警（可能订阅链路异常）。待与平台运维确认监控指标。

### 2.4 设备可用性缓存与过点设备校验

本上下文**不定义**设备可用性事实（归台账上下文），仅**订阅** `eam.asset.availability` 维护本地缓存。

```
【外部系统:设备工装台账上下文】
        -- 发布 AssetAvailabilityChanged(eam.asset.availability)
           (asset_id, available, blocking_reasons[])
                |
                v
【读模型:EquipmentAvailabilityCache】 --【命令:刷新设备状态缓存 RefreshEquipmentAvailabilityCache】
                |
                v
【事件:设备状态缓存已刷新 EquipmentAvailabilityCacheRefreshed】(asset_id, available, source_event_id)
```

**过点设备准入校验**（防错校验维度之一）：

```
过点时校验目标工位设备
        |
        v
EquipmentAvailabilityCache 查询 (equipment_id)
        |
        ├── available=true 且在目标工位 --> 准入通过
        |
        ├── available=false --> CheckpointBlocked(blocking_reason=EquipmentUnavailable)
        |
        └── 缓存未命中 --> 降级查询台账上下文(REST)
                                |
                                ├── 回填缓存 + 重新校验
                                └── 超时 --> CheckpointBlocked(blocking_reason=EquipmentCacheMiss)
```

**过点设备实时数据校验**（按需查询，非缓存）：

```
过点需校验设备实时状态（如当前温度、当前程序版本）
        |
        v
【命令:查询设备实时数据 QueryEquipmentRealtimeData】--> 【外部系统:设备数据接入上下文(REST, ≤200ms)】
        |
        v
返回实时数据点 --> 与工艺要求比较
        |
        ├── 符合 --> 该防错项通过
        ├── 不符 --> CheckpointBlocked(blocking_reason=EquipmentDataOutOfRange)
        └── 超时 --> CheckpointBlocked(blocking_reason=EquipmentDataTimeout)（保守拦截, INV-10）
```

> 🔴 **热点：设备实时数据校验范围**——哪些工艺步序需要在过点时校验设备实时数据（而非仅准入校验）？建议仅**关键工序**（如回流焊温区、波峰焊链速）启用实时数据校验，其余仅做准入校验。校验项配置归工艺管理上下文，本上下文只执行。待与工艺管理上下文 / 质量上下文确认关键工序清单。

> 🔴 **热点：设备数据超时的保守拦截**——设备实时数据查询超时（>200ms）时，是拦截还是放行？建议**保守拦截**（INV-10），避免无数据放行导致质量风险；但需提供工程师**授权放行**通道（带原因记录），防止单点故障阻塞全线。

### 2.5 测试数据接入适配层

测试数据接入适配层是本上下文的**端口适配层**（防腐层性质），从异构渠道汇集数据并结构化为 `TestResult`，与过点判定同事务（领域总览 §5.3）。它**不是独立限界上下文**。

```
┌──── 异构数据源 ────┐
│                   │
│ 【来源1:FTP 文件】 │  AOI/SPI/测试设备按板输出 CSV/XML 到共享目录
│ 【来源2:HTTP 回调】│  烧录电脑/测试电脑通过 HTTP 回调推送结果
│ 【来源3:手工填报】 │  无自动采集的工位，操作员终端填报
│                   │
└────────┬──────────┘
         |
         v
【聚合:TestResult(适配层)】 --【命令:结构化测试结果 StructureTestResult】
         |
         +-- 解析 + 字段映射 + 单位归一化 + 质量打标(原始, 不翻译 PASS/FAIL 业务语义)
         |
         v
【事件:测试结果已结构化 TestResultStructured】
 (test_id, sn, station_id, test_type, raw_verdict, measured_items[], source_channel, source_ts, structured_ts)
         |
         v
落入 TestResult 查询表（供过点引擎同事务查询）
         |
         v
过点引擎查询 TestResult(sn, station_id) --> 作为防错校验/质量门禁输入
```

**适配层职责边界**：

| 职责 | 归属 | 说明 |
|------|------|------|
| 文件/回调解析、字段映射、单位归一化 | **本适配层** | 纯结构化，不做业务判定 |
| 原始判定字段透传（raw_verdict） | **本适配层** | 保留设备原始判定，不翻译 |
| PASS/FAIL 业务语义解释 | **质量上下文** | 缺陷字典、判定规则归质量上下文 |
| TestResult 与过点判定的事务一致性 | **本上下文** | 同事务写入（INV-11） |
| 设备原始信号采集、补传、协议适配 | **设备数据接入上下文** | 本适配层消费其已投递的报文，不重复采集 |

> **与设备数据接入上下文的边界**：设备数据接入上下文负责"原始信号从设备到平台的可靠搬运"（`dc.*`），本适配层负责"把已到达平台的数据结构化为过点可用的 `TestResult`"。两者是**搬运 → 结构化**的接力关系，不重叠。对于 FTP 文件/HTTP 回调这类**不经过设备数据接入上下文**的渠道，本适配层直接对接数据源。

> 🔴 **热点：TestResult 与过点的时间窗匹配**——`TestResult` 写入与过点扫码可能存在时序差（设备出结果晚于扫码，或反之）。建议：过点扫码后开启一个**等待窗口**（如 5s），窗口内 `TestResult` 到达则同事务判定；窗口超时则 `CheckpointBlocked(blocking_reason=TestResultTimeout)`。窗口长度待与 SME 确认。

> 🔴 **热点：补传数据的过点回填**——设备数据接入上下文断连补传后到达的 `TestResult`，若对应过点已拦截关单，是否回填修正？建议：**不回填已关单过点记录**（不可变， INV-12），而是发布 `TestResultLateArrived` 事件供返修/质量上下文评估是否需要补救。待与返修上下文确认。

【命令:执行质量门禁判定 EvaluateQualityGate】

### 2.7 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动缓存刷新、首件门禁、返修/返工再入，**不重发**这些事件——必须包装成本上下文语义的事件（INV-13）。

| 上下文外事件 | 来源限界上下文 | 触发命令 | 状态变更 | 幂等 / 去重策略 |
|---|---|---|---|---|
| ProcessRouteActivated | 工艺管理上下文 | RefreshProcessRouteCache | ProcessRouteCache 刷新 | source_event_id 去重表（INV-09） |
| AssetAvailabilityChanged | 设备工装台账上下文 (`eam.asset.availability`) | RefreshEquipmentAvailabilityCache | EquipmentAvailabilityCache 刷新 | asset_id + version 乐观锁 + source_event_id 去重 |
| FirstArticleReleased | 首件处理上下文 (`fai.article.released`) | EnableBatchProduction（解除首件门禁） | 工单首件门禁状态 → 放行 | work_order_id 去重（INV-14） |
| FirstArticleBlocked | 首件处理上下文 (`fai.article.blocked`) | EnforceFirstArticleGate（强制首件门禁拦截） | 工单首件门禁状态 → 拦截 | work_order_id 去重 |
| UnitReworkCompleted | 返修上下文 (`rework.task.completed`) | ReenterFromRework | RoutingProgress 再入点校验 + 推进 | sn + rework_task_id 去重（INV-07） |
| BatchReworkOrderReleased | 返工上下文 (`brework.order.released`) | ReenterFromBatchRework | 批量再入点校验 + 推进 | rework_order_id + sn 去重 |
| 质量门禁规则变更 | 质量上下文 | RefreshQualityGateCache | QualityGateCache 刷新 | rule_id + version 去重 |
| KitStatusChanged | 在制品追踪上下文 (`wip.kit.status`) | EvaluateKitStatus（首次过点齐套校验） | 工单齐套状态缓存刷新；首次过点 kit_ready=false 则拦截 | work_order_id + source_event_id 去重 |
| UnitScrappedAfterRework | 返修上下文 (`rework.task.scrapped`) | MarkUnitScrapped（标记单件报废，不再推进） | RoutingProgress 标记终态；WorkOrderProgress scrapped_qty 累计 | sn + task_id 去重 |
| BatchReworkOrderScrapped | 返工上下文 (`brework.order.scrapped`) | MarkBatchScrapped（标记批量报废，不再推进） | 批量 RoutingProgress 标记终态；WorkOrderProgress scrapped_qty 累计 | rework_order_id + sn 去重 |

> **再入点校验**（INV-07）：消费 `UnitReworkCompleted(reentry_point)` 后，本上下文校验 `reentry_point` 是否在当前 `routeVersion` 的合法步序内；合法则 `RoutingProgress` 跳转至 `reentry_point` 并标记 `reentered_from=Rework`；非法则发布 `ReentryRejected` 事件，由返修上下文重新判定再入点（读取返修规则与再入点配置，领域总览 §4.2）。

> **降级查询不算"消费事件"**：过点时 ProcessRouteCache / EquipmentAvailabilityCache 未命中触发的 REST 查询是**同步降级路径**，不在本表事件消费范畴，但需保证不引入分布式事务（领域总览 §5.3）。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **CheckpointRecord** | 单次过点记录 | INV-01 必须绑定 SN × 工单 × 工位；INV-12 过点记录不可变，一经写入不可修改（含 TestResult 同事务写入）；INV-03 放行/拦截决策与记录同事务 |
| **CheckpointDecision** | 单次过点判定 | INV-02 BLOCK 决策必带 `blocking_reason` + `failed_checks`；放行需所有防错维度通过 + 质量门禁 PASS + 设备可用 |
| **RoutingProgress** | 单件工艺路线推进 | INV-05 COMPLETED 为终态，之后任何推进均拒收；INV-06 推进需下一站步序正确（或授权跳站）；INV-07 再入需 `reentry_point` 合法；INV-09 跟随单件首次过点锁定的 `routeVersion` |
| **WorkOrderProgress** | 单工单执行进度 | INV-08 `good_qty` 仅由 CheckpointReleased 累计；`bad_qty` 由 CheckpointBlocked 累计；返修再入后放行计入 `reworked_qty`（不重复计入 good） |
| **TestResult**（适配层） | 单件测试结果 | INV-11 TestResult 写入与过点判定同事务（领域总览 §5.3）；保留 `raw_verdict` 原始判定，不翻译业务语义 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-09**（工艺版本一致性，变更仅影响变更后首次过点）→ `ProcessRouteCache` 投影 + `CheckpointRecord.route_version` 快照 + `RoutingProgress` 锁定版本，由过点引擎在首次过点时固化版本保证
> - **INV-10**（设备实时数据超时保守拦截）→ `AntiErrorCheckService` 领域服务 + 查询超时策略
> - **INV-13**（上下文外事件只消费不重发）→ 主题前缀隔离（本上下文只发 `mes.*`，订阅 `eam.*`/工艺管理/`fai.*`/`rework.*`/`brework.*`）+ 事件命名约定
> - **INV-14**（首件门禁幂等）→ `FirstArticleGateProjection` 读模型 + work_order_id 去重表
>
> 注：`ProcessRouteCache` / `EquipmentAvailabilityCache` / `QualityGateCache` 是**读模型/投影**，不是聚合根——它们由订阅上下文外事件维护，最终一致，不承担事务一致性责任。

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|
| CheckpointScanned | 扫码后自动防错校验 | RunAntiErrorChecks（INV-01） |
| AntiErrorChecked(passed=true) + 质量门禁 PASS + 设备可用 | 判定放行 | DecideCheckpoint(release) → CheckpointReleased |
| AntiErrorChecked(passed=false) / 质量门禁 BLOCK/HOLD / 设备不可用 / 跳站违规 | 判定拦截 | DecideCheckpoint(block) → CheckpointBlocked（INV-02） |
| CheckpointReleased | 同事务记录 + 站序校验 + 推进 + 进度累计 + 工装上报 | RecordCheckpoint + CheckStationSequence + AdvanceRouting + AccrueWorkOrderProgress + ReportFixtureUsage（INV-03） |
| CheckpointReleased 且工位绑定工装 | 上报工装使用 | ReportFixtureUsage → 发布 FixtureUsed |
| CheckpointBlocked(blocking_reason ∈ 质量类) | 自动分流返修 | RouteToRework → 发布 UnitRoutedToRework |
| 过点发现设备异常（产线视角） | 上报设备故障 | ReportMalfunction → 发布 MalfunctionReported |
| ProcessRouteCache 未命中 | 降级 REST 查询制造资源服务 | QueryRouteFallback（≤500ms） |
| EquipmentAvailabilityCache 未命中 | 降级 REST 查询台账上下文 | （同步降级） |
| 关键工序过点 | 按需查询设备实时数据 | QueryEquipmentRealtimeData（≤200ms，超时拦截, INV-10） |

> **关键**：本上下文发布 `FixtureUsed` / `MalfunctionReported` / `UnitRoutedToRework` 后，**不直接调用**台账上下文 / 维修上下文 / 返修上下文的接口——通过事件异步解耦。下游上下文消费后自行决策（如台账累积度量、维修创建工单、返修创建任务）。

主题命名约定：
- 本上下文**对外发布**：`mes.<aggregate>.<event-class>`（`mes` = manufacturing execution 缩写），表示"已经过过点执行语义解释"。

> 🔴 **热点：制造资源服务各上下文的主题前缀**——工艺管理上下文与质量上下文的事件风暴尚未编写，其主题前缀（建议 `process.*` / `quality.*`）待制造资源服务限界上下文事件风暴落地后对齐。本上下文订阅 `ProcessRouteActivated` 与质量门禁规则变更事件，前缀以制造资源服务最终约定为准。

## 与工艺管理上下文、质量上下文的协作边界

本上下文是工艺/质量配置数据的**消费方**，通过"本地缓存 + 降级查询"解耦：

| 维度 | 本上下文（过点执行上下文） | 工艺管理上下文 / 质量上下文 |
|------|------------------------|---------------------------|
| 视角 | 执行视角（"过点瞬间怎么判定、怎么记录"） | 配置视角（"工艺怎么定义、质量规则怎么配"） |
| 存储 | 过点记录、路线推进、工单进度、本地缓存 | 工序定义、工艺路线、版本、参数模板、检验标准、缺陷分类、门禁规则 |
| 谁判定 | 执行判定（综合多维度放行/拦截） | 规则定义（不执行过点判定） |
| 事件性质 | 过点执行事实：CheckpointReleased/Blocked | 配置生效事实：ProcessRouteActivated / 质量规则变更 |
| 流程所有权 | 扫码 → 防错 → 决策 → 记录 → 推进 → 累计 | 工序定义 → 路线编排 → 版本生效发布 → 质量规则配置 |
| 版本一致性 | 消费生效事件刷新缓存；过点记录锁定 routeVersion（INV-09） | 发布生效事件为权威源 |

**事件流向**：
- **工艺管理/质量上下文 → 本上下文**：`ProcessRouteActivated`（刷新工艺缓存）、质量门禁规则变更（刷新质量门禁缓存）
- **本上下文 → 工艺管理/质量上下文**：`mes.testresult.structured`（供质量上下文做业务判定/缺陷统计）
- **降级查询**：缓存未命中时本上下文 REST 查询制造资源服务（工艺 ≤500ms），不引入分布式事务

## 与设备工装台账上下文、设备数据接入上下文的协作边界

本上下文是设备可用性与实时数据的**消费方**，工装使用与设备故障的**上报方**：

| 维度 | 本上下文（过点执行上下文） | 设备工装台账上下文 | 设备数据接入上下文 |
|------|------------------------|------------------|-------------------|
| 视角 | 执行视角（"过点时设备能不能用、数据对不对"） | 资产管理视角（"资产能不能用"） | 通信视角（"连接通不通、报文进没进来"） |
| 存储 | 过点记录、设备可用性缓存（投影） | 资产身份/状态/可用性事实源、累积度量 | 原始采集报文、连接状态 |
| 谁判定 | 过点设备准入校验（读缓存） | 资产可用性综合判定（权威源） | 连接通不通、报文是否进来 |
| 事件性质 | 过点执行事实 | 资产生命周期与可用性事实 | 通信与采集事实 |
| 流程所有权 | 扫码 → 设备准入校验 → 实时数据校验 → 决策 | 入账 → 投用 → 停用 → 报废 | 网关绑定 → 协议适配 → 在线监控 → 上行投递 |

**事件流向**：
- **台账上下文 → 本上下文**：`eam.asset.availability`（刷新设备可用性缓存）
- **本上下文 → 台账上下文**：`mes.fixture.used`（台账转换为 `FixtureCycleAccrued` 累积工装使用次数）
- **本上下文 → 维修上下文**：`mes.malfunction.reported`（产线视角设备故障，维修上下文创建工单）
- **设备数据接入上下文 → 本上下文**：过点时 REST 查询设备实时数据（≤200ms，按需非订阅）
- **不互相调用的操作**：本上下文不调用台账上下文的"停用资产"接口——资产状态变更归台账上下文；台账上下文不调用本上下文的"过点"接口——全部通过事件 + 降级查询异步解耦

> 🔴 **热点：设备实时数据查询的归属**——过点设备实时数据校验（如当前温度）REST 查询设备数据接入上下文，还是台账上下文？建议查**设备数据接入上下文**（它持有实时采集数据），台账上下文只提供可用性状态。待与设备管理服务确认查询入口。

## 与首件处理上下文、返修上下文、返工上下文、在制品追踪上下文的协作边界

这四个限界上下文是本上下文在生产执行服务内的紧邻协作方，通过事件契约解耦：

| 维度 | 本上下文（过点执行） | 首件处理上下文 | 返修上下文（单件） | 返工上下文（批量） | 在制品追踪上下文 |
|------|------------------|--------------|----------------|----------------|----------------|
| 视角 | 过站瞬间执行 | 批量放行门禁 | 单件离线返修 | 批量返工工单 | 在制品位置投影 |
| 存储 | 过点记录/路线推进/工单进度 | 首件标识/检验流程/放行门禁 | 返修任务/诊断/再入点 | 返工工单/返工路线 | WIP 身份/位置/流转历史/齐套 |
| 事件性质 | 过点执行事实 | 首件门禁事实 | 返修执行结果 | 返工工单再入 | 位置投影（消费本上下文事件） |
| 流程所有权 | 扫码→防错→决策→记录→推进→累计 | 换线/ECN/换料触发→首件检验→放行门禁 | 分流→诊断→修理→再入点判定→完成 | 返工触发→返工工单→返工路线→批量再入 | 消费过点事件投影位置/齐套 |

**事件流向**：
- **本上下文 → 在制品追踪上下文**：`mes.routing.progress` / `mes.checkpoint.lifecycle`（WIP 投影位置变更）、`mes.unit.routed-to-rework`（WIP 标记离线）、`mes.reentry.reworked`（WIP 标记再入）
- **在制品追踪上下文 → 本上下文**：`wip.kit.status`（齐套状态，首次过点防错校验消费）
- **首件处理上下文 → 本上下文**：`fai.article.released` / `fai.article.blocked`（首件门禁放行/拦截）
- **返修上下文 → 本上下文**：`rework.task.completed`（返修完成再入，含 reentry_point）、`rework.task.scrapped`（单件已报废，不再推进）
- **返工上下文 → 本上下文**：`brework.order.released`（返工工单再入，含 reentry_point）、`brework.order.scrapped`（批量 SN 已报废，不再推进）
- **本上下文 → 返修上下文**：`mes.unit.routed-to-rework`（质量类拦截分流，返修上下文创建任务）
- **本上下文 → 返工上下文**：（无直接事件，返工工单由质量/人工触发，非本上下文发起）
- **不互相调用的操作**：本上下文不创建首件/返修/返工任务，不直接修改 WIP 位置——各限界上下文自治，仅通过事件协作

> 🔴 **热点：首件门禁的作用域**——`FirstArticleReleased` 的作用域是工单级还是产线级？换线后首件放行，是仅放行当前工单，还是放行该产线该产品的所有在制工单？建议**工单级**（按工单 × 产品 × 产线维度），待与首件处理上下文确认门禁粒度。
