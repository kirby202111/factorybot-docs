# 返修上下文（Rework Bounded Context）

> **限界上下文**：返修上下文
> **统一语言**：ReworkTask（返修任务·单件级）, FaultDiagnosis（故障诊断·产品缺陷定位）, RepairAction（返修动作·产品修理执行）, ReworkVerification（返修验证）, ReentryPoint（再入点·修后回主线的工位）, ReworkRule（返修规则·工艺管理上下文只读引用）, ReentryRule（再入点配置·工艺管理上下文只读引用）, ScrapDecision（单件报废决策·返修后判废）
> **核心关注**：以"事件风暴"方式刻画**单件离线返修**——过点拦截后单件被分流至返修区，经故障诊断、修理执行、再入点判定后返回主线再入。本上下文负责"**单件产品怎么修、修后从哪再入、修不好怎么报废**"；**不负责**过点判定与拦截（归过点执行上下文，本上下文消费其分流事件）、不负责批量返工工单（归返工上下文）、**不负责设备维修（归设备管理服务维修上下文 repair\*，那是修机器，本上下文修的是产品单件）**、不负责返修规则与再入点配置的维护（归工艺管理上下文，本上下文只读引用）、不负责在制品位置投影（归在制品追踪上下文）。

---

## 0. 设计原则与边界澄清

返修上下文是生产执行服务下的**单件离线再处理子域**，是 PCBA + 整机组装混合型车间所有"单件返修活动"的执行权威源。它接收过点执行上下文的质量类拦截分流信号，离线完成诊断—修理—再入判定的闭环，再把修后的单件交回过点执行上下文做再入校验与推进。

**上下文内职责**（本上下文负责）：
- **返修任务接单**——消费过点执行上下文 `UnitRoutedToRework` 创建单件级 `ReworkTask`（一个 SN 一个任务），幂等去重
- **故障诊断**——离线定位单件缺陷（结合过点拦截原因 + 质量上下文缺陷分类辅助），记录可修复性判定
- **返修执行**——记录返修动作（补焊/更换元器件/重做工序/调整等），产品级修理而非设备修理
- **返修验证**——修后验证（复测/复检），VERIFIED / FAILED 分流
- **再入点判定**——读取工艺管理上下文的"返修规则与再入点配置"，判定该单件修后应从哪个工位再入主线，随 `UnitReworkCompleted` 发布给过点执行上下文
- **返修后单件报废**——诊断后若判定无法修复 / 不经济，发布 `UnitScrappedAfterRework`（单件报废），不修机器
- **再入点被拒重判**——过点执行上下文拒绝再入点时（`ReentryRejected`），本上下文重新读取规则并重新判定再入点

**上下文外职责**（本上下文**不**负责）：
- 过点判定与拦截决策（放行/拦截） → 过点执行上下文（本上下文仅消费其 `mes.unit.routed-to-rework`，向其发布 `rework.task.completed` / `rework.task.scrapped`）
- 工单执行进度累计 → 过点执行上下文（本上下文不直接扣减 `WorkOrderProgress`，是否扣减由过点执行上下文按 INV-08 口径处理）
- 工艺路线推进与再入校验 → 过点执行上下文（本上下文只提供 `reentry_point`，是否合法由过点执行上下文按 INV-07 校验）
- 批量返工工单、返工工艺路线、批量再入点 → 返工上下文（批量级，与单件返修完全不同）
- 返修规则定义与再入点配置维护 → 工艺管理上下文（本上下文只读引用，REST 查询）
- 缺陷分类字典、检验标准 → 质量上下文（本上下文只读引用辅助诊断）
- 在制品身份、位置投影、流转历史 → 在制品追踪上下文（本上下文只发布完成/报废事件供其投影）
- **设备维修（机器/设备的故障报修→派工→诊断→维修→验证→关单）** → 设备管理服务维修上下文（`repair.*`）——**与本上下文完全不同**，详见 §"与设备管理服务维修上下文的协作边界"
- 设备报废审批与执行 → 设备工装台账上下文（本上下文发布的是单件报废，不调用台账报废审批）

**为什么不把单件返修并入过点执行上下文**：
1. **生命周期与节奏不同**——过点是毫秒级实时判定，返修是分钟到小时级的离线任务流（诊断/修理/验证各有独立 SLA），混入过点引擎会使其职责膨胀、SLA 混乱。
2. **聚合边界清晰**——过点聚合是 `CheckpointRecord` / `RoutingProgress`；返修聚合是 `ReworkTask`（含 `FaultDiagnosis` / `RepairAction` / `ReentryPoint`）。无共用聚合，强合等于耦合。
3. **流程所有权不同**——过点管"过站瞬间的判定与推进"，返修管"离线再处理的诊断—修理—再入判定"。两者通过事件契约解耦即可协作。
4. **离线特性**——返修任务在返修区独立执行，可能跨班次、跨操作员，与过点的在线同步交互模型完全不同。

**为什么不把单件返修并入返工上下文**：
1. **粒度根本不同**——返修是单件级（一个 SN 一个任务，事件驱动接单），返工是订单/批次级（一张返工工单覆盖一批 SN，含返工工艺路线）。强行合并会把"单件任务流"与"批量工单流"两套模型互相污染。
2. **触发机制不同**——返修由过点质量类拦截分流自动触发；返工由质量异常分析/人工决策触发批量再处理。
3. **再入机制不同**——返修再入点是单件级判定（读取返修规则），返工再入点是批量级（返工工单定义返工路线起点）。两者配置与判定逻辑不同。
4. **聚合不同**——返修聚合是 `ReworkTask`，返工聚合是 `BatchReworkOrder`。无共用聚合。

**为什么单件返修与设备维修是两个上下文**（关键边界，避免事件混淆）：
1. **修理对象根本不同**——返修（rework）修的是**产品单件**（SN 级，一块 PCBA / 一台半成品）；维修（repair）修的是**机器/设备**（asset_id 级，一台贴片机/回流焊）。两者前缀不同（`rework.*` vs `repair.*`）、聚合不同（`ReworkTask` vs `RepairOrder`）、职责不同。
2. **触发机制不同**——返修由过点质量类拦截分流触发（`mes.unit.routed-to-rework`）；维修由故障报告触发（产线报修/通信故障/点检异常）。
3. **流程终点不同**——返修终点是"单件修后从再入点回主线"（`rework.task.completed`）或"单件报废"（`rework.task.scrapped`）；维修终点是"设备修复恢复可用"（`repair.order.completed` → 台账 ReinstateAsset）或"设备报废"（`repair.scrap.recommendation` → 台账报废审批）。
4. **报废语义不同**——返修后报废是**单件产品报废**（一个 SN 报废，不影响设备）；维修后报废是**设备资产报废**（一台设备退役，需台账审批）。两者事件名、主题前缀、消费方均不同，**严禁混用**。

## 1. 主语清单与触发入口

返修上下文的入口是上下文外事件 `UnitRoutedToRework`（来自过点执行上下文），统一建模单件返修任务。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 上下文外触发源

```
┌──────────────────────────────────────────────────────────────────┐
│ 返修任务入口                                                      │
│                                                                  │
│  【来源1:过点执行上下文 (mes.*)】                                  │
│    UnitRoutedToRework（质量类拦截分流，单件被送到返修区）           │
│    → 本上下文消费后创建 ReworkTask（幂等：source_event_id 去重）   │
│                                                                  │
│  【来源2:过点执行上下文 (mes.*)】                                  │
│    ReentryRejected(reason=InvalidReentryPoint)                    │
│    → 本上下文发布的 reentry_point 被过点执行拒绝，重新判定再入点   │
│      （读取返修规则与再入点配置）                                   │
│                                                                  │
│  【来源3:工艺管理上下文 (process.*，REST 只读)】                   │
│    返修规则与再入点配置（非事件订阅，按需 REST 查询）               │
│    → 诊断后判定可修复性、修理后判定再入点时读取                     │
│                                                                  │
│  【来源4:质量上下文（REST 只读）】                                 │
│    缺陷分类字典（辅助诊断，非事件订阅）                             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 聚合根主语清单

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **ReworkTask**（返修任务·单件级） | 过点拦截分流触发的单件返修任务 | 一个 SN 一个任务；携带 source_event_id 幂等去重；状态单向流转 | 过点执行上下文 `UnitRoutedToRework` |
| **FaultDiagnosis**（故障诊断） | 首次诊断、再入点被拒后重诊断 | 定位产品缺陷根因；记录可修复性判定；驱动修理/报废分流 | 由 ReworkTask 触发 |
| **RepairAction**（返修动作） | 补焊、更换元器件、重做工序、调整 | 关联 IN_REPAIR 任务；记录产品级修理（非设备修理） | 由返修工程师执行 |
| **ReworkVerification**（返修验证） | 复测、复检 | 关联有 RepairAction 的任务；VERIFIED/FAILED 分流 | 由返修工程师/质检执行 |
| **ReentryPoint**（再入点判定） | 修后回主线的工位判定 | 读取工艺管理上下文配置；随完成事件发布给过点执行 | 由 ReworkTask 触发 |

**返修任务模型**：

```
ReworkTask
├── task_id          : 返修任务唯一标识
├── sn               : 单件序列号（一个 SN 一个任务, INV-01）
├── work_order_id    : 来源工单
├── defect_reason    : 缺陷原因（来自过点拦截事件）
├── source_station   : 拦截工位（来自过点拦截事件）
├── source_event_id  : 来源事件 ID（UnitRoutedToRework 事件 ID, 幂等去重）
├── status           : CREATED | DIAGNOSED | IN_REPAIR | PENDING_REENTRY | COMPLETED | SCRAPPED
├── diagnosis        : FaultDiagnosis?（诊断结论）
├── repair_actions[] : RepairAction[]（返修动作列表）
├── verification     : ReworkVerification?（验证结论）
├── reentry_point    : ReentryPoint?（再入点判定）
├── created_at       : 创建时间
└── completed_at     : 完成时间（COMPLETED/SCRAPPED 时填）
```

> 🔴 **热点：返修任务的 SLA**——单件返修是否有响应/修复 SLA？建议：**有**——按缺陷严重度分级（如关键缺陷 4 小时、一般缺陷 24 小时），超时升级到返修主管。SLA 阈值是否归工艺管理上下文还是本上下文配置，待与 SME 确认。

> 🔴 **热点：返修任务是否关联工单进度扣减**——单件分流至返修后，`WorkOrderProgress.bad_qty` 是否立即 +1？返修完成再入后再次过点放行是否计入 `reworked_qty`？建议：**本上下文不直接扣减**——进度累计口径归过点执行上下文（其 INV-08 已定义：分流计入 bad_qty，再入后放行计入 reworked_qty 不重复计入 good_qty）。本上下文只发布完成/报废事件，过点执行上下文消费后按其口径累计。待与过点执行上下文确认事件载荷是否需携带 qty 提示字段。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 接单与诊断（主链第一段）

```
【外部系统:过点执行上下文】
        -- 发布 UnitRoutedToRework(sn, from_station, defect_reason, work_order_id)
                |
                v
【策略:消费分流事件自动接单】（幂等：source_event_id 去重, INV-02）
                |
                v
【命令:接返修单 AcceptReworkTask】--> 【聚合:ReworkTask】
                                        |
                                        v
                                【事件:返修任务已创建 ReworkTaskCreated】
                                 (task_id, sn, defect_reason, source_station, source_event_id)
                                        |
                                        v
                                发布到 rework.task.created
                                        |
                                        v
【参与者:返修工程师】
        --【命令:开始诊断 StartDiagnosis】--> 【聚合:FaultDiagnosis】
                                                |
                                ┌──── 读取辅助 ────┐
                                v                   v
                  【外部系统:质量上下文】     过点拦截 defect_reason
                  （缺陷分类字典, REST 只读）  （已在任务上）
                                |                   |
                                └───────┬───────────┘
                                        v
        --【命令:记录诊断结论 RecordDiagnosisResult】-->
                                        |
                                        v
                                【事件:诊断结论已记录 DiagnosisResultRecorded】
                                 (diagnosis_id, task_id, fault_cause, repair_plan,
                                  repair_feasibility: REPAIRABLE | NOT_ECONOMICAL | NOT_REPAIRABLE)
                                        |
                            ┌──── repair_feasibility ────┐
                            v                v                   v
                      REPAIRABLE       NOT_ECONOMICAL      NOT_REPAIRABLE
                            |                |                   |
                            v                v                   v
                    进入返修执行       发起单件报废          发起单件报废
                                     （§2.4 报废分支）    （§2.4 报废分支）
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| ReworkTaskCreated | 消费分流事件接单 | task_id, sn, defect_reason, source_station, source_event_id | 触发诊断（INV-02 幂等去重） |
| DiagnosisResultRecorded | 诊断完成 | diagnosis_id, task_id, fault_cause, repair_plan, repair_feasibility | REPAIRABLE→修理；NOT_ECONOMICAL/NOT_REPAIRABLE→报废（INV-03） |

> 🔴 **热点：诊断是否需要复测确认缺陷**——过点拦截的 `defect_reason` 是否可直接采信进入修理，还是返修区需复测确认？建议：**关键缺陷（如 FCT 不良）需返修区复测确认**，避免误判导致无效修理；一般缺陷（如 AOI 标记的焊点不良）可直接采信。复测确认是否作为独立的 `DefectConfirmed` 事件，待与质量上下文确认缺陷确认流程。

> 🔴 **热点：可修复性判定的阈值**——"返修不经济"的判定标准（如返修成本 vs 单件价值）不在本上下文硬编码，应由工程标准/财务规则定义。本上下文只记录诊断结论和返修预估成本，判定逻辑由策略引擎执行（参考维修上下文同款热点，但语义是产品而非资产）。

### 2.2 状态流转

```
[CREATED] ──DiagnosisResultRecorded(REPAIRABLE)──▶ [DIAGNOSED] ──StartRepair──▶ [IN_REPAIR] ──ReworkVerificationSubmitted(VERIFIED)──▶ [PENDING_REENTRY] ──CompleteReworkTask──▶ [COMPLETED]
                                                                                                                                                    (终态—单件修后回主线, INV-05)

[DIAGNOSED] ──RecommendScrap──▶ [PENDING_SCRAP_DECISION] ──ScrapApproved──▶ [SCRAPPED]
                                                                 (终态—单件报废, INV-05)

[PENDING_REENTRY] ──ReentryRejected(来自过点执行)──▶ [IN_REPAIR]（回退, 重新判定再入点, INV-06）
                          ▲
                          │
                          └─ ReentryPointRejected 消费 mes.reentry.rejected 后触发 ReevaluateReentryPoint

[IN_REPAIR] ──ReworkVerificationSubmitted(FAILED)──▶ [IN_REPAIR]（回退继续修理, INV-07）
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| CREATED → DIAGNOSED | RecordDiagnosisResult(repair_feasibility=REPAIRABLE) | 当前 CREATED；诊断结论已记录（INV-03） | 返修工程师 |
| DIAGNOSED → IN_REPAIR | StartRepair | 当前 DIAGNOSED；feasibility=REPAIRABLE | 返修工程师 |
| IN_REPAIR → PENDING_REENTRY | SubmitReworkVerification(result=VERIFIED) + DetermineReentryPoint | 当前 IN_REPAIR；至少一条 RepairAction（INV-04）；再入点已判定 | 返修工程师 / 质检 + 再入点判定策略 |
| PENDING_REENTRY → COMPLETED | CompleteReworkTask | 当前 PENDING_REENTRY；再入点未被过点执行拒绝 | 系统（再入点确认后自动完成） |
| DIAGNOSED → PENDING_SCRAP_DECISION | RecommendScrap | 当前 DIAGNOSED；feasibility ∈ {NOT_ECONOMICAL, NOT_REPAIRABLE} | 返修工程师 |
| PENDING_SCRAP_DECISION → SCRAPPED | ApproveScrap | 单件报废审批通过（INV-08） | 返修主管 / 质量主管 |
| PENDING_SCRAP_DECISION → DIAGNOSED | RejectScrapRediagnosis | 报废审批驳回/撤回（INV-09） | 报废审批投影消费审批结果后触发 |
| PENDING_REENTRY → IN_REPAIR | ReevaluateReentryPoint | 消费过点执行 `ReentryRejected`，重新判定再入点（INV-06） | 再入点重判策略 |
| IN_REPAIR → IN_REPAIR（回退） | ResumeRepair | 当前 IN_REPAIR；验证 FAILED 且带原因（INV-07） | 返修工程师 |
| → COMPLETED / SCRAPPED 之后 | （任何） | 终态，拒收（INV-05） | — |

> **可逆边说明**：仅 `ReevaluateReentryPoint`（再入点被拒回退重判）、`ResumeRepair`（验证失败回退继续修理）、`RejectScrapRediagnosis`（报废驳回回退诊断）允许回退，其余状态单向流转（INV-10）。

> 🔴 **热点：再入点被拒后的回退粒度**——`ReentryRejected` 后是回退到 `IN_REPAIR` 还是单独的 `PENDING_REENTRY_REEVAL` 状态？建议：回退到 `IN_REPAIR`（复用状态，简化状态机），通过 `reentry_reeval_count` 字段记录重判次数防止死循环（如重判 3 次仍被拒则升级人工处理）。待与过点执行上下文确认再入点校验失败的常见原因，是否多数可通过重判解决。

**返修动作模型**：

```
RepairAction
├── action_id    : 动作唯一标识
├── task_id      : 关联返修任务
├── action_type  : REPAIR（补焊/修复） | REPLACE_COMPONENT（更换元器件） | REDO_PROCESS（重做工序） | ADJUST（调整） | OTHER
├── description  : 动作描述
├── performed_by : 执行人
├── started_at   : 开始时间
├── completed_at : 完成时间
└── evidence_urls[] : 照片/视频证据
```

> **关键区分**：本上下文的 `RepairAction` 是**产品级修理**（修一块 PCBA 的焊点/换一个元器件），与设备管理服务维修上下文的 `RepairAction`（修一台设备、可能消耗备件）**完全不同**——前缀不同（`rework.*` vs `repair.*`）、聚合归属不同、动作类型不同。本上下文的返修动作**不消耗备件库存**（元器件由返修区物料管理，不归备件管理上下文），若涉及元器件领用另走物料上下文，不在本上下文展开。

> 🔴 **热点：返修区物料/元器件管理**——返修更换的元器件是否需要领用记录与库存扣减？建议：**需要**——但归物料上下文（或返修区物料子域），本上下文只在 `RepairAction` 记录 `parts_replaced[]` 用于单件维修历史追溯，不直接扣减库存。待与物料上下文确认返修区物料管理模式。

> 🔴 **热点：返修动作是否需关联原工位/设备**——返修动作记录是否需追溯到是在哪台返修设备上做的（如返修工作站/BGA 返修台）？建议：**记录** `repair_station_id`（返修区工位），便于质量追溯与返修设备利用率分析。但不与设备工装台账上下文的 asset_id 强绑定（返修设备可能不纳入台账管理），待与台账上下文确认返修设备是否纳入资产台账。

**再入点判定模型**：

```
ReentryPoint
├── task_id          : 关联返修任务
├── sn               : 单件序列号
├── reentry_point    : 再入工位（来自返修规则与再入点配置）
├── rule_id          : 命中的返修规则 ID
├── route_id         : 工艺路线 ID
├── route_version    : 工艺路线版本
├── defect_reason    : 原缺陷原因（用于规则匹配）
├── repair_result    : 返修结果（REPAIRED，用于规则匹配）
├── determined_at    : 判定时间
└── reeval_count     : 重判次数（防止 ReentryRejected 死循环）
```

> 🔴 **热点：返修规则与再入点配置的具体形态**——工艺管理上下文的"返修规则与再入点配置"尚未定义事件风暴，其数据结构（如 `ReworkRule(route_id, defect_reason, repair_action_type) → reentry_point`）与查询接口形态未确定。建议：本上下文定义一个 `ReworkRuleQuery` 端口接口（输入 route_id/route_version/defect_reason/repair_result，输出 reentry_point + rule_id），具体实现由工艺管理上下文提供。待工艺管理上下文事件风暴落地后对齐。

> 🔴 **热点：再入点判定的默认回退**——若返修规则未命中（如该缺陷原因无配置再入点），如何处理？建议：**回退到拦截工位的下一工位**（保守再入，避免跳站），并发布 `ReentryPointFallbackApplied` 事件记录默认回退，便于工艺管理上下文补全规则。待与工艺管理上下文确认默认回退策略。

> 🔴 **热点：返修是否总能回到原工艺路线**——某些严重返修（如更换主芯片后需重新走全流程）是否需指定再入点为路线起点而非拦截工位？建议：返修规则支持 `reentry_point = ROUTE_START`（回路线起点重做全流程）作为特殊配置，由工艺管理上下文维护。待与工艺管理上下文确认是否支持该配置项。

### 2.6 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动接单与再入点重判，**不重发**这些事件——必须包装成本上下文语义的事件（INV-11）。返修规则与再入点配置通过 REST 查询工艺管理上下文，不属于事件订阅范畴。

| 上下文外事件 | 来源限界上下文 | 触发命令 | 状态变更 | 幂等 / 去重策略 |
|---|---|---|---|---|
| UnitRoutedToRework | 过点执行上下文 (`mes.unit.routed-to-rework`) | AcceptReworkTask | 新建 ReworkTask（CREATED） | source_event_id 去重表（INV-02） |
| ReentryRejected(reason=InvalidReentryPoint) | 过点执行上下文 (`mes.reentry.rejected`) | ReevaluateReentryPoint | PENDING_REENTRY → IN_REPAIR（回退重判再入点, INV-06） | task_id + rework_task_id 去重（防止同一拒绝重复触发） |

**再入点重判流程**（消费 `ReentryRejected`）：

```
【外部系统:过点执行上下文】
        -- 发布 ReentryRejected(sn, reason=InvalidReentryPoint, rework_task_id)
                |
                v
【策略:再入点被拒后重新判定】（幂等：task_id 去重, INV-06）
                |
                v
【命令:重新判定再入点 ReevaluateReentryPoint】--> 【聚合:ReentryPoint】
                |
                v
        reeval_count + 1
                |
                v
重新 REST 查询返修规则与再入点配置
（可能命中不同规则或回退默认再入点）
                |
                ├── reeval_count < 阈值（如 3）且新再入点判定成功
                |       |
                |       v
                |   【事件:再入点已重判 ReentryPointReevaluated】
                |    (task_id, sn, new_reentry_point, reeval_count)
                |       |
                |       v
                |   重新 CompleteReworkTask → 发布 UnitReworkCompleted(new_reentry_point)
                |   （过点执行上下文再次校验）
                |
                └── reeval_count ≥ 阈值 或 仍无合法再入点
                        |
                        v
                    【事件:再入点重判已耗尽 ReentryPointExhausted】
                     (task_id, sn, reeval_count)
                        |
                        v
                    升级人工处理（返修主管介入），任务停留在 IN_REPAIR
```

> 🔴 **热点：再入点重判的死循环防护**——`ReentryRejected` 后重判再入点，若新再入点仍被过点执行拒绝，可能形成死循环。建议：**reeval_count 阈值=3**，超过则发布 `ReentryPointExhausted` 升级人工处理，不再自动重判。阈值是否可配置待与 SME 确认。

> 🔴 **热点：ReentryRejected 的常见原因**——过点执行上下文拒绝再入点的常见原因（`reason=InvalidReentryPoint`）是什么？是再入点不在当前 routeVersion 合法步序内，还是 routeVersion 已变更导致原再入点失效？建议：过点执行上下文在 `ReentryRejected` 载荷中补充 `rejection_detail`（如 RouteVersionChanged / StepNotFound），便于本上下文针对性重判。待与过点执行上下文确认载荷扩展。

> 🔴 **热点：工艺版本变更对在制返修任务的影响**——单件在返修区期间，工艺路线版本发生变更（`ProcessRouteActivated`），原再入点判定是否仍有效？建议：**返修任务在 CREATED 时锁定 routeVersion**（参考过点执行上下文 INV-09 模式），再入点判定基于锁定版本；若过点执行上下文因版本变更拒绝再入点，本上下文重判时读取**最新版本**的返修规则。待与过点执行上下文确认再入点校验是用单件锁定的 routeVersion 还是最新版本。

## 3. 事件风暴 — 返修验证子流程

返修验证是返修完成前的质量门，验证通过才能进入再入点判定。

```
[参与者:返修工程师/质检]
        --【命令:提交返修验证 SubmitReworkVerification】--> 【聚合:ReworkVerification】
                                                            |
                                                            v
                                                    【事件:返修验证已提交 ReworkVerificationSubmitted】
                                                     (verification_id, task_id, method, result: VERIFIED | FAILED)
                                                            |
                                                ┌──── 验证结果 ────┐
                                                v                   v
                                            VERIFIED           FAILED
                                                |                   |
                                                v                   v
                                    【命令:判定再入点           【命令:继续返修
                                     DetermineReentryPoint】     ResumeRepair】
                                                |                   |
                                                v                   v
                                    进入 PENDING_REENTRY        回到 IN_REPAIR
                                    （§2.5 完成流程）            （INV-07）
```

**验证方式**：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| RETEST | 返修区复测（同工位测试设备复跑） | 电气/功能类返修 |
| REINSPECT | 返修区复检（AOI/X-Ray 复检） | 焊点/外观类返修 |
| LINE_RETRY | 回主线工位复测（再入点测试） | 需主线设备验证的返修 |

> 🔴 **热点：LINE_RETRY 与再入点的关系**——`LINE_RETRY` 验证方式与"再入点判定"是否重叠？建议：**不重叠**——`LINE_RETRY` 是返修区**主动选择**让单件回主线某工位复测作为验证手段（验证失败则不完成再入）；再入点判定是验证通过后**根据规则**确定修后从哪再入主线。两者顺序：先验证（可能 LINE_RETRY），通过后再判再入点。若 LINE_RETRY 的工位与再入点恰好一致，可合并流程。待与 SME 确认是否合并。

> 🔴 **热点：返修验证失败的重试上限**——`ResumeRepair` 回退后可无限次重试吗？建议：**重试上限=3 次**，超过则升级返修主管评估是否转报废（`repair_feasibility` 重判为 NOT_REPAIRABLE）。待与 SME 确认重试上限。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **ReworkTask** | 单件返修任务 | INV-01 一个 SN 同时只能有一个未终态的 ReworkTask（同一 SN 不并行返修）；INV-05 COMPLETED / SCRAPPED 为终态，之后任何事件均拒收；INV-10 状态单向流转，仅 ReevaluateReentryPoint / ResumeRepair / RejectScrapRediagnosis 允许回退 |
| **FaultDiagnosis** | 单次诊断记录 | INV-03 一个 ReworkTask 可多次诊断（首次诊断后报废驳回可再诊断）；每次诊断必须记录 repair_feasibility |
| **RepairAction** | 单个返修动作 | INV-04 必须关联一个 IN_REPAIR 状态的 ReworkTask；产品级修理，不消耗备件库存（与设备维修的 RepairAction 区分） |
| **ReworkVerification** | 单次验证 | INV-07 必须关联一个有 RepairAction 记录的 ReworkTask；FAILED 验证必须有原因说明 |
| **ReentryPoint** | 单次再入点判定 | INV-06 再入点必须来自工艺管理上下文的返修规则配置（或默认回退）；reeval_count 超阈值则升级人工，不再自动重判 |
| **ScrapDecision** | 单件报废决策 | INV-08 单件报废需审批通过才能 SCRAPPED；INV-09 报废驳回/撤回后任务退回 DIAGNOSED 重新评估 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同一 UnitRoutedToRework 不重复创建返修任务）→ `ReworkTaskService` 领域服务 + source_event_id 事件去重表
> - **INV-11**（上下文外事件只消费不重发）→ 主题前缀隔离（本上下文只发 `rework.*`，订阅 `mes.*`；严禁发 `repair.*` 或 `mes.*`）+ 事件命名约定
> - **INV-12**（单件报废与设备报废不混淆）→ 事件名区分（`UnitScrappedAfterRework` vs `ScrapRecommendation`）+ 主题前缀区分（`rework.task.scrapped` vs `repair.scrap.recommendation`）+ 消费方区分（过点执行/WIP vs 台账上下文）
> - **INV-13**（再入点配置一致性）→ 本上下文 REST 查询工艺管理上下文为权威源，本地不缓存规则（每次判定实时查询，避免配置漂移）；若性能要求需缓存，待与工艺管理上下文确认缓存失效机制

### N.1 上下文外事件 → 本上下文响应

### N.2 本上下文内部策略

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|
| ReworkTaskCreated | 触发诊断 | StartDiagnosis |
| DiagnosisResultRecorded(feasibility=REPAIRABLE) | 进入返修执行 | StartRepair |
| DiagnosisResultRecorded(feasibility ∈ {NOT_ECONOMICAL, NOT_REPAIRABLE}) | 建议单件报废 | RecommendScrap |
| RepairActionPerformed（至少一条）+ 验证通过 | 判定再入点 | DetermineReentryPoint |
| ReentryPointDetermined | 完成返修任务 | CompleteReworkTask → 发布 UnitReworkCompleted |
| ReworkVerificationSubmitted(result=FAILED) | 退回继续修理 | ResumeRepair（INV-07，重试上限 3 次） |
| ScrapRecommended | 单件报废审批 | ApproveScrap / RejectScrapRediagnosis |
| ScrapApproved | 单件报废完成 | 发布 UnitScrappedAfterRework |
| ReentryRejected（来自过点执行） | 重新判定再入点 | ReevaluateReentryPoint（reeval_count 阈值 3, INV-06） |
| ReentryPointExhausted | 升级人工处理 | 通知返修主管（任务停留 IN_REPAIR） |

| `rework.verification` | ReworkVerificationSubmitted(task_id, method, result) | 数据应用上下文（返修质量统计） |

主题命名约定：
- 本上下文**对外发布**：`rework.<aggregate>.<event-class>`（`rework` 为本上下文缩写，**与设备管理服务维修上下文的 `repair` 前缀严格区分**），表示"已经过单件返修语义解释"。
- 本上下文**订阅**：使用各发布限界上下文自己的前缀，不重写——`mes.*`（过点执行上下文）、工艺管理上下文（`process.*` 或制造资源服务约定前缀，REST 查询为主）、质量上下文（REST 查询为主）。

> 🔴 **热点：工艺管理上下文/质量上下文的主题前缀**——工艺管理上下文与质量上下文的事件风暴尚未编写，其主题前缀（建议 `process.*` / `quality.*`）待制造资源服务限界上下文事件风暴落地后对齐。本上下文对这两个上下文以 REST 查询为主，事件订阅较少，前缀影响有限。

## 与过点执行上下文的协作边界

本上下文与过点执行上下文是"离线返修"与"在线过点"的接力关系，通过事件契约解耦。这是本上下文最紧密的协作方，必须对齐避免事件混淆。

| 维度 | 本上下文（返修上下文） | 过点执行上下文 |
|------|------------------|--------------|
| 视角 | 单件离线返修视角（"单件怎么修、修后从哪再入"） | 在线过点执行视角（"过站瞬间怎么判定、怎么推进"） |
| 存储 | 返修任务 / 诊断 / 返修动作 / 验证 / 再入点判定 | 过点记录 / 路线推进 / 工单进度 / 本地缓存 |
| 谁判定 | 缺陷可修复性判定、再入点判定（读返修规则配置） | 过点放行/拦截决策、再入点合法性校验（INV-07） |
| 事件性质 | 返修执行结果事实：UnitReworkCompleted / UnitScrappedAfterRework | 过点执行事实：UnitRoutedToRework / ReentryRejected |
| 流程所有权 | 接单 → 诊断 → 修理 → 验证 → 再入点判定 → 完成/报废 | 扫码 → 防错 → 决策 → 记录 → 推进 → 累计；再入校验 → 推进 |

**事件流向**：

```
过点执行上下文 ──mes.unit.routed-to-rework──▶ 本上下文（创建返修任务）
本上下文 ──rework.task.completed──▶ 过点执行上下文（再入点校验 + RoutingProgress 推进）
本上下文 ──rework.task.scrapped──▶ 过点执行上下文（单件已报废，不再回主线）
过点执行上下文 ──mes.reentry.rejected──▶ 本上下文（再入点被拒，重新判定）
```

**职责切分要点**：
- **本上下文管**：单件离线修复（诊断/修理/验证）+ 再入点判定（读规则配置给出 reentry_point）
- **过点执行上下文管**：再入点合法性校验（reentry_point 是否在 routeVersion 合法步序内, INV-07）+ 路线推进 + 工单进度累计
- **不互相调用的操作**：本上下文不直接修改 `RoutingProgress`（归过点执行）；过点执行上下文不直接创建 `ReworkTask`（归本上下文）。全部通过事件异步解耦。
- **再入点的"判定"与"校验"分离**：本上下文**判定**再入点（基于返修规则配置），过点执行上下文**校验**再入点（基于当前 routeVersion 步序合法性）。两者职责不重叠——本上下文不持有 routeVersion 的步序定义，无法自校验；过点执行上下文不持有返修规则，无法自判定。

> 🔴 **热点：再入点判定与校验的职责切分是否过细**——本上下文判定再入点、过点执行校验再入点，是否会导致频繁 `ReentryRejected` 往返？建议：本上下文判定时尽量读取与过点执行**同一份** routeVersion 步序信息（REST 查询工艺管理上下文时一并校验步序合法性），减少往返。但最终校验权仍在过点执行（它是路线推进的权威源）。待与过点执行上下文确认是否可在判定阶段预校验。

## 与设备管理服务维修上下文的协作边界（关键区分表）

本上下文（返修 rework）与设备管理服务维修上下文（维修 repair）**名称相似但完全不同**，必须严格区分，避免事件名、主题前缀、聚合、消费方混淆。这张表是防止"rework 与 repair 混淆"的核心边界。

| 维度 | 本上下文（返修上下文 rework） | 设备管理服务维修上下文（repair） |
|------|------------------|--------------|
| **修理对象** | **产品单件**（SN 级，一块 PCBA / 一台半成品） | **机器/设备**（asset_id 级，一台贴片机/回流焊/测试设备） |
| **服务前缀** | `rework.*` | `repair.*` |
| **聚合根** | ReworkTask（含 FaultDiagnosis / RepairAction / ReentryPoint） | RepairOrder（含 FaultDiagnosis / RepairAction / RepairVerification） |
| **触发机制** | 过点质量类拦截分流（`mes.unit.routed-to-rework`） | 故障报告（产线报修/通信故障/点检异常，多来源） |
| **流程起点** | 接单（消费分流事件） | 报修 → 派工 |
| **流程终点** | 单件修后从再入点回主线（`rework.task.completed`）或单件报废（`rework.task.scrapped`） | 设备修复恢复可用（`repair.order.completed` → 台账 ReinstateAsset）或设备报废建议（`repair.scrap.recommendation` → 台账审批） |
| **报废语义** | **单件产品报废**（一个 SN 报废，事件 `UnitScrappedAfterRework`，主题 `rework.task.scrapped`，消费方：过点执行/WIP） | **设备资产报废建议**（一台设备退役，事件 `ScrapRecommendation`，主题 `repair.scrap.recommendation`，消费方：台账上下文审批） |
| **备件消耗** | 不消耗备件库存（元器件由返修区物料管理，归物料上下文） | 消耗备件库存（`PartsConsumed` → 备件管理上下文扣减） |
| **SLA 性质** | 单件返修 SLA（小时级，影响工单良率） | 设备维修 SLA（可能停线，CRITICAL 分钟级） |
| **所属服务** | 生产执行服务 | 设备管理服务 |
| **与生产的关系** | 直接影响工单良率与在制品流转 | 间接影响（设备不可用导致过点拦截） |

**关键澄清**：
- 两者**无直接事件往来**——本上下文不订阅 `repair.*`，维修上下文不订阅 `rework.*`。它们通过过点执行上下文间接关联（过点执行既消费返修完成事件，又发布设备故障上报给维修上下文）。
- **事件名严格区分**：`UnitReworkCompleted` / `UnitScrappedAfterRework`（本上下文，单件级）vs `RepairCompleted` / `ScrapRecommendation`（维修上下文，设备级）。**严禁混用或互相引用**。
- **RepairAction 同名不同义**：两个上下文都有 `RepairAction` 聚合，但本上下文的是产品级修理动作（补焊/换元器件），维修上下文的是设备级维修动作（更换设备部件/调整设备参数）。通过主题前缀（`rework.action.*` vs `repair.*`）区分。

> 🔴 **热点：返修过程中发现设备故障的联动**——返修工程师在返修设备（如 BGA 返修台）上发现设备本身故障时，是否需上报到维修上下文？建议：**是**——但走**过点执行上下文的 `MalfunctionReported`**（产线视角报修，`mes.malfunction.reported`）或返修区独立报修通道，**本上下文不直接发布 `repair.*` 事件**。返修设备故障上报与单件返修任务解耦，各自独立处理。待与过点执行/维修上下文确认返修区设备故障上报通道。

## 与返工上下文、在制品追踪上下文、工艺管理上下文的协作边界

这四个限界上下文是本上下文在生产执行服务内及制造资源服务内的协作方，通过事件契约与 REST 查询解耦：

| 维度 | 本上下文（返修） | 返工上下文（批量） | 在制品追踪上下文 | 工艺管理上下文 |
|------|----------------|----------------|----------------|--------------|
| 视角 | 单件离线返修 | 批量返工工单 | 在制品位置投影 | 工艺配置定义 |
| 粒度 | 单件级（一个 SN 一个任务） | 订单/批次级（一张工单覆盖一批 SN） | 单件/批次位置 | 工序/路线/规则配置 |
| 存储 | 返修任务/诊断/再入点 | 返工工单/返工路线 | WIP 身份/位置/流转历史 | 工序定义/路线/返修规则/再入点配置 |
| 事件性质 | 单件返修执行结果 | 批量返工工单再入 | 位置投影（消费本上下文事件） | 配置生效事实（本上下文 REST 查询） |
| 流程所有权 | 接单→诊断→修理→再入点判定→完成 | 返工触发→返工工单→返工路线→批量再入 | 消费过点/返修事件投影位置 | 工序定义→路线编排→版本生效→返修规则配置 |
| 与本上下文关系 | — | 互不直接事件往来（粒度不同） | 消费本上下文完成/报废事件投影 WIP | 本上下文 REST 查询其返修规则配置 |

**事件流向**：
- **本上下文 → 在制品追踪上下文**：`rework.task.created`（WIP 标记离线返修）、`rework.task.completed`（WIP 标记再入）、`rework.task.scrapped`（WIP 标记报废）
- **本上下文 → 工艺管理上下文**：（无事件，REST 查询返修规则与再入点配置）
- **本上下文 → 返工上下文**：（无直接事件往来，单件返修与批量返工各自独立）
- **不互相调用的操作**：本上下文不创建返工工单，不直接修改 WIP 位置，不修改工艺规则配置——各限界上下文自治，仅通过事件 + REST 查询协作

> 🔴 **热点：单件返修与批量返工的边界场景**——当一批 SN 因同一缺陷需返修时，是逐个走单件返修（本上下文）还是开批量返工工单（返工上下文）？建议：**按缺陷影响范围判定**——若缺陷是单件偶发（如个别焊点不良），走单件返修；若缺陷是批次性/系统性（如某工序参数偏差导致整批不良），开批量返工工单。判定归质量上下文/人工决策，本上下文只接收已判定为单件返修的分流。待与返工上下文/质量上下文确认边界判定机制。
