# 返工上下文（Batch Rework Bounded Context）

> **限界上下文**：返工上下文
> **统一语言**：BatchReworkOrder（返工工单·批次/订单级）, ReworkRoute（返工工艺路线·工艺管理上下文只读引用）, ReentryPoint（批量再入点·整批回主线的工位）, ReworkTrigger（返工触发·质量异常/工程变更/人工决策）, ReworkProgress（返工进度·按 SN 维度累计）, ScrapDecision（批量报废决策·返工后整批或部分判废）
> **核心关注**：以"事件风暴"方式刻画**批量返工**——当一批产品出现共性质量问题（批次性/系统性缺陷），生成订单/批次级返工工单，定义返工工艺路线，批量再入主线。本上下文负责"**一批产品怎么返工、走什么返工路线、从哪批量再入**"；**不负责**过点判定与拦截（归过点执行上下文，本上下文向其发布 `brework.order.released` 由其做再入校验+推进）、**不负责单件返修**（归返修上下文 `rework.*`，单件级离线修复，过点拦截分流触发，与本上下文粒度/触发/聚合完全不同）、**不负责设备维修**（归设备管理服务维修上下文 `repair.*`，修机器而非修产品）、不负责工艺路线定义与版本生命周期（归工艺管理上下文，本上下文只读引用返工工艺路线）、不负责在制品位置投影（归在制品追踪上下文，本上下文只发布再入事件供其投影）。

---

## 0. 设计原则与边界澄清

返工上下文是生产执行服务下的**批量再处理子域**，是 PCBA + 整机组装混合型车间所有"订单/批次级返工活动"的执行权威源。它接收批量质量异常/工程变更/人工决策触发，生成返工工单并定义返工工艺路线，返工完成后整批从指定再入点回主线，把批量再入信号交给过点执行上下文做再入校验与推进。

**上下文内职责**（本上下文负责）：
- **返工工单生成**——消费质量上下文批量异常/工程变更/人工决策，生成 `BatchReworkOrder`（一张工单覆盖一批 SN），关联源工单 + SN 清单 + 返工工艺路线 + 再入点
- **返工工艺路线引用**——从工艺管理上下文 REST 查询/引用返工工艺路线（与正常工艺路线不同，返工走独立路线），返工工单按返工路线推进
- **批量再入点定义**——返工工单定义整批回主线的再入点（`reentry_point`），随 `brework.order.released` 发布给过点执行上下文
- **返工进度跟踪**——按 SN 维度跟踪返工进度（已返工/待返工/已报废数量），不重复过点执行上下文的工单进度累计
- **批量再入放行**——返工完成后发布 `brework.order.released`，触发过点执行上下文批量再入校验+推进
- **返工后批量报废**——若返工后整批或部分 SN 判废，发布 `brework.order.scrapped`（批量报废）
- **再入点被拒重判**——过点执行上下文拒绝批量再入点时（`mes.reentry.rejected`），本上下文重新判定再入点

**上下文外职责**（本上下文**不**负责）：
- 过点判定与拦截决策（放行/拦截） → 过点执行上下文（本上下文向其发布 `brework.order.released`，由其按 INV-07 做再入校验+推进）
- 工单执行进度累计（good/bad/reworked qty） → 过点执行上下文（本上下文不直接扣减 `WorkOrderProgress`，进度口径归过点执行 INV-08）
- 工艺路线推进与再入校验 → 过点执行上下文（本上下文只提供 `reentry_point`，是否合法由过点执行校验）
- 单件离线返修任务、单件再入点判定 → 返修上下文（`rework.*`，单件级，过点拦截分流触发，详见 §"与返修上下文的协作边界"）
- 返工工艺路线定义与版本生命周期 → 工艺管理上下文（本上下文只读引用，REST 查询）
- 缺陷分类字典、质量异常判定 → 质量上下文（本上下文消费其批量异常信号，不重判质量）
- 在制品身份、位置投影、流转历史 → 在制品追踪上下文（本上下文发布创建/报废事件供其直接投影，再入事件通过过点执行上下文中转）
- **设备维修（机器/设备的故障修复）** → 设备管理服务维修上下文（`repair.*`）——**与本上下文完全不同**，详见 §"与设备管理服务维修上下文的协作边界"
- 设备报废审批与执行 → 设备工装台账上下文（本上下文发布的是批量产品报废，不调用台账报废审批）

**为什么不把批量返工并入过点执行上下文**：
1. **生命周期与节奏不同**——过点是毫秒级实时判定，批量返工是小时到天级的离线工单流（生成/路线分配/进度跟踪/批量再入各有独立节奏），混入过点引擎会使其职责膨胀。
2. **聚合边界清晰**——过点聚合是 `CheckpointRecord` / `RoutingProgress`；返工聚合是 `BatchReworkOrder`（含 `ReworkRoute` / `ReentryPoint` / `ReworkProgress`）。无共用聚合，强合等于耦合。
3. **流程所有权不同**——过点管"过站瞬间的判定与推进"，返工管"批量再处理的工单生成—路线引用—进度跟踪—批量再入"。两者通过事件契约解耦即可协作。
4. **批量特性**——返工工单覆盖一批 SN，含返工工艺路线与批量再入点，与过点的单件在线同步交互模型完全不同。

**为什么不把批量返工并入单件返修上下文（关键边界）**：
1. **粒度根本不同**——返修（`rework.*`）是单件级（一个 SN 一个任务，过点拦截分流触发）；返工（`brework.*`）是订单/批次级（一张返工工单覆盖一批 SN，质量异常/工程变更/人工决策触发）。强行合并会把"单件任务流"与"批量工单流"两套模型互相污染。
2. **触发机制不同**——返修由过点质量类拦截分流自动触发（`mes.unit.routed-to-rework`）；返工由批量质量异常分析/工程变更/人工决策触发批量再处理（消费质量上下文批量异常信号或人工指令）。
3. **再入机制不同**——返修再入点是单件级判定（读取返修规则配置）；返工再入点是批量级（返工工单定义返工路线起点/再入点，整批统一再入）。两者配置与判定逻辑不同。
4. **聚合不同**——返修聚合是 `ReworkTask`（单件），返工聚合是 `BatchReworkOrder`（批量）。无共用聚合。
5. **服务前缀不同**——`rework.*` vs `brework.*`，事件名、主题、消费方严格区分，**严禁混用**。

**为什么批量返工与设备维修是两个上下文**（关键边界，避免事件混淆）：
1. **修理对象根本不同**——返工（brework）修的是**产品批量**（一批 SN，一批 PCBA/半成品）；维修（repair）修的是**机器/设备**（asset_id 级）。两者前缀不同（`brework.*` vs `repair.*`）、聚合不同（`BatchReworkOrder` vs `RepairOrder`）、职责不同。
2. **流程终点不同**——返工终点是"整批从再入点回主线"（`brework.order.released`）或"批量报废"（`brework.order.scrapped`）；维修终点是"设备修复恢复可用"（`repair.order.completed` → 台账 ReinstateAsset）或"设备报废建议"（`repair.scrap.recommendation` → 台账审批）。
3. **报废语义不同**——返工后报废是**批量产品报废**（一批 SN 报废）；维修后报废是**设备资产报废建议**（一台设备退役，需台账审批）。两者事件名、主题前缀、消费方均不同，**严禁混用**。

## 1. 主语清单与触发入口

返工上下文的入口是上下文外触发（质量异常/工程变更/人工决策），统一建模为 `BatchReworkOrder`。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 上下文外触发源

```
┌──────────────────────────────────────────────────────────────────┐
│ 返工工单入口                                                      │
│                                                                  │
│  【来源1:质量上下文】                                             │
│    批量质量异常信号（批次性/系统性缺陷，如某工序参数偏差          │
│    导致整批不良）                                                 │
│    → 本上下文消费后评估生成 BatchReworkOrder                     │
│      （🔴 质量上下文批量异常事件未定义，热点标注）                │
│                                                                  │
│  【来源2:工程变更】                                               │
│    ECN 触发已生产批次返工（如 BOM 变更/工艺修正需返工在制品）     │
│    → 本上下文消费后生成 BatchReworkOrder                         │
│      （🔴 ECN 事件来源未定义，热点标注）                          │
│                                                                  │
│  【来源3:人工决策】                                               │
│    生产主管/质量主管/工程师通过终端决策批量返工                   │
│    → 本上下文接收指令生成 BatchReworkOrder                       │
│                                                                  │
│  【来源4:过点执行上下文 (mes.*)】                                  │
│    ReentryRejected(reason=InvalidReentryPoint)                   │
│    → 本上下文发布的 reentry_point 被过点执行拒绝，重新判定再入点 │
│                                                                  │
│  【来源5:工艺管理上下文 (process.*，REST 只读)】                  │
│    返工工艺路线（非事件订阅，按需 REST 查询）                     │
│    → 生成返工工单时引用返工路线，进度跟踪时读取路线步序           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 聚合根主语清单

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **BatchReworkOrder**（返工工单·批次/订单级） | 质量异常返工、ECN 返工、人工决策返工 | 一张工单覆盖一批 SN；关联源工单+返工路线+再入点；状态单向流转 | 质量上下文 / 工程变更 / 人工决策 |
| **ReworkRoute**（返工工艺路线·只读引用） | 返工独立工艺路线 | 从工艺管理上下文 REST 查询引用；与正常工艺路线不同；返工工单按返工路线推进 | 工艺管理上下文（REST 只读） |
| **ReentryPoint**（批量再入点） | 整批回主线的工位判定 | 返工工单定义的批量再入点；随 `brework.order.released` 发布给过点执行 | 由 BatchReworkOrder 触发 |
| **ReworkProgress**（返工进度） | 按 SN 维度的返工进度累计 | 已返工/待返工/已报废数量；不重复过点执行的工单进度 | 由 BatchReworkOrder 驱动 |

**返工工单模型**：

```
BatchReworkOrder
├── rework_order_id     : 返工工单唯一标识
├── source_work_order_id: 源工单（原生产工单）
├── sn_list[]           : 返工 SN 清单（一批 SN, INV-01）
├── rework_route_id     : 返工工艺路线 ID（引用工艺管理上下文）
├── rework_route_version: 返工工艺路线版本（锁定, INV-09）
├── reentry_point       : 批量再入点（整批回主线的工位）
├── reason              : 返工原因（BatchDefect / ECN / ManualDecision / ...）
├── trigger_source      : 触发来源（QUALITY / ECN / MANUAL）
├── source_event_id     : 来源事件 ID（幂等去重，若来自事件）
├── status              : CREATED | ROUTE_ASSIGNED | IN_REWORK | PENDING_REENTRY | RELEASED | COMPLETED | SCRAPPED
├── progress            : ReworkProgress（返工进度）
├── created_at          : 创建时间
└── released_at         : 放行时间（RELEASED 时填）
```

> 🔴 **热点：返工工单的 SN 清单来源**——返工 SN 清单是来自源工单的全部 SN、还是质量上下文标记的不良 SN 子集、还是人工圈定的 SN 范围？建议：**三者皆可**——`trigger_source=QUALITY` 时由质量上下文提供不良 SN 清单；`trigger_source=ECN` 时由源工单的全部在制 SN；`trigger_source=MANUAL` 时由人工圈定。SN 清单的准确性由触发方保证，本上下文只接收并固化。待与质量上下文确认批量异常信号是否携带 SN 清单。

> 🔴 **热点：返工工单是否支持部分 SN 报废**——一批 SN 返工后，部分判废部分再入，还是整批统一处理？建议：**支持部分报废**——`ReworkProgress` 区分 `reworked_qty` / `scrapped_qty` / `pending_qty`，返工完成后部分 SN 走 `brework.order.scrapped`、部分走 `brework.order.released`。但状态机简化为整批统一状态，部分报废通过进度字段表达，不拆分多张工单。待与 SME 确认是否需拆单。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 返工工单生成与路线分配（主链第一段）

```
【外部系统:质量上下文 / 工程变更 / 人工决策】
        -- 触发批量返工（批量异常信号 / ECN / 人工指令）
                |
                v
【策略:消费触发信号评估生成返工工单】（幂等：source_event_id 去重, INV-02）
                |
                v
【命令:生成返工工单 CreateBatchReworkOrder】--> 【聚合:BatchReworkOrder】
        |
        ┌──── 读取辅助 ────┐
        v                   v
【外部系统:工艺管理上下文】  源工单 + SN 清单
（返工工艺路线, REST 只读）  （来自触发信号）
        |                   |
        └───────┬───────────┘
                v
        【事件:返工工单已创建 BatchReworkOrderCreated】
         (rework_order_id, source_work_order_id, sn_list[], rework_route_id,
          reentry_point, reason, trigger_source, source_event_id)
                |
                v
        发布到 brework.order.created
                |
                v
【策略:返工工单创建后自动分配返工路线】
                |
                v
【命令:分配返工路线 AssignReworkRoute】--> 【聚合:BatchReworkOrder】
                |
                v
        【事件:返工路线已分配 ReworkRouteAssigned】
         (rework_order_id, rework_route_id, rework_route_version)
                |
                v
        进入 IN_REWORK 状态，开始按 SN 维度跟踪返工进度
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| BatchReworkOrderCreated | 消费触发信号生成返工工单 | rework_order_id, source_work_order_id, sn_list[], rework_route_id, reentry_point, reason, trigger_source, source_event_id | 自动分配返工路线（INV-02 幂等去重） |
| ReworkRouteAssigned | 返工路线分配完成 | rework_order_id, rework_route_id, rework_route_version | 进入 IN_REWORK，开始返工进度跟踪 |

> 🔴 **热点：返工触发来源的事件形态**——质量上下文的"批量质量异常信号"与工程变更的"ECN 返工触发"事件尚未定义。建议：本上下文定义一个 `ReworkTriggerPort` 端口接口（输入 source_work_order_id/sn_list/reason/trigger_source，输出是否生成返工工单），具体触发事件由质量上下文/工程变更模块提供。待质量上下文与工程变更模块事件风暴落地后对齐触发事件契约。

> 🔴 **热点：返工工艺路线在工艺管理上下文的形态**——工艺管理上下文的"返工工艺路线"尚未定义事件风暴，其数据结构（如 `ReworkRoute(route_id, steps[], reentry_point)`）与查询接口形态未确定。建议：本上下文定义一个 `ReworkRouteQuery` 端口接口（输入 source_route_id/reason，输出 rework_route_id + rework_route_version + steps + reentry_point），具体实现由工艺管理上下文提供。返工工艺路线是否与正常工艺路线同表管理还是独立管理，待工艺管理上下文确认。待工艺管理上下文事件风暴落地后对齐。

> 🔴 **热点：返工工艺路线的版本锁定**——返工工单在 ROUTE_ASSIGNED 时锁定 `rework_route_version`（参考过点执行上下文 INV-09 模式），后续返工进度跟踪与再入点判定基于锁定版本；若返工工艺路线在返工期间变更，已生成返工工单不受影响（沿用锁定版本），新返工工单用新版本。待与工艺管理上下文确认返工工艺路线版本管理机制。

### 2.2 状态流转

```
[CREATED] ──AssignReworkRoute──▶ [ROUTE_ASSIGNED] ──StartBatchRework──▶ [IN_REWORK] ──CompleteBatchRework──▶ [PENDING_REENTRY] ──ReleaseBatchRework──▶ [RELEASED] ──CompleteReworkOrder──▶ [COMPLETED]
                                                                                                                                                                                (终态—整批回主线, INV-05)

[IN_REWORK] ──RecommendBatchScrap──▶ [PENDING_SCRAP_DECISION] ──ScrapApproved──▶ [SCRAPPED]
                                                                      (终态—批量报废, INV-05)

[PENDING_REENTRY] ──ReentryRejected(来自过点执行)──▶ [IN_REWORK]（回退, 重新判定再入点, INV-06）
                          ▲
                          │
                          └─ ReentryPointRejected 消费 mes.reentry.rejected 后触发 ReevaluateReentryPoint

[RELEASED] ──ReentryRejected(来自过点执行, 部分SN再入失败)──▶ [IN_REWORK]（回退, 重新判定再入点, INV-06）
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| CREATED → ROUTE_ASSIGNED | AssignReworkRoute | 当前 CREATED；返工工艺路线已从工艺管理上下文查询成功并锁定版本（INV-03） | 系统（工单创建后自动分配） |
| ROUTE_ASSIGNED → IN_REWORK | StartBatchRework | 当前 ROUTE_ASSIGNED；返工路线已分配 | 系统/返工主管（开始批量返工） |
| IN_REWORK → PENDING_REENTRY | CompleteBatchRework | 当前 IN_REWORK；所有待返工 SN 已完成返工（`pending_qty=0`, INV-04）；再入点已定义 | 系统（进度完成后自动转态）/ 返工主管 |
| PENDING_REENTRY → RELEASED | ReleaseBatchRework | 当前 PENDING_REENTRY；再入点未被过点执行拒绝 | 系统（再入点确认后自动放行） |
| RELEASED → COMPLETED | CompleteReworkOrder | 当前 RELEASED；过点执行上下文已消费 `brework.order.released` 并完成批量再入（`BatchReworkReentered`） | 系统（消费过点执行再入完成信号后关单） |
| IN_REWORK → PENDING_SCRAP_DECISION | RecommendBatchScrap | 当前 IN_REWORK；返工后判定整批或部分 SN 无法修复/不经济 | 返工主管 / 质量主管 |
| PENDING_SCRAP_DECISION → SCRAPPED | ApproveBatchScrap | 批量报废审批通过（INV-08） | 返工主管 / 质量主管 |
| PENDING_SCRAP_DECISION → IN_REWORK | RejectScrapRework | 报废审批驳回/撤回（INV-09） | 报废审批投影消费审批结果后触发 |
| PENDING_REENTRY → IN_REWORK | ReevaluateReentryPoint | 消费过点执行 `ReentryRejected`，重新判定再入点（INV-06） | 再入点重判策略 |
| RELEASED → IN_REWORK | ReevaluateReentryPoint | 消费过点执行 `ReentryRejected`（部分 SN 再入失败），重新判定再入点（INV-06） | 再入点重判策略 |
| → COMPLETED / SCRAPPED 之后 | （任何） | 终态，拒收（INV-05） | — |

> **可逆边说明**：仅 `ReevaluateReentryPoint`（再入点被拒回退重判）、`RejectScrapRework`（报废驳回回退返工）允许回退，其余状态单向流转（INV-10）。

> 🔴 **热点：RELEASED 状态的语义**——`RELEASED` 表示本上下文已发布 `brework.order.released` 给过点执行上下文，但过点执行尚未完成批量再入。`COMPLETED` 才表示整批已成功再入主线。是否需要 `RELEASED` 这个中间状态，还是直接从 `PENDING_REENTRY` 跳到 `COMPLETED`？建议：**保留 RELEASED**——区分"本上下文已放行"与"过点执行已再入完成"，便于跟踪再入进度与处理再入失败回退。待与过点执行上下文确认是否回传再入完成信号（`BatchReworkReentered`）以驱动 RELEASED→COMPLETED。

> 🔴 **热点：部分 SN 再入失败的处理**——整批再入时，若部分 SN 再入点校验失败（`ReentryRejected`），是整批回退还是仅失败 SN 回退？建议：**整批回退到 IN_REWORK 重新判定再入点**（简化状态机），通过 `ReworkProgress` 记录失败 SN 列表，重判后再次整批放行。若频繁部分失败，考虑拆分返工工单。待与过点执行上下文确认批量再入是逐 SN 校验还是整批校验。

### 2.3 返工进度跟踪

返工进度按 SN 维度累计，由返工区的返工动作驱动（返工动作本身归返修上下文或返工区执行，本上下文只跟踪进度）。

```
[IN_REWORK] 状态下
        |
        v
【参与者:返工区操作员/系统】
        --【命令:记录返工进度 RecordReworkProgress】--> 【聚合:ReworkProgress】
        |
        v
【事件:返工进度已累计 ReworkProgressAccrued】
 (rework_order_id, sn, reworked_qty, pending_qty, scrapped_qty, accrued_at)
        |
        v
发布到 brework.order.progress
        |
        v
【策略:所有待返工 SN 完成后自动转态】
        |
        ├── pending_qty > 0 --> 继续 IN_REWORK
        |
        └── pending_qty = 0 --> 【命令:完成批量返工 CompleteBatchRework】
                                        |
                                        v
                                进入 PENDING_REENTRY（§2.2）
```

**返工进度模型**：

```
ReworkProgress
├── rework_order_id    : 关联返工工单
├── total_qty          : 返工 SN 总数
├── reworked_qty       : 已返工 SN 数
├── scrapped_qty       : 已报废 SN 数
├── pending_qty        : 待返工 SN 数（= total - reworked - scrapped）
├── sn_status[]        : 各 SN 的返工状态（PENDING / REWORKED / SCRAPPED）
└── last_accrued_at    : 最后累计时间
```

> 🔴 **热点：返工进度跟踪的粒度**——返工进度是按 SN 维度逐个跟踪，还是按数量批量累计？建议：**按 SN 维度跟踪**（`sn_status[]` 记录每个 SN 的状态），便于精确追溯哪些 SN 已返工、哪些待返工、哪些报废。但若批量大（如 1000+ SN），SN 维度跟踪可能有性能压力，待与 SME 确认典型批量规模与跟踪粒度。

> 🔴 **热点：返工动作的归属**——批量返工的"返工动作"（如整批补焊/整批重做工序）是归本上下文还是返修上下文？建议：**返工动作归返修上下文或返工区执行系统**，本上下文只跟踪进度（消费返修上下文的 `UnitReworkCompleted` 或返工区执行系统的进度上报）。本上下文不重复建模返工动作的细节（诊断/修理/验证），那是单件级执行，归返修上下文。批量返工工单是"工单级编排+进度跟踪"，单件返修任务是"单件级执行"。待与返修上下文确认是否消费其完成事件驱动本上下文进度累计。

> 🔴 **热点：返工进度与工单进度的关系**——本上下文的 `ReworkProgress`（返工工单维度）与过点执行上下文的 `WorkOrderProgress`（源工单维度）是两个独立进度，不混淆。返工进度累计"这批 SN 返工完成多少"，工单进度累计"源工单完成多少良品/不良品"。返工完成再入后，过点执行上下文按 INV-08 口径将再入放行的 SN 计入 `reworked_qty`（不重复计入 good_qty）。待与过点执行上下文确认再入后进度累计口径。

        |
        v
【事件:返工工单已放行 BatchReworkOrderReleased】
 (rework_order_id, reentry_point, sn_list[], rework_route_version)
        |

发布到 brework.order.released
        |
        v
【外部系统:过点执行上下文】订阅

### 2.6 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动再入点重判，**不重发**这些事件——必须包装成本上下文语义的事件（INV-11）。返工工艺路线通过 REST 查询工艺管理上下文，不属于事件订阅范畴。

| 上下文外事件 | 来源限界上下文 | 触发命令 | 状态变更 | 幂等 / 去重策略 |
|---|---|---|---|---|
| ReentryRejected(reason=InvalidReentryPoint) | 过点执行上下文 (`mes.reentry.rejected`) | ReevaluateReentryPoint | PENDING_REENTRY / RELEASED → IN_REWORK（回退重判再入点, INV-06） | rework_order_id 去重（防止同一拒绝重复触发） |
| 批量质量异常信号 | 质量上下文（🔴 事件未定义） | CreateBatchReworkOrder | 新建 BatchReworkOrder（CREATED） | source_event_id 去重表（INV-02） |
| ECN 返工触发 | 工程变更（🔴 来源未定义） | CreateBatchReworkOrder | 新建 BatchReworkOrder（CREATED） | source_event_id 去重表（INV-02） |
| BatchReworkReentered（再入完成回传） | 过点执行上下文 (`mes.reentry.reworked`, 🔴 是否回传统一) | CompleteReworkOrder | RELEASED → COMPLETED | rework_order_id 去重（INV-07） |

**再入点重判流程**（消费 `ReentryRejected`）：

```
【外部系统:过点执行上下文】
        -- 发布 ReentryRejected(rework_order_id, reason=InvalidReentryPoint)
                |
                v
【策略:再入点被拒后重新判定】（幂等：rework_order_id 去重, INV-06）
                |
                v
【命令:重新判定再入点 ReevaluateReentryPoint】--> 【聚合:BatchReworkOrder】
                |
                v
        reeval_count + 1
                |
                v
重新 REST 查询返工工艺路线的再入点配置
（可能命中不同再入点或回退默认再入点）
                |
                ├── reeval_count < 阈值（如 3）且新再入点判定成功
                |       |
                |       v
                |   【事件:再入点已重判 ReentryPointReevaluated】
                |    (rework_order_id, new_reentry_point, reeval_count)
                |       |
                |       v
                |   重新 ReleaseBatchRework → 发布 BatchReworkOrderReleased(new_reentry_point)
                |   （过点执行上下文再次校验）
                |
                └── reeval_count ≥ 阈值 或 仍无合法再入点
                        |
                        v
                    【事件:再入点重判已耗尽 ReentryPointExhausted】
                     (rework_order_id, reeval_count)
                        |
                        v
                    升级人工处理（返工主管介入），工单停留在 IN_REWORK
```

> 🔴 **热点：再入点重判的死循环防护**——`ReentryRejected` 后重判再入点，若新再入点仍被过点执行拒绝，可能形成死循环。建议：**reeval_count 阈值=3**，超过则发布 `ReentryPointExhausted` 升级人工处理，不再自动重判。阈值是否可配置待与 SME 确认。参考返修上下文同款热点。

> 🔴 **热点：ReentryRejected 的载荷扩展**——过点执行上下文拒绝批量再入点的常见原因（`reason=InvalidReentryPoint`）是什么？是再入点不在当前 routeVersion 合法步序内，还是 routeVersion 已变更导致原再入点失效？建议：过点执行上下文在 `ReentryRejected` 载荷中补充 `rejection_detail`（如 RouteVersionChanged / StepNotFound），便于本上下文针对性重判。待与过点执行上下文确认载荷扩展。参考返修上下文同款热点。

> 🔴 **热点：工艺版本变更对在制返工工单的影响**——批量返工期间，工艺路线版本发生变更（`ProcessRouteActivated`），原再入点判定是否仍有效？建议：**返工工单在 ROUTE_ASSIGNED 时锁定 `rework_route_version`**（参考过点执行上下文 INV-09 模式），再入点判定基于锁定版本；若过点执行上下文因版本变更拒绝再入点，本上下文重判时读取**最新版本**的返工工艺路线。待与过点执行上下文确认再入点校验是用返工工单锁定的 routeVersion 还是最新版本。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **BatchReworkOrder** | 单个返工工单 | INV-01 一张返工工单覆盖一批 SN（sn_list 非空）；INV-05 COMPLETED / SCRAPPED 为终态，之后任何事件均拒收；INV-10 状态单向流转，仅 ReevaluateReentryPoint / RejectScrapRework 允许回退 |
| **ReworkRoute**（只读引用） | 返工工艺路线引用 | INV-03 返工工艺路线必须从工艺管理上下文 REST 查询获得，本上下文不缓存路线（每次引用实时查询，避免配置漂移）；ROUTE_ASSIGNED 时锁定 `rework_route_version` |
| **ReentryPoint** | 批量再入点判定 | INV-06 再入点必须来自工艺管理上下文的返工工艺路线配置（或默认回退）；reeval_count 超阈值则升级人工，不再自动重判 |
| **ReworkProgress** | 返工工单进度累计 | INV-04 `pending_qty=0` 才能从 IN_REWORK 转 PENDING_REENTRY（所有待返工 SN 已终态：返工或报废）；`reworked_qty + scrapped_qty + pending_qty = total_qty` |
| **ScrapDecision** | 批量报废决策 | INV-08 批量报废需审批通过才能 SCRAPPED；INV-09 报废驳回/撤回后工单退回 IN_REWORK 重新评估 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-02**（同一触发信号不重复创建返工工单）→ `BatchReworkOrderService` 领域服务 + source_event_id 事件去重表
> - **INV-07**（再入点合法性校验）→ 过点执行上下文 `RoutingProgress` 校验 `reentry_point` 是否在 routeVersion 合法步序内（本上下文只判定不校验，校验权归过点执行）
> - **INV-11**（上下文外事件只消费不重发）→ 主题前缀隔离（本上下文只发 `brework.*`，订阅 `mes.*`；严禁发 `rework.*` / `repair.*` / `mes.*`）+ 事件命名约定
> - **INV-12**（批量报废与单件报废/设备报废不混淆）→ 事件名区分（`BatchReworkOrderScrapped` vs `UnitScrappedAfterRework` vs `ScrapRecommendation`）+ 主题前缀区分（`brework.order.scrapped` vs `rework.task.scrapped` vs `repair.scrap.recommendation`）+ 消费方区分（过点执行/WIP vs 过点执行/WIP vs 台账上下文）
> - **INV-13**（返工工艺路线一致性）→ 本上下文 REST 查询工艺管理上下文为权威源，本地不缓存路线（每次引用实时查询，避免配置漂移）；`rework_route_version` 在 ROUTE_ASSIGNED 时锁定

### N.2 本上下文内部策略

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|

## 与过点执行上下文的协作边界

本上下文与过点执行上下文是"批量返工"与"在线过点"的接力关系，通过事件契约解耦。这是本上下文最紧密的协作方，必须对齐避免事件混淆。

| 维度 | 本上下文（返工上下文） | 过点执行上下文 |
|------|------------------|--------------|
| 视角 | 批量离线返工视角（"一批怎么返工、走什么返工路线、从哪批量再入"） | 在线过点执行视角（"过站瞬间怎么判定、怎么推进"） |
| 存储 | 返工工单 / 返工路线引用 / 返工进度 / 批量再入点 | 过点记录 / 路线推进 / 工单进度 / 本地缓存 |
| 谁判定 | 返工工艺路线引用（读工艺管理配置）、批量再入点判定（读返工路线配置） | 过点放行/拦截决策、再入点合法性校验（INV-07） |
| 事件性质 | 批量返工执行结果事实：BatchReworkOrderReleased / BatchReworkOrderScrapped | 过点执行事实：ReentryRejected / BatchReworkReentered（🔴 是否回传） |
| 流程所有权 | 触发→返工工单→返工路线→进度跟踪→批量再入放行→完成/报废 | 扫码→防错→决策→记录→推进→累计；批量再入校验→推进 |

**事件流向**：

```
本上下文 ──brework.order.released──▶ 过点执行上下文（批量再入校验 + RoutingProgress 推进）
本上下文 ──brework.order.scrapped──▶ 过点执行上下文（批量 SN 已报废，不再回主线）
过点执行上下文 ──mes.reentry.rejected──▶ 本上下文（再入点被拒，重新判定）
过点执行上下文 ──BatchReworkReentered（🔴 是否回传）──▶ 本上下文（再入完成，驱动 RELEASED→COMPLETED）
```

**职责切分要点**：
- **本上下文管**：批量返工工单生成（关联源工单+SN清单+返工路线）+ 返工工艺路线引用 + 返工进度跟踪 + 批量再入点判定（读返工路线配置给出 reentry_point）
- **过点执行上下文管**：再入点合法性校验（reentry_point 是否在 routeVersion 合法步序内, INV-07）+ 路线推进 + 工单进度累计
- **不互相调用的操作**：本上下文不直接修改 `RoutingProgress`（归过点执行）；过点执行上下文不直接创建 `BatchReworkOrder`（归本上下文）。全部通过事件异步解耦。
- **再入点的"判定"与"校验"分离**：本上下文**判定**批量再入点（基于返工工艺路线配置），过点执行上下文**校验**再入点（基于当前 routeVersion 步序合法性）。两者职责不重叠——本上下文不持有 routeVersion 的步序定义，无法自校验；过点执行上下文不持有返工工艺路线，无法自判定。

> 🔴 **热点：批量再入与单件再入的校验差异**——过点执行上下文 §5.2 提到"返工再入 BatchReworkOrderReleased 同构，发布 BatchReworkReentered / ReentryRejected"，但批量再入是逐 SN 推进还是整批推进？建议：**整批校验再入点合法性，逐 SN 推进 RoutingProgress**（每个 SN 独立跳转至 reentry_point）。若整批校验失败则发布 `ReentryRejected`（不带具体 SN），本上下文整批重判。待与过点执行上下文确认批量再入校验与推进粒度。

## 与返修上下文的协作边界（关键区分表）

本上下文（返工 brework）与返修上下文（返修 rework）**名称相似但完全不同**，必须严格区分，避免事件名、主题前缀、聚合、消费方混淆。这张表是防止"brework 与 rework 混淆"的核心边界。

| 维度 | 本上下文（返工上下文 brework） | 返修上下文（rework） |
|------|------------------|--------------|
| **粒度** | **订单/批次级**（一张返工工单覆盖一批 SN） | **单件级**（一个 SN 一个任务） |
| **服务前缀** | `brework.*` | `rework.*` |
| **聚合根** | BatchReworkOrder（含 ReworkRoute / ReentryPoint / ReworkProgress） | ReworkTask（含 FaultDiagnosis / RepairAction / ReentryPoint） |
| **触发机制** | 批量质量异常/工程变更/人工决策（消费质量上下文批量异常信号或人工指令） | 过点质量类拦截分流（`mes.unit.routed-to-rework`，单件自动触发） |
| **流程起点** | 生成返工工单（消费批量触发信号） | 接单（消费单件分流事件） |
| **工艺路线** | 返工走**独立返工工艺路线**（从工艺管理上下文引用，与正常路线不同） | 返修走**返修规则配置**（单件级再入点判定，读返修规则） |
| **再入机制** | 批量再入点（返工工单定义，整批统一再入） | 单件再入点（读取返修规则配置，单件判定） |
| **流程终点** | 整批从再入点回主线（`brework.order.released`）或批量报废（`brework.order.scrapped`） | 单件修后从再入点回主线（`rework.task.completed`）或单件报废（`rework.task.scrapped`） |
| **报废语义** | **批量产品报废**（一批 SN 报废，事件 `BatchReworkOrderScrapped`，主题 `brework.order.scrapped`） | **单件产品报废**（一个 SN 报废，事件 `UnitScrappedAfterRework`，主题 `rework.task.scrapped`） |
| **进度跟踪** | 返工工单维度（`ReworkProgress`，按 SN 维度累计返工进度） | 单件任务维度（`ReworkTask` 状态机） |
| **所属服务** | 生产执行服务 | 生产执行服务 |

**关键澄清**：
- 两者**无直接事件往来**——本上下文不订阅 `rework.*`，返修上下文不订阅 `brework.*`。它们通过过点执行上下文间接关联（过点执行既消费返工工单再入事件，又消费返修任务完成事件）。
- **事件名严格区分**：`BatchReworkOrderReleased` / `BatchReworkOrderScrapped`（本上下文，批量级）vs `UnitReworkCompleted` / `UnitScrappedAfterRework`（返修上下文，单件级）。**严禁混用或互相引用**。
- **触发边界判定**：当一批 SN 因同一缺陷需处理时，是逐个走单件返修（返修上下文）还是开批量返工工单（本上下文）？按缺陷影响范围判定——若缺陷是单件偶发（如个别焊点不良），走单件返修；若缺陷是批次性/系统性（如某工序参数偏差导致整批不良），开批量返工工单。判定归质量上下文/人工决策，本上下文只接收已判定为批量返工的触发。参考返修上下文同款热点。

> 🔴 **热点：单件返修与批量返工的边界判定机制**——质量上下文如何判定缺陷是单件偶发还是批次性系统性？判定信号如何分别触发返修上下文（单件）与本上下文（批量）？建议：**质量上下文做缺陷影响范围分析**——单件偶发缺陷通过过点拦截分流触发返修上下文（`mes.unit.routed-to-rework`）；批次性系统性缺陷通过批量异常信号触发本上下文（`brework.order.created`）。判定逻辑归质量上下文，本上下文只接收已判定为批量返工的触发。待与质量上下文确认缺陷影响范围分析机制与触发信号契约。

> 🔴 **热点：批量返工中单件执行是否走返修上下文**——批量返工工单生成后，每个 SN 的实际返工执行（诊断/修理/验证）是否复用返修上下文的单件返修任务？还是返工区独立执行？建议：**返工动作归返修上下文或返工区执行系统**，本上下文只跟踪返工工单维度进度（消费返修上下文的 `UnitReworkCompleted` 或返工区执行系统的进度上报）。本上下文不重复建模单件返工动作细节。待与返修上下文确认是否消费其完成事件驱动本上下文 `ReworkProgress` 累计。
