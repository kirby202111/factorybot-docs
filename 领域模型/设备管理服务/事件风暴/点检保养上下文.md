# 点检保养限界上下文（Preventive Maintenance Bounded Context）

> **限界上下文**：点检保养
> **统一语言**：InspectionRule（点检规则）, MaintenanceRule（保养规则）, ItemTemplate（项目模板）, TriggerCondition（触发条件）, InspectionPlan（点检计划）, MaintenancePlan（保养计划）, InspectionTask（点检任务）, MaintenanceTask（保养任务）, TargetAsset（目标资产·抽象引用）, SOPReference（SOP 引用）, OverdueState（超期状态）, FixtureLifeJudgement（工装寿命判定）, MetricResetDirective（度量重置指令）
> **核心关注**：以"事件风暴"方式刻画**预防性运维**从规则定义、计划调度、任务执行到结果签核的完整流程。本上下文负责"**何时该做、谁来做、怎么做**"；**不负责**资产的身份数据与可用性状态——那归设备工装台账上下文（下文简称「台账上下文」）。

---

## 0. 设计原则与边界澄清

点检保养上下文是设备管理服务下的**预防性运维子域**，是 PCBA + 整机组装混合型车间所有"预防性运维活动"的调度权威源。

**上下文内职责**（本上下文负责）：
- 点检/保养**规则定义**——触发条件（周期/阈值）、检查/维护项模板、SOP 绑定、责任人/角色指派
- 点检/保养**计划生成**——根据规则与触发条件，自动或手动生成执行计划
- 点检/保养**任务调度与下发**——计划拆解为可执行任务，分配到人/班次
- 点检/保养**执行登记与签核**——逐项记录结果、照片、签名，判定 PASS/FAIL
- **超期检测与预警**——未按时执行的任务标记超期，发布 Overdue 事件
- **工装寿命判定**——消费台账上下文累积度量，与寿命上限规则比较，发布 FixtureLifeExceeded

**上下文外职责**（本上下文**不**负责）：
- 资产身份、归属、状态、可用性聚合 → 台账上下文
- 设备通信与数据采集 → 设备数据接入上下文
- 故障报修、维修执行、备件消耗 → 维修上下文
- 计量器具检定/校准/证书管理 → 计量检定上下文
- 资产报废审批流程 → 台账上下文（本上下文仅发布寿命超限事件，由台账上下文发起报废申请）
- 移动端 UI 交互、消息推送机制 → 应用层
- SOP 文档版本管理 → 文档管理（本上下文仅引用 SOP ID）

**为什么不拆"设备点检保养上下文 + 工装点检保养上下文"**：
1. **流程骨架完全同构**——规则→计划→任务→执行→签核，拆开等于双份代码。
2. **规则模型可统一抽象**——差异仅在触发条件（运行时长 vs 使用次数）和检查项内容，通过 `TargetAsset.asset_kind` + 模板库即可区分。
3. **调度引擎共用**——计划生成、任务分配、超期检测逻辑无差别。
4. **组织一致**——设备和工装的点检保养由同一工程团队执行，无组织鸿沟。

**为什么不拆"点检上下文 + 保养上下文"**：
1. **触发机制同构**——都是周期/阈值驱动的预防性活动。
2. **调度模型一致**——Plan → Task → Execute → Sign-off 骨架完全相同。
3. **执行经常联动**——点检发现异常时可能触发即时保养（如润滑），拆成两个上下文增加协调成本。
4. **区别仅在内容**——点检项是"检查 pass/fail"，保养项是"执行动作 + 可能重置累积值"，结构差异不足以支撑独立限界上下文。

## 1. 聚合清单与规则模型

### 1.1 主语清单

本上下文处理的对象按"管理特征"分类。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

| 聚合根 | 典型实例 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|----------------|
| **InspectionRule**（点检规则） | 回流焊日点检规则、钢网每 5000 次点检规则 | 触发条件 + 检查项模板 + 执行角色；可启停、可变更 | 日历定时器；台账上下文累积度量 |
| **MaintenanceRule**（保养规则） | 设备每 500 运行小时保养规则、钢网清洁规则、治具翻修规则 | 触发条件 + 维护项模板 + 是否重置度量；含寿命上限 | 台账上下文累积度量 |
| **InspectionPlan**（点检计划） | 2026-W24 回流焊点检计划 | 单周期计划，按资产拆明细；快照自规则 | 规则到期事件 |
| **MaintenancePlan**（保养计划） | 2026-W24 钢网清洁计划 | 同上 | 规则到期事件 |
| **InspectionTask**（点检任务） | 单台设备一次点检 | 逐项登记 + 签核 + verdict | 计划拆解；维修上下文加急请求 |
| **MaintenanceTask**（保养任务） | 单台设备一次保养 | 逐项执行 + 签核 + 度量重置 | 计划拆解；维修上下文加急请求 |

> 阈值（保养周期、点检周期、寿命上限）**存本上下文**——它们是运维规则，归 MaintenanceRule / InspectionRule。台账上下文只存"当前累积值"，不做阈值判定（对应台账上下文 INV-22 / INV-23）。

### 1.2 规则统一模型

点检规则和保养规则共享同一规则骨架，差异通过字段语义区分：

```
┌─────────────────────────────────────────────────────────────┐
│ Rule（规则骨架）                                              │
│                                                              │
│  rule_id            : 规则唯一标识                             │
│  rule_kind          : INSPECTION | MAINTENANCE               │
│  target_asset_kind  : EQUIPMENT | FIXTURE | *（通配）         │
│  target_classification : 资产分类（如"回流焊""钢网"）           │
│  trigger            : TriggerCondition                       │
│  item_templates[]   : ItemTemplate[]                         │
│  responsible_role   : 执行角色                                 │
│  sop_references[]   : SOPReference[]                         │
│  plan_horizon       : 计划周期（如每周/每月）                   │
│  task_deadline      : 任务执行时限（从计划生成起算）             │
│  life_limit         : 寿命上限（仅 MAINTENANCE + 工装）         │
│  enabled            : 是否启用                                 │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 触发条件

| 触发类型 | 适用资产 | 说明 | 来源 |
|----------|---------|------|------|
| **CALENDAR_PERIOD** | 设备 + 工装 | 日历周期：每班/每天/每周/每月 | 本上下文内计算 |
| **RUN_HOUR_THRESHOLD** | 设备 | 运行时长累积达阈值：如每 500 小时 | 消费台账上下文 `EquipmentRunHourAccrued` |
| **CYCLE_THRESHOLD** | 工装 | 使用次数累积达阈值：如每 5000 次 | 消费台账上下文 `FixtureCycleAccrued` |
| **LIFE_UPPER_LIMIT** | 工装 | 使用次数达到寿命上限 | 消费台账上下文 `FixtureCycleAccrued` + 本上下文规则中的 `life_limit` |

> 🔴 **热点：双重触发**——一台设备可能同时配置"日历周期点检"和"运行时长保养"，两者独立触发、独立生成计划。规则引擎必须支持同一资产绑定多条不同触发条件的规则。

### 1.4 项目模板

```
ItemTemplate
├── item_id          : 模板项唯一标识
├── item_kind        : CHECK（检查项，输出 pass/fail + 实测值）
│                     | ACTION（执行项，输出 done/skipped + 可能重置度量）
├── name             : 项目名称
├── description      : 描述/标准
├── standard_value   : 标准值（如"温度 250±5℃"）
├── measurement_unit : 单位
├── check_method     : 检查方式（目视/仪器/软件自检）
├── required_evidence: 是否必须拍照
├── metric_reset     : 保养项专有——执行后是否重置累积度量
│                     （如保养后 run_hours 基准清零）
└── sort_order       : 排序
```

**设备 vs 工装的项目模板差异**：

| 维度 | 设备点检项示例 | 工装点检项示例 |
|------|--------------|--------------|
| 检查项 | 温度校准值、气压值、传送带张力、光源亮度 | 张力值、开口磨损度、定位销偏移、清洁度 |
| 维度 | 设备保养项示例 | 工装保养项示例 |
| 执行项 | 清洁传动系统、更换滤芯、润滑导轨、校准传感器 | 清洗、翻修、张力恢复、补胶 |
| 度量重置 | 保养后可能重置 `run_hours` 基准 | 翻修后可能重置 `accumulated_cycles`（由 `metric_reset` 字段控制） |

## 2. 事件风暴 — 主链：规则定义与触发

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐/有争议点（正文用 🔴 前缀）

### 2.1 规则生命周期

```
[参与者:设备工程师/工艺工程师]
        --【命令:定义点检规则 DefineInspectionRule】--> 【聚合:InspectionRule】
        --【命令:定义保养规则 DefineMaintenanceRule】--> 【聚合:MaintenanceRule】
                                                    |
                                                    v
                            【事件:点检规则已定义 InspectionRuleDefined】
                            【事件:保养规则已定义 MaintenanceRuleDefined】
                                                    |
                                            【策略:规则生效后立即开始监听触发条件】
                                                    |
                                                    v
                            【命令:激活规则 ActivateRule】
                            【事件:点检规则已激活 InspectionRuleActivated】
                            【事件:保养规则已激活 MaintenanceRuleActivated】
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| InspectionRuleDefined | 首次定义点检规则 | rule_id, target_asset_kind, trigger, item_templates, responsible_role | 进入 DRAFT，待激活 |
| MaintenanceRuleDefined | 首次定义保养规则 | rule_id, target_asset_kind, trigger, item_templates, metric_reset_info, life_limit? | 进入 DRAFT，待激活 |
| InspectionRuleUpdated / MaintenanceRuleUpdated | 规则内容变更 | changed_fields, effective_from（立即/下个周期生效） | 已生成计划/任务不受影响（快照语义） |
| InspectionRuleActivated / MaintenanceRuleActivated | 启用规则 | rule_id | 激活后开始监听触发条件 |
| InspectionRuleDeactivated / MaintenanceRuleDeactivated | 停用规则 | rule_id | 暂停监听 |
| InspectionRuleObsoleted / MaintenanceRuleObsoleted | 规则废弃（不再使用） | replacement_rule_id（可选） | 终态 |

> 🔴 **热点：规则变更的生效策略**——变更已有规则时，已生成的计划和任务是否受影响？建议：**已生成的计划/任务不受影响**（快照语义），变更从下个周期生效。避免执行中途规则变更导致混乱。

### 2.2 状态流转

**InspectionRule / MaintenanceRule 状态机**：

```
                Define
                  |
                  v
              +--------+   Activate    +--------+   Deactivate   +------------+
              | DRAFT  |-------------->| ACTIVE |---------------->| SUSPENDED  |
              +--------+               +--------+                 +------------+
                  |                        ^                          |
                  |                        |    Activate              |
                  |                        +--------------------------+
                  |
                  |  Obsolete（终态）
                  v
              +----------+
              | OBSOLETE |  终态——之后任何事件均拒收
              +----------+
```

- DRAFT → ACTIVE：`ActivateRule`，前置 = 规则项模板非空
- ACTIVE ⇄ SUSPENDED：`ActivateRule` / `DeactivateRule`，可逆
- 任意非终态 → OBSOLETE：`ObsoleteRule`，终态——之后任何事件均拒收

**InspectionTask / MaintenanceTask 状态机**：

```
   Create
     |
     v
 +---------+  Start    +-------------+  Confirm(all items)   +------------+
 | PENDING |---------->| IN_PROGRESS |---------------------->| COMPLETED  | (终态)
 +---------+           +-------------+                         +------------+
     |                       |
     | Skip                  | 过 deadline 未完成
     v                       v
 +---------+           +---------+
 | SKIPPED |           | OVERDUE |---Confirm---> COMPLETED (终态)
 +---------+           +---------+
   (终态)                (可逆)
```

- PENDING → IN_PROGRESS：`StartInspection` / `StartMaintenance`
- IN_PROGRESS → COMPLETED：`ConfirmInspection` / `ConfirmMaintenance`，前置 = 所有 item 已登记结果
- PENDING → SKIPPED：`SkipTask(reason)`，终态（资产停用/外转等）
- IN_PROGRESS → OVERDUE：`MarkInspectionOverdue` / `MarkMaintenanceOverdue`（定时扫描）
- OVERDUE → COMPLETED：`ConfirmInspection` / `ConfirmMaintenance`（补做后签核闭环）

> 🔴 **热点：OVERDUE 是否终态**——建议 OVERDUE 为可逆中间态（补做后 Confirm 转 COMPLETED），不设为终态；SKIPPED 为终态。待 SME 确认是否允许"超期后仍可签核闭环"。

### 2.3 触发与 Due 事件生成

```
【外部系统:台账上下文】
        -- 发布 eam.asset.metric (EquipmentRunHourAccrued / FixtureCycleAccrued / FixtureRefurbished)
                |
                v
        【聚合:InspectionRule / MaintenanceRule】
        --【策略:度量值与阈值比较】
                |
                ├── 达到点检周期阈值 ──► 【命令:标记点检到期 MarkInspectionDue】
                │                           |
                │                           v
                │                    【事件:点检已到期 InspectionDue】(asset_id, rule_id, due_reason)
                │
                ├── 达到保养周期阈值 ──► 【命令:标记保养到期 MarkMaintenanceDue】
                │                           |
                │                           v
                │                    【事件:保养已到期 MaintenanceDue】(asset_id, rule_id, due_reason)
                │
                └── 工装达到寿命上限 ──► 【命令:判定工装寿命超限 JudgeFixtureLife】
                                            |
                                            v
                                     【事件:工装寿命已超限 FixtureLifeExceeded】(fixture_id, accumulated_cycles, life_limit)
```

**触发判定逻辑**：

| 触发类型 | 判定逻辑 | 事件 |
|----------|---------|------|
| CALENDAR_PERIOD | 本上下文定时器扫描规则 + 资产列表，距上次执行达到周期 | InspectionDue / MaintenanceDue |
| RUN_HOUR_THRESHOLD | 消费 `EquipmentRunHourAccrued`，使用 `accumulated_run_hours - reset_baseline ≥ threshold` 判定 | MaintenanceDue |
| CYCLE_THRESHOLD | 消费 `FixtureCycleAccrued`，使用 `accumulated_cycles - reset_baseline ≥ threshold` 判定 | MaintenanceDue |
| LIFE_UPPER_LIMIT | 消费 `FixtureCycleAccrued`，使用 `accumulated_cycles ≥ life_limit` 判定 | FixtureLifeExceeded |

> **度量重置机制（重要）**：`reset_baseline`（重置基准值）由台账上下文维护并透传——台账上下文发布的度量事件中同时携带 `accumulated_value` 和 `reset_baseline`。当保养项包含 `metric_reset=true` 时，本上下文在保养完成后发布 `MaintenanceCompleted(metric_reset=true, reset_metric_kind, reset_baseline=当前累积值)` 通知台账上下文更新基准，台账上下文消费后发布 `MetricBaselineReset`，后续发布的度量事件将携带新的 `reset_baseline`。本上下文只需使用 `accumulated_value - reset_baseline` 即可计算出相对于上次重置的偏移量，与阈值比较。

### 2.4 上下文外事件消费与投影

本上下文被动响应的所有上下文外事件，统一在此登记。每条明确：触发命令、状态变更、可用性/投影影响、幂等/去重策略。

| 事件源（上下文外） | 来源限界上下文 | 触发命令 | 状态变更 | 可用性/投影影响 | 幂等/去重 |
|-------------------|--------------|----------|---------|----------------|----------|
| `EquipmentRunHourAccrued` | 台账上下文 (`eam.asset.metric`) | MarkMaintenanceDue（达阈值时） | 无（只读比较） | 重算保养到期视图 | 按 (asset_id, accumulated_run_hours) 去重，重复值不重复触发 |
| `FixtureCycleAccrued` | 台账上下文 (`eam.asset.metric`) | MarkMaintenanceDue / JudgeFixtureLife | 无（只读比较） | 重算保养到期 + 寿命消耗视图 | 按 (fixture_id, accumulated_cycles) 去重 |
| `FixtureRefurbished(life_reset=true)` | 台账上下文 (`eam.asset.metric`) | RefreshLifeConsumption（更新本上下文寿命消耗快照） | 无 | 寿命消耗按新 `accumulated_cycles=0` 重算，避免误判超限 | 按 fixture_id + 翻修序号去重 |
| `MetricBaselineReset` | 台账上下文 (`eam.asset.metric`) | （内部更新缓存的 reset_baseline） | 无 | 阈值比较基准刷新 | 按 (asset_id, metric_kind) 取最新基准 |
| `AssetCommissioned` | 台账上下文 (`eam.asset.lifecycle`) | ActivateMatchingRules | 激活该资产匹配的所有规则 | 资产纳入调度范围 | 按 asset_id 幂等 |
| `AssetSuspended` | 台账上下文 (`eam.asset.lifecycle`) | SkipPendingTasks(reason) | 该资产待执行任务 → SKIPPED | 暂停调度 | 按 asset_id 幂等 |
| `AssetRetired` | 台账上下文 (`eam.asset.lifecycle`) | DeactivateRules + CancelPendingTasks | 停用该资产所有规则，取消未执行任务 | 移出调度范围 | 按 asset_id 幂等 |
| `InspectionRequested` | 维修上下文 (`repair.suggestion`) | CreateInspectionTask(urgency=POST_REPAIR) | 新建加急点检任务，关联维修工单 | — | 若同资产已有未执行任务则合并，不重复创建 |
| `MaintenanceSuggested` | 维修上下文 (`repair.suggestion`) | GenerateMaintenancePlan(urgency=POST_REPAIR) | 生成加急保养计划 | — | 若已有未执行保养计划则合并 |

> 🔴 **热点：`MaintenanceSuggested` 方向**——本上下文按"维修上下文 → 本上下文"消费（维修中发现/完成后建议保养）。维修上下文事件命名速查表中存在一处反向标注（误标为本上下文发布），需与维修上下文对齐统一为"维修上下文发布、本上下文消费"。
>
> 🔴 **热点：`FixtureRefurbished` vs `FixtureCycleAccrued` 双通道**——台账上下文翻修时既重置 `accumulated_cycles=0` 又发布 `FixtureRefurbished(life_reset=true)`。本上下文理论上靠 `FixtureCycleAccrued` 携带的新 `reset_baseline` 即可重算，`FixtureRefurbished` 作为显式信号用于寿命视图刷新。建议两者都消费、以 `FixtureCycleAccrued.reset_baseline` 为权威计算源，`FixtureRefurbished` 仅触发视图刷新，待 SME 确认是否冗余。

## 3. 事件风暴 — 计划生成与任务调度

### 3.1 Due → Plan

```
InspectionDue / MaintenanceDue
        |
        v
【策略:到期自动生成计划】
        |
        v
【命令:生成点检计划 GenerateInspectionPlan】
【命令:生成保养计划 GenerateMaintenancePlan】
        |
        v
```

**关键事件**：

| 事件 | 何时发生 | 关键载荷 | 下游消费 |
|------|---------|---------|----------|
| MaintenanceStarted | 保养员开始执行 | task_id, started_at, started_by | — |
| MaintenanceItemPerformed | 逐项执行 | item_id, action_taken, evidence_url, metric_reset | — |
| MaintenanceCompleted(metric_reset=false) | 执行完成，无度量重置 | task_id, asset_id, completed_by | — |

| MaintenanceSkipped | 跳过 | task_id, skip_reason | — |

**度量重置机制**：

保养项如果配置了 `metric_reset=true`，签核时本上下文发布 `MaintenanceCompleted` 携带重置信息。台账上下文消费后，**不修改绝对累积值**，而是记录新的重置基准点。下次阈值比较时，本上下文使用 `accumulated_value - reset_baseline` 计算偏移量。

| 资产类型 | 重置场景 | 重置度量 | 说明 |
|---------|---------|---------|------|
| 设备 | 大保养后 | run_hours 基准清零 | 如每 2000 小时大保养后 |
| 工装 | 翻修后 | accumulated_cycles 清零 | 如治具大修后（由工程标准判定） |
| 工装 | 钢网清洁后 | **不清零** | 钢网清洁是常规保养，不重置寿命 |

> 🔴 **热点：谁决定是否重置**——`metric_reset` 字段来自规则模板定义，不是执行时临时决定。但执行人可以在签核时标注"本次跳过重置"（如发现翻修不合格）。策略归工程标准，本上下文记录执行事实。

## 7. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **InspectionRule** | 单条点检规则 | INV-01 同一 `target_asset_kind + classification + trigger` 组合不重复（由唯一索引保证）；INV-02 启用状态下至少 1 个 item_template；INV-03 OBSOLETE 后任何事件均拒收（终态） |
| **MaintenanceRule** | 单条保养规则 | INV-04 同 INV-01；INV-05 含 `metric_reset=true` 的 item 必须指定 `reset_metric_kind`；INV-06 `life_limit` 仅工装 + MAINTENANCE 规则可设 |
| **InspectionPlan** | 单个周期点检计划 | INV-07 每条 plan_item 必须关联一条 ACTIVE 的 InspectionRule；INV-08 plan 内 asset 不重复（由领域服务 + 唯一索引保证跨计划去重） |
| **MaintenancePlan** | 单个周期保养计划 | INV-09 同 INV-07；INV-10 同 INV-08 |
| **InspectionTask** | 单次点检任务 | INV-11 必须关联一个 InspectionPlan 的 plan_item；INV-12 所有 item 必须登记结果后才能 Confirm；INV-13 SKIPPED 后终态 |
| **MaintenanceTask** | 单次保养任务 | INV-14 同 INV-11；INV-15 同 INV-12；INV-16 含 metric_reset 的任务签核时必须携带 reset_baseline |

> 跨聚合不变式：
> - INV-08/INV-10「同一资产同一周期不重复生成计划」由 **MaintenanceSchedulingService + 唯一索引 (asset_id, rule_id, plan_horizon)** 保证，不挂在聚合根上。
> - INV-17「`FixtureLifeExceeded` 与 `MaintenanceDue(LIFE_UPPER_LIMIT)` 不重复发布」由 **事件去重表 (fixture_id, accumulated_cycles, life_limit)** 保证。

## 8. 关键策略（Policy）汇总

### 8.1 上下文外事件 → 本上下文响应

| 触发事件（上下文外） | 来源限界上下文 | 策略 | 触发命令 |
|---------------------|---------------|------|----------|
| `EquipmentRunHourAccrued` | 台账上下文 | 运行时长与保养阈值比较 | MarkMaintenanceDue（达阈值时） |
| `FixtureCycleAccrued` | 台账上下文 | 使用次数与保养阈值/寿命上限比较 | MarkMaintenanceDue / JudgeFixtureLife |
| `FixtureRefurbished(life_reset=true)` | 台账上下文 | 刷新寿命消耗视图 | RefreshLifeConsumption |
| `MetricBaselineReset` | 台账上下文 | 更新本上下文基准值缓存 | （内部更新，不发布新事件） |
| `AssetCommissioned` | 台账上下文 | 激活该资产匹配的所有规则 | ActivateMatchingRules |
| `AssetSuspended` | 台账上下文 | 标记该资产待执行任务为 SKIPPED | SkipPendingTasks |
| `AssetRetired` | 台账上下文 | 停用该资产所有规则，取消未执行任务 | DeactivateRules + CancelPendingTasks |
| `InspectionRequested` | 维修上下文 | 评估是否创建加急点检任务 | CreateInspectionTask(urgency=POST_REPAIR)，关联维修工单 |
| `MaintenanceSuggested` | 维修上下文 | 评估是否生成加急保养计划 | GenerateMaintenancePlan(urgency=POST_REPAIR)，已有未执行保养计划则合并 |

> **设计意图**：维修上下文发布的 `InspectionRequested` 和 `MaintenanceSuggested` 是**建议性事件**，本上下文有权根据当前调度状态决定是否执行。若已有同资产的未执行任务，可合并而非重复创建。`MetricBaselineReset` 仅用于更新本上下文的基准值缓存，确保阈值计算与台账上下文保持一致。

### 8.2 本上下文内部策略

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|
| InspectionRuleDefined / MaintenanceRuleDefined | 规则生效后开始监听触发条件 | ActivateRule |

| MaintenanceDue | 自动生成保养计划 | GenerateMaintenancePlan |
| InspectionPlanGenerated | 拆解为点检任务 | CreateInspectionTask（per plan_item） |
| MaintenancePlanGenerated | 拆解为保养任务 | CreateMaintenanceTask（per plan_item） |

## 10. 与台账上下文的协作边界

| 维度 | 本上下文（点检保养） | 台账上下文 |
|------|-------------------|-----------|
| 视角 | 运维活动**怎么管**（何时做、谁做、怎么做） | 资产**能不能用**（身份/状态/可用性） |
| 存储 | 阈值规则：保养周期 / 寿命上限 / 点检周期 / 项目模板 | 累积值：run_hours / accumulated_cycles / reset_baseline |
| 判定 | **判定并触发**——订阅度量值，与阈值比较 | **不判定**——只发布度量值 |
| 事件性质 | 调度信号：MaintenanceDue / InspectionDue / FixtureLifeExceeded | 度量事实：EquipmentRunHourAccrued / FixtureCycleAccrued |
| 流程所有权 | 规则→计划→任务→执行→签核 全流程 | 资产入账→投用→停用→报废 全生命周期 |

**事件流向**：

```
台账上下文 ──→ eam.asset.metric ──→ 本上下文（阈值比较 → 触发调度）
台账上下文 ──→ eam.asset.metric(MetricBaselineReset / FixtureRefurbished) ──→ 本上下文（更新基准值缓存/寿命视图）
台账上下文 ──→ eam.asset.lifecycle ──→ 本上下文（Commissioned → 激活规则 / Suspended → 跳过任务 / Retired → 停用规则）
本上下文  ──→ pm.inspection.completed ──→ 台账上下文（FAIL → SuspendAsset）
本上下文  ──→ pm.inspection.overdue  ──→ 台账上下文（MarkAssetUnavailable）
本上下文  ──→ pm.maintenance.completed ──→ 台账上下文（metric_reset → UpdateMetricBaseline → MetricBaselineReset）
本上下文  ──→ pm.maintenance.overdue ──→ 台账上下文（MarkAssetUnavailable）
本上下文  ──→ pm.fixture.life ──→ 台账上下文（SuspendAsset）
```

## 11. 与维修上下文的协作边界

点检保养上下文与维修上下文是"预防"与"纠正"的关系：

| 协作场景 | 方向 | 事件 | 说明 |
|---------|------|------|------|
| 点检异常建议报修 | 本上下文 → 维修上下文 | `RepairSuggested` | 点检 FAIL 后建议进入维修流程 |
| 维修完成后恢复 | 维修上下文 → 台账上下文 | `RepairCompleted` | 维修上下文发布后台账上下文 Reinstate，本上下文无需额外动作 |
| 维修后建议保养 | 维修上下文 → 本上下文 | `MaintenanceSuggested` | 维修过程中/完成后发现潜在保养需求（如润滑不足） |
| 维修后触发点检验证 | 维修上下文 → 本上下文 | `InspectionRequested` | 维修验证方式选择"点检验证"时 |

> 本上下文**不直接创建维修工单**——仅发布 `RepairSuggested` 建议，由维修上下文决定是否创建 RepairOrder。避免预防上下文直接指挥纠正上下文，保持各上下文自治。
