# 维修上下文（Repair Bounded Context）

> **限界上下文**：维修上下文
> **统一语言**：MalfunctionReport（故障报告）, RepairOrder（维修工单）, FaultDiagnosis（故障诊断）, RepairAction（维修动作）, PartsConsumption（备件消耗）, RepairVerification（维修验证）, RepairEscalation（维修升级）, ScrapRecommendation（报废建议）, ExternalRepairDispatch（外协维修派工）, RepairKnowledgeEntry（维修知识条目）
> **核心关注**：以"事件风暴"方式刻画**纠正性运维**从故障报告、派工、诊断、维修执行、验收到关单的完整流程。本上下文负责"**故障怎么修、备件怎么用、修不好怎么办**"；**不负责**资产的身份数据与可用性状态——那归台账上下文。

---

## 0. 设计原则与边界澄清

维修上下文是设备管理服务下的**纠正性运维子域**，是 PCBA + 整机组装混合型车间所有"故障修复活动"的执行权威源。

**上下文内职责**（本上下文负责）：
- 故障**报告受理**——多来源入口统一建模，故障分级分类
- 维修**工单管理**——派工、优先级、SLA、超时升级
- 故障**诊断记录**——诊断结论、根因分析、维修方案选择
- 维修**执行与备件消耗**——维修动作记录、备件领用与消耗回写
- 维修**验证与关单**——修复后验证、部分修复、关单判定
- 维修**升级与外协**——超时/超能力升级、外部维修商对接
- **报废建议**——维修不经济时，向台账上下文发起报废建议

**上下文外职责**（本上下文**不**负责）：
- 资产身份、归属、状态 → 台账上下文
- 预防性点检/保养 → 点检保养上下文
- 计量器具检定/校准 → 计量检定上下文
- 备件库存管理、备件 BOM → 后续「备件管理上下文」（本上下文仅发布消耗事件）
- 设备通信与数据采集 → 设备数据接入上下文
- 移动端 UI 交互 → 应用层

**为什么不把维修放进点检保养上下文**：
1. **触发机制根本不同**——点检保养是"主动/计划性"（规则驱动），维修是"被动/事件驱动"（故障触发）。混合后"预防"和"纠正"两套模型互相污染。
2. **流程骨架差异大**——点检保养是 规则→计划→任务→执行→签核；维修是 报修→派工→诊断→维修→验证→关单。无共用聚合。
3. **复杂度不同**——维修包含诊断推理、备件消耗、升级外协、报废建议等点检保养不具备的逻辑，合在一起会使点检保养上下文膨胀。
4. **备件关联**——只有维修涉及备件消耗，点检保养不涉及（或仅耗材），职责重心不同。
5. **对生产影响不同**——点检保养可安排停机窗口，维修往往紧急、可能停线，SLA 管控逻辑完全不同。

## 1. 故障报告入口与主语清单

维修上下文的入口是 `MalfunctionReport`，统一建模来自多个来源的故障信息。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

### 1.1 上下文外触发源

```
┌──────────────────────────────────────────────────────────────┐
│ 故障报告入口                                                  │
│                                                              │
│  【来源1:生产执行服务】                                        │
│    MalfunctionReported（设备故障/产线异常触发报修）             │
│                                                              │
│  【来源2:台账上下文】                                          │
│    AssetSuspended(reason=CommunicationLost)（通信故障停用资产）  │
│    → 本上下文独立订阅后评估是否生成故障报告                      │
│    （设备数据接入上下文 ChannelEscalated 先由台账停用资产，       │
│      本上下文再评估是否报修，不直接消费 ChannelEscalated）        │
│                                                              │
│  【来源3:点检保养上下文】                                      │
│    RepairSuggested（点检异常建议报修）                          │
│                                                              │
│  【来源4:人工报修】                                            │
│    操作员/工程师通过终端直接报修                                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 聚合根主语清单

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **MalfunctionReport**（故障报告） | 产线报修、点检异常报修、通信故障评估报修、人工报修 | 多来源入口统一建模；故障分级分类；幂等去重 | 生产执行服务 / 台账上下文 / 点检保养上下文 / 人工 |
| **RepairOrder**（维修工单） | 紧急工单、普通工单、外协工单 | 关联故障报告；SLA 管控；状态单向流转；升级外协 | 由 MalfunctionReport 创建 |
| **FaultDiagnosis**（故障诊断） | 首次诊断、升级后再诊断 | 记录根因与可修复性判定；驱动维修/报废分流 | 由 RepairOrder 触发 |
| **RepairAction**（维修动作） | 更换、修复、调整、清洁、润滑 | 关联 IN_REPAIR 工单；记录备件消耗 | 由维修工程师执行 |
| **RepairVerification**（维修验证） | 人工试运行、自动测试、试产、点检验证 | 关联有 RepairAction 的工单；VERIFIED/FAILED 分流 | 由设备工程师/生产主管执行 |

**故障报告模型**：

```
MalfunctionReport
├── report_id       : 报告唯一标识
├── source          : PRODUCTION | DATA_COLLECTION | INSPECTION | MANUAL
├── asset_id        : 故障资产（设备/工装）
├── asset_kind      : EQUIPMENT | FIXTURE | INSTRUMENT
├── symptom         : 故障现象描述
├── severity        : CRITICAL（停线） | HIGH（降级运行） | MEDIUM（可带病运行） | LOW（轻微）
├── reported_by     : 报告人
├── reported_at     : 报告时间
├── production_impact: 是否影响生产（停线/降级/不影响）
├── attachments[]   : 现场照片/视频
└── source_event_id : 来源事件ID（用于追溯，如 RepairSuggested 的 task_id）
```

> 🔴 **热点：自动报修 vs 人工确认**——台账上下文 `AssetSuspended(reason=CommunicationLost)` 是否自动生成 MalfunctionReport？建议：**自动生成草稿**（severity=MEDIUM, source=DATA_COLLECTION），由值班工程师**确认后**才创建 RepairOrder。避免通信抖动产生大量误报工单。

## 2. 事件风暴 — 维修工单生命周期

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点

### 2.1 报修到派工

```
[参与者:操作员/设备工程师/系统]
        --【命令:报告故障 ReportMalfunction】--> 【聚合:MalfunctionReport】
                                                    |
                                                    v
                                            【事件:故障已报告 MalfunctionReported】(report_id, asset_id, severity)
                                                    |
                                            【策略:故障分级 → 派工优先级映射】
                                                    |
                                            ┌─── CRITICAL ───┐
                                            v                v
                                    【命令:创建紧急维修工单    【命令:创建普通维修工单
                                     CreateUrgentRepairOrder   CreateRepairOrder
                                     priority=URGENT】         priority=NORMAL】
                                            |                        |
                                            v                        v
                                    【聚合:RepairOrder】      【聚合:RepairOrder】
                                            |                        |
                                            v                        v
                                    【事件:维修工单已创建 RepairOrderCreated】
                                     (order_id, report_id, asset_id, priority, sla_deadline)
                                            |
                                    【策略:自动派工或等待人工派工】
                                            |
                                            v
                                    【命令:分配维修人员 DispatchRepairOrder】
                                            |
                                            v
                                    【事件:维修工单已派工 RepairOrderDispatched】
                                     (order_id, assignee, dispatched_at)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| MalfunctionReported | 故障报告受理 | report_id, asset_id, severity, source | 故障分级 → 创建工单（INV-01 幂等去重） |
| RepairOrderCreated | 故障确认后创建工单 | order_id, report_id, asset_id, priority, sla_deadline | 自动/人工派工 |
| RepairOrderDispatched | 分配维修人员 | order_id, assignee, dispatched_at | 启动 SLA 响应计时 |

**SLA 映射规则**：

| severity | priority | SLA 响应时间 | SLA 修复时间 | 超时升级 |
|----------|----------|-------------|-------------|---------|
| CRITICAL | URGENT | 15 分钟 | 2 小时 | 30 分钟未响应升级到主管 |
| HIGH | HIGH | 30 分钟 | 4 小时 | 1 小时未响应升级 |
| MEDIUM | NORMAL | 2 小时 | 24 小时 | 4 小时未响应升级 |
| LOW | LOW | 8 小时 | 72 小时 | 无自动升级 |

> 🔴 **热点：CRITICAL 停线处理**——CRITICAL 级别故障导致产线停机，SLA 极短。是否需要与异常处理上下文联动（自动触发停线异常流程）？建议：**是的**——CRITICAL 级别同时发布 `LineStoppageRequested` 到异常处理上下文，两个限界上下文并行处理。

### 2.2 状态流转

```
[CREATED] ──DispatchRepairOrder──▶ [DISPATCHED] ──DiagnosisResultRecorded(REPAIRABLE)──▶ [DIAGNOSED]
                                                                                  │
                                                                                  │ StartRepair
                                                                                  v
                                                                               [IN_REPAIR] ──RepairVerificationSubmitted(VERIFIED)──▶ [VERIFIED] ──CloseRepairOrder──▶ [CLOSED]
                                                                                  │ ▲                                                                                              (终态—之后任何事件均拒收, INV-03)
                                                       RepairVerificationSubmitted │ │ ReopenRepairOrder
                                                                  (FAILED)        │ │
                                                                                  v │
                                                                              [IN_REPAIR]

[DIAGNOSED] ──RecommendScrap──▶ [PENDING_SCRAP_DECISION] ─┬─ ScrapApplicationApproved(资产已报废) ──▶ [CLOSED](reason=AssetScrapped)
                                                            └─ ScrapApplicationRejected/Withdrawn ──▶ [DIAGNOSED](重新评估方案)

[DISPATCHED]/[IN_REPAIR] ──SLA 超时──▶ [ESCALATED] ──(重新派工/外协决策)──▶ [DISPATCHED] 或 [IN_REPAIR]
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| CREATED → DISPATCHED | DispatchRepairOrder | 当前 CREATED；已分配维修人员 | 派工员 / 自动派工策略 |
| DISPATCHED → DIAGNOSED | RecordDiagnosisResult(repair_feasibility=REPAIRABLE) | 当前 DISPATCHED；诊断结论已记录（INV-04） | 维修工程师 |
| DIAGNOSED → IN_REPAIR | PerformRepairAction | 当前 DIAGNOSED；feasibility=REPAIRABLE | 维修工程师 |
| IN_REPAIR → VERIFIED | SubmitRepairVerification(result=VERIFIED) | 当前 IN_REPAIR；至少一条 RepairAction（INV-05） | 设备工程师 / 生产主管 |
| VERIFIED → CLOSED | CloseRepairOrder | 当前 VERIFIED | 系统（验证通过后自动关单） |
| IN_REPAIR → IN_REPAIR（回退） | ReopenRepairOrder | 当前 IN_REPAIR；验证 FAILED 且带原因（INV-06） | 设备工程师 |
| DIAGNOSED → PENDING_SCRAP_DECISION | RecommendScrap | 当前 DIAGNOSED；feasibility ∈ {NOT_ECONOMICAL, NOT_REPAIRABLE} | 维修工程师 |
| PENDING_SCRAP_DECISION → CLOSED | CloseRepairOrder | 台账上下文 ScrapApplicationApproved（资产已报废）（INV-08） | ScrapDecisionProjection 消费台账事件后触发 |
| PENDING_SCRAP_DECISION → DIAGNOSED | ReopenForRediagnosis | 台账上下文 ScrapApplicationRejected/Withdrawn（INV-09） | ScrapDecisionProjection 消费台账事件后触发 |
| DISPATCHED/IN_REPAIR → ESCALATED | EscalateResponse / EscalateRepair | SLA 响应/修复超时 | SLA 监控策略 |
| ESCALATED → DISPATCHED/IN_REPAIR | DispatchRepairOrder / PerformRepairAction | 升级决策完成（内部接手或外协回填） | 派工员 / 维修工程师 |
| → CLOSED 之后 | （任何） | 终态，拒收（INV-03） | — |

> **可逆边说明**：仅 `ReopenRepairOrder`（验证失败回退维修）与 `ScrapApplicationRejected/Withdrawn`（报废驳回回退诊断）允许回退，其余状态单向流转（INV-02）。

### 2.3 诊断

```
[参与者:维修工程师]
        --【命令:开始诊断 StartDiagnosis】--> 【聚合:FaultDiagnosis】
                                                |
                                                v
                                        【事件:诊断已开始 DiagnosisStarted】(order_id, started_at)
                                                |
        --【命令:记录诊断结论 RecordDiagnosisResult】-->
                                                |
                                                v
                                        【事件:诊断结论已记录 DiagnosisResultRecorded】
                                         (diagnosis_id, fault_cause, repair_plan,
                                          estimated_parts[], estimated_duration,
                                          repair_feasibility: REPAIRABLE / NOT_ECONOMICAL / NOT_REPAIRABLE)
                                                |
                                    ┌──── repair_feasibility ────┐
                                    v                v                   v
                              REPAIRABLE       NOT_ECONOMICAL      NOT_REPAIRABLE
                                    |                |                   |
                                    v                v                   v
                            继续维修流程     发起报废建议          发起报废建议
```

**诊断模型**：

```
FaultDiagnosis
├── diagnosis_id    : 诊断唯一标识
├── order_id        : 关联维修工单
├── fault_cause     : 故障原因（分类编码 + 自由描述）
├── repair_plan     : 维修方案
├── estimated_parts[]: 预估所需备件（part_code, quantity）
├── estimated_duration: 预估维修时长
├── repair_feasibility: REPAIRABLE | NOT_ECONOMICAL | NOT_REPAIRABLE
├── diagnosed_by    : 诊断人
└── diagnosed_at    : 诊断时间
```

**repair_feasibility 判定逻辑**：

| 判定 | 条件 | 后续动作 |
|------|------|---------|
| REPAIRABLE | 维修成本 < 资产残值 × 阈值（如 60%） | 进入维修执行 |
| NOT_ECONOMICAL | 维修成本 ≥ 资产残值 × 阈值 | 发布 `ScrapRecommendation` 到台账上下文 |
| NOT_REPAIRABLE | 技术上无法修复（如核心部件停产） | 发布 `ScrapRecommendation` 到台账上下文 |

> 🔴 **热点：可修复性判定的阈值**——"维修不经济"的判定标准不在本上下文硬编码，应由工程标准/财务规则定义。本上下文只记录诊断结论和维修预估成本，判定逻辑由策略引擎执行。

### 2.4 维修执行

```
【聚合:RepairAction】
        |
        v
[参与者:维修工程师]
        --【命令:执行维修动作 PerformRepairAction】--> 【聚合:RepairAction】
                                                        |
                                                        v
                                                【事件:维修动作已执行 RepairActionPerformed】
                                                 (action_id, order_id, action_description,
                                                  parts_consumed[], duration)
                                                        |
                                        ┌──── 维修完成 ────┐
                                        v                   v
                                    含备件消耗           无备件消耗
                                        |                   |
                                        v                   v
                                【事件:备件已消耗          （无额外事件）
                                 PartsConsumed】
                                 (part_code, quantity, order_id, asset_id)
```

**维修动作模型**：

```
RepairAction
├── action_id       : 动作唯一标识
├── order_id        : 关联维修工单
├── action_type     : REPLACE（更换） | REPAIR（修复） | ADJUST（调整） | CLEAN（清洁） | LUBRICATE（润滑） | OTHER
├── description     : 动作描述
├── parts_consumed[]: PartsConsumption[]
│   ├── part_code   : 备件编码
│   ├── part_name   : 备件名称
│   ├── quantity    : 消耗数量
│   └── from_location: 领用库位
├── performed_by    : 执行人
├── started_at      : 开始时间
├── completed_at    : 完成时间
└── evidence_urls[] : 照片/视频证据
```

> 🔴 **热点：备件消耗的实时性**——维修过程中备件是"先领后用"还是"用后回写"？建议：**先领后用**——维修开始时从备件管理上下文领用（预留库存），维修完成后确认消耗或退回。本上下文发布 `PartsConsumed` 事件供备件管理上下文扣减库存。

### 2.5 验证与关单

```
[参与者:设备工程师/生产主管]
        --【命令:提交维修验证 SubmitRepairVerification】--> 【聚合:RepairVerification】
                                                            |
                                                            v
                                                    【事件:维修验证已提交 RepairVerificationSubmitted】
                                                     (verification_id, order_id, method, result)
                                                            |
                                                ┌──── 验证结果 ────┐
                                                v                   v
                                            VERIFIED           FAILED
                                                |                   |
                                                v                   v
                                    【命令:关闭维修工单      【命令:退回维修
                                     CloseRepairOrder】        ReopenRepairOrder】
                                                |                   |
                                                v                   v
                                    【事件:维修工单已关闭      【事件:维修工单已退回
                                     RepairOrderClosed】       RepairOrderReopened】
                                                |                   |
                                                v                   v
                                    发布 RepairCompleted     回到维修执行阶段
                                    台账上下文消费：ReinstateAsset
```

**验证方式**：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| MANUAL_TEST | 人工试运行验证 | 机械类维修 |
| AUTO_TEST | 自动测试程序验证 | 电气/软件类维修 |
| PRODUCTION_TRIAL | 试产验证（过若干产品） | 关键设备维修后 |
| INSPECTION | 点检验证（触发一次点检任务） | 建议联动点检保养上下文 |

> 🔴 **热点：PRODUCTION_TRIAL 的边界**——试产验证需要生产执行服务配合（投少量产品验证），涉及跨上下文协调。建议：本上下文发布 `RepairVerificationRequiresProductionTrial` 事件，由生产执行服务决定是否安排试产。本上下文不直接指挥生产。

### 2.6 上下文外事件消费与投影

本上下文**只消费**下列上下文外事件以驱动故障报告生成与工单联动，**不重发**这些事件——必须包装成本上下文语义的事件（INV-10）。报废审批结果由台账上下文的 `eam.asset.scrap` 主题送达，本上下文通过 `ScrapDecisionProjection` 读模型订阅并驱动工单状态回退/关单（INV-08, INV-09）。

| 上下文外事件 | 来源限界上下文 | 触发命令 | 状态变更 | 幂等 / 去重策略 |
|---|---|---|---|---|
| MalfunctionReported | 生产执行服务 | ReportMalfunction(source=PRODUCTION) | 新建 MalfunctionReport | source_event_id 去重表（INV-01） |
| AssetSuspended(reason=CommunicationLost) | 台账上下文 | ReportMalfunction(draft=true, source=DATA_COLLECTION) | 生成草稿故障报告，待人工确认 | asset_id + 通信故障时间窗口去重 |
| RepairSuggested | 点检保养上下文 | ReportMalfunction(source=INSPECTION) | 新建 MalfunctionReport | source_event_id（点检 task_id）去重 |
| AssetRetired | 台账上下文 | CloseRepairOrder(reason=AssetRetired) | 该资产所有未完成工单 → CLOSED | 批量按 asset_id 关闭，单工单幂等 |
| ScrapApplicationApproved | 台账上下文 | CloseRepairOrder(reason=AssetScrapped) | PENDING_SCRAP_DECISION → CLOSED | application_id 去重表（INV-08） |
| ScrapApplicationRejected / Withdrawn | 台账上下文 | ReopenForRediagnosis | PENDING_SCRAP_DECISION → DIAGNOSED | application_id 去重表（INV-09） |
| AssetAvailabilityChanged（只读） | 台账上下文 | （无命令，仅读模型更新） | — | 投影定期对账，不触发状态变更 |

> 通信故障（`ChannelEscalated`）由台账上下文先停用资产（发布 `AssetSuspended`），本上下文订阅 `AssetSuspended(reason=CommunicationLost)` 后再评估是否报修，**不直接消费**设备数据接入上下文的 `ChannelEscalated`——避免通信视角与维修视角混淆。

## 3. 事件风暴 — 维修升级与外协

### 3.1 升级

```
【策略:SLA超时自动升级】
        |
        ├── 响应超时（派工后未开始）
        │       |
        │       v
        │   【命令:升级响应 EscalateResponse】
        │       |
        │       v
        │   【事件:维修已升级 RepairEscalated】
        │    (order_id, escalation_level: L1→L2, reason: ResponseTimeout)
        │       |
        │       v
        │   通知上级主管，重新分配维修人员
        │
        └── 修复超时（开始后未在 SLA 内完成）
                |
                v
            【命令:升级修复 EscalateRepair】
                |
                v
            【事件:维修已升级 RepairEscalated】
             (order_id, escalation_level, reason: RepairTimeout)
                |
                v
            ┌──── 升级决策 ────┐
            v                   v
        内部升级            外协维修
        (高级工程师接手)    (委外)
            |                   |
            v                   v
        回到 DISPATCHED    【命令:派外协 DispatchExternalRepair】
                                |
                                v
                           【事件:外协维修已派工 ExternalRepairDispatched】
                            (order_id, vendor, estimated_completion)
```

**升级矩阵**：

| 升级级别 | 触发条件 | 动作 |
|---------|---------|------|
| L1 → L2 | 一线工程师 SLA 超时 | 通知维修主管，重新分配高级工程师 |
| L2 → L3 | 高级工程师 SLA 超时 | 通知工程经理，决定是否外协 |
| L3 → 外协 | 内部无法修复 | 派工外部维修商 |

### 3.2 外协维修

```
ExternalRepairDispatched
        |
        v
【外部系统:外协维修商】
        -- 维修执行（上下文外黑盒）
        |
        v
[参与者:设备工程师]
        --【命令:登记外协维修结果 RecordExternalRepairResult】--> 【聚合:RepairOrder】
                                                                    |
                                                                    v
                                                            【事件:外协维修已完成 ExternalRepairCompleted】
                                                             (order_id, result, cost, parts_replaced[])
                                                                    |
                                                                    v
                                                            进入验证阶段（同 §2.5）
```

> 外协维修的执行过程对本上下文是黑盒，只记录结果和费用。备件消耗由外协方自带，本上下文仅记录 `parts_replaced` 用于资产维护历史追溯。

FaultDiagnosis (repair_feasibility = NOT_ECONOMICAL | NOT_REPAIRABLE)
        |
        v
【命令:建议报废 RecommendScrap】--> 【聚合:RepairOrder】
                                        |

                                        |

> **关键**：本上下文**仅发布建议**，不执行报废。报废审批与执行归台账上下文。本上下文的维修工单在建议报废后进入 `PENDING_SCRAP_DECISION` 状态，等待台账上下文的报废审批结果（§2.6）。若审批驳回/撤回，维修工单退回 `DIAGNOSED` 重新评估方案（INV-09）；若审批通过，资产已被台账报废，工单关闭（INV-08）。

## 5. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **MalfunctionReport** | 单条故障报告 | INV-01 同一故障不可重复报告（幂等判定：asset_id + symptom + 时间窗口 / source_event_id 去重表） |
| **RepairOrder** | 单个维修工单 | INV-02 必须关联一条 MalfunctionReport；状态单向流转，仅 ReopenRepairOrder / ReopenForRediagnosis 允许回退；INV-03 `RepairOrderClosed` 后任何事件均拒收（终态） |
| **FaultDiagnosis** | 单次诊断记录 | INV-04 一个 RepairOrder 可多次诊断（首次诊断后升级再诊断）；每次诊断必须记录 repair_feasibility |
| **RepairAction** | 单个维修动作 | INV-05 必须关联一个 IN_REPAIR 状态的 RepairOrder；parts_consumed 中的备件必须来自有效库位 |
| **RepairVerification** | 单次验证 | INV-06 必须关联一个有 RepairAction 记录的 RepairOrder；FAILED 验证必须有原因说明 |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-07**（同一上下文外事件不重复触发故障报告）→ `MalfunctionReportService` 领域服务 + source_event_id 事件去重表
> - **INV-08**（报废审批通过后工单关闭）→ `ScrapDecisionProjection` 读模型订阅 `eam.asset.scrap` + application_id 去重表
> - **INV-09**（报废驳回/撤回后工单退回诊断）→ `ScrapDecisionProjection` 读模型订阅 `eam.asset.scrap` + application_id 去重表
> - **INV-10**（上下文外事件只消费不重发）→ 本上下文将订阅事件包装为 `MalfunctionReported` / `RepairOrderClosed` 等本上下文语义事件后发布，下游可凭主题前缀区分来源

| RepairSuggested | 点检保养上下文 | 点检异常建议报修 | ReportMalfunction(source=INSPECTION)（确认需维修时） |

### 6.2 本上下文内部策略

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|
| MalfunctionReported(severity=CRITICAL) | 创建紧急工单 + 通知异常处理上下文 | CreateUrgentRepairOrder + 发布 LineStoppageRequested |
| MalfunctionReported(severity≠CRITICAL) | 创建普通工单 | CreateRepairOrder |
| RepairOrderCreated | 自动/人工派工 | DispatchRepairOrder |
| 派工后 SLA 响应超时 | 升级到 L2 | EscalateResponse |
| 维修中 SLA 修复超时 | 升级到 L3 或外协 | EscalateRepair / DispatchExternalRepair |

## 8. 与台账上下文的协作边界

本上下文与台账上下文共享 `asset_id` 但**视角不同**，必须对齐避免事件混淆——同一个 `asset_id` 在两个限界上下文里讲的不是同一件事。

| 维度 | 本上下文（维修上下文） | 台账上下文 |
|------|--------------|--------|
| 视角 | 纠正性运维视角（"故障怎么修、备件怎么用、修不好怎么办"） | 资产管理视角（"资产能不能用"） |
| 存储 | 维修工单 / 诊断记录 / 维修动作 / 备件消耗 / 验证记录 | 资产身份 / 归属 / 状态 / 可用性投影 / 累积度量 |
| 判定 | 故障可修复性判定（REPAIRABLE / NOT_ECONOMICAL / NOT_REPAIRABLE） | 资产可用性综合判定（综合所有阻塞维度） |
| 事件性质 | 维修执行结果事实：RepairCompleted / ScrapRecommendation | 资产生命周期与可用性事实：AssetSuspended / AssetRetired |
| 流程所有权 | 报修 → 派工 → 诊断 → 维修 → 验证 → 关单 | 入账 → 投用 → 停用 → 复用 → 报废 |

**事件流向**：

```
本上下文 ──→ repair.order.completed ──→ 台账上下文（ReinstateAsset）
本上下文 ──→ repair.scrap.recommendation ──→ 台账上下文（发起报废审批）
台账上下文 ──→ eam.asset.lifecycle(AssetSuspended) ──→ 本上下文（评估是否报修）
台账上下文 ──→ eam.asset.lifecycle(AssetRetired) ──→ 本上下文（关闭未完成工单）
台账上下文 ──→ eam.asset.scrap(Approved/Rejected/Withdrawn) ──→ 本上下文（关单/退回诊断）
台账上下文 ──→ eam.asset.availability ──→ 本上下文（只读，辅助判断资产是否已停用）
```

> **重要**：报废建议的**审批与执行**归台账上下文（`ScrapApplication` 聚合），本上下文只发布 `ScrapRecommendation` 建议，并消费审批结果驱动工单状态流转。

## 9. 与点检保养上下文的协作边界

本上下文与点检保养上下文是"纠正"与"预防"的关系，两者通过建议性事件协作，互不直接指挥。

| 维度 | 本上下文（维修上下文） | 点检保养上下文 |
|------|--------------|--------|
| 视角 | 纠正性运维视角（故障后修复） | 预防性运维视角（计划性检查/保养） |
| 存储 | 维修工单 / 诊断 / 备件消耗 | 点检保养规则 / 计划 / 任务 / 结果 |
| 判定 | 故障可修复性判定 | 度量值与阈值比较，判定 Due / Overdue / LifeExceeded |
| 事件性质 | 维修执行结果事实 | 调度信号与执行结果 |
| 流程所有权 | 报修 → 派工 → 诊断 → 维修 → 验证 → 关单 | 规则 → 计划 → 任务 → 执行 → 签核 |

**协作场景**：

| 协作场景 | 方向 | 事件 | 说明 |
|---------|------|------|------|
| 点检异常建议报修 | 点检保养上下文 → 本上下文 | `RepairSuggested` | 点检 FAIL 后建议维修 |
| 维修完成建议保养 | 本上下文 → 点检保养上下文 | `MaintenanceSuggested` | 维修过程中发现潜在保养需求 |
| 维修后触发点检验证 | 本上下文 → 点检保养上下文 | `InspectionRequested` | 维修验证方式选择"点检验证"时 |

> 本上下文**不直接创建点检任务**——仅发布 `InspectionRequested` 建议，由点检保养上下文决定是否创建。点检保养上下文**不直接创建维修工单**——仅发布 `RepairSuggested`，由本上下文决定是否创建 RepairOrder。保持各限界上下文自治。
