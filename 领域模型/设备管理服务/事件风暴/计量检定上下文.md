# 计量检定上下文（Calibration Bounded Context）

> **限界上下文**：计量检定上下文
> **统一语言**：CalibrationRule（检定规则）, CalibrationPlan（检定计划）, CalibrationTask（检定任务）, CalibrationCertificate（检定证书）, CalibrationResult（检定结果）, ExternalLab（外部检定机构）, CalibrationHistory（检定历史）, RepairFeasibility（处置可行性）, LeadTime（送检提前期）
> **核心关注**：以"事件风暴"方式刻画**计量器具**从检定规则定义、周期检定计划、送检 / 收回流程、证书管理到过期预警的完整流程。本上下文负责"**何时该检、怎么检、证书怎么管**"；**不负责**计量器具的身份数据与可用性状态——那归设备工装台账限界上下文（下文简称「台账上下文」）。

## 0. 设计原则与边界澄清

计量检定上下文是设备管理服务下的**计量管理子域**，是 PCBA + 整机组装混合型车间所有"计量器具检定活动"的管理权威源。

**上下文内职责**（本上下文负责）：
- 检定**规则定义**——检定周期、检定方式（内检 / 外检）、检定项目、外部机构绑定
- 检定**计划生成**——根据规则与证书有效期，自动生成周期检定计划
- **送检 / 收回流程**——计量器具送检（暂停使用）、检定执行、收回（恢复使用）
- **证书管理**——证书签发、有效期维护、过期检测（本上下文为证书有效期权威源）
- **过期预警**——证书即将过期 / 已过期的预警与通知

**上下文外职责**（本上下文**不**负责）：
- 计量器具身份、归属、可用性状态 → 台账上下文
- 设备 / 工装的点检 / 保养 → 点检保养上下文
- 故障报修 / 维修执行 / 备件消耗 → 维修上下文
- 设备通信与数据采集 → 设备数据接入上下文
- 资产报废审批流程 → 台账上下文（本上下文仅发布检定不合格的处置可行性，由台账上下文发起报废申请）
- 移动端 UI 交互、消息推送机制 → 应用层
- SOP 文档版本管理 → 文档管理（本上下文仅引用 SOP ID）

**为什么不放点检保养上下文**：
1. **法规属性不同**——计量检定受国家计量法规强制约束，证书具法律效力；点检保养是内部管理行为，无强制法规。两者合规要求完全不同。
2. **流程特征不同**——检定涉及外部实验室、法定证书、追溯链；点检保养是纯内部活动。送检 / 收回流程是本上下文独有。
3. **证书管理独有**——检定证书有唯一编号、法定有效期、签发机构，是本上下文的核心聚合，点检保养上下文无此概念。
4. **判定结果的法律含义不同**——检定不合格意味着"法定不可用"，必须立即停用；点检 FAIL 是内部判定，处理流程更灵活。

**为什么不放台账上下文**：
1. **职责分离**——台账上下文关注"资产是谁、归谁、能不能用"，本上下文关注"检定活动怎么管"。混在一起台账上下文变成"台账 + 计量调度"混合体。
2. **生命周期节奏不同**——资产生命周期以年为单位，检定周期以月为单位，变更频率差异大。
3. **与台账上下文已有设计一致**——台账上下文 §0 / §5 已明确"检定周期设定、送检 / 收回流程、证书签发 / 过期判定全部归计量检定上下文"，台账上下文仅存 `cert_valid_until` 当前值并消费本上下文的证书事件。

## 1. 计量器具管理特征与检定规则

### 1.1 主语清单

本上下文处理的对象按"聚合根"分类，决定了事件链与触发源的差异。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

| 聚合根 | 典型对象 | 关键管理特征 | 上下文外触发源 |
|--------|---------|-------------|---------------|
| **CalibrationRule**（检定规则） | 卡尺检定规则、扭矩扳手检定规则、温度计检定规则 | 按器具分类定义周期 / 方式 / 项目；ABC 分类决定严格度 | 台账上下文 `AssetCommissioned(asset_kind=INSTRUMENT)` 激活匹配规则 |
| **CalibrationPlan**（检定计划） | 2026-W24 周期检定计划 | 单周期内多器具汇总；每条明细关联规则与器具 | 由本上下文 Due 事件触发生成 |
| **CalibrationTask**（检定任务） | 单台卡尺本季度送检任务 | 单次检定执行单元；内检 / 外检分支；送检即暂停使用 | — |
| **CalibrationCertificate**（检定证书） | 内检证书 CI-2026-00012、外检证书 CE-2026-00007 | 法定有效期；本上下文为权威源；过期触发冻结 | — |

### 1.2 计量器具的特殊管理要求

在 PCBA + 整机组装混合型车间，计量器具具有以下特殊性：

| 特征 | 说明 | 管理含义 |
|------|------|---------|
| **强制周期检定** | 国家计量法规定必须按周期送检 | 到期必须送检，不可跳过 |
| **证书有效期** | 每次检定签发证书，有法定有效期 | 超期即不可使用，必须冻结 |
| **内检 vs 外检** | 部分器具可内部检定，部分必须外检 | 流程分支：内检在厂完成，外检需送出 |
| **ABC 分类** | A 类（强制检定）、B 类（依法管理）、C 类（一次性 / 免检） | 不同分类管理严格度不同 |
| **追溯链** | 检定用标准器本身也需检定 | 形成检定溯源链 |

### 1.3 检定规则模型

```
CalibrationRule
├── rule_id              : 规则唯一标识
├── target_classification: 计量器具分类（如"卡尺""扭矩扳手""温度计"）
├── abc_category         : A | B | C
├── calibration_cycle    : 检定周期（月）
├── calibration_method   : INTERNAL（内检） | EXTERNAL（外检）
├── external_lab_id      : 外部检定机构（外检时必填）
├── calibration_items[]  : 检定项目模板
│   ├── item_name        : 项目名称
│   ├── standard_value   : 标准值
│   ├── tolerance        : 允差
│   └── measurement_unit : 单位
├── certificate_template : 证书模板（内检时）
├── lead_time_days       : 送检提前期（天）——证书到期前多久开始送检
├── responsible_role     : 负责角色
└── enabled              : 是否启用
```

**ABC 分类管理策略**：

| 分类 | 检定方式 | 证书管理 | 超期处理 |
|------|---------|---------|---------|
| A 类（强制检定） | 必须外检到法定机构 | 法定证书，有效期严格 | 超期立即冻结，法规违规告警 |
| B 类（依法管理） | 可内检或外检 | 内部证书或外部证书 | 超期冻结，无法规违规告警 |
| C 类（免检 / 一次性） | 首件检定后免周期检定 | 仅首次证书 | 无周期检定要求 |

> 🔴 **热点：ABC 分类来源**——`abc_category` 由谁维护？建议：分类属计量管理属性，归本上下文的 `CalibrationRule`（按 `target_classification` 绑定）；台账上下文的 `MeasuringInstrument` 仅存当前分类快照供查询，不做法定判定。待与计量管理员确认是否允许同分类下个别器具例外（如某台 A 类降级为 B 类）。

## 2. 事件风暴 — 规则定义与触发

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点（正文用 🔴 前缀）

### 2.1 规则生命周期

```
[参与者:计量管理员]
        --【命令:定义检定规则 DefineCalibrationRule】--> 【聚合:CalibrationRule】
                                                        |
                                                        v
                                                【事件:检定规则已定义 CalibrationRuleDefined】
                                                 (rule_id, target_classification, calibration_cycle, method)
                                                        |
                                                【策略:规则生效后监听证书有效期与日历周期】
                                                        |
                                                        v
                                                【命令:激活规则 ActivateRule】
                                                【事件:规则已激活 RuleActivated】
```

**规则变更事件**：

| 事件 | 触发场景 | 载荷要点 |
|------|---------|---------|
| CalibrationRuleDefined | 首次定义检定规则 | rule_id, target_classification, abc_category, cycle, method, external_lab_id? |
| CalibrationRuleUpdated | 规则内容变更 | changed_fields, effective_from（立即 / 下个周期生效） |
| RuleActivated / RuleDeactivated | 启用 / 停用规则 | — |

> 🔴 **热点：规则变更的生效策略**——变更已有规则时，已生成的计划和任务是否受影响？建议：**已生成的计划 / 任务不受影响**（快照语义），变更从下个周期生效，避免执行中途规则变更导致混乱。

### 2.2 状态流转

**CalibrationRule 状态机**：

```
                Define
                  |
                  v
              +--------+   Activate    +--------+   Deactivate   +--------------+
              | DRAFT  |-------------->| ACTIVE |---------------->| DEACTIVATED  |
              +--------+               +--------+                 +--------------+
                  |                        ^                          |
                  |                        |    Activate              |
                  |                        +--------------------------+
                  |
                  |  Obsolete（终态）
                  v
              +----------+
              | OBSOLETE |  终态——之后任何事件均拒收
              +----------+
```

- DRAFT → ACTIVE：`ActivateRule`，前置 = 检定项目模板非空
- ACTIVE ⇄ DEACTIVATED：`ActivateRule` / `DeactivateRule`，可逆
- 任意非终态 → OBSOLETE：`ObsoleteRule`，终态——之后任何事件均拒收

**CalibrationTask 状态机**：见 §4.5（CalibrationTask 在计划生成与任务调度后才有完整语义，状态机随送检 / 检定执行流程一并给出）。

> 上下文外事件消费与投影清单见 §6（本上下文被动响应台账上下文的资产生命周期事件，用于规则激活 / 触发判定 / 任务取消）。

### 2.3 触发与 Due 事件生成

检定计划的触发源有两个：**证书有效期临近**和**日历周期到达**。两者均由本上下文自身的调度引擎判定，本上下文是证书有效期与检定周期的权威源。

```
【触发源1:证书临期】本上下文定时器扫描证书
        -- 策略: valid_until - lead_time_days ≤ now
                |
                v
        【命令:标记检定到期 MarkCalibrationDue】
                |
                v
        【事件:检定已到期 CalibrationDue】(instrument_id, rule_id, due_reason: CertExpiring)

【触发源2:日历周期扫描】本上下文定时器扫描规则 + 器具列表
        -- 策略: 距上次检定达到检定周期
                |
                v
        【命令:标记检定到期 MarkCalibrationDue】
                |
                v
        【事件:检定已到期 CalibrationDue】(instrument_id, rule_id, due_reason: CycleReached)
```

**触发判定逻辑**：

| 触发类型 | 判定逻辑 | 事件 |
|----------|---------|------|
| CERT_EXPIRING | 本上下文定时器扫描证书，`valid_until - lead_time_days ≤ now` | CalibrationDue(due_reason=CertExpiring) |
| CYCLE_REACHED | 本上下文定时器扫描规则 + 器具列表，距上次检定 ≥ `calibration_cycle` | CalibrationDue(due_reason=CycleReached) |
| CERT_EXPIRED | `valid_until ≤ now`，紧急检定 | CalibrationDue(due_reason=CertExpired) |

> 🔴 **热点：证书有效期数据源**——本上下文签发证书后自身拥有 `valid_until`，过期检测由本上下文定时器独立执行，**不依赖**台账上下文事件触发。台账上下文发布的 `InstrumentCertValidityUpdated` 仅作为**补充校验**（台账上下文可能经 ERP / 批量导入更新 `cert_valid_until`，本上下文需感知以避免漏检）。主触发逻辑仍使用本上下文自身的证书数据。是否保留该补充订阅待与 SME 确认（见 §6）。

## 3. 事件风暴 — 计划生成与任务调度

### 3.1 Due → Plan

```
CalibrationDue
        |
        v
【策略:到期自动生成检定计划】
        |
        v
【命令:生成检定计划 GenerateCalibrationPlan】--> 【聚合:CalibrationPlan】
                                                    |
                                                    v
                                            【事件:检定计划已生成 CalibrationPlanGenerated】
                                             (plan_id, plan_items[], deadline)
```

**计划结构**：

```
CalibrationPlan
├── plan_id          : 计划唯一标识
├── plan_items[]     : PlanItem[]（每条对应一台器具的检定明细）
│   ├── instrument_id: 目标器具
│   ├── rule_id      : 来源规则
│   ├── due_reason   : 到期原因
│   └── deadline     : 该条明细的执行截止时间
├── plan_horizon     : 计划周期（如 2026-W24）
├── status           : DRAFT → ACTIVE → PARTIALLY_COMPLETED → COMPLETED / OVERDUE
└── generated_at     : 生成时间
```

### 3.2 Plan → Task

```
CalibrationPlanGenerated
        |
        v
【策略:计划激活后拆解为检定任务】（每条 plan_item → 一个 task）
        |
        v
【命令:创建检定任务 CreateCalibrationTask】--> 【聚合:CalibrationTask】
                                                    |
                                                    v
                                            【事件:检定任务已创建 CalibrationTaskCreated】
                                             (task_id, instrument_id, method: INTERNAL/EXTERNAL,
                                              assignee, deadline)
```

> 🔴 **热点：同一器具重复未完成任务**——同一器具可能因证书临期与日历周期双触发产生两条 Due。建议：生成任务前由 `CalibrationDispatchService` 查询读模型，若该器具已有未完成检定任务则合并而非重复创建（跨聚合不变式 INV-C3，由领域服务 + 唯一索引保证）。

## 4. 事件风暴 — 送检与检定执行

### 4.1 送检（暂停使用）

```
CalibrationTaskCreated
        |
        v
【命令:送检 DispatchCalibration】--> 【聚合:CalibrationTask】
                                        |
                                        v
                                【事件:已送检 CalibrationDispatched】
                                 (task_id, instrument_id, dispatch_method: INTERNAL/EXTERNAL,
                                  dispatched_at, expected_return_date)
                                        |
                                        v
                                发布到 calibration.task.dispatched
                                  → 台账上下文消费：SuspendAsset(reason=Calibrating)
```

> **关键**：送检意味着计量器具**暂停使用**。本上下文发布 `CalibrationDispatched`，台账上下文消费后执行 `SuspendAsset(reason=Calibrating)`。外检时器具物理离开车间，内检时器具在车间内但标记为"检定中"——两种情况对台账上下文而言都是"不可派工"，统一由 `Calibrating` 阻塞原因表达。

### 4.2 内检执行

```
[参与者:计量员/内检员]
        --【命令:执行内检 PerformInternalCalibration】--> 【聚合:CalibrationTask】
                                                            |
                                                            v
                                                    【事件:内检已执行 InternalCalibrationPerformed】
                                                     (task_id, instrument_id, items[], results[])
                                                            |
                                                ┌──── 检定结果 ────┐
                                                v                   v
                                            PASS                FAIL
                                                |                   |
                                                v                   v
                                    【命令:签发内检证书         【命令:记录检定不合格
                                     IssueInternalCertificate】  RecordCalibrationFailed】
                                                |                   |
                                                v                   v
                                    【事件:检定证书已签发          【事件:检定不合格
                                     CalibrationCertificateIssued  CalibrationFailed】
                                     (cert_no, instrument_id,      (instrument_id, task_id,
                                      valid_until, issued_by,       failed_items[],
                                      issued_at)                    repair_feasibility:
                                                |                    RECALIBRATABLE | NOT_ECONOMICAL | NOT_REPAIRABLE)
                                                v                        |
                                    发布到 calibration.certificate.issued   v
                                    → 台账上下文消费：                ┌──── repair_feasibility ────┐
                                      UpdateCertValidity              v                v                   v
                                      + ClearBlocking(CALIBRATION_*) RECALIBRATABLE   NOT_ECONOMICAL    NOT_REPAIRABLE
                                      + RecomputeAndDecideReinstate   |                |                   |
                                                                    可重新调校       台账上下文消费：       台账上下文消费：
                                                                    安排下次检定     SuspendAsset           SuspendAsset
                                                                                   (reason=CalibrationFailed) (reason=CalibrationFailed)
                                                                                   + 可发起报废审批        + 可发起报废审批
```

```

                                                        v               v
                                                    PASS            FAIL
                                                        |               |
                                                        v               v
                                        【命令:录入外部证书          【命令:记录外检不合格
                                         RegisterExternalCertificate】 RecordExternalCalibrationFailed】
                                                        |               |
                                                        v               v
                                        【事件:检定证书已签发          【事件:检定不合格
                                         CalibrationCertificateIssued  CalibrationFailed】
                                         (cert_no, instrument_id,        (instrument_id, task_id,
                                          valid_until, issued_by,         failed_items[],
                                          issued_at, external_lab_id)      external_lab_id,
                                                                           repair_feasibility)

                                        发布到 calibration.certificate.issued  发布到 calibration.task.failed
```

### 4.4 收回

```
CalibrationCertificateIssued / CalibrationFailed
        |
        v
【命令:收回器具 ReceiveCalibratedInstrument】--> 【聚合:CalibrationTask】
                                                    |
                                                    v
                                            【事件:检定器具已收回 InstrumentCalibrationReceived】
                                             (task_id, instrument_id, received_at, received_by)
                                                    |
                                            仅对 PASS 的情况：
                                            台账上下文已通过 CalibrationCertificateIssued 消费
                                            执行了 ReinstateAsset，收回事件为补充记录（物理回位登记）
```

> **设计说明**：`InstrumentCalibrationReceived` 对 PASS 场景是"物理回位"补充记录——资产的可用性恢复已由 `CalibrationCertificateIssued` 驱动台账上下文完成；对 FAIL 场景，器具收回但资产仍保持冻结（`reason=CalibrationFailed`），等待处置决策。

### 4.5 状态流转（CalibrationTask）

```
                         [PENDING]
                            |
                  ┌─────────┴────────── 报废/退役 Cancel（任意非终态, INV-C5）──────────┐
                  │                     │                                              │
                  │           送检 Dispatch                                            │
                  │            (前置: PENDING)                                         │
                  │                     │                                              │
                  │                     v                                              │
                  │             [DISPATCHED] ─────────────────── 报废/退役 Cancel ─────┤
                  │                     │                                              │
                  │   内检执行 / 登记外检结果                                           │
                  │   (前置: DISPATCHED)                                               │
                  │                     │                                              │
                  │              ┌──────┴───────┐                                      │
                  │              v               v                                     │
                  │        [CERT_ISSUED]     [FAILED]                                  │
                  │     (PASS, 已签发证书)   (检定不合格)                                │
                  │              │               │                                     │
                  │       收回 Receive      收回 Receive                                │
                  │       (前置: CERT_ISSUED) (前置: FAILED)                           │
                  │              │               │                                     │
                  │              v               v                                     │
                  └────────▶ [RECEIVED] ◀─────────────────────────────────────────────┘
                                  (终态—之后任何事件均拒收, INV-C2)
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| PENDING → DISPATCHED | DispatchCalibration | 当前 PENDING | 调度引擎（计划激活后）/ 计量管理员 |
| DISPATCHED → CERT_ISSUED | IssueInternalCertificate / RegisterExternalCertificate | 当前 DISPATCHED 且检定结果 PASS | 内检员 / 计量管理员 |
| DISPATCHED → FAILED | RecordCalibrationFailed / RecordExternalCalibrationFailed | 当前 DISPATCHED 且检定结果 FAIL | 内检员 / 计量管理员 |
| CERT_ISSUED → RECEIVED | ReceiveCalibratedInstrument | 当前 CERT_ISSUED | 计量管理员（物理回位登记） |
| FAILED → RECEIVED | ReceiveCalibratedInstrument | 当前 FAILED | 计量管理员 |
| 任意非终态 → RECEIVED 之后 | （任何） | 终态，拒收（INV-C2） | — |
| 任意非终态 → CANCELLED | CancelCalibrationTask | 资产退役 / 报废（INV-C5） | 台账上下文 `AssetRetired` 消费后触发 |

> 🔴 **热点：FAIL 后是否允许复检**——`repair_feasibility=RECALIBRATABLE` 时，器具调校后可重新检定。建议：原任务进入 `RECEIVED` 终态，调校后由新一轮 `CalibrationDue` 触发新任务，不在原任务上回退状态（保持状态机单向、终态不可逆）。

## 5. 事件风暴 — 证书管理与过期预警

### 5.1 证书生命周期

```
CalibrationCertificateIssued
        |
        v
【聚合:CalibrationCertificate】
        |
        v
    [VALID] ──(valid_until - lead_time_days ≤ now)──► [EXPIRING_SOON] ──(valid_until ≤ now)──► [EXPIRED]
                     |                                           |
                     v                                           v
            【事件:证书即将过期                                 【事件:证书已过期
             CalibrationExpiringSoon】                          CalibrationExpired】
             (instrument_id, cert_no, valid_until,              (instrument_id, cert_no,
              days_remaining)                                    expired_since)
                     |                                           |
                     v                                           v
            发布到 calibration.certificate.expiring            发布到 calibration.certificate.expired
            → 通知服务（不触发状态变更）                       → 台账上下文消费：SuspendAsset(reason=CalibrationExpired)
                                                              + 本上下文自动触发 CalibrationDue(due_reason=CertExpired)
```

**每条边的可触发性**：

| 边 | 触发条件 | 谁来发命令 | 后续 |
|----|---------|-----------|------|
| VALID → EXPIRING_SOON | `valid_until - lead_time_days ≤ now` | 本上下文定时器 | 发布 CalibrationExpiringSoon，通知 |
| EXPIRING_SOON → EXPIRED | `valid_until ≤ now` | 本上下文定时器 | 发布 CalibrationExpired；台账上下文冻结；本上下文触发紧急检定 |
| → EXPIRED 之后 | （任何） | 终态，拒收（INV-C4） | — |

### 5.2 证书模型

```
CalibrationCertificate
├── cert_no         : 证书编号（全局唯一）
├── instrument_id   : 计量器具 ID
├── issued_by       : 签发机构（内部 / 外部实验室名称）
├── issued_at       : 签发日期
├── valid_from      : 有效起始日期
├── valid_until     : 有效截止日期
├── calibration_result: PASS（仅 PASS 才签发证书）
├── calibration_items[]: 检定项目与结果
├── external_lab_id : 外部实验室 ID（外检时）
├── status          : VALID | EXPIRING_SOON | EXPIRED | REVOKED
└── attachment_urls[]: 证书扫描件 / 电子证书
```

> 🔴 **热点：证书编号规则**——内检证书与外检证书是否统一编号？建议：**统一前缀区分**——内检 `CI-YYYY-NNNNN`，外检 `CE-YYYY-NNNNN`，便于区分但统一管理。`cert_no` 全局唯一由数据库唯一索引保证（INV-C1）。

### 5.3 过期预警策略

| 预警级别 | 触发条件 | 动作 |
|---------|---------|------|
| 预警（黄色） | `valid_until - 30 天 ≤ now` | 通知计量管理员，准备送检 |
| 临期（橙色） | `valid_until - lead_time_days ≤ now` | 自动触发 `CalibrationDue(due_reason=CertExpiring)`，生成检定计划 |
| 过期（红色） | `valid_until ≤ now` | 发布 `CalibrationExpired`，台账上下文冻结资产；本上下文自动触发紧急检定 |

| InstrumentCertValidityUpdated(cert_no, valid_until) | 台账上下文 | `eam.asset.metric` | MarkCalibrationDue（仅当新有效期触发临期 / 过期判定时） | — | 补充触发证书过期检测（主触发为本上下文定时器） |

>
> 🔴 **热点：InstrumentCertValidityUpdated 补充订阅是否保留**——本上下文已是证书有效期权威源，订阅台账上下文回写的 `InstrumentCertValidityUpdated` 存在循环风险。建议：**保留为补充校验**，仅当台账上下文经 ERP / 批量导入更新了 `cert_valid_until` 时用于感知；本上下文自身签发证书后不依赖该事件触发。若确认所有证书变更都经本上下文签发，可移除该订阅。待 SME 决策。

**幂等与去重**（INV-C6）：
- `InstrumentCertValidityUpdated` 可能重复投递——本上下文以 `(instrument_id, cert_no, valid_until)` 去重，相同值不重复触发 `MarkCalibrationDue`。
- `AssetCommissioned` / `AssetRetired` 以 `asset_id + event_id` 幂等消费，重复事件不重复激活 / 取消。

## 7. 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **CalibrationRule** | 单条检定规则 | INV-R1 同一 `target_classification` 不重复；INV-R2 检定周期 > 0；INV-R3 外检时 `external_lab_id` 必填 |
| **CalibrationPlan** | 单个周期检定计划 | INV-P1 每条 plan_item 必须关联已激活的 CalibrationRule；INV-P2 plan 内 instrument 不重复 |
| **CalibrationTask** | 单次检定任务 | INV-C1（同下）；INV-C2 `RECEIVED` / `CANCELLED` 为终态，之后任何事件均拒收；INV-C4 送检后必须记录结果才能收回；PASS 必须签发证书 |
| **CalibrationCertificate** | 单张检定证书 | INV-T1 `cert_no` 全局唯一；INV-T2 签发时 `calibration_result` 必须为 PASS；INV-T3 `valid_until > issued_at`；INV-T4 `EXPIRED` 为终态 |

> 跨聚合不变式由谁保证：
> - **INV-C1**（`cert_no` 全局唯一）→ 数据库唯一索引 `(cert_no)`，非聚合根自身可校验。
> - **INV-C3**（同一器具同一时间不可有重复未完成检定任务）→ `CalibrationDispatchService` 领域服务在创建任务前查询读模型 + 数据库唯一索引 `(instrument_id, status ∈ {PENDING, DISPATCHED})` 共同保证。
> - **INV-C5**（资产退役时取消未执行任务）→ 由 §6 消费 `AssetRetired` 后 `CancelCalibrationTask` 触发，跨上下文不变式，由事件订阅 + 任务状态机共同保证。
> - **INV-C6**（上下文外事件幂等消费）→ 事件去重表 `(source, event_id)` + 业务键去重。

## 10. 与台账上下文的协作边界

本上下文与 [设备工装台账上下文.md](设备工装台账上下文.md) 共享 `instrument_id` 但**视角不同**，必须对齐避免事件混淆：

| 维度 | 本上下文（计量检定上下文） | 台账上下文 |
|------|-------------------|--------|
| 视角 | 检定活动**怎么管**、证书**怎么发** | 资产**能不能用** |
| 存储 | 检定规则、检定周期、证书全量数据（含 `valid_until`） | `cert_valid_until`（仅当前值快照） |
| 判定 | **本上下文判定**——证书有效期归本上下文管理，过期检测由本上下文定时器执行 | **台账上下文消费**——收到 `CalibrationExpired` 后冻结 |
| 事件性质 | 证书事实：Issued / Failed / Expired；调度信号：Due | 资产生命周期与可用性事实 |
| 流程所有权 | 规则 → 计划 → 送检 → 检定 → 证书 → 收回 | 资产建档 → 投用 → 停用 → 报废 |

**事件流向**：

```
本上下文 ──→ calibration.task.dispatched        ──→ 台账上下文（SuspendAsset(reason=Calibrating)）
本上下文 ──→ calibration.certificate.issued     ──→ 台账上下文（UpdateCertValidity + ClearBlocking(CALIBRATION_*) + RecomputeAndDecideReinstate）
本上下文 ──→ calibration.task.failed            ──→ 台账上下文（SuspendAsset(reason=CalibrationFailed)；不可修复→可发起报废审批）
本上下文 ──→ calibration.certificate.expiring   ──→ 通知服务（不变更状态）
本上下文 ──→ calibration.certificate.expired    ──→ 台账上下文（SuspendAsset(reason=CalibrationExpired)）
台账上下文 ──→ eam.asset.metric(InstrumentCertValidityUpdated) ──→ 本上下文（辅助触发判定，见 §6 热点）
台账上下文 ──→ eam.asset.lifecycle              ──→ 本上下文（Commissioned→激活规则 / Suspended→记录 / Retired→停用规则+取消任务）
```

> **关于证书有效期事件中转的设计说明**：本上下文签发证书后自身拥有 `valid_until`，但仍订阅台账上下文的 `InstrumentCertValidityUpdated` 事件作为**双重保障**：(1) 台账上下文可能经 ERP / 批量导入更新 `cert_valid_until`，本上下文需感知这些变更避免漏检；(2) 证书过期检测由本上下文定时器独立执行，**不依赖**台账上下文事件触发，`InstrumentCertValidityUpdated` 仅作补充校验。是否保留该补充订阅见 §6 🔴 热点。
