# 设备工装台账限界上下文（Asset Registry Bounded Context）

> **限界上下文**：设备工装台账
> **统一语言**：Asset（资产·抽象）, Equipment（设备）, Fixture（工装夹具）, MeasuringInstrument（计量器具）, AssetIdentifier（资产编码）, AssetSpecificationProfile（能力规格画像）, FixtureMatchingRule（工装匹配规则）, AvailabilityState（可用性状态）, AvailabilityProjection（可用性投影）, RunHourAccrual（运行时长累积）, CycleAccrual（使用次数累积）, CertValidity（检定证书有效期·只读引用）, BlockingReason（阻塞原因）, ScrapApplication（报废申请）
> **核心关注**：以"事件风暴"方式刻画**实物资产从入账到报废**的全生命周期事件链。本上下文只对资产的**身份、归属、状态、生命周期**主数据，以及**可用性聚合视图**负责；**不负责**点检 / 保养 / 检定 / 维修的调度与执行流程——那些归「点检保养限界上下文」「计量检定限界上下文」「维修限界上下文」独立建模，本上下文仅**消费其结果事件**以驱动资产状态变更与可用性重算。

---

## 0. 设计原则与边界澄清

设备工装台账限界上下文是设备管理服务下的**主数据子域**，是 PCBA + 整机组装混合型车间所有"实物资产"运行态的权威源。

**上下文内职责**（本上下文负责）：
- 设备、工装夹具、计量器具的**入账、赋码、建档、变更、报废**
- 资产**位置、归属、责任人、状态**的全生命周期记录
- 资产**可用性聚合视图**的维护与发布——消费上下文外运维事件，重算可用性，对外发布
- 资产**能力规格画像**与**工装匹配规则**的发布（供制造资源服务工艺设计引用）
- 资产**累积度量**的维护（运行时长、使用次数、检定证书有效期）——仅存储当前值，阈值与触发规则归点检保养限界上下文 / 计量检定限界上下文

**上下文外职责**（本上下文**不**负责）：
- 设备数据采集行为本身 → 设备数据接入限界上下文
- 工艺路线 / BOM / 工序定义 → 制造资源服务（仅引用本上下文规格读模型）
- 派工时的资产匹配与占用 → 生产执行服务（订阅本上下文可用性视图）
- OEE / 设备效率 / 故障率分析 → 数据应用限界上下文
- **点检 / 保养 / 检定 / 维修的完整流程**——规则定义、计划生成、任务下发、执行登记、签核、SOP → **点检保养限界上下文 / 计量检定限界上下文 / 维修限界上下文**（本上下文仅消费其结果事件）
- 备件库存、备件 BOM → 后续「备件管理限界上下文」
- 资产折旧、财务接口 → 本上下文的财务折旧适配层（ACL）

**为什么不拆"设备台账限界上下文 + 工装台账限界上下文"两个并列限界上下文**：
1. 两类资产**生命周期同构**——入账 → 投用 → 停用 → 报废，事件骨架完全可复用，拆开等于写两份。
2. **可用性引擎共用**——无论哪类资产，可用性重算逻辑同构；拆开会逼出双份。
3. **对外契约一致**——制造资源服务工艺路线绑定的是"资源能力规格"抽象，不区分设备还是工装；本上下文应以**统一资产规格读模型**对外暴露。
4. **内部聚合区分即可**——同一上下文内并存 Equipment / Fixture / MeasuringInstrument 三个聚合根，共享主链 + 各自特定事件。
5. **组织一致**——本车间形态下，设备和工装由同一工程团队管理，没有拆分所必需的组织鸿沟。

**为什么点检 / 保养 / 检定 / 维修不放本上下文**：
- `InspectionPlan` / `MaintenancePlan` / `CalibrationPlan` 是**调度聚合**——关心"何时该做、谁来做、怎么做"，不是资产属性。
- 台账上下文关注"**资产能不能用**"，运维限界上下文关注"**运维活动怎么管**"——生命周期、责任人、变更频率完全不同。
- 混在一起，台账上下文就变成"台账 + 运维调度"混合体，边界模糊，修改点检规则就要动台账代码。

## 1. 资产分类与台账形态

按"管理特征"分类，决定了事件链与累积度量的差异。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

| 聚合根 | 典型对象 | 关键管理特征 | 累积度量 | 上下文外触发源 |
|--------|---------|-------------|----------|-----------|
| **Equipment**（设备） | 锡膏印刷机、AOI、波峰焊、镭雕机、烧录电脑、测试电脑、智能电批控制器、线边仓库、老化房 | 位置相对固定；绑定采集通道 | run_hours（运行时长） | 设备数据接入限界上下文时长聚合 |
| **Fixture**（工装夹具） | 钢网、SMT 治具、FCT 测试治具、烧录治具、装配 / 对位夹具、工装托盘 | 配套使用（绑产品 / 工序 / 设备）；有借出归还 | accumulated_cycles（使用次数） | 生产执行服务 / 设备数据接入限界上下文上报 |
| **MeasuringInstrument**（计量器具） | 卡尺、千分尺、扭矩扳手、张力计、温度计、湿度计、绝缘电阻仪 | 强制周期检定；有证书有效期；超期即冻结 | cert_valid_until（证书有效期） | 计量检定限界上下文签发 |

> 说明：智能电批的"枪头 / 批头"按工装管理；老化房的"载具 / 治具"按工装管理；老化房温控按设备管理。三类聚合在本上下文内并存，不互相继承，仅共享主链事件名。
>
> **阈值（保养周期、点检周期、寿命上限、检定周期）不在本上下文存储**——它们是运维规则，归点检保养限界上下文 / 计量检定限界上下文（INV-22）。本上下文只存"当前累积值"，上下文外规则引擎根据累积值判定是否触发。

### 2.1 入账与建档

```
【参与者:设备工程师 / 工装管理员 / 计量员】
        --【命令:登记资产 RegisterAsset】--> 【聚合:Asset】
                                                  |
                                                  v
                          【事件:资产已登记 AssetRegistered】(asset_kind: Equipment / Fixture / Instrument)
                                                  |
                          【策略:首次登记自动赋码 + 建档】（AssetRegistrationService 编排）
                                                  |
                                                  v
                          【命令:分配资产编码 AssignAssetIdentifier】
                          【事件:资产编码已分配 AssetIdentifierAssigned】(code, code_scheme)
                                                  |
                                                  v
                          【事件:能力规格已建立 AssetSpecificationProfileEstablished】(spec_keys, spec_values)
                                                  |
                                                  v
                          【命令:初始化可用性投影 InitAvailabilityProjection】
                          【事件:可用性已变更 AssetAvailabilityChanged】(available=false, blocking_reasons=[])
                                                  |
                                            (供制造资源服务读)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| AssetRegistered | 工程师登记新资产 | asset_id, asset_kind, model, vendor, custodian_org | 触发赋码 + 建档（INV-01, INV-02） |
| AssetIdentifierAssigned | 系统分配 / 导入外部码 | asset_code, code_scheme（自编 / 厂商 / 资产标签） | 进入 NEW 状态 |
| AssetAttributeChanged | 属性变更（型号 / 规格 / 责任部门等） | changed_fields, before, after | 触发规格画像更新 |
| AssetClassificationUpdated | 资产分类变更 | new_classification, reason | — |
| AssetSpecificationProfileEstablished / Updated | 能力规格画像首次建立或更新 | spec_keys, spec_values（如：回流焊·峰值 ≥260℃） | 发布到 `eam.asset.specification`（INV-05 前置） |

> 编码方案**——是用统一的"资产标签"（如 `EQ-202606-00123`）还是按类型前缀分段（`EQ-` / `FX-` / `MI-`）？建议**单一标签前缀 + 类型字段**，避免重码冲突；与 ERP / 财务资产编码的对齐归本上下文的 ERP 资产主数据适配层（ACL）。

### 2.2 状态流转

```
                         【NEW】
                           |
            ┌──────────────┼────────────── 报废 Retire（任意非终态均可，INV-08）──────────────┐
            │              │                                                              │
            │      投用 Commission                                                        │
            │       (前置: AssetSpecificationProfileEstablished, INV-05)                   │
            │              │                                                              │
            │              v                                                              │
            │        【IN_SERVICE】 ◀──── 复用 Reinstate ──── (前置: 无任何阻塞维度, INV-07) │
            │              │                              ▲                                 │
            │              │ 停用 Suspend                  │                                 │
            │              │ (前置: IN_SERVICE, 载荷带 reason, INV-06)                       │
            │              v                              │                                 │
            │         【SUSPENDED】 ───────────────────────┘                                 │
            │              │                                                                │
            │              │ 报废 Retire                                                    │
            │              v                                                                v
            └──────────▶ 【RETIRED】 ◀────────────────────────────────────────────────── 【RETIRED】
                                 (终态—之后任何事件均拒收, INV-03)
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| NEW → IN_SERVICE | Commission | 规格画像已建立（INV-05） | 设备工程师（验收完成后） |
| IN_SERVICE → SUSPENDED | Suspend(reason) | 当前 IN_SERVICE；载荷带 reason（INV-06） | 本上下文 `AvailabilityRecomputeService`（消费上下文外事件）/ 工程师（外转、借出） |
| 已 SUSPENDED → 叠加阻塞 | AddBlockingReason(reason) | 当前 SUSPENDED；仅追加 blocking_reasons，不重复状态变更（INV-26） | `AvailabilityRecomputeService`（上下文外事件到达且资产已停用） |
| SUSPENDED → IN_SERVICE | Reinstate | 当前 SUSPENDED 且**所有**阻塞维度均已消除（INV-07） | `AvailabilityRecomputeService.recomputeAndDecideReinstate` |
| 任意非终态 → RETIRED | Retire | 当前 ∈ {NEW, IN_SERVICE, SUSPENDED} | `ScrapApprovalService`（审批通过）/ 工程师（直接退役） |
| → RETIRED 之后 | （任何） | 终态，拒收（INV-03） | — |

**关键事件**

| 事件 | 触发场景 | 不变式 |
|------|---------|--------|
| AssetCommissioned | 验收完成、首次投入产线 | 必须先有 `AssetSpecificationProfileEstablished`（INV-05） |
| AssetSuspended | 故障 / 检定不合格 / 寿命超限 / 通信中断 / 关键报警 / 外转 | 载荷必带 `reason`；前置状态为 IN_SERVICE（INV-06）。已 SUSPENDED 时改用 AddBlockingReason 追加原因（INV-26） |
| AssetReinstated | 维修结束 / 检定通过 / 所有阻塞消除 | 必须有对应的 Suspended 才能复用；综合所有阻塞维度判定，不直接由单一事件触发（INV-07） |
| AssetRetired | 报废、转出、丢失 | 终态——**之后任何事件均拒收**（INV-03）；允许从 NEW / IN_SERVICE / SUSPENDED 任意非终态转入（INV-08） |

> **Suspend / Reinstate 的命令来源**：本上下文自身可发起（外转、借出归还），也可由**上下文外运维事件消费后自动触发**（详见 §2.6）。
>
> **资产已 SUSPENDED 时叠加新阻塞**：当资产已处于 SUSPENDED 状态时，若新的上下文外事件（如已停用设备又通信中断）到达，不重复调用 `Suspend`（会违反状态机），而是通过 `AddBlockingReason` 追加阻塞原因并更新可用性投影（INV-26）。恢复时需所有阻塞维度均消除才执行 `Reinstate`。

### 2.4 累积度量维护

台账上下文维护三类资产的"当前累积值"，供上下文外规则引擎判定触发条件。**本上下文不修改绝对累积值**——只存当前值并透传，阈值判定归点检保养限界上下文 / 计量检定限界上下文（INV-22, INV-23）。

```
【外部系统:设备数据接入限界上下文】
        -- 发布 EquipmentOnline / Offline 区间聚合后的运行时长
                |
                v
        【聚合:Equipment】--【命令:累计运行时长 AccrueRunHour】
                |
                v
        【事件:运行时长已累计 EquipmentRunHourAccrued】(run_hours_delta, accumulated_run_hours, reset_baseline, source)
                |
                v
        发布到 eam.asset.metric（供点检保养限界上下文订阅）
        点检保养限界上下文使用 accumulated_run_hours - reset_baseline 计算偏移量与保养阈值比较

【外部系统:生产执行服务 / 设备数据接入限界上下文】
        -- 上报工装使用次数
                |
                v
        【聚合:Fixture】--【命令:累计使用次数 AccrueCycle】
                |
                v
        【事件:使用次数已累计 FixtureCycleAccrued】(cycles_delta, accumulated_cycles, reset_baseline, source)
                |
                v
        发布到 eam.asset.metric（供点检保养限界上下文订阅）
        点检保养限界上下文使用 accumulated_cycles - reset_baseline 与寿命预警阈值比较，
        使用 accumulated_cycles 直接与 life_limit 比较判定寿命

【外部系统:计量检定限界上下文】
        -- 发布检定证书签发事件
                |
                v
        【聚合:MeasuringInstrument】--【命令:刷新证书有效期 RefreshCertValidity】
                |
                v
        【事件:检定证书有效期已更新 InstrumentCertValidityUpdated】(cert_no, valid_until)
```

> 🔴 **热点：运行时长来源**——`EquipmentRunHourAccrued` 由谁计算？建议由**设备数据接入限界上下文**根据 `EquipmentOnline / Offline` 区间聚合后**周期性发布** `EquipmentRunHourAggregated` 到本上下文订阅；本上下文消费后执行 `AccrueRunHour` 累积为 `EquipmentRunHourAccrued`，不直接采集状态流。
>
> 🔴 **热点：cycles 来源**——`FixtureCycleAccrued` 谁来报？建议**优先由生产执行服务在派工 / 过站时上报**（最权威）；当工装与设备绑定且通过设备采集事件可推断时，由设备数据接入限界上下文通过策略派生上报。
>
> 🔴 **热点：阈值判定归谁**——累积值在本上下文，但"是否达到阈值"的判定与触发归**点检保养限界上下文**。本上下文只发布 `EquipmentRunHourAccrued` / `FixtureCycleAccrued`，不发布 `MaintenanceDue` / `FixtureLifeThresholdReached`——后两者是调度事件，不是资产度量事件（INV-23）。

**度量重置机制**（INV-11, INV-12）：

当点检保养限界上下文完成包含 `metric_reset=true` 的保养任务后，发布 `MaintenanceCompleted(metric_reset=true, reset_metric_kind, reset_baseline)` 到本上下文。本上下文消费后更新内部基准值，后续发布的度量事件将携带新的 `reset_baseline`：

```
【外部系统:点检保养限界上下文】
        -- 发布 MaintenanceCompleted(metric_reset=true, reset_metric_kind, reset_baseline)
                |
                v
        【聚合:Equipment / Fixture】
        --【命令:重置度量基准 ResetMetricBaseline】
                |
                v
        【事件:度量基准已更新 MetricBaselineReset】
         (asset_id, metric_kind: RUN_HOURS | CYCLES, new_baseline, reset_at, source_task_id)
                |
                v
        后续发布的 EquipmentRunHourAccrued / FixtureCycleAccrued 将携带新的 reset_baseline
```

> **关键**：本上下文**不修改绝对累积值**——`accumulated_run_hours` 和 `accumulated_cycles` 单调递增，永远反映设备 / 工装自入账以来的真实运行总量（INV-09, INV-10）。`reset_baseline` 是一个"快照标记"，由点检保养限界上下文在保养完成时提供，本上下文仅存储并透传（INV-11）。点检保养限界上下文使用 `accumulated_value - reset_baseline` 计算相对于上次重置的偏移量，与保养阈值比较；使用 `accumulated_value` 直接与 `life_limit` 比较判定寿命。偏移量 ≥ 0 由 INV-12 保证。

**翻修清零的特殊处理**（INV-10, INV-12）：

当 `FixtureRefurbished(life_reset=true)` 发生时，本上下文**同步重置** `accumulated_cycles=0` 和 `reset_baseline=0`，保证偏移量 = 累积值 − 基准值 ≥ 0 不变式不被破坏。该事件同时发布到 `eam.asset.metric` 主题，点检保养限界上下文订阅后用新的 `accumulated_cycles` 重新计算寿命消耗——**否则点检保养限界上下文会继续用旧累积值误判寿命超限**。

```
【本上下文内部】FixtureRefurbished(life_reset=true)
        |
        v
  accumulated_cycles := 0
  reset_baseline := 0
        |
        v
  发布 FixtureRefurbished(life_reset=true, reset_baseline=0) 到 eam.asset.metric
        |
        v
【点检保养限界上下文】消费后重置寿命消耗计算基线
```

### 2.5 报废审批子流程

报废不是独立限界上下文，而是**资产生命周期的终态变更**，归属本上下文。但报废可由多个限界上下文发起建议，本上下文统一审批执行。

**报废发起来源**（多个限界上下文）：

- 【来源1:点检保养限界上下文】FixtureLifeExceeded（工装寿命超限）→ 建议报废
- 【来源2:维修限界上下文】ScrapRecommendation（维修不经济 / 无法修复）→ 建议报废
- 【来源3:计量检定限界上下文】CalibrationFailed（检定不合格且无法修复）→ 建议报废
- 【来源4:管理决策】设备工程师 / 管理层直接发起报废申请

```
                           v
        【聚合:ScrapApplication】
        --【命令:提交报废申请 SubmitScrapApplication】
                |
                v
        【事件:报废申请已提交 ScrapApplicationSubmitted】
         (application_id, asset_id, source, source_event_id, reason, applicant)
                |
                v
        【策略:根据来源自动填充审批流】（ScrapApprovalService，INV-04 跨实例校验）
                |
                v
        ┌──── 审批决策 / 撤回 ────┐
        v           v              v
    逐步批准      驳回            撤回
```

**审批流匹配规则**：

| 来源 | 审批流 | 说明 |
|------|--------|------|
| INSPECTION（寿命超限） | 工程主管 → 厂级 | 工装寿命到期，审批较简 |
| REPAIR（维修不经济） | 工程主管 → 财务 → 厂级 | 涉及维修成本 vs 残值比较，需财务确认 |
| CALIBRATION（检定不合格） | 计量管理员 → 工程主管 | 法规相关，计量管理员先确认 |
| ADMINISTRATIVE（管理决策） | 工程主管 → 财务 → 厂级 | 完整审批链 |

> **驳回与撤回语义**：
> - **驳回**：任意审批步骤驳回即结束整个申请（发出 `ScrapApplicationRejected`），不退回上一步——简化流程，避免审批回环（INV-19）。申请人若需重新申请，需重新提交。
> - **撤回**：申请人在审批中可主动撤回（发出 `ScrapApplicationWithdrawn`），释放该资产上的"未决报废申请"约束，允许重新发起。
> - **INV-04 跨聚合保证**："同一资产不可有未决报废申请"是跨实例不变式，由 `ScrapApprovalService` 领域服务在提交时查询读模型 + 数据库唯一索引 (asset_id, status=SUBMITTED) 共同保证，非聚合根自身可校验。

> 🔴 **热点：报废审批是否阻塞维修工单**——维修限界上下文发布 `ScrapRecommendation` 后，维修工单进入 `PENDING_SCRAP_DECISION` 状态。本上下文驳回报废申请后，需通知维修限界上下文（发布 `ScrapApplicationRejected`），维修限界上下文将工单退回 `DIAGNOSED` 重新评估方案（INV-20）。撤回场景同样需通知来源限界上下文。

**可用性不变式**（INV-13）：`AssetAvailabilityChanged.available = true` 当且仅当：
- 状态为 IN_SERVICE（SUSPENDED 状态必为不可用）
- 且无未解除的 InspectionOverdue / MaintenanceOverdue（条件性阻塞）
- 且（若为计量器具）cert_valid_until > now()（INV-15）
- 且无 ACTIVE 的 MalfunctionReported（未 RepairCompleted）
- 且（若为工装）accumulated_cycles 未超过点检保养限界上下文判定的寿命上限

> 🔴 **热点：FixtureLifeExceeded 由谁判定**？——建议由**点检保养限界上下文**根据 `FixtureCycleAccrued` 累积值 + 寿命上限规则判定后发布 `FixtureLifeExceeded` 事件到本上下文；本上下文消费后执行 SuspendAsset。本上下文自身不存"寿命上限"阈值，不做判定（INV-22, INV-23）。

**三种使资产不可用机制的语义澄清**（INV-16, INV-17, INV-25）：

| 机制 | 触发场景 | 状态变更 | 解除方式 | 语义 |
|------|---------|---------|---------|------|
| **SuspendAsset** | 故障 / 检定不合格 / 通信中断 / 寿命超限等"硬阻塞" | IN_SERVICE → SUSPENDED | 需显式 ReinstateAsset | 资产被明确停用，不可再派工（双重不可用，INV-16） |
| **AddBlockingReason** | 资产已 SUSPENDED 后叠加新硬阻塞原因 | 状态保持 SUSPENDED，仅追加 blocking_reasons | 移除该阻塞原因后综合判定是否 Reinstate | 多阻塞维度并存场景（如设备同时通信中断 + 报修 + 检定超期）（INV-26） |
| **MarkAssetUnavailable** | 点检 / 保养超期等"条件性阻塞" | 状态保持 IN_SERVICE，仅 available=false | 超期条件消除后自动恢复 available=true | 资产仍在线但被标记不可派工（INV-17） |

> **设计意图**：SuspendAsset 是"显式停用"——资产因物理或法定原因不可用，需要人工确认恢复；AddBlockingReason 是"叠加阻塞"——资产已停用后又有新阻塞原因，避免重复状态变更；MarkAssetUnavailable 是"条件阻塞"——资产本身可用但运维合规性不满足，补做后自动恢复。三者最终都导致 `AssetAvailabilityChanged(available=false)`，但下游消费方可根据 `blocking_reasons[]` 区分原因类型并采取不同应对策略。
>
> **阻塞原因职责划分**（INV-25）：资产聚合根的 `blocking_reasons` 仅记录"导致状态从 IN_SERVICE 变为 SUSPENDED 的硬阻塞"，与 `availability_state` 联动；`AvailabilityProjection.blocking_reasons` 记录"全部影响可用性的阻塞原因"——既含硬阻塞，也含条件性阻塞。两者由 `AvailabilityRecomputeService` 同步维护，投影定期与资产聚合根的硬阻塞原因对账。
>
> **恢复决策统一走 recomputeAndDecideReinstate**：所有"可能恢复"的事件（检定通过、维修完成、保养完成、点检通过）都不直接调用 ReinstateAsset，而是先 clearBlocking 再 recompute，仅当所有阻塞维度均不成立且当前为 SUSPENDED 时才恢复。避免在仍有其他阻塞原因时错误恢复。
>
> **并发与幂等**（INV-27）：多事件并发到达时，`AvailabilityProjection` 通过乐观锁（asset_id + version）+ 事件去重表保证 `recompute` 幂等；`AssetAvailabilityChanged` 发布前比较新旧状态，无变化则不发布。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **Asset**（抽象：Equipment / Fixture / MeasuringInstrument） | 单件资产从入账到报废 | INV-01 同一资产编码全局唯一；INV-02 资产类型一经确定不可变更；INV-03 `AssetRetired` 后任何事件均拒收（终态）；INV-08 允许从 NEW / IN_SERVICE / SUSPENDED 任意非终态直接 Retire |

| **AvailabilityProjection** | 单件资产的当前可用性 | INV-13 available=true 当且仅当所有阻塞维度均不成立；INV-14 任意影响可用性的事件必须触发重算并幂等发布；INV-15 计量器具可用需 cert_valid_until > now()；INV-16 SuspendAsset 致 SUSPENDED + available=false；INV-17 MarkAssetUnavailable 仅 available=false，状态保持 IN_SERVICE；INV-25 阻塞原因职责划分（资产聚合根仅硬阻塞，投影含全部）；INV-26 已 SUSPENDED 时叠加阻塞不重复状态变更；INV-27 重算幂等 + 乐观锁 + 事件去重表 |
| **FixtureMatchingRule** | 工装 × 适用范围（产品 / 工序 / 设备型号） | 规则变更必须发布 `FixtureMatchingRuleUpdated` 通知制造资源服务 |
| **ScrapApplication** | 单次报废申请 | INV-04 同一资产不可重复提交未决报废申请（**跨实例**，由 `ScrapApprovalService` + 唯一索引保证）；INV-18 审批通过后必须执行 RetireAsset；INV-19 步骤按序推进，任意步骤驳回即结束整个申请；INV-20 驳回需通知来源限界上下文 |

> 跨聚合不变式由谁保证：
> - **INV-04**（同一资产无未决报废申请）→ `ScrapApprovalService` 领域服务查询读模型 + 数据库唯一索引 (asset_id, status=SUBMITTED)
> - **INV-12**（偏移量 = 累积值 − 基准值 ≥ 0）→ 单调递增语义 + 翻修时两者同步清零，由 Fixture 聚合根 `refurbish(life_reset=true)` 行为保证
> - **INV-25 / INV-26 / INV-27**（阻塞原因一致性、叠加阻塞、重算幂等）→ `AvailabilityRecomputeService` 领域服务 + 乐观锁 + 事件去重表
>
> 注：`InspectionPlan` / `MaintenancePlan` / `CalibrationPlan` **不在本上下文**——它们是调度聚合，归点检保养限界上下文 / 计量检定限界上下文。

| FixtureLifeExceeded | 点检保养限界上下文 | 寿命超限冻结 | SuspendAsset(reason=LifeExceeded)（若已 SUSPENDED 则 AddBlockingReason） |

### 本上下文内部策略

| 触发事件 | 策略 | 触发命令 |
|---------|------|----------|
| AssetTransferredOut | 外转后自动停用 | 若 IN_SERVICE 则 SuspendAsset(reason=TransferredOut)；若已 SUSPENDED 则 AddBlockingReason(TRANSFERRED_OUT) |
| 任意可用性影响事件（含上下文外消费后的状态变更） | 重算可用性 | RecomputeAvailability → 发布 AssetAvailabilityChanged（无变化则不发布，幂等，INV-14） |
| MaintenanceCompleted(metric_reset=true) | 重置度量基准 | ResetMetricBaseline → 发布 MetricBaselineReset（INV-11） |
| FixtureRefurbished(life_reset=true) | 翻修清零寿命基准 | 重置 accumulated_cycles=0, reset_baseline=0 + 发布到 eam.asset.metric 通知点检保养限界上下文（INV-10, INV-12） |
| 设备数据接入限界上下文 ChannelEscalated（重连超阈值） | 通信故障停用资产 | SuspendAsset(reason=CommunicationLost)（若已 SUSPENDED 则 AddBlockingReason） |
| 设备数据接入限界上下文 EquipmentAlarmRaised(severity=CRITICAL) | 关键报警停用资产 | SuspendAsset(reason=CriticalAlarm)（若已 SUSPENDED 则 AddBlockingReason） |
| ScrapApplicationApproved | 审批通过执行报废 | RetireAsset → 发布 AssetRetired（INV-18） |
| ScrapApplicationRejected | 审批驳回通知来源限界上下文 | 通知来源限界上下文（如维修限界上下文工单退回）（INV-20） |
| ScrapApplicationWithdrawn | 审批撤回释放约束 | 释放资产报废约束；通知来源限界上下文 |

> 注：`EquipmentRunHourAccrued 达阈值 → GenerateMaintenancePlan` 和 `FixtureCycleAccrued 达阈值 → RaiseFixtureLifeWarning` 这类**调度策略**不在本上下文——由点检保养限界上下文订阅本上下文发布的度量事件后自行判定与触发（INV-23）。

## 与「设备数据接入限界上下文」的协作边界

本上下文与 [设备数据接入上下文.md](设备数据接入上下文.md) 共享 `equipment_id` 但**视角不同**，必须对齐避免事件混淆：

| 维度 | 本上下文（台账限界上下文） | 设备数据接入限界上下文（数据接入限界上下文） |
|------|---------------|---------------------|
| 视角 | 资产管理视角（"它是谁、归谁、能不能用"） | 通信视角（"它的连接通不通、报文进没进来"） |
| 存储 | 累积值：run_hours / accumulated_cycles / cert_valid_until | 原始采集报文、连接状态、通道配置 |
| 判定 | 资产能不能用（综合所有阻塞维度） | 连接通不通、报文是否进来 |
| 事件性质 | 资产生命周期与可用性事实 | 通信与采集事实 |
| 流程所有权 | 资产建档 → 投用 → 停用 → 报废 | 网关绑定 → 协议适配 → 在线监控 → 报警升级 |
| 主链事件 | AssetCommissioned / Suspended / Retired | EquipmentOnline / Offline / Recovered |
| `EquipmentRegistered` | **本上下文为权威源**（资产建档） | 设备数据接入限界上下文订阅，用于绑定网关与协议适配器 |
| `equipment_id` | 由本上下文 `AssetIdentifierAssigned` 颁发 | 设备数据接入限界上下文使用同一 ID，不另行分配 |

**事件流向**：
- **本上下文 → 设备数据接入限界上下文**：`AssetRegistered` / `AssetSpecificationProfileUpdated` / `AssetRetired`（触发通道挂载与卸载）
- **设备数据接入限界上下文 → 本上下文**：`ChannelEscalated`（直接 SuspendAsset(reason=CommunicationLost)）、`EquipmentRunHourAggregated`（基于 Online / Offline 区间聚合后的运行时长，本上下文消费后执行 AccrueRunHour 累积为 `EquipmentRunHourAccrued`）、`EquipmentAlarmRaised(severity=CRITICAL)`（关键报警停用资产）

**工装计数映射规则**：

设备数据接入限界上下文上报的原始采集事件与台账上下文累积度量事件的映射关系：

| 设备数据接入限界上下文原始事件 | 台账上下文累积度量事件 | 映射逻辑 |
|---------------|-------------------|---------|
| StencilUsageIncremented(stencil_id, total_count) | FixtureCycleAccrued | 每次增量 +1，台账上下文自行累加 cycles_delta |
| TestFixtureCycleCounted(cycles_total) | FixtureCycleAccrued | 设备数据接入限界上下文已累计，台账上下文以 cycles_delta = cycles_total - last_known 计算增量 |

> 映射规则：设备数据接入限界上下文发布原始采集事件到 `dc.station.event.raw`，台账上下文订阅后根据映射规则转换为 `FixtureCycleAccrued`。设备数据接入限界上下文**不直接发布** `FixtureCycleAccrued`——累积度量的维护权归台账上下文。

> 🔴 **热点：EquipmentOnline ≠ AssetCommissioned**（INV-21）——一台资产可能 Commissioned 但暂时 Offline（断网 / 停机），也可能 Online 但 Suspended（在维修自检）。**两者必须独立维护**，不可互为充要条件。

## 与点检保养限界上下文、维修限界上下文、计量检定限界上下文的协作边界

这是本上下文与运维限界上下文的职责分界线——**本上下文管"能不能用"，运维限界上下文管"怎么管"**：

| 维度 | 本上下文（台账限界上下文） | 点检保养限界上下文 | 维修限界上下文 | 计量检定限界上下文 |
|------|---------------|-----------|--------|-----------|
| 关心什么 | 资产**能不能用** | 预防性运维**怎么管** | 纠正性运维**怎么修** | 检定活动**怎么管** |
| 存储什么 | 累积值：run_hours / accumulated_cycles / cert_valid_until | 阈值规则：保养周期 / 寿命上限 / 点检周期 | 维修工单 / 诊断记录 / 备件消耗 | 检定规则 / 证书全量 / 检定周期 |
| 谁判定触发 | **不判定**——只发布度量值 | **判定并触发**——订阅度量值，与阈值比较 | **故障事件驱动**——不比较阈值 | **判定**——证书有效期与周期比较 |
| 事件性质 | 度量事实 | 调度信号：Due / Overdue / Completed | 执行结果：Completed / ScrapRecommendation | 证书事实：Issued / Failed / Expired |
| 流程所有权 | 报废审批子流程 | 规则 → 计划 → 任务 → 执行 → 签核 | 报修 → 派工 → 诊断 → 维修 → 验证 → 关单 | 规则 → 计划 → 送检 → 检定 → 证书 → 收回 |

**事件流向**：
- **本上下文 → 运维限界上下文**：`eam.asset.metric` 主题（累积度量变更，供阈值判定）；`eam.asset.lifecycle`（资产生命周期变更）
- **点检保养限界上下文 → 本上下文**：点检 / 保养的**结果事件**（§2.6 消费表），本上下文消费后驱动状态变更
- **维修限界上下文 → 本上下文**：`RepairCompleted`（恢复资产）、`ScrapRecommendation`（触发报废审批）
- **计量检定限界上下文 → 本上下文**：`CalibrationCertificateIssued` / `CalibrationFailed` / `CalibrationExpired` / `CalibrationDispatched`
- **不互相调用的操作**：本上下文不调用运维限界上下文的"生成计划"接口；运维限界上下文不调用本上下文的"停用资产"接口——全部通过事件异步解耦

## 与制造资源服务、生产执行服务的协作边界

- **制造资源服务**仅消费 `eam.asset.specification`：工艺路线引用"能力规格 key / value"和"工装匹配规则"，**不复制**台账实体；本上下文资产的 ID / 编码可作为只读防腐层快照保留，但累积度量、运维记录一律回查本上下文或运维限界上下文。
- **生产执行服务**消费 `eam.asset.availability` + `eam.asset.location`：派工时检查"当前可用 + 在目标工位"；**不直接修改**资产状态——派工占用通过自身的"工单 × 资产占用"模型记录，资产状态变更必须发命令回本上下文（INV-24）。
- **资产报废、外转**等强生命周期变更只能从本上下文发起，避免下游服务越权（INV-24）。

> **工装借出维度**（仅 Fixture）：工装借出到工位 A 后，对工位 A 仍可用、对其他工位不可用。但 `available` 是全局布尔值，借出不直接置 available=false。`listAvailableAssetsByStation(station_id)` 查询时由查询服务结合 `issued_to.station_id` 过滤——若 `issued_to.station_id != target_station_id`，则该工装不在候选集。借出与归还通过 `FixtureIssuedToStation` / `FixtureReturnedToStorage` 事件维护，不进入可用性阻塞维度。
