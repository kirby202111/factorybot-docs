# 设备数据接入上下文（Equipment Data Ingestion Bounded Context）

> **限界上下文**：设备数据接入上下文
> **统一语言**：Equipment（设备）, EdgeGateway（边缘网关）, CollectionChannel（采集通道）, DataPoint（数据点）, DataPacket（数据报文）, CollectionSession（采集会话）, RawSignal（原始信号）, ProtocolAdapter（协议适配器）, BackfillCursor（补传游标）, DataQualityFlag（数据质量标记）
> **核心关注**：以"事件风暴"方式刻画从设备产生原始信号到被MES平台可靠接收为止的完整事件链；只关心**采集行为**本身，不解释设备产生的业务语义（如"是否合格"由质量上下文负责，"是否过站"由工序上下文负责）。

---

## 0. 设计原则与边界澄清

数据采集上下文是 PCBA + 整机组装混合型车间MES的**支撑域**，仅对"数据如何从设备汇集到平台"负责。

**域内职责**（本域负责）：
- 设备/工装/控制器的接入、握手、心跳、断连重连
- 原始信号采集、协议解析、报文封装、质量打标
- 边缘缓存、断点续传、重复消除、乱序矫正
- 上行投递到平台并提供消费契约（Topic/Schema）

**域外职责**（本域**不**负责）：
- 业务判定（PASS/FAIL 的工艺含义）→ 质量上下文
- 过站推进、工单关联 → 工序上下文 / 生产执行上下文
- 物料消耗扣减 → 物料上下文
- 设备 OEE 计算、报警业务化处理 → 设备管理上下文 / 设备效率域
- 数据后续分析、SPC、追溯查询 → 数据应用上下文

**为什么这样切**：在我经手过的多家电子厂中，"采集"和"业务解释"耦合是最常见的腐败源——把焊接温度阈值写进采集器、把NG判定写进网关，会导致工艺一改就要改驱动。本域坚持"采集只搬运、不解释"。

## 1. 设备清单与采集形态分类

针对你列出的9类设备，按**接入形态**重新归为四类，决定了事件链的差异：

| 类别 | 设备 | 接入形态 | 数据节奏 | 典型协议 | 事件特征 |
|------|------|---------|----------|----------|----------|
| A. 工业设备（标准协议） | 锡膏印刷机、AOI、波峰焊 | 设备主动上报 / 网关订阅 | 秒级状态 + 单板事件 | SECS/GEM、OPC-UA、Modbus TCP | 连续状态流 + 离散过板事件 |
| B. 标识/打码设备 | 镭雕机 | 网关请求-应答 / 文件回执 | 单板事件 | TCP Socket、串口、共享目录 | "标识落版"是关键事件，绑定SN |
| C. PC型工站（软件输出） | 烧录电脑、测试工装电脑 | Agent / 文件落盘 / HTTP回调 | 单件事件 + 日志文件 | 本地Agent + HTTP/REST、共享盘CSV/XML、串口 | 一次烧录/测试 = 一次会话，附带日志包 |
| D. IoT/控制器型 | 智能电批、线边仓库、老化房控制器 | 长连接持续上报 | 事件驱动 + 周期心跳 | MQTT、WebSocket、PLC over OPC-UA | 拧紧每枪/出入库每笔/老化每时刻 |

## 2. 数据采集事件风暴（核心）

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 域外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐/有争议点

### 2.1 设备接入与会话生命周期（横切，所有设备共用）

```
[参与者:运维/产线工程师] --【命令:登记设备】--> 【聚合:Equipment】
                                                    |
                                                    v
                                      【事件:设备已登记 EquipmentRegistered】
                                                    |
                              【策略:首次登记自动分派网关与协议适配器】
                                                    |
                                                    v
                            【命令:启动采集通道 OpenCollectionChannel】
                                                    |
                                                    v
                          【聚合:CollectionChannel(网关侧)】
                                                    |
                                       --【事件:采集通道已建立】
                                                    |
                                                    v
                                       --【事件:设备已上线 EquipmentOnline】
                                                    |
                                       周期性 --【事件:心跳已收到 HeartbeatReceived】
                                                    |
                                       异常时 --【事件:心跳超时 HeartbeatTimedOut】
                                                          |
                                       【策略:超时则标记下线并启动重连】
                                                          |
                                                          v
                                              【事件:设备已离线 EquipmentOffline】
                                              【事件:重连尝试中 ReconnectAttempted】
                                                          |
                                              成功 --【事件:设备已恢复 EquipmentRecovered】
                                              失败N次 --【事件:通道告警升级 ChannelEscalated】
```

**关键事件清单**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|----------|----------|
| EquipmentRegistered | 工程师在主数据登记设备 | equipment_id、类别、协议、网关绑定 | 触发协议适配器加载 |
| CollectionChannelOpened | 网关与设备首次握手成功 | channel_id、protocol、cipher_suite | 进入心跳监控 |
| EquipmentOnline | 握手通过且能解析首帧 | equipment_id、firmware_ver、program_ver | 标记可投产 |
| HeartbeatReceived | 周期心跳到达 | rtt、序号 | 滚动更新最近活跃时间戳 |
| HeartbeatTimedOut | 超过宽限窗口未到 | last_seen、超时窗口 | 标记 SUSPECT，进入降级 |
| EquipmentOffline | 多次心跳丢失或主动断 | reason、entered_buffer_mode | 切换到边缘缓存模式 |
| ReconnectAttempted | 重连尝试 | attempt_no、backoff | — |
| EquipmentRecovered | 重连成功且补传完成 | gap_window、backfilled_msg_count | 解除 SUSPECT |
| ChannelEscalated | 重连超阈值 | level（产线/车间/工厂） | 通知设备管理上下文 |

### 2.2 通用数据帧采集事件链（设备 → 网关 → 平台）

无论哪类设备，原始信号到达MES都走同一条骨干链路；差异在事件**载荷**而非**链路**。

```
[设备] --【事件:原始信号已发出 RawSignalEmitted】(在网关侧观测)
            |
            v
[聚合:ProtocolAdapter] --【命令:解析帧】
            |
            +-- 解析成功 --> 【事件:数据点已解析 DataPointDecoded】
            |                      |
            |                      v
            |          【聚合:DataPacket(网关侧)】
            |                      |
            |          【命令:打质量标 StampQuality】
            |                      |
            |                      v
            |          【事件:质量已打标 QualityStamped】(GOOD/SUSPECT/BAD/LATE)
            |                      |
            |          【命令:封装报文】
            |                      |
            |                      v
            |          【事件:数据报文已封装 DataPacketSealed】(含 msg_id, source_ts, ingest_ts)
            |                      |
            |                      v
            |          【命令:推入上行队列】
            |                      |
            |                      v
            |          【事件:已写入边缘缓存 EdgeBuffered】
            |                      |
            |     连接正常 --【事件:已上行投递 PacketDispatched】
            |     连接异常 --【事件:进入断连缓冲 BackfillEnqueued】
            |                      |
            |                      v
            |          恢复后 --【事件:补传中 BackfillResuming】
            |                      |
            |                      v
            |                   --【事件:补传完成 BackfillCompleted】(含 cursor)
            |
            +-- 解析失败 --> 【事件:解析失败 FrameDecodeFailed】
                                   |
                       【策略:连续失败N次升级】--> 【事件:协议异常告警 ProtocolAnomalyRaised】
                                   |
                                   v
                              【命令:落入死信队列】
                              【事件:死信入库 DeadLetterStored】

[平台层]
   --【事件:报文已被Broker接收 PacketReceived】
   --【命令:去重】
   --【事件:重复已丢弃 DuplicateDiscarded】 / 【事件:已纳入处理 PacketAccepted】
   --【命令:乱序窗口缓冲】
   --【事件:乱序已矫正 LateArrivalReordered】
   --【命令:落库 + 分发主题】
   --【事件:已持久化 PacketPersisted】
   --【事件:已发布到下游主题 PacketPublished】

[下游消费方:工序上下文/质量上下文/物料上下文...] (域外，本域只暴露主题契约)
```

**热点 🔴**
- 🔴 **source_ts 与 ingest_ts 二选一作为事件时间**：建议以设备 source_ts 为准，但设备时钟漂移>3s 时按 ingest_ts 修正并记录 `clock_skew`，待与设备管理上下文对齐"时钟同步"职责归属。
- 🔴 **死信回放**：解码失败的原始字节是否保留以便事后回放？建议保留 7 天滚动窗口，但存储成本需与运维确认。

### 2.3 设备特定事件（按四类形态展开）

下列每个设备只列**采集层关心**的事件——业务语义事件（如"印刷不良"）属于质量上下文，本域不做。

#### 2.3.1 锡膏印刷机（A 类：工业设备 + 标准协议）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 订阅设备状态 | **PrinterStatusReported** | running/idle/fault, recipe_id | 秒级 / 状态变化触发 |
| 拉取过板循环 | **PrinterCycleObserved** | cycle_index, cycle_duration | 每板 |
| 拉取关键工艺采样 | **PrinterProcessParamSampled** | speed/pressure/angle/demold（不解释好坏） | 每板 |
| 接收钢网计数 | **StencilUsageIncremented** | stencil_id, total_count | 每板 |
| 接收清洁触发 | **StencilCleanTriggered** | clean_type, board_count_since_last | 事件触发 |
| 接收报警帧 | **PrinterAlarmRaised / Cleared** | alarm_code, severity（**不**做SOP分发） | 事件触发 |

#### 2.3.2 AOI（A 类）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 接收检测开始信号 | **AOIInspectionStarted** | sn, program_ver, fixture_id | 每板 |
| 拉取原始判定结果 | **AOIInspectionResultReceived** | raw_verdict, defect_list[]（仅原文，不翻译缺陷字典） | 每板 |
| 接收图像索引 | **AOIImageReferenceCaptured** | image_uri, sha256, retain_until | 每板 |
| 接收人工复核回写 | **AOIFalseCallMarked** | original_defect_id, reviewer_id | 复核后回写 |

> 注：AOI 把判定字典、误判率分析等留给质量上下文；本域只保证"AOI 说了什么"被忠实搬运。

#### 2.3.3 波峰焊（A 类）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 订阅温区与链速 | **WavePreheatTempSampled / WaveSolderTempSampled / WaveConveyorSpeedSampled** | zone, value, source_ts | 秒级 |
| 接收过板事件 | **WaveBoardPassed** | sn（若可读条码）、contact_time | 每板 |
| 接收消耗品状态 | **WaveFluxRefilled / SolderBarReplenished** | qty, operator | 事件触发 |
| 接收报警帧 | **WaveAlarmRaised / Cleared** | alarm_code | 事件触发 |

#### 2.3.4 镭雕机（B 类：标识落版）

镭雕机是 PCBA SN 体系的"源头"，事件采集语义比工艺设备更敏感。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 接收镭雕请求回执 | **MarkingJobAccepted** | job_id, requested_sn, board_position | 每板 |
| 接收镭雕完成 | **SerialNumberMarked** | sn, panel_id, position_xy, marked_at（**这是SN在物理世界存在的瞬间**） | 每板 |
| 接收镭雕失败/重打 | **MarkingFailed / SerialNumberRemarked** | reason, original_sn, new_sn | 异常时 |
| 接收质量自检结果 | **MarkQualityRead** | grade/contrast 原始值 | 每板 |

> 🔴 **热点：SN 池策略**——SN 是平台预分配下发，还是镭雕机本地生成上报？这是与"工序上下文 / 物料标识上下文"的交界，本域只承诺：无论哪种，**SerialNumberMarked** 必须先到平台、再被下游消费，否则后续过站事件会找不到主键。

#### 2.3.5 烧录电脑（C 类：PC 工站，常无标准协议）

烧录电脑通常运行厂商烧录工具（J-Link、UniFlash、自研Loader），输出 CSV/log/数据库行；接入靠 **本地 Agent 或共享盘扫描**。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| Agent 启动 | **ProgrammerAgentOnline** | host_id, station_id | — |
| 触发烧录会话 | **ProgrammingSessionStarted** | sn, firmware_id, firmware_sha256 | 每件 |
| 接收烧录完成回调 | **ProgrammingSessionCompleted** | result, duration, log_uri | 每件 |
| 上传烧录日志 | **ProgrammingLogUploaded** | log_uri, size, retention | 每件 |
| 校验固件指纹 | **FirmwareFingerprintVerified** | expected_sha vs actual_sha | 每件 |

> 设计要点：**固件文件不进采集域主流**——大文件走对象存储，主流只承载 URI + 校验和。`FirmwareFingerprintVerified` 让后续追溯能锁"这块板烧的是哪个二进制"。

#### 2.3.6 测试工装与测试电脑（C 类：FCT/老化前测试）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 工装握手 | **TestFixtureBound** | fixture_id, station_id, calibration_due | 上下电 |
| 测试会话开启 | **TestSessionStarted** | sn, test_program_ver, fixture_id | 每件 |
| 测试项采样 | **TestItemMeasured** | item_name, raw_value, unit（**不带PASS/FAIL语义**，仅带原始测量值与原始判定字段） | 每测试项 |
| 测试会话结束 | **TestSessionCompleted** | raw_verdict, duration, log_uri | 每件 |
| 工装计数 | **TestFixtureCycleCounted** | cycles_total（用于校准提醒，但提醒动作属设备管理上下文） | 每件 |

> 🔴 **热点：测试项粒度**——是按"每测试项一条事件"还是"一次会话打包一条"？建议会话级事件 + 详情走对象存储 JSON；详情打开式查询，不进时序库。

#### 2.3.7 智能电批（D 类：IoT，每枪即事件）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 电批配对 | **TorqueToolBound** | tool_id, station_id, program_id | 上下电 |
| 接收每枪结果 | **FastenerTightened** | sn(或工件id)、seq_no、torque、angle、result（**原始判定**）、timestamp | 每紧固点 |
| 接收程序切换 | **TorqueProgramChanged** | from_program → to_program, operator | 切换时 |
| 接收异常 | **TorqueAnomalyReported** | code, raw_curve_uri | 异常时 |

> 设计要点：电批的扭矩曲线（采样数百点/枪）**不**实时进时序库——主流只携带峰值/最终值，曲线作为附件 URI；这是吞吐与成本的核心折衷。

#### 2.3.8 线边仓库（D 类：物料事件源）

线边仓属于物料上下文，但**它向 MES 投递事件**这件事在采集层完成。本域只保证事件不丢、不重，**不**做物料账目计算。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 仓控系统对接 | **LineSideWarehouseLinked** | warehouse_id, vendor, api_ver | — |
| 接收入库扫码 | **MaterialBinInRegistered** | bin_id, lot_no, qty, operator, scan_ts | 事件触发 |
| 接收出库扫码 | **MaterialBinOutRegistered** | bin_id, target_station, qty | 事件触发 |
| 接收盘点 | **MaterialStockCounted** | snapshot_id, items[]（大事件，单独通道） | 周期/触发 |
| 接收 MSD 倒计时 | **MoistureExposureTimerStarted / Expired** | lot_no, exposure_seconds | 湿敏物料 |

> 🔴 **热点**：扫码枪偶发"一次按键多次扫描"，是否在采集域去重？建议网关侧做 800ms 内同 bin_id+lot_no 去重，并打 `dedup_reason` 标记。

#### 2.3.9 老化房控制器（D 类：长周期连续监控）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 控制器接入 | **AgingChamberOnline** | chamber_id, channels, capacity | — |
| 装载 | **AgingBatchLoaded** | batch_id, sn_list[], load_time | 装载触发 |
| 周期采样 | **AgingEnvironmentSampled** | chamber_temp, set_point, humidity（**仅原始值**） | 分钟级（建议5min） |
| 单件电气量 | **AgingDeviceMeasured** | sn, channel, voltage, current（若有） | 分钟级 |
| 失效事件 | **AgingFailureSignalled** | sn, fail_code（原文）, signalled_at | 事件触发 |
| 卸载 | **AgingBatchUnloaded** | batch_id, total_duration | 卸载触发 |

> 设计要点：老化数据极易撑爆时序库——本域**只**承诺原始数据落入冷热分层，是否降采样、聚合，由数据存储策略章节统一规则，不在事件载荷里"提前聚合"。

### 2.4 设备运行时长聚合事件

接入域除了搬运原始信号，还承担一项**轻量聚合**职责：根据设备的 Online/Offline 区间，周期性计算运行时长并发布到台账上下文。这是台账上下文维护 `EquipmentRunHourAccrued` 的数据源头。

```
EquipmentOnline / EquipmentOffline 事件流
        |
        v
【聚合:EquipmentRunHourAggregation】
        -- 按设备分组，累加 Online→Offline 区间时长
        |
        v
【策略:每班/每天周期性发布运行时长】
        |
        v
【事件:设备运行时长已聚合 EquipmentRunHourAggregated】
 (equipment_id, period_start, period_end, run_hours_in_period,
  total_online_segments, source)
        |
        v
发布到 dc.equipment.runtime → 台账上下文订阅后累计为 EquipmentRunHourAccrued
```

**聚合规则**：

| 参数 | 取值 | 说明 |
|------|------|------|
| 聚合周期 | 每班（8h） | 与车间排班对齐，便于跨班次追溯 |
| 在线判定 | EquipmentOnline 时刻 ~ EquipmentOffline 时刻 | 以采集域自身观测的 Online/Offline 为准，不依赖台账上下文状态 |
| 心跳宽限 | 心跳超时窗口内的 SUSPECT 时段仍计为在线 | SUSPECT 表示"可能断连但未确认"，保守计为在线 |
| 重连补传 | EquipmentRecovered 回填时段不重复计算 | 补传数据的时间戳已在原始区间内，去重处理 |

> 🔴 **热点：聚合精度**——秒级 Online/Offline 区间累加到小数精度是否足够？建议保留 1 位小数（如 7.5h），与台账上下文 `run_hours` 精度一致。台账上下文收到后执行 AccrueRunHour，将 `run_hours_in_period` 累加到 `accumulated_run_hours` 并发布 `EquipmentRunHourAccrued`。

> 🔴 **热点：工装使用次数的聚合归属**——工装的使用次数（`FixtureCycleAccrued`）**不在本域聚合**。接入域只上报原始采集事件（如 `StencilUsageIncremented`、`TestFixtureCycleCounted`），由台账上下文订阅后自行累加。原因：工装使用次数的权威来源是生产执行服务（派工/过站时上报），本域的采集事件仅作为辅助补充。

## 3. 跨设备共享聚合

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **Equipment** | 单台设备的元数据与在线状态 | 同一 equipment_id 同时只能有一个活跃 CollectionChannel |
| **CollectionChannel** | 网关到设备的一次连接 | channel_id 与 (gateway_id, equipment_id) 一一对应；缓冲区单调递增游标 |
| **DataPacket** | 一帧封装好的上行报文 | msg_id 全局唯一；source_ts ≤ ingest_ts；quality 必须打标 |
| **BackfillCursor** | 每条通道的补传位点 | 单调递增；恢复后必须先补齐再放新 |
| **EquipmentRunHourAggregation** | 单台设备一个班次内的运行时长 | 同一 equipment_id + period 不重复聚合；在线区间不可重叠（去重） |
| **DeadLetterEnvelope** | 解码失败的原始字节 | 保留原始字节 + 失败原因 + retain_until |

## 4. 关键策略（Policy）汇总

| 触发事件 | 策略 | 触发命令 |
|----------|------|----------|
| HeartbeatTimedOut | 心跳超时降级 | 标记 SUSPECT、启动 Reconnect |
| ReconnectAttempted ×N | 升级告警 | 通知设备管理上下文（发布 ChannelEscalated） |
| EquipmentRecovered | 自动补传 | StartBackfill |
| FrameDecodeFailed | 死信落地 | StoreDeadLetter |
| FrameDecodeFailed ×N | 协议异常 | RaiseProtocolAnomaly |
| BackfillCompleted | 解除降级 | ClearSuspect |
| MarkingFailed | 阻塞下游 | 不向工序上下文发布 SerialNumberMarked，避免脏 SN 流入 |
| FirmwareFingerprintVerified=mismatch | 阻塞投递 | 不发布 ProgrammingSessionCompleted，仅发 ProgrammingFingerprintMismatch |
| 每班结束 | 运行时长聚合 | 发布 EquipmentRunHourAggregated 到 dc.equipment.runtime |
| AlarmRaised(severity=CRITICAL) | 关键报警标记 | 在 AlarmRaised 事件中携带 severity=CRITICAL，供台账上下文消费后 SuspendAsset |
