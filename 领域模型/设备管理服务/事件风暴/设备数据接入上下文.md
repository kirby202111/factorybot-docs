# 设备数据接入上下文（Equipment Data Ingestion Bounded Context）

> **限界上下文**：设备数据接入上下文
> **统一语言**：Equipment（设备·采集侧记录）, EdgeGateway（边缘网关）, CollectionChannel（采集通道）, ProtocolAdapter（协议适配器）, DataPoint（数据点）, DataPacket（数据报文）, RawSignal（原始信号）, BackfillCursor（补传游标）, DataQualityFlag（数据质量标记）, EquipmentRunHourAggregation（运行时长聚合）, DeadLetterEnvelope（死信信封）, EquipmentAlarm（设备报警·归一化）
> **核心关注**：以"事件风暴"方式刻画从设备产生原始信号到被 MES 平台可靠接收为止的完整采集链路。本上下文只对**采集行为本身**负责——搬运而不解释；**不负责**设备资产身份与建档（归设备工装台账上下文，下文简称「台账上下文」）、不负责业务判定（PASS/FAIL 归质量上下文）、不负责过站推进（归过点执行上下文）。

---

## 0. 设计原则与边界澄清

设备数据接入上下文是设备管理服务下的**支撑域**，仅对"数据如何从设备汇集到平台"负责。

**上下文内职责**（本上下文负责）：
- 设备 / 控制器的接入、握手、心跳、断连重连（**通信视角**，非资产视角）
- 原始信号采集、协议解析、报文封装、质量打标
- 边缘缓存、断点续传、重复消除、乱序矫正
- 上行投递到平台并提供消费契约（Topic + 关键载荷）
- 基于自身观测的 Online / Offline 区间做**运行时长轻量聚合**（供台账上下文累积）
- 所有设备报警帧的**归一化**（统一为 `EquipmentAlarmRaised / Cleared`）

**上下文外职责**（本上下文**不**负责）：
- 设备资产身份、建档、赋码、退役 → **台账上下文**（本上下文订阅其 `AssetRegistered` / `AssetRetired` 来绑定 / 解绑通道）
- 业务判定（PASS/FAIL 的工艺含义） → 质量上下文
- 过站推进、工单关联 → 过点执行上下文 / 生产执行服务
- 物料消耗扣减、物料账目 → 物料上下文（本上下文只搬运线边仓事件）
- 资产可用性判定、OEE、报警业务化处理 → 台账上下文 / 数据应用上下文
- 工装使用次数累积、寿命判定 → 台账上下文 / 点检保养上下文（本上下文只上报原始采集事件）
- 数据后续分析、SPC、追溯查询 → 数据应用上下文 / 追溯上下文

**为什么不并入台账上下文**：
1. **视角不同**——台账上下文是"资产管理视角"（它是谁、归谁、能不能用），本上下文是"通信视角"（连接通不通、报文进没进来）。`EquipmentOnline ≠ AssetCommissioned`（INV-C2 / 台账 INV-21）：一台资产可能 Commissioned 但暂时 Offline，也可能 Online 但 Suspended。两者必须独立维护，合并必生腐败。
2. **变化频率不同**——协议适配、边缘缓存、断点续传是高频技术演化点，与资产主数据的低频变更解耦才不会互相拖累。
3. **"采集只搬运、不解释"是硬约束**——把焊接温度阈值、NG 判定写进采集器是最常见的腐败源，工艺一改就要改驱动。本上下文坚持纯搬运，业务语义一律留给下游上下文。

> 🔴 **热点：`Equipment` 聚合的语义**——本上下文的 `Equipment` 是"采集侧设备记录"（在线状态、通道绑定、协议画像），**不是资产主数据**。`equipment_id` 由台账上下文 `AssetIdentifierAssigned` 颁发，本上下文不另行分配（INV-C1）。建议在统一语言中始终称"设备·采集侧记录"以示区别，待 SME 确认是否改用 `EquipmentEndpoint` 之类更无歧义的名称。

## 1. 设备清单与采集形态分类

按**接入形态**将车间设备归为四类，决定了事件链的差异。这一节是后续事件链的"主语清单"——没有它，事件就悬空。

| 类别 | 典型设备 | 接入形态 | 数据节奏 | 典型协议 | 事件特征 |
|------|---------|---------|----------|----------|----------|
| A. 工业设备（标准协议） | 锡膏印刷机、AOI、波峰焊 | 设备主动上报 / 网关订阅 | 秒级状态 + 单板事件 | SECS/GEM、OPC-UA、Modbus TCP | 连续状态流 + 离散过板事件 |
| B. 标识 / 打码设备 | 镭雕机 | 网关请求-应答 / 文件回执 | 单板事件 | TCP Socket、串口、共享目录 | "标识落版"是关键事件，绑定 SN |
| C. PC 型工站（软件输出） | 烧录电脑、测试工装电脑 | Agent / 文件落盘 / HTTP 回调 | 单件事件 + 日志文件 | 本地 Agent + HTTP/REST、共享盘 CSV/XML、串口 | 一次烧录 / 测试 = 一次会话，附带日志包 |
| D. IoT / 控制器型 | 智能电批、线边仓库、老化房控制器 | 长连接持续上报 | 事件驱动 + 周期心跳 | MQTT、WebSocket、PLC over OPC-UA | 拧紧每枪 / 出入库每笔 / 老化每时刻 |

> 说明：无论哪类设备，原始信号到达 MES 都走同一条骨干链路（§2.3）；差异在事件**载荷**而非**链路**。资产身份与建档不在本表——那归台账上下文。

## 2. 事件风暴 — 主链

事件风暴使用以下色块语义，文中以 `【】` 标注：

- **【事件】** 已发生的事实（过去时）
- **【命令】** 触发事件的意图（动词原形）
- **【参与者】** 触发命令的人或系统
- **【聚合】** 一致性边界
- **【策略】** 当某事件发生时自动触发某命令
- **【外部系统】** 上下文外协作方
- **【读模型】** 投影出的视图
- **【热点】** 待对齐 / 有争议点，正文用 🔴 前缀

### 2.1 设备接入与会话生命周期（横切，所有设备共用）

本上下文不发布 `AssetRegistered`——资产建档归台账上下文。本上下文**订阅**台账上下文的 `AssetRegistered(asset_kind=Equipment)` 来驱动网关绑定与通道开启。

```
【外部系统:台账上下文】
        -- 发布 AssetRegistered(asset_kind=Equipment, asset_id, model, vendor)
                |
                v
【聚合:Equipment(采集侧)】 --【命令:绑定网关与协议适配器 BindGateway】
                |
                v
【事件:网关已绑定 GatewayBound】(equipment_id, gateway_id, protocol_profile)
                |
                v
【命令:启动采集通道 OpenCollectionChannel】
                |
                v
【聚合:CollectionChannel】 -- 网关与设备首次握手
                |
                +-- 握手成功 --> 【事件:采集通道已建立 CollectionChannelOpened】(channel_id, protocol, cipher_suite)
                |                       |
                |                       v
                |                 【事件:设备已上线 EquipmentOnline】(equipment_id, firmware_ver, program_ver)
                |                       |
                |                 周期 --【事件:心跳已收到 HeartbeatReceived】(rtt, seq_no)
                |                       |
                |                 异常 --【事件:心跳超时 HeartbeatTimedOut】(last_seen, grace_window)
                |                       |
                |                       v
                |                 【策略:超时则标记 SUSPECT 并启动重连】
                |                       |
                |                 成功 --【事件:设备已恢复 EquipmentRecovered】(gap_window, backfilled_msg_count)
                |                 失败N次 --【事件:通道告警升级 ChannelEscalated】(level: 产线/车间/工厂)
                |                       |
                |                       v
                |                 【事件:设备已离线 EquipmentOffline】(reason, entered_buffer_mode)
                |
                +-- 握手失败 --> 【事件:采集通道建立失败 CollectionChannelRejected】(reason)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| GatewayBound | 消费 `AssetRegistered` 后绑定网关 | equipment_id, gateway_id, protocol_profile | 触发 OpenCollectionChannel |
| CollectionChannelOpened | 网关与设备首次握手成功 | channel_id, protocol, cipher_suite | 进入心跳监控 |
| CollectionChannelRejected | 握手失败（协议不符 / 鉴权失败） | equipment_id, reason | 通知平台运维，等待人工介入 |
| EquipmentOnline | 握手通过且能解析首帧 | equipment_id, firmware_ver, program_ver | 发布到 `dc.equipment.lifecycle` |
| HeartbeatReceived | 周期心跳到达 | rtt, seq_no | 滚动更新最近活跃时间戳 |
| HeartbeatTimedOut | 超过宽限窗口未到 | last_seen, grace_window | 标记 SUSPECT，进入降级 |
| EquipmentOffline | 多次心跳丢失或主动断 | reason, entered_buffer_mode | 切换到边缘缓存模式 |
| EquipmentRecovered | 重连成功且补传完成 | gap_window, backfilled_msg_count | 解除 SUSPECT |
| ChannelEscalated | 重连超阈值 | level（产线/车间/工厂） | 发布到 `dc.equipment.lifecycle`，供台账上下文 SuspendAsset |

> 🔴 **热点：`AssetRegistered` 的资产类型过滤**——台账上下文 `AssetRegistered` 的 `asset_kind` 含 Equipment / Fixture / MeasuringInstrument。本上下文只对 `asset_kind=Equipment` 响应；工装 / 计量器具的采集（如测试工装、扭矩扳手）是否也走本上下文？建议工装类采集走"工位事件"通道（§3.6 / §3.7），不绑定长连接通道。待与台账上下文确认 `asset_kind` 枚举是否覆盖"工装采集设备"。

### 2.2 状态流转

#### 2.2.1 Equipment（采集侧）在线状态机

```
                         [UNBOUND]
                            |
              消费 AssetRegistered → BindGateway
                            |
                            v
                         [OPENING]
                            |
                    握手成功(首帧可解析)
                            |
                            v
                 ┌──▶ [ONLINE] ◀─── 心跳恢复 / 重连+补传完成 ───┐
                 │        │                                        │
                 │   心跳超时                                       │
                 │        v                                        │
                 │   [SUSPECT] ───── 重连失败 N 次 ─────▶ [OFFLINE]│
                 │        │                                        │
                 │   心跳恢复                                       │
                 └────────┘                                        │
                                                                   │
        消费 AssetRetired(台账) ── 关闭通道 ──▶ [CLOSED 终态] ◀──────┘
                              (终态—之后任何采集事件均拒收, INV-03)
```

**每条边的可触发性**：

| 边 | 命令 | 前置条件 | 谁来发命令 |
|----|------|---------|-----------|
| UNBOUND → OPENING | BindGateway | 已消费 `AssetRegistered(asset_kind=Equipment)` | `GatewayBindingService`（消费台账事件） |
| OPENING → ONLINE | OpenCollectionChannel | 握手成功且首帧可解析 | `ProtocolAdapter` |
| OPENING → CLOSED | RejectChannel | 握手失败（不可恢复） | `ProtocolAdapter`（人工介入后重试则重新 BindGateway） |
| ONLINE → SUSPECT | MarkSuspect | 心跳超时宽限窗口内未收到 | `HeartbeatMonitor` |
| SUSPECT → ONLINE | ClearSuspect | 心跳恢复 或 重连成功且补传完成 | `ReconnectService` |
| SUSPECT → OFFLINE | MarkOffline | 重连失败达 N 次 | `ReconnectService` |
| OFFLINE → ONLINE | RecoverEquipment | 重连成功且补传完成 | `ReconnectService` |
| 任意非终态 → CLOSED | CloseChannel | 已消费 `AssetRetired`（台账上下文） | `GatewayBindingService`（消费台账事件） |
| → CLOSED 之后 | （任何） | 终态，拒收（INV-03） | — |

> **SUSPECT 语义**：表示"可能断连但未确认"，保守计为在线（运行时长聚合仍计入，§2.4）。`ChannelEscalated` 在 SUSPECT→OFFLINE 转移时发布，台账上下文消费后执行 `SuspendAsset(reason=CommunicationLost)`。

#### 2.2.2 CollectionChannel 连接状态机

```
[CLOSED] --OpenCollectionChannel--> [OPENING] --握手成功--> [ESTABLISHED]
                                              |                  |
                                        握手失败                 心跳超时
                                              v                  v
                                          [CLOSED]          [DEGRADED]
                                                                 |
                                                          心跳恢复 / 重连成功
                                                                 |
                                                                 v
                                                            [ESTABLISHED]
```

> `CollectionChannel` 与 `Equipment` 状态联动但独立维护：一个 `equipment_id` 同时只有一个活跃 `CollectionChannel`（INV-01）。资产退役时通道进入 CLOSED 终态。

### 2.3 通用数据帧采集事件链（设备 → 网关 → 平台）

无论哪类设备，原始信号到达 MES 都走同一条骨干链路。

```
[设备] --【事件:原始信号已发出 RawSignalEmitted】(网关侧观测)
            |
            v
【聚合:ProtocolAdapter】 --【命令:解析帧 DecodeFrame】
            |
            +-- 解析成功 --> 【事件:数据点已解析 DataPointDecoded】
            |                      |
            |                      v
            |          【聚合:DataPacket(网关侧)】
            |                      |
            |          【命令:打质量标 StampQuality】
            |                      |
            |                      v
            |          【事件:质量已打标 QualityStamped】(GOOD/SUSPECT/BAD/LATE)
            |                      |
            |          【命令:封装报文 SealPacket】
            |                      |
            |                      v
            |          【事件:数据报文已封装 DataPacketSealed】(msg_id, source_ts, ingest_ts)
            |                      |
            |          【命令:推入上行队列 EnqueueUplink】
            |                      |
            |                      v
            |          【事件:已写入边缘缓存 EdgeBuffered】
            |                      |
            |     连接正常 --【事件:已上行投递 PacketDispatched】
            |     连接异常 --【事件:进入断连缓冲 BackfillEnqueued】
            |                      |
            |                      v
            |          恢复后 --【事件:补传中 BackfillResuming】
            |                      |
            |                      v
            |                   --【事件:补传完成 BackfillCompleted】(cursor)
            |
            +-- 解析失败 --> 【事件:解析失败 FrameDecodeFailed】
                                   |
                       【策略:连续失败 N 次升级】--> 【事件:协议异常告警 ProtocolAnomalyRaised】
                                   |
                                   v
                              【命令:落入死信 StoreDeadLetter】
                              【事件:死信入库 DeadLetterStored】

[平台层]
   --【事件:报文已被 Broker 接收 PacketReceived】
   --【命令:去重 Deduplicate】
   --【事件:重复已丢弃 DuplicateDiscarded】 / 【事件:已纳入处理 PacketAccepted】
   --【命令:乱序窗口缓冲 BufferReorder】
   --【事件:乱序已矫正 LateArrivalReordered】
   --【命令:落库 + 分发主题 PersistAndPublish】
   --【事件:已持久化 PacketPersisted】
   --【事件:已发布到下游主题 PacketPublished】

[下游消费方:过点执行上下文/质量上下文/物料上下文...] (上下文外，本上下文只暴露主题契约)
```

**关键事件**

| 事件 | 何时发生 | 关键载荷 | 后续策略 |
|------|---------|---------|----------|
| RawSignalEmitted | 网关侧观测到设备发出信号 | equipment_id, frame_ref | 进入解析 |
| DataPointDecoded | 协议解析成功 | data_point_id, raw_value, unit | 打质量标 |
| QualityStamped | 质量标记完成 | quality: GOOD/SUSPECT/BAD/LATE | 封装报文 |
| DataPacketSealed | 报文封装完成 | msg_id, source_ts, ingest_ts | 推入上行队列 |
| EdgeBuffered | 写入边缘缓存 | msg_id, cursor | 等待上行或进入断连缓冲 |
| PacketDispatched | 连接正常时上行投递 | msg_id | 平台去重 |
| BackfillEnqueued | 连接异常时进入断连缓冲 | msg_id, cursor | 恢复后补传 |
| BackfillCompleted | 补传完成 | cursor, backfilled_count | 解除降级 |
| FrameDecodeFailed | 解析失败 | equipment_id, reason | 连续失败 N 次升级 |
| ProtocolAnomalyRaised | 连续解析失败超阈值 | equipment_id, fail_streak | 通知平台运维 |
| DeadLetterStored | 死信落地 | raw_bytes_ref, reason, retain_until | 保留窗口后清理 |
| DuplicateDiscarded | 平台去重命中 | msg_id | 丢弃 |
| PacketPersisted | 落库完成 | msg_id, partition | 分发主题 |
| PacketPublished | 发布到下游主题 | msg_id, topic | 供上下文外消费 |

> 🔴 **热点：`source_ts` 与 `ingest_ts` 二选一作为事件时间**——建议以设备 `source_ts` 为准，但设备时钟漂移 > 3s 时按 `ingest_ts` 修正并记录 `clock_skew`。"时钟同步"职责归属待与台账上下文 / 平台运维对齐。
>
> 🔴 **热点：死信回放**——解码失败的原始字节是否保留以便事后回放？建议保留 7 天滚动窗口，存储成本需与运维确认。

### 2.4 设备运行时长聚合

本上下文除了搬运原始信号，还承担一项**轻量聚合**职责：根据自身观测的 `EquipmentOnline` / `EquipmentOffline` 区间，周期性计算运行时长并发布。这是台账上下文维护 `EquipmentRunHourAccrued` 的数据源头——本上下文发布 `EquipmentRunHourAggregated`，台账上下文订阅后执行 `AccrueRunHour` 累加（不本上下文代劳）。

```
EquipmentOnline / EquipmentOffline 事件流
        |
        v
【聚合:EquipmentRunHourAggregation】
        -- 按设备分组，累加 Online→Offline 区间时长
        |
        v
【策略:每班结束周期性发布运行时长】
        |
        v
【事件:设备运行时长已聚合 EquipmentRunHourAggregated】
 (equipment_id, period_start, period_end, run_hours_in_period,
  total_online_segments, source)
        |
        v
发布到 dc.equipment.runtime → 台账上下文订阅后累计为 EquipmentRunHourAccrued
```

**聚合规则**：

| 参数 | 取值 | 说明 |
|------|------|------|
| 聚合周期 | 每班（8h） | 与车间排班对齐，便于跨班次追溯 |
| 在线判定 | EquipmentOnline 时刻 ~ EquipmentOffline 时刻 | 以本上下文自身观测的 Online/Offline 为准，不依赖台账上下文资产状态（INV-C2） |
| 心跳宽限 | SUSPECT 时段仍计为在线 | SUSPECT 表示"可能断连但未确认"，保守计为在线 |
| 重连补传 | EquipmentRecovered 回填时段不重复计算 | 补传数据的时间戳已在原始区间内，去重处理 |

> 🔴 **热点：聚合精度**——秒级 Online/Offline 区间累加到小数精度是否足够？建议保留 1 位小数（如 7.5h），与台账上下文 `run_hours` 精度一致。
>
> 🔴 **热点：工装使用次数的聚合归属**——工装使用次数（`FixtureCycleAccrued`）**不在本上下文聚合**。本上下文只上报原始采集事件（如 `StencilUsageIncremented`、`TestFixtureCycleCounted`），由台账上下文订阅后自行累加。原因：工装使用次数的权威来源是生产执行服务（派工/过站时上报 `FixtureUsed`），本上下文的采集事件仅作辅助补充。

### 2.5 上下文外事件消费与投影

本上下文被动响应**台账上下文**的资产生命周期事件，用于绑定 / 解绑采集通道。本上下文**只消费、不重发**——订阅后包装为本上下文语义事件（`GatewayBound` / `CloseChannel`），不把 `AssetRegistered` 原样再发一次（INV-C3）。

| 上下文外事件 | 来源限界上下文 | 触发命令 | 状态变更 | 采集影响 |
|---|---|---|---|---|
| AssetRegistered(asset_kind=Equipment) | 台账上下文 (`eam.asset.lifecycle`) | BindGateway + OpenCollectionChannel | UNBOUND → OPENING → ONLINE | 新建设备采集侧记录并开启通道 |
| AssetSpecificationProfileUpdated | 台账上下文 (`eam.asset.specification`) | ReloadProtocolProfile | 状态保持 | 协议参数 / 节点映射重载（不中断通道） |
| AssetRetired | 台账上下文 (`eam.asset.lifecycle`) | CloseChannel | 任意非终态 → CLOSED（终态） | 停止采集，关闭通道，拒收后续事件（INV-03） |

> 🔴 **热点：`AssetSpecificationProfileUpdated` 是否影响采集**——规格画像变更（如温区数、传送宽度）多数不影响协议层。建议仅当画像含协议相关字段时触发 `ReloadProtocolProfile`，其余仅记录。待与台账上下文确认规格 key 集合是否区分"协议相关 / 工艺相关"。

**幂等 / 去重策略**：
- `AssetRegistered` 消费幂等：同一 `asset_id` 重复到达时不重复绑定（INV-01 保证同设备单活跃通道），通过 `equipment_id` 去重表实现。
- `AssetRetired` 幂等：已 CLOSED 状态再次到达直接忽略。
- 所有上下文外事件消费均记录 `source_event_id`，用于跨上下文追溯与幂等判定。

## 3. 设备特定采集事件（按接入形态展开）

下列每个设备只列**采集层关心**的事件——业务语义事件（如"印刷不良"）属于质量上下文，本上下文不做。所有设备报警帧统一归一化为 `EquipmentAlarmRaised` / `EquipmentAlarmCleared`（§3.10），不按设备类型各发一套报警事件。

### 3.1 锡膏印刷机（A 类：工业设备 + 标准协议）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 订阅设备状态 | **PrinterStatusReported** | running/idle/fault, recipe_id | 秒级 / 状态变化触发 |
| 拉取过板循环 | **PrinterCycleObserved** | cycle_index, cycle_duration | 每板 |
| 拉取关键工艺采样 | **PrinterProcessParamSampled** | speed/pressure/angle/demold（不解释好坏） | 每板 |
| 接收钢网计数 | **StencilUsageIncremented** | stencil_id, total_count | 每板 |
| 接收清洁触发 | **StencilCleanTriggered** | clean_type, board_count_since_last | 事件触发 |
| 接收报警帧 | 归一化为 **EquipmentAlarmRaised/Cleared** | alarm_code, severity | 事件触发 |

### 3.2 AOI（A 类）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 接收检测开始信号 | **AOIInspectionStarted** | sn, program_ver, fixture_id | 每板 |
| 拉取原始判定结果 | **AOIInspectionResultReceived** | raw_verdict, defect_list[]（仅原文，不翻译缺陷字典） | 每板 |
| 接收图像索引 | **AOIImageReferenceCaptured** | image_uri, sha256, retain_until | 每板 |
| 接收人工复核回写 | **AOIFalseCallMarked** | original_defect_id, reviewer_id | 复核后回写 |

> 注：AOI 把判定字典、误判率分析等留给质量上下文；本上下文只保证"AOI 说了什么"被忠实搬运。

### 3.3 波峰焊（A 类）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 订阅温区与链速 | **WavePreheatTempSampled / WaveSolderTempSampled / WaveConveyorSpeedSampled** | zone, value, source_ts | 秒级 |
| 接收过板事件 | **WaveBoardPassed** | sn（若可读条码）、contact_time | 每板 |
| 接收消耗品状态 | **WaveFluxRefilled / SolderBarReplenished** | qty, operator | 事件触发 |
| 接收报警帧 | 归一化为 **EquipmentAlarmRaised/Cleared** | alarm_code, severity | 事件触发 |

### 3.4 镭雕机（B 类：标识落版）

镭雕机是 PCBA SN 体系的"源头"，事件采集语义比工艺设备更敏感。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 接收镭雕请求回执 | **MarkingJobAccepted** | job_id, requested_sn, board_position | 每板 |
| 接收镭雕完成 | **SerialNumberMarked** | sn, panel_id, position_xy, marked_at（**这是 SN 在物理世界存在的瞬间**） | 每板 |
| 接收镭雕失败/重打 | **MarkingFailed / SerialNumberRemarked** | reason, original_sn, new_sn | 异常时 |
| 接收质量自检结果 | **MarkQualityRead** | grade/contrast 原始值 | 每板 |

> 🔴 **热点：SN 池策略**——SN 是平台预分配下发，还是镭雕机本地生成上报？这是与过点执行上下文 / 物料标识上下文的交界，本上下文只承诺：无论哪种，`SerialNumberMarked` 必须先到平台、再被下游消费，否则后续过站事件会找不到主键。

### 3.5 烧录电脑（C 类：PC 工站，常无标准协议）

烧录电脑通常运行厂商烧录工具（J-Link、UniFlash、自研 Loader），输出 CSV/log/数据库行；接入靠**本地 Agent 或共享盘扫描**。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| Agent 启动 | **ProgrammerAgentOnline** | host_id, station_id | — |
| 触发烧录会话 | **ProgrammingSessionStarted** | sn, firmware_id, firmware_sha256 | 每件 |
| 接收烧录完成回调 | **ProgrammingSessionCompleted** | result, duration, log_uri | 每件 |
| 上传烧录日志 | **ProgrammingLogUploaded** | log_uri, size, retention | 每件 |
| 校验固件指纹 | **FirmwareFingerprintVerified** | expected_sha vs actual_sha | 每件 |

> 设计要点：**固件文件不进采集主流**——大文件走对象存储，主流只承载 URI + 校验和。`FirmwareFingerprintVerified` 让后续追溯能锁"这块板烧的是哪个二进制"。

### 3.6 测试工装与测试电脑（C 类：FCT/老化前测试）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 工装握手 | **TestFixtureBound** | fixture_id, station_id, calibration_due | 上下电 |
| 测试会话开启 | **TestSessionStarted** | sn, test_program_ver, fixture_id | 每件 |
| 测试项采样 | **TestItemMeasured** | item_name, raw_value, unit（**不带 PASS/FAIL 语义**，仅带原始测量值与原始判定字段） | 每测试项 |
| 测试会话结束 | **TestSessionCompleted** | raw_verdict, duration, log_uri | 每件 |
| 工装计数 | **TestFixtureCycleCounted** | cycles_total（用于校准提醒，但提醒动作属台账/点检保养上下文） | 每件 |

> 🔴 **热点：测试项粒度**——是按"每测试项一条事件"还是"一次会话打包一条"？建议会话级事件 + 详情走对象存储 JSON；详情打开式查询，不进时序库。

### 3.7 智能电批（D 类：IoT，每枪即事件）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 电批配对 | **TorqueToolBound** | tool_id, station_id, program_id | 上下电 |
| 接收每枪结果 | **FastenerTightened** | sn(或工件 id)、seq_no、torque、angle、result（**原始判定**）、timestamp | 每紧固点 |
| 接收程序切换 | **TorqueProgramChanged** | from_program → to_program, operator | 切换时 |
| 接收异常 | **TorqueAnomalyReported** | code, raw_curve_uri | 异常时 |

> 设计要点：电批的扭矩曲线（采样数百点/枪）**不**实时进时序库——主流只携带峰值/最终值，曲线作为附件 URI；这是吞吐与成本的核心折衷。

### 3.8 线边仓库（D 类：物料事件源）

线边仓属于物料上下文，但**它向 MES 投递事件**这件事在采集层完成。本上下文只保证事件不丢、不重，**不**做物料账目计算。

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 仓控系统对接 | **LineSideWarehouseLinked** | warehouse_id, vendor, api_ver | — |
| 接收入库扫码 | **MaterialBinInRegistered** | bin_id, lot_no, qty, operator, scan_ts | 事件触发 |
| 接收出库扫码 | **MaterialBinOutRegistered** | bin_id, target_station, qty | 事件触发 |
| 接收盘点 | **MaterialStockCounted** | snapshot_id, items[]（大事件，单独通道） | 周期/触发 |
| 接收 MSD 倒计时 | **MoistureExposureTimerStarted / Expired** | lot_no, exposure_seconds | 湿敏物料 |

> 🔴 **热点**：扫码枪偶发"一次按键多次扫描"，是否在采集层去重？建议网关侧做 800ms 内同 bin_id+lot_no 去重，并打 `dedup_reason` 标记。

### 3.9 老化房控制器（D 类：长周期连续监控）

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 控制器接入 | **AgingChamberOnline** | chamber_id, channels, capacity | — |
| 装载 | **AgingBatchLoaded** | batch_id, sn_list[], load_time | 装载触发 |
| 周期采样 | **AgingEnvironmentSampled** | chamber_temp, set_point, humidity（**仅原始值**） | 分钟级（建议 5min） |
| 单件电气量 | **AgingDeviceMeasured** | sn, channel, voltage, current（若有） | 分钟级 |

| 卸载 | **AgingBatchUnloaded** | batch_id, total_duration | 卸载触发 |

> 设计要点：老化数据极易撑爆时序库——本上下文**只**承诺原始数据落入冷热分层，是否降采样、聚合由数据存储策略统一规则，不在事件载荷里"提前聚合"。

### 3.10 设备报警归一化

所有设备（A/B/C/D 类）的报警帧统一归一化为一对事件，发布到 `dc.equipment.alarm.raw`，供台账上下文按 `severity=CRITICAL` 消费后 `SuspendAsset`、供安灯上下文消费：

| 命令 | 事件 | 载荷要点 | 节奏 |
|------|------|---------|------|
| 接收设备报警帧 | **EquipmentAlarmRaised** | equipment_id, alarm_code, severity(CRITICAL/WARNING/INFO), source_protocol | 事件触发 |
| 接收报警消除帧 | **EquipmentAlarmCleared** | equipment_id, alarm_code, cleared_at | 事件触发 |

> 🔴 **热点：severity 映射规则**——不同设备协议的报警分级（如 SECS/GEM 的 alarm code、PLC 的 bit 位）如何映射到 CRITICAL/WARNING/INFO？建议本上下文维护一份"设备型号 × alarm_code → severity"映射表（配置驱动），映射规则归工程标准，不在事件载荷里硬编码。待与台账上下文 / 安灯上下文确认 CRITICAL 的判定口径。

## 聚合与不变式

| 聚合 | 一致性边界 | 不变式 |
|------|-----------|--------|
| **Equipment**（采集侧记录） | 单台设备的在线状态与通道绑定 | INV-01 同一 equipment_id 同时只能有一个活跃 CollectionChannel；INV-03 资产退役（CLOSED）后任何采集事件均拒收（终态） |
| **CollectionChannel** | 网关到设备的一次连接 | INV-02 channel_id 与 (gateway_id, equipment_id) 一一对应；缓冲区单调递增游标 |
| **DataPacket** | 一帧封装好的上行报文 | INV-04 msg_id 全局唯一；source_ts ≤ ingest_ts；quality 必须打标 |
| **BackfillCursor** | 每条通道的补传位点 | INV-05 单调递增；恢复后必须先补齐再放新 |
| **EquipmentRunHourAggregation** | 单台设备一个班次内的运行时长 | INV-06 同一 equipment_id + period 不重复聚合；在线区间不可重叠（去重） |
| **DeadLetterEnvelope** | 解码失败的原始字节 | INV-07 保留原始字节 + 失败原因 + retain_until |

> 跨聚合 / 跨上下文不变式由谁保证：
> - **INV-C1**（`equipment_id` 由台账上下文颁发，本上下文不另行分配）→ 跨上下文，由台账上下文 `AssetIdentifierAssigned` 颁发 + 本上下文消费 `AssetRegistered` 绑定时校验，**不由本上下文聚合根自校**
> - **INV-C2**（`EquipmentOnline ≠ AssetCommissioned`，通信视角与资产视角独立维护）→ 跨上下文对齐，由两侧各自独立状态机保证；台账上下文 INV-21 对应
> - **INV-C3**（上下文外事件只消费不重发）→ 由主题前缀隔离（本上下文只发 `dc.*`，订阅 `eam.*`）+ 事件命名约定保证
> - **INV-01**（同设备单活跃通道）→ `GatewayBindingService` 领域服务 + (equipment_id, status=ACTIVE) 唯一索引保证
> - **INV-04**（msg_id 全局唯一）→ 平台层去重表 + 幂等消费保证

## 与「设备工装台账上下文」的协作边界

本上下文与 [设备工装台账上下文.md](设备工装台账上下文.md) 共享 `equipment_id` 但**视角不同**，必须对齐避免事件混淆：

| 维度 | 本上下文（设备数据接入上下文） | 台账上下文 |
|------|---------------|---------------------|
| 视角 | 通信视角（"连接通不通、报文进没进来"） | 资产管理视角（"它是谁、归谁、能不能用"） |
| 存储 | 原始采集报文、连接状态、通道配置、边缘缓存 | 累积值：run_hours / accumulated_cycles / cert_valid_until |
| 判定 | 连接通不通、报文是否进来、数据质量打标 | 资产能不能用（综合所有阻塞维度） |
| 事件性质 | 通信与采集事实 | 资产生命周期与可用性事实 |
| 流程所有权 | 网关绑定 → 协议适配 → 在线监控 → 报警归一化 → 上行投递 | 资产建档 → 投用 → 停用 → 报废 |
| 主链事件 | EquipmentOnline / Offline / Recovered | AssetCommissioned / Suspended / Retired |
| `AssetRegistered` | **订阅**（用于绑定网关与协议适配器） | **权威源**（资产建档） |
| `equipment_id` | 使用台账上下文 `AssetIdentifierAssigned` 颁发的 ID，不另行分配 | 颁发者 |

**事件流向**：
- **台账上下文 → 本上下文**：`AssetRegistered(asset_kind=Equipment)` / `AssetSpecificationProfileUpdated` / `AssetRetired`（触发通道挂载、参数重载、卸载）
- **本上下文 → 台账上下文**：`ChannelEscalated`（台账 `SuspendAsset(reason=CommunicationLost)`）、`EquipmentRunHourAggregated`（台账 `AccrueRunHour` 累积为 `EquipmentRunHourAccrued`）、`EquipmentAlarmRaised(severity=CRITICAL)`（台账 `SuspendAsset(reason=CriticalAlarm)`）

**工装计数映射规则**：

本上下文上报的原始采集事件与台账上下文累积度量事件的映射关系：

| 本上下文原始事件 | 台账上下文累积度量事件 | 映射逻辑 |
|---------------|-------------------|---------|
| StencilUsageIncremented(stencil_id, total_count) | FixtureCycleAccrued | 每次增量 +1，台账上下文自行累加 cycles_delta |
| TestFixtureCycleCounted(cycles_total) | FixtureCycleAccrued | 本上下文已累计，台账上下文以 cycles_delta = cycles_total − last_known 计算增量 |

> 映射规则：本上下文发布原始采集事件到 `dc.station.event.raw`，台账上下文订阅后根据映射规则转换为 `FixtureCycleAccrued`。本上下文**不直接发布** `FixtureCycleAccrued`——累积度量的维护权归台账上下文。

> 🔴 **热点：`EquipmentOnline ≠ AssetCommissioned`**（INV-C2 / 台账 INV-21）——一台资产可能 Commissioned 但暂时 Offline（断网 / 停机），也可能 Online 但 Suspended（在维修自检）。**两者必须独立维护**，不可互为充要条件。

## 与「过点执行上下文」的协作边界

本上下文与 [过点执行上下文](../../生产执行服务/事件风暴/过点执行上下文.md) 是"搬运 → 结构化"的接力关系：

| 维度 | 本上下文（设备数据接入上下文） | 过点执行上下文 |
|------|---------------|---------------------|
| 视角 | 通信视角（"原始信号怎么从设备到平台"） | 执行视角（"过点瞬间怎么判定、怎么记录"） |
| 存储 | 原始采集报文、连接状态、边缘缓存 | 过点记录、TestResult（适配层产出）、本地缓存 |
| 事件性质 | 通信与采集事实（`dc.*`） | 过点执行事实（`mes.*`） |
| 流程所有权 | 网关绑定 → 协议适配 → 在线监控 → 上行投递 | 扫码 → 防错 → 决策 → 记录 → 推进 → 累计 |
| 实时数据交互 | 提供实时采集数据查询入口 | 过点时按需 REST 查询设备实时数据（≤200ms，关键工序） |

**事件流向**：
- **本上下文 → 过点执行上下文**：`dc.station.event.raw`（会话级采集事件，过点适配层可消费结构化为 `TestResult`）、过点时按需 REST 查询实时数据点
- **过点执行上下文 → 本上下文**：无事件订阅（过点不反向通知采集层）；过点上下文发布的 `mes.malfunction.reported`（产线视角设备故障）由维修上下文消费，不经本上下文

> **边界澄清**：对于 FTP 文件 / HTTP 回调这类**不经过本上下文**的渠道，过点适配层直接对接数据源；本上下文只负责"经过网关的长连接设备"的采集。两者不重叠，多源数据由过点适配层去重（按 sn + station + testItem 幂等）。

> 🔴 **热点：设备实时数据查询入口**——过点时校验设备实时状态（如当前温度、当前程序版本）REST 查询本上下文，还是台账上下文？建议查**本上下文**（它持有实时采集数据），台账上下文只提供可用性状态。待与设备管理服务确认查询入口。
