## 0. 实现总览

本上下文是设备管理服务的**主数据子域**，以三类资产聚合根（`Equipment` / `Fixture` / `MeasuringInstrument`）保护单件资产全生命周期状态与度量不变式，以 `AvailabilityProjection` 投影聚合根综合各维度阻塞得出可用性结论，以 `ScrapApplication` 管理报废审批流，以 `FixtureMatchingRule` 发布工装匹配关系。

**核心代码目标**：
- 三类资产共享主链行为（`register / commission / suspend / addBlockingReason / removeBlockingReason / reinstate / retire / relocate / changeCustodian / transferOut / updateAttribute / updateClassification / updateSpecificationProfile`），通过抽象基类 `AbstractAsset` 落地公共状态机与阻塞原因维护，特化行为各自下沉子类。
- `AvailabilityRecomputeService` 作为**事件消费中枢**，把域外运维事件翻译为资产状态变更命令 + 投影重算，统一走 `recomputeAndDecideReinstate` 决策恢复。
- 写侧事件同事务写 Outbox，异步发布到 `eam.*` 主题；消费侧经 ACL 翻译、`consumed_event` 去重表保证幂等。

**关键事务**：
- 单资产状态变更：单聚合事务（资产聚合根 + 投影同事务保存，Outbox 同事务写入）。
- 报废审批：`ScrapApplication` 聚合事务；最终审批通过同事务触发 `RetireAsset`（跨聚合由 `ScrapApprovalService` 编排，置于同一应用服务事务内）。

**关键集成点**：
- 订阅 `pm.*` / `calibration.*` / `repair.*` / `mes.*` / `dc.*` 域外主题 → `AssetEventConsumer`。
- 对外发布 `eam.asset.lifecycle / availability / specification / metric / location / scrap`。

### 分层边界

```text
interfaces   → REST Controller / MQ Listener / Scheduler / DTO
application  → ApplicationService / Command / Query / 事务 / 幂等 / Outbox 收集
domain       → AbstractAsset + 三资产聚合根 / 投影聚合根 / 报废申请 / 匹配规则 / 领域服务 / 领域事件 / Repository Port
infrastructure → MyBatis Adapter / MyBatis-Plus 基础表 / Outbox Publisher / ACL Translator / Query 投影
```

```text
interfaces  -> application  -> domain
                  |             ^
                  v             |
            infrastructure ------+
```

**本实现坚持**：
- `domain` 不依赖 Spring / MyBatis / MyBatis-Plus / Kafka / HTTP。聚合根、值对象、领域事件、领域服务均为纯 POJO。
- `application` 开启事务、加载保存聚合、写 Outbox，不写 INV 业务规则。
- `infrastructure` 实现 Repository / Outbox / ACL / Query 端口，技术细节不出现在领域层。
- `interfaces` 只做协议转换与入参校验，不直接调 Mapper 改状态。

## 2. 聚合代码落地

三类资产共享主链行为，采用**抽象基类 + 特化子类**落地，避免 21 个公共方法在三个类中重复。基类持有公共状态（`assetId / identifier / model / vendor / classification / availabilityState / location / custodianOrg / custodianPerson / blockingReasons`）与公共行为及状态机校验；子类持有特化字段（`runHourAccrual / cycleAccrual / certValidity` 等）与特化行为。

### 2.1 AbstractAsset（资产聚合根基类）

**来源**：领域建模 §1.1 / §1.2 / §1.3 的公共部分
**一致性边界**：单件资产从入账到报废的全生命周期状态与度量。
**实现形态**：`domain.model.asset.AbstractAsset`（抽象类，不实例化）

```java
public abstract class AbstractAsset extends AggregateRoot<AssetId> {
    protected AssetId id;
    protected AssetIdentifier identifier;          // 赋码后生成；未赋码前为 null
    protected String model;
    protected String vendor;
    protected AssetClassification classification;
    protected AvailabilityState availabilityState; // NEW / IN_SERVICE / SUSPENDED / RETIRED
    protected AssetLocation location;
    protected String custodianOrg;
    protected String custodianPerson;
    protected List<BlockingReason> blockingReasons = new ArrayList<>();
    protected boolean specificationProfileEstablished = false; // INV-05 投用前置
    protected long version;                         // 乐观锁

    // ── 生命周期主链 ──
    public void assignIdentifier(AssetIdentifier identifier) {
        require(this.identifier == null, "资产已赋码，不可重复赋码");
        this.identifier = identifier;
        registerEvent(new AssetIdentifierAssigned(id, identifier.code(), identifier.codeScheme()));
    }

    public void commission() {
        requireState(AvailabilityState.NEW);
        require(specificationProfileEstablished, "投用前必须已建立能力规格画像"); // INV-05
        this.availabilityState = AvailabilityState.IN_SERVICE;
        registerEvent(new AssetCommissioned(id, clock.now()));
    }

    public void suspend(BlockingReasonType reason) {
        requireState(AvailabilityState.IN_SERVICE);            // 仅 IN_SERVICE 可 suspend
        this.availabilityState = AvailabilityState.SUSPENDED;
        appendBlockingReason(reason);                          // INV-06 reason 由枚举承载
        registerEvent(new AssetSuspended(id, reason, clock.now()));
    }

    public void addBlockingReason(BlockingReasonType reason) {
        requireState(AvailabilityState.SUSPENDED);             // INV-26 仅 SUSPENDED 时叠加
        appendBlockingReason(reason);                          // 不变更状态
    }

    public void removeBlockingReason(BlockingReasonType reasonType) {
        blockingReasons.removeIf(r -> r.reasonType() == reasonType);
        // 不自动状态变更；恢复统一走 reinstate()
    }

    public void reinstate() {
        requireState(AvailabilityState.SUSPENDED);             // INV-07
        this.availabilityState = AvailabilityState.IN_SERVICE;
        registerEvent(new AssetReinstated(id, clock.now()));
    }

    public void retire() {
        require(availabilityState != AvailabilityState.RETIRED, "已报废，拒收命令"); // INV-03
        this.availabilityState = AvailabilityState.RETIRED;
        registerEvent(new AssetRetired(id, clock.now()));
    }

    // ── 位置与归属 ──
    public void relocate(AssetLocation to) { /* 校验非终态 → 变更 → AssetRelocated */ }
    public void changeCustodian(String newCustodianOrg, String newCustodianPerson) { /* ... */ }

    public void transferOut(String targetOrg) {
        require(availabilityState != AvailabilityState.RETIRED, "已报废不可外转");
        if (availabilityState == AvailabilityState.IN_SERVICE) {
            suspend(BlockingReasonType.TRANSFERRED_OUT);       // IN_SERVICE → suspend
        } else if (availabilityState == AvailabilityState.SUSPENDED) {
            addBlockingReason(BlockingReasonType.TRANSFERRED_OUT); // 已 SUSPENDED → 仅追加
        }
        registerEvent(new AssetTransferredOut(id, targetOrg, clock.now()));
    }

    // ── 属性与画像 ──
    public void updateAttribute(Map<String, String> changedFields) { /* AssetAttributeChanged */ }
    public void updateClassification(AssetClassification next, String reason) { /* AssetClassificationUpdated */ }
    public void markSpecificationProfileEstablished() { this.specificationProfileEstablished = true; }
    public void updateSpecificationProfile(List<String> keys, Map<String,String> values) { /* AssetSpecificationProfileUpdated */ }

    // ── 内部工具 ──
    protected void appendBlockingReason(BlockingReasonType type) {
        if (blockingReasons.stream().noneMatch(r -> r.reasonType() == type)) { // 同类型去重
            blockingReasons.add(new BlockingReason(type, null, clock.now()));
        }
    }
    protected void requireState(AvailabilityState expected) {
        require(availabilityState == expected, "非法状态流转：当前=" + availabilityState + "，期望=" + expected);
    }
    // 子类提供特化画像类型与度量累积
}
```

> **suspend vs addBlockingReason**（领域建模 §1.1 备注 / INV-26）：`suspend()` 是状态变更方法，仅在 `IN_SERVICE` 时调用；资产已 `SUSPENDED` 时叠加新阻塞原因走 `addBlockingReason()`，避免重复状态变更抛错。`appendBlockingReason` 内部对同类型去重，保证阻塞原因集合语义。

### 2.3 Fixture（工装夹具）

**来源**：领域建模 §1.2
**实现形态**：`domain.model.asset.Fixture`

```java
public final class Fixture extends AbstractAsset {
    private IssuanceRecord issuedTo;               // 当前借出记录（可空=在库）
    private CycleAccrual cycleAccrual;             // 使用次数累积

    public void issueToStation(String stationId, String issuedTo, LocalDateTime expectedReturn) {
        require(availabilityState == AvailabilityState.IN_SERVICE, "仅投用工装可借出");
        require(this.issuedTo == null, "工装已借出，不可重复借出");
        this.issuedTo = new IssuanceRecord(stationId, issuedTo, clock.now(), expectedReturn);
        registerEvent(new FixtureIssuedToStation(id, stationId, issuedTo, expectedReturn));
    }
    public void returnToStorage(ReturnCondition condition) {
        require(this.issuedTo != null, "工装未借出，无需归还");
        this.issuedTo = null;
        registerEvent(new FixtureReturnedToStorage(id, clock.now(), condition));
    }

    public void accrueCycles(int delta, String source) {
        require(availabilityState != AvailabilityState.RETIRED, "已报废拒收度量");
        require(delta >= 0, "使用次数增量不可为负");               // INV-10
        this.cycleAccrual = cycleAccrual.accrue(delta);
        registerEvent(new FixtureCycleAccrued(id, delta,
                cycleAccrual.accumulated(), cycleAccrual.resetBaseline(), source));
    }

    public void refurbish(String log, boolean lifeReset) {
        require(availabilityState != AvailabilityState.RETIRED, "已报废拒收翻修");
        if (lifeReset) {
            // INV-10 / INV-12：同步重置 accumulated_cycles=0 和 reset_baseline=0
            this.cycleAccrual = cycleAccrual.resetLife();
        }
        BigDecimal newBaseline = lifeReset ? BigDecimal.ZERO : null;
        registerEvent(new FixtureRefurbished(id, log, lifeReset, newBaseline));
    }

    public void clean(String method) { /* FixtureCleaned */ }
}
```

> **FixtureRefurbished 与 life_reset**（领域建模 §1.2 / §2.8）：`lifeReset=true` 时同步重置 `accumulated_cycles=0` 且 `reset_baseline=0`，由 `CycleAccrual.resetLife()` 不可变返回新值对象，保证 INV-12 偏移量 ≥ 0。事件载荷携带新的 `reset_baseline=0`，应用服务将其额外发布到 `eam.asset.metric` 主题供点检保养上下文订阅重算寿命消耗。

### 2.5 AvailabilityProjection（可用性投影聚合根）

**来源**：领域建模 §1.4
**一致性边界**：单件资产的当前可用性判定结果。
**实现形态**：`domain.model.availability.AvailabilityProjection`

```java
public final class AvailabilityProjection extends AggregateRoot<AssetId> {
    private AssetId assetId;
    private boolean available;
    private List<BlockingReason> blockingReasons = new ArrayList<>();
    // 八个维度布尔
    private boolean lifecycleBlocked, inspectionOverdue, maintenanceOverdue,
                    calibrationBlocked, malfunctionActive, lifeExceeded,
                    commLost, criticalAlarm;
    private long version;   // INV-27 乐观锁

    public void markUnavailable(BlockingReasonType type) {
        applyBlocking(type, true);          // 设置对应维度 + 追加阻塞原因
        recompute(null);                    // 重算 available
    }
    public void clearBlocking(BlockingReasonType type) {
        applyBlocking(type, false);
        recompute(null);
    }
    public void clearBlockingByCategory(String prefix) {   // CALIBRATION_* 批量清除
        blockingReasons.removeIf(r -> r.reasonType().name().startsWith(prefix));
        calibrationBlocked = false;
        recompute(null);
    }
    public boolean recompute(String triggeringEvent) {
        boolean before = available;
        this.available = !lifecycleBlocked && !inspectionOverdue && !maintenanceOverdue
                      && !calibrationBlocked && !malfunctionActive && !lifeExceeded
                      && !commLost && !criticalAlarm;        // INV-13 合取
        if (available) blockingReasons.clear();
        if (available != before) {                            // INV-14 / INV-27 无变化不发布
            registerEvent(new AssetAvailabilityChanged(assetId, available, List.copyOf(blockingReasons)));
            return true;
        }
        return false;
    }
    // applyBlocking 把 BlockingReasonType 映射到对应维度布尔
}
```

> **投影与资产的 1:1 独立聚合**（领域建模 §1.4 设计说明）：投影独立聚合根，因为可用性重算消费多个域外事件、一致性边界与发布节奏不同于资产主数据。`AvailabilityProjection.blockingReasons` 记录**全部阻塞**（含条件性 `INSPECTION_OVERDUE` / `MAINTENANCE_OVERDUE`），资产聚合根 `blockingReasons` 仅记录**硬阻塞**（INV-25），由 `AvailabilityRecomputeService` 同步维护。

### 2.6 ScrapApplication（报废申请聚合根）

**来源**：领域建模 §1.5
**一致性边界**：单次报废申请从提交到审批完结。
**实现形态**：`domain.model.scrap.ScrapApplication`，内聚 `ApprovalFlow` 实体与 `ApproverStep` 实体。

```java
public final class ScrapApplication extends AggregateRoot<ApplicationId> {
    private ApplicationId id;
    private AssetId assetId;
    private AssetKind assetKind;
    private ScrapSource source;
    private String sourceEventId;
    private String reason;
    private String applicant;
    private ApprovalFlow approvalFlow;     // 聚合内实体
    private ScrapApplicationStatus status; // SUBMITTED / APPROVED / REJECTED
    private LocalDateTime submittedAt, decidedAt;
    private String decidedBy;

    public void submit(AssetId assetId, AssetKind kind, ScrapSource source,
                       String sourceEventId, String reason, String applicant, ApprovalFlow flow) {
        this.status = ScrapApplicationStatus.SUBMITTED;
        this.approvalFlow = flow;
        registerEvent(new ScrapApplicationSubmitted(id, assetId, kind, source, sourceEventId, reason, applicant));
    }

    public boolean approve(String approver) {
        approvalFlow.advanceCurrentStep(approver);              // INV-19 按序推进
        if (approvalFlow.isFinalStepDecided()) {
            this.status = ScrapApplicationStatus.APPROVED;
            this.decidedBy = approver;
            registerEvent(new ScrapApplicationApproved(id, assetId, approver, clock.now(), approvalFlow.allApproverIds()));
            return true;                                        // 通知应用服务触发 RetireAsset（INV-18）
        }
        registerEvent(new ScrapApprovalStepAdvanced(id, assetId, approvalFlow.currentStepOrder(), approver, clock.now()));
        return false;
    }

    public void reject(String approver, String reason) {
        require(status == ScrapApplicationStatus.SUBMITTED, "申请已结束，不可驳回");
        this.status = ScrapApplicationStatus.REJECTED;
        registerEvent(new ScrapApplicationRejected(id, assetId, reason, approver, clock.now(), approvalFlow.currentStepOrder()));
    }

    public void withdraw(String applicant) {
        require(status == ScrapApplicationStatus.SUBMITTED, "仅审批中申请可撤回");
        this.status = ScrapApplicationStatus.REJECTED; // 终态归类，约束释放见 INV-04
        registerEvent(new ScrapApplicationWithdrawn(id, assetId, applicant, clock.now()));
    }
}
```

> `ApprovalFlow` / `ApproverStep` 为聚合内实体，不暴露独立 Repository，生命周期由 `ScrapApplication` 控制。`submit` 仅校验参数，**不查询其他实例**——INV-04 跨实例约束由 `ScrapApprovalService` + 唯一索引保证（领域建模 §1.5 跨聚合约束说明）。

### 2.7 FixtureMatchingRule（工装匹配规则聚合根）

**来源**：领域建模 §1.6

### 4.2 持久化实现

**技术选择**：写模型聚合装配用 **MyBatis**（手写 ResultMap，处理阻塞原因 JSON 数组、累积度量两列、子实体装配）；基础设施表（`outbox_event` / `consumed_event` / `idempotent_command`）与读模型投影表用 **MyBatis-Plus**。

**选择理由**：三类资产聚合根字段差异大且包含 JSON 值对象（`AssetLocation` / `blockingReasons` / `SpecificationProfile`），手写 ResultMap + `PersistenceAssembler` 能精确控制聚合重建；读模型为单表投影与列表查询，MyBatis-Plus 足矣且开发快。

| 领域对象 | DO / 表 | Mapper | 转换器 | 说明 |
|----------|---------|--------|--------|------|
| `Equipment` | `EquipmentDO` / `eam_equipment` | `EquipmentMapper` | `EquipmentPersistenceAssembler` | 主表，含 `version` 乐观锁、`spec_profile` JSON |
| `Fixture` | `FixtureDO` / `eam_fixture` | `FixtureMapper` | `FixturePersistenceAssembler` | 含 `cycle_accrual`、`issuance_record` JSON |
| `MeasuringInstrument` | `MeasuringInstrumentDO` / `eam_measuring_instrument` | `MeasuringInstrumentMapper` | `InstrumentPersistenceAssembler` | 含 `cert_validity` |
| `AvailabilityProjection` | `AvailabilityProjectionDO` / `eam_availability_projection` | `AvailabilityProjectionMapper` | `AvailabilityProjectionAssembler` | 八维度布尔 + `blocking_reasons` JSON + `version` |
| `ScrapApplication` | `ScrapApplicationDO` / `eam_scrap_application` + `eam_approver_step` | `ScrapApplicationMapper` | `ScrapApplicationAssembler` | 主表 + 审批步骤子表，同 Repository 装配 |
| `FixtureMatchingRule` | `FixtureMatchingRuleDO` / `eam_fixture_matching_rule` | `FixtureMatchingRuleMapper` | `MatchingRuleAssembler` | `rule_keys` JSON |

**装配规则**：Repository Adapter `save` 时，由 Assembler 把聚合根拆为 DO/PO，调用 Mapper upsert；`findById` 时按 ResultMap 装配聚合根（含子实体 `ApproverStep`、JSON 值对象反序列化）。**聚合内子实体（`ApproverStep`）不单独暴露 Repository**，由 `ScrapApplicationRepository` 统一保存。

### 4.3 数据库约束对应 INV

| 约束 | 对应 INV | 实现机制 |
|------|----------|----------|
| `uk_asset_identifier_code` on `eam_equipment/eam_fixture/eam_measuring_instrument.code` | INV-01 | 跨三表的编码唯一性由应用层赋码时查读模型 + 各表唯一索引共同保证（赋码为低频操作，跨表查读模型可接受） |
| `uk_scrap_application_asset_open` on `(asset_id) WHERE status = 'SUBMITTED'` | INV-04 | 部分唯一索引兜底跨实例未决申请约束 |

## 5. 应用服务与事务边界

### 5.1 AssetCommandService（资产命令应用服务）

```java
@Service
public class AssetCommandService {
    private final AssetRegistrationService registrationService;     // 领域服务
    private final AvailabilityRecomputeService recomputeService;    // 领域服务
    private final AssetRepositoryResolver repositoryResolver;       // 按 AssetKind 选 Repository
    private final AvailabilityProjectionRepository projectionRepository;
    private final OutboxCollector outboxCollector;

    @Transactional
    public AssetId registerAsset(RegisterAssetCommand cmd) {
        AssetId assetId = registrationService.registerAsset(   // 跨聚合编排：Asset + Projection + 赋码 + 画像
                cmd.assetKind(), cmd.model(), cmd.vendor(), cmd.custodianOrg());
        outboxCollector.collectFrom(registrationService);       // 同事务写 Outbox
        return assetId;
    }

    @Transactional
    public void suspendAsset(SuspendAssetCommand cmd) {
        AbstractAsset asset = loadAsset(cmd.assetId());
        asset.suspend(cmd.reason());                            // 聚合内 INV-06/INV-08
        saveAsset(asset);
        // 同事务更新投影：硬阻塞双重不可用（INV-16）
        AvailabilityProjection p = projectionRepository.findByAssetId(cmd.assetId()).orElseThrow();
        p.markUnavailable(cmd.reason());
        projectionRepository.save(p);
        outboxCollector.collectFrom(asset, p);
    }

    @Transactional
    public void reinstateAsset(AssetId assetId) {
        AbstractAsset asset = loadAsset(assetId);
        asset.reinstate();
        saveAsset(asset);
        AvailabilityProjection p = projectionRepository.findByAssetId(assetId).orElseThrow();
        p.recompute("manual-reinstate");                        // 重算，可能仍因其他维度不可用
        projectionRepository.save(p);
        outboxCollector.collectFrom(asset, p);
    }

    @Transactional
    public void refurbishFixture(RefurbishFixtureCommand cmd) {
        Fixture fixture = fixtureRepository.findById(cmd.assetId()).orElseThrow();
        fixture.refurbish(cmd.log(), cmd.lifeReset());          // INV-10/INV-12
        fixtureRepository.save(fixture);
        // life_reset=true 额外发布到 eam.asset.metric，由 OutboxEventAssembler 标记 topic
        outboxCollector.collectFrom(fixture);
    }

    // retireAsset / relocateAsset / changeCustodian / transferOutAsset / deployEquipmentToStation
    // / accrueRunHours / accrueCycles / issueFixture / returnFixture / updateCertValidity
    // / updateMetricBaseline / updateSpecificationProfile ... 同构：load → 聚合方法 → save → 投影重算 → Outbox
}
```

| 用例 | Command | 事务边界 | 调用领域对象 | 产出事件 | 幂等策略 |
|------|---------|----------|--------------|----------|----------|
| 登记资产 | `RegisterAssetCommand` | 单事务（Asset+Projection+Outbox） | `AssetRegistrationService` | Registered/IdentifierAssigned/SpecProfileEstablished | `command_id` 幂等表 |
| 投用 | `CommissionAssetCommand` | 单聚合事务 | `asset.commission()` | Commissioned | 乐观锁 + 状态机拒非法 |
| 停用 | `SuspendAssetCommand` | 资产+投影同事务 | `asset.suspend()` + `projection.markUnavailable()` | Suspended + AvailabilityChanged | 乐观锁 |
| 恢复 | `ReinstateAssetCommand` | 资产+投影同事务 | `asset.reinstate()` + `projection.recompute()` | Reinstated + AvailabilityChanged | 乐观锁 |
| 退役 | `RetireAssetCommand` | 资产+投影同事务 | `asset.retire()` + `projection.recompute()` | Retired + AvailabilityChanged | 乐观锁 |
| 外转 | `TransferOutAssetCommand` | 资产+投影同事务 | `asset.transferOut()`（内部分支） | TransferredOut (+Suspended) | 乐观锁 |
| 工装翻修 | `RefurbishFixtureCommand` | 单聚合事务 | `fixture.refurbish()` | Refurbished（life_reset 走 metric topic） | `command_id` 幂等表 |
| 累积运行时长 | `AccrueRunHoursCommand` | 单聚合事务 | `equipment.accrueRunHours()` | RunHourAccrued | 源事件 ID 去重表（dc.* 重复上报） |

**边界要求**：
- `@Transactional` 放应用服务，不放领域对象。
- 应用服务只编排 load → 聚合方法 → save → 投影重算 → Outbox；INV 判断下沉聚合根或领域服务。
- `transferOut` 的"IN_SERVICE 则 suspend、已 SUSPENDED 则 addBlockingReason"分支由聚合根方法内部表达（§2.1），应用服务不写 if/else 业务。

## 6. 领域服务实现

| 领域服务 | Java 类 | 使用场景 | 依赖端口 | 为什么不在聚合根内 |
|----------|---------|----------|----------|--------------------|
| `AssetRegistrationService` | `domain.service.AssetRegistrationService` | 资产登记：创建 Asset + 赋码 + 画像 + 初始 Projection | `*Repository` Port、`AssetIdentifierGenerator` Port | 跨聚合创建多个聚合根（Asset + Projection） |
| `AvailabilityRecomputeService` | `domain.service.AvailabilityRecomputeService` | 域外事件消费中枢：翻译为状态变更 + 投影重算 + 恢复决策 | `*Repository` Port、`ProjectionRepository`、`ScrapApprovalService` | 跨聚合（资产+投影）、需查询当前状态决定 suspend vs addBlockingReason（INV-26） |
| `ScrapApprovalService` | `domain.service.ScrapApprovalService` | 报废审批流编排：INV-04 校验、审批流匹配、推进 | `ScrapApplicationRepository`、`ApprovalFlowFactory`、读模型 | INV-04 跨实例约束无法由聚合根自身保证；审批流匹配是策略 |

```java
public class AvailabilityRecomputeService {
    private final AssetRepositoryResolver repositoryResolver;
    private final AvailabilityProjectionRepository projectionRepository;
    private final ScrapApprovalService scrapApprovalService;     // initiateScrapApplication

    public void onInspectionCompleted(AssetId assetId, InspectionVerdict verdict) {
        AbstractAsset asset = loadAsset(assetId);
        AvailabilityProjection p = projectionRepository.findByAssetId(assetId).orElseThrow();
        if (verdict == InspectionVerdict.FAIL) {
            suspendOrAddBlocking(asset, BlockingReasonType.INSPECTION_ANOMALY);  // INV-26
            p.markUnavailable(BlockingReasonType.INSPECTION_ANOMALY);
        } else {
            asset.removeBlockingReason(BlockingReasonType.INSPECTION_ANOMALY);
            p.clearBlocking(BlockingReasonType.INSPECTION_ANOMALY);
        }
        saveAsset(asset);
        recomputeAndDecideReinstate(asset, p);                   // 统一恢复决策
        projectionRepository.save(p);
    }

    // onCalibrationCertificateIssued / onCalibrationFailed / onCalibrationExpired / onCalibrationDispatched
    //   均走 suspendOrAddBlocking + markUnavailable / clearBlockingByCategory + recomputeAndDecideReinstate
    // onRepairCompleted → clearBlocking(UNDER_REPAIR/MALFUNCTION_ACTIVE) + asset.removeBlockingReason + recomputeAndDecideReinstate
    // onScrapRecommendation → 不停用资产，委托 scrapApprovalService.initiateScrapApplication
    // onMaintenanceCompleted → clearBlocking(MAINTENANCE_OVERDUE) + 若 metric_reset 则 asset.updateMetricBaseline + recomputeAndDecideReinstate

    private void suspendOrAddBlocking(AbstractAsset asset, BlockingReasonType type) {
        if (asset.getAvailabilityState() == AvailabilityState.IN_SERVICE) {
            asset.suspend(type);                                 // 硬阻塞 + 状态变更（INV-16）
        } else if (asset.getAvailabilityState() == AvailabilityState.SUSPENDED) {
            asset.addBlockingReason(type);                       // INV-26 仅追加
        }
    }

    private void recomputeAndDecideReinstate(AbstractAsset asset, AvailabilityProjection p) {
        p.recompute(null);                                       // INV-13/INV-14
        if (p.isAvailable() && asset.getAvailabilityState() == AvailabilityState.SUSPENDED) {
            asset.reinstate();                                   // 仅当所有维度均不成立才恢复
        }
    }
}
```

> 领域服务依赖 Repository Port 与其它领域服务端口，**不依赖 Mapper / Kafka Client / REST Client**。`AvailabilityRecomputeService` 不含阈值判定（INV-22/INV-23），阈值归点检保养/计量检定上下文，本服务只接收已判定的结论事件。该服务不持有事务注解——事务由调用它的 `AssetEventConsumer` 应用服务开启。

| `FixtureMatchingRuleDefined` / `FixtureMatchingRuleUpdated` | `domain.event.matching.*` | `FixtureMatchingRule` | rule_id, fixture_asset_id, applies_to, rule_keys | 是（specification） |

> 事件名与领域建模 §5 逐字一致。事件类为不可变 `record`，置于 `domain.event`，不依赖 Jackson 注解（序列化由基础设施层 `OutboxEventAssembler` 用 ObjectMapper 处理）。

### 7.2 Outbox 设计

```text
聚合根 registerEvent(domainEvent)
        ↓
应用服务 save aggregate（同事务）
        ↓
OutboxCollector.collectFrom(aggregate) 读取聚合 pullDomainEvents()
        ↓
OutboxEventAssembler 组装 outbox_event 行（同事务写）
        ↓
OutboxPublisher 定时/异步扫描 status=INIT → 发 Kafka → 标记 PUBLISHED
        ↓
发布失败重试；超过阈值标记 FAILED 并告警
```

`outbox_event` 表（MyBatis-Plus）：

| 字段 | 说明 |
|------|------|
| `event_id` | 全局唯一事件 ID（聚合内 `registerEvent` 时生成） |
| `aggregate_id` | 聚合根 ID |
| `aggregate_type` | EQUIPMENT / FIXTURE / MEASURING_INSTRUMENT / AVAILABILITY_PROJECTION / SCRAP_APPLICATION / FIXTURE_MATCHING_RULE |
| `event_type` | 事件类型（如 `AssetSuspended`） |
| `topic` | 对外主题（`eam.asset.lifecycle` / `availability` / `specification` / `metric` / `location` / `scrap`） |
| `payload` | JSON 载荷 |
| `status` | INIT / PUBLISHED / FAILED |
| `occurred_at` | 发生时间 |
| `published_at` | 发布时间 |
| `retry_count` | 重试次数 |

**Topic 路由规则**（`OutboxEventAssembler`）：
- 生命周期类（Registered/Commissioned/Suspended/Reinstated/Retired/TransferredOut/IdentifierAssigned/AttributeChanged/ClassificationUpdated/SpecProfileEstablished/Updated）→ `eam.asset.lifecycle`
- `AssetAvailabilityChanged` → `eam.asset.availability`
- `FixtureMatchingRuleDefined/Updated` + `AssetSpecificationProfileEstablished/Updated` → `eam.asset.specification`
- `EquipmentRunHourAccrued` / `FixtureCycleAccrued` / `InstrumentCertValidityUpdated` / `MetricBaselineReset` / `FixtureRefurbished(life_reset=true)` → `eam.asset.metric`
- `AssetRelocated` / `EquipmentDeployedToStation` / `EquipmentUndeployed` / `FixtureIssuedToStation` / `FixtureReturnedToStorage` → `eam.asset.location`
- 报废类 5 个 → `eam.asset.scrap`

> **FixtureRefurbished 特例**：`life_reset=true` 时该事件**必须**发布到 `eam.asset.metric`（点检保养上下文订阅重置寿命消耗）；`life_reset=false` 时归 `eam.asset.lifecycle`。由 `OutboxEventAssembler` 根据载荷字段路由。

**取舍说明**：默认不用分布式事务；使用 Outbox 保证数据库状态与消息发布最终一致。`AssetAvailabilityChanged` 在 `recompute` 内部已做新旧状态比较（INV-27），无变化不 `registerEvent`，从源头避免无效发布。

## 10. 异常、幂等与并发控制

| 场景 | 实现策略 | 对应 INV / 风险 |
|------|----------|-----------------|
| 重复登记/翻修命令 | `idempotent_command` 表（`command_id` 唯一索引）+ 应用服务先查后执行 | 防重复执行 |
| 并发修改同一资产聚合 | `version` 乐观锁：`UPDATE ... WHERE id=? AND version=?`，影响 0 行抛 `OptimisticLockException`，应用层可重试或拒绝 | INV-08 状态机、防状态覆盖 |
| 并发重算投影 | `AvailabilityProjection.version` 乐观锁（INV-27）+ 事件去重表；`recompute` 新旧状态比较无变化不发布 | INV-27 幂等发布 |
| 外部事件重复投递 | `consumed_event` 表（`event_id` 唯一），消费前检查，已消费直接 ack | 防重复过账 |
| 设备采集重复/乱序 | `dc.station.event.raw` 以原始序列号去重；`CycleAccrual` / `RunHourAccrual` 单调递增仅累加正向增量 | INV-09/INV-10 |
| 跨实例未决报废申请 | `ScrapApprovalService` 查读模型 + `uk_scrap_application_asset_open` 部分唯一索引兜底 | INV-04 |
| 审批通过 → 退役 跨聚合一致 | `ScrapApplicationService.approveApplication` 与 `retireAsset` 同事务（INV-18） | 防中间态 |
| 证书过期时间驱动 | `CalibrationExpiryScheduler` 每日扫描 `cert_valid_until <= now()` 的计量器具，触发 `onCalibrationExpired`；同时消费 `calibration.expiry` 事件 | INV-15 双重触发防遗漏 |
| Outbox 发布失败 | `OutboxPublisher` 重试 `retry_count`，超阈值标 `FAILED` 并告警；人工介入或对账 | 防事件丢失 |

**MES 特别注意**：设备上报（`dc.*`）、扫码过点（工装计数）、维修/检定事件普遍存在重复、乱序、迟到。本实现以"事件去重表 + 度量单调递增 + 乐观锁 + 新旧状态比较"四重机制覆盖：去重表防重复消费，单调递增防乱序回退，乐观锁防并发覆盖，状态比较防无效发布。

## 暂缓与扩展点

- **暂不引入 Event Sourcing**：仅保留 Outbox + 操作流水（`outbox_event` 即事件日志）。未来如审计要求提升，可从 `outbox_event` 扩展为事件溯源源，聚合重建改为回放，不影响当前聚合状态存储实现。
- **暂不引入 CQRS 独立读库**：读模型先用同库投影表（`eam_asset_view` / `eam_asset_metric_view`）。后续追溯查询/报表压力上来再拆 Elasticsearch / ClickHouse，QueryService 接口不变，仅替换实现。
- **暂不引入 Saga 框架**：报废审批通过 → 退役的跨聚合一致由同事务保证（强一致，INV-18）；若未来审批流异步化或跨服务，再引入 Saga/补偿。
- **`dc.station.event.raw` 工装计数映射规则**：当前按 `StencilUsageIncremented` / `TestFixtureCycleCounted` 两种原始事件映射为 `accrueCycles`；未来若新增采集事件类型，在 `DcEventAcl` 扩展映射，不污染领域层。
- **资产编码跨表全局唯一**（INV-01）：当前依赖应用层查读模型 + 各表唯一索引；若未来赋码频率上升或要求强一致，可引入独立 `eam_asset_identifier` 编码登记表统一约束。
- **审批流配置化**：当前 `ApprovalFlowFactory` 按 `ScrapSource` 硬编码匹配（领域建模 §2.14）；未来若审批流需可配置，抽取为规则配置聚合，本实现 `ScrapApprovalService` 依赖 `ApprovalFlowFactory` 端口，替换实现即可。
