## 0. 建模总览

本域是设备管理服务下的**主数据子域**，是所有实物资产（设备、工装夹具、计量器具）运行态的权威源。核心职责是管理资产的**身份、归属、状态、生命周期**主数据，以及维护**可用性聚合视图**供下游消费。

## 1. 聚合根

### 1.1 Equipment（设备）

**一致性边界**：单台设备从入账到报废的全生命周期状态与度量。

**职责**：管理设备的身份建档、状态流转、位置归属、采集通道绑定、运行时长累积。

```yaml
Equipment:
  # ─── 标识 ───
  asset_id:              AssetId                    # 资产唯一标识（聚合根ID）
  identifier:            AssetIdentifier            # 资产编码（赋码后生成）

  # ─── 基本属性 ───
  model:                 String                     # 设备型号
  vendor:                String                     # 制造商
  classification:        AssetClassification        # 资产分类
  specification_profile: EquipmentSpecificationProfile  # 能力规格画像

  # ─── 状态 ───
  availability_state:    AvailabilityState          # 可用性状态枚举

  # ─── 位置与归属 ───
  location:              AssetLocation              # 当前位置
  custodian_org:         String                     # 责任部门
  custodian_person:      String                     # 责任人

  # ─── 设备特有 ───
  station_id:            StationReference?          # 部署工位（可为空）
  data_channel_id:       ChannelReference?          # 关联采集通道（可为空）

  # ─── 累积度量 ───
  run_hour_accrual:      RunHourAccrual             # 运行时长累积

  # ─── 可用性阻塞原因 ───
  blocking_reasons:      List<BlockingReason>       # 当前阻塞原因列表
```

**行为**：
- `register(asset_kind, model, vendor, custodian_org)` → 发出 `AssetRegistered`
- `assignIdentifier(code, code_scheme)` → 发出 `AssetIdentifierAssigned`
- `commission()` → 发出 `AssetCommissioned`（前置：规格画像已建立）
- `suspend(reason)` → 发出 `AssetSuspended`（前置：当前为 `IN_SERVICE`）
- `addBlockingReason(reason)` → 追加硬阻塞原因（前置：当前为 `SUSPENDED`，仅追加 `blocking_reasons`，不重复状态变更；用于资产已停用后叠加新阻塞场景）
- `removeBlockingReason(reason_type)` → 移除指定类型阻塞原因（不自动状态变更）
- `reinstate()` → 发出 `AssetReinstated`（前置：当前为 SUSPENDED）
- `retire()` → 发出 `AssetRetired`（终态，之后拒收任何命令；**允许从任意非 RETIRED 状态转入**）
- `relocate(to_location)` → 发出 `AssetRelocated`
- `changeCustodian(new_custodian)` → 发出 `AssetCustodianChanged`
- `transferOut(target_org)` → 发出 `AssetTransferredOut`；若当前为 `IN_SERVICE` 则触发 `suspend(TransferredOut)`，若已 `SUSPENDED` 则调用 `addBlockingReason(TRANSFERRED_OUT)`
- `updateAttribute(changed_fields)` → 发出 `AssetAttributeChanged`
- `updateClassification(new_classification, reason)` → 发出 `AssetClassificationUpdated`
- `deployToStation(station_id)` → 发出 `EquipmentDeployedToStation`
- `undeploy(reason)` → 发出 `EquipmentUndeployed`
- `linkDataChannel(channel_id)` → 发出 `EquipmentDataChannelLinked`
- `unlinkDataChannel(reason)` → 发出 `EquipmentDataChannelUnlinked`
- `accrueRunHours(delta, source)` → 发出 `EquipmentRunHourAccrued`
- `updateMetricBaseline(new_baseline, source_task_id)` → 发出 `MetricBaselineReset`
- `updateSpecificationProfile(spec_keys, spec_values)` → 发出 `AssetSpecificationProfileUpdated`

> **suspend 与 addBlockingReason 的关系**：`suspend(reason)` 是状态变更方法（IN_SERVICE → SUSPENDED + 追加原因），仅在 IN_SERVICE 时调用；资产已 SUSPENDED 时叠加新阻塞原因应使用 `addBlockingReason(reason)`，避免重复状态变更报错。这是事件消费场景（如资产已停用后又被报修）的关键设计。

### 1.4 AvailabilityProjection（可用性投影）

**一致性边界**：单件资产的当前可用性判定结果——综合所有阻塞因素后得出"可用/不可用"结论。

**职责**：聚合所有影响可用性的因素，计算并发布可用性变更事件。这是一个**投影聚合**——它不拥有资产本身的数据，而是消费各类事件后重算可用性结论。

```yaml
AvailabilityProjection:
  # ─── 标识 ───
  asset_id:              AssetId                    # 关联的资产ID（聚合根ID）

  # ─── 投影结果 ───
  available:             Boolean                    # 当前是否可用
  blocking_reasons:      List<BlockingReason>       # 当前阻塞原因列表

  # ─── 各维度阻塞状态 ───
  lifecycle_blocked:     Boolean                    # 生命周期状态阻塞（非IN_SERVICE）
  inspection_overdue:    Boolean                    # 点检超期阻塞
  maintenance_overdue:   Boolean                    # 保养超期阻塞
  calibration_blocked:   Boolean                    # 计量阻塞（超期/不合格/送检中）
  malfunction_active:    Boolean                    # 活跃故障未修复
  life_exceeded:         Boolean                    # 寿命超限（工装）
  comm_lost:             Boolean                    # 通信中断（设备）
  critical_alarm:        Boolean                    # 关键报警（设备）
```

**行为**：
- `recompute(triggering_event)` → 综合所有维度重算 available 值，发出 `AssetAvailabilityChanged`
- `markUnavailable(reason)` → 添加阻塞原因，重算可用性
- `clearBlocking(reason_type)` → 移除指定类型的阻塞原因，重算可用性
- `clearBlockingByCategory(reason_type_prefix)` → 按类别前缀批量移除阻塞原因（如 `CALIBRATION_*` 清除 `CALIBRATING/CALIBRATION_FAILED/CALIBRATION_EXPIRED`），重算可用性

> **设计说明**：`AvailabilityProjection` 与三类资产聚合根是**1:1 关系**，但作为独立聚合根，因为：
> 1. 可用性重算需要消费**多个域外事件**，一致性边界不同于资产主数据；
> 2. 可用性变更的发布节奏和下游消费方与资产生命周期事件不同；
> 3. 未来可用性判定逻辑可能独立演进（如增加新的阻塞维度），不污染资产聚合根。

> **阻塞原因职责划分**（INV-25）：
> - **资产聚合根的 `blocking_reasons`**：仅记录"导致状态从 IN_SERVICE 变为 SUSPENDED 的硬阻塞原因"，与 `availability_state` 联动维护，是状态变更的审计证据。
> - **`AvailabilityProjection.blocking_reasons`**：记录"全部影响可用性的阻塞原因"——既包括硬阻塞（与资产聚合根一致），也包括条件性阻塞（如 `INSPECTION_OVERDUE` / `MAINTENANCE_OVERDUE`，资产仍 IN_SERVICE 但 available=false）。
> - **一致性保证**：`AvailabilityRecomputeService` 在处理域外事件时，若触发状态变更（suspend/reinstate），同时操作资产聚合根和投影；若仅触发条件性阻塞（markUnavailable/clearBlocking），只操作投影。投影定期与资产聚合根的硬阻塞原因对账（通过消费生命周期事件）。
> - **证书有效期维度**（仅 MeasuringInstrument）：`calibration_blocked` 的判定依赖 `cert_valid_until > now()`。`now()` 取查询时间。证书过期是时间驱动的，由 `CalibrationExpired` 事件 + 定时扫描双重触发 `recompute`，避免遗漏。

> **工装借出维度**（仅 Fixture）：工装借出到工位 A 后，对工位 A 仍可用、对其他工位不可用。但 `available` 是全局布尔值，借出不直接置 available=false。`listAvailableAssetsByStation(station_id)` 查询时由查询服务结合 `issued_to.station_id` 过滤——若 `issued_to.station_id != target_station_id`，则该工装不在候选集。借出与归还通过 `FixtureIssuedToStation` / `FixtureReturnedToStorage` 事件维护，不进入可用性阻塞维度。

### 1.5 ScrapApplication（报废申请）

**一致性边界**：单次报废申请从提交到审批完结的全流程。

**职责**：管理报废申请的提交、审批流推进、审批决策。

```yaml
ScrapApplication:
  # ─── 标识 ───
  application_id:        ApplicationId              # 申请唯一标识（聚合根ID）

  # ─── 关联资产 ───
  asset_id:              AssetId                    # 报废资产
  asset_kind:            AssetKind                  # 资产类型（设备/工装/计量器具）

  # ─── 来源 ───
  source:                ScrapSource                # 发起来源枚举
  source_event_id:       String?                    # 来源事件ID（如维修工单号）
  reason:                String                     # 报废原因

  # ─── 辅助信息 ───
  residual_value:        Decimal?                   # 资产残值
  applicant:             String                     # 申请人
  attachments:           List<Attachment>           # 附件

  # ─── 审批流 ───
  approval_flow:         ApprovalFlow               # 审批流实例
  status:                ScrapApplicationStatus     # 申请状态

  # ─── 审计 ───
  submitted_at:          DateTime
  decided_at:            DateTime?
  decided_by:            String?
```

**行为**：
- `submit(asset_id, source, reason, applicant)` → 发出 `ScrapApplicationSubmitted`
- `approve(approver)` → 推进审批流当前步骤；若当前步骤为最终步骤，发出 `ScrapApplicationApproved`；否则发出 `ScrapApprovalStepAdvanced`（中间步骤通过，不结束申请）
- `reject(approver, reason)` → 任意步骤驳回即结束整个申请，发出 `ScrapApplicationRejected`（不允许仅驳回单步退回上一步——简化流程，避免回环）
- `withdraw(applicant)` → 申请人在审批中主动撤回，发出 `ScrapApplicationWithdrawn`（释放该资产上的"未决报废申请"约束，允许重新发起）

> **跨聚合约束说明**：INV-04"同一资产不可有未决报废申请"是**跨实例不变式**，无法由 `ScrapApplication` 聚合根自身保证。该约束由 `ScrapApprovalService` 领域服务在 `submitScrapApplication` 时通过查询读模型 + 数据库唯一索引（asset_id + status∈{SUBMITTED, APPROVED_PENDING_RETIRE}）共同保证。聚合根 `submit` 行为仅校验参数合法性，不查询其他实例。

> **多步审批与事件载荷**：多步审批流（如 工程主管 → 财务 → 厂级）的每步审批人记录在 `ApprovalFlow.approvers[].approver_id`。最终 `ScrapApplicationApproved` 事件的 `approved_by` 字段记录**最终步骤审批人**，全部审批历史通过 `ApprovalFlow` 实体审计查询。

### 1.6 FixtureMatchingRule（工装匹配规则）

**一致性边界**：单条工装匹配规则的定义——规定某工装适用于哪些产品/工序/设备型号。

**职责**：发布工装与产品/工序/设备的匹配关系，供制造资源服务工艺设计引用。

```yaml
FixtureMatchingRule:
  # ─── 标识 ───
  rule_id:               RuleId                     # 规则唯一标识（聚合根ID）

  # ─── 关联工装 ───
  fixture_asset_id:      AssetId                    # 关联的工装资产ID

  # ─── 适用范围 ───
  applies_to:            MatchingScope              # 适用范围（产品族/工序/设备型号）

  # ─── 匹配键 ───
  rule_keys:             Map<String, String>        # 匹配键值对（开口规格、Pin数、张力范围等）
```

**行为**：
- `define(fixture_asset_id, applies_to, rule_keys)` → 发出 `FixtureMatchingRuleDefined`
- `update(rule_keys)` → 发出 `FixtureMatchingRuleUpdated`（通知制造资源服务重新加载匹配规则）

## 2. 值对象

### 2.1 AssetId（资产唯一标识）

```yaml
AssetId:
  value: String    # UUID 或数据库序列生成的唯一标识
```

> 不可变。由系统在 `AssetRegistered` 事件时生成。

### 2.2 AssetIdentifier（资产编码）

```yaml
AssetIdentifier:
  code:        String       # 资产编码，如 EQ-202606-00123
  code_scheme: CodeScheme   # 编码方案枚举
```

```yaml
CodeScheme:  # 编码方案
  - SELF_ASSIGNED    # 系统自编
  - VENDOR           # 厂商编码
  - ASSET_TAG        # 资产标签
```

> 不可变。一旦赋码不可变更。编码方案为单一标签前缀 + 类型字段，避免重码冲突。

### 2.3 AssetKind（资产类型）

```yaml
AssetKind:
  - EQUIPMENT              # 设备
  - FIXTURE                # 工装夹具
  - MEASURING_INSTRUMENT   # 计量器具
```

> 不可变枚举。在 `AssetRegistered` 时确定，之后不变。

### 2.4 AssetClassification（资产分类）

```yaml
AssetClassification:
  primary_category:   String    # 一级分类
  secondary_category: String    # 二级分类
```

> 可变——通过 `AssetClassificationUpdated` 事件变更。

### 2.5 AvailabilityState（可用性状态）

```yaml
AvailabilityState:
  - NEW        # 已登记，未投用
  - IN_SERVICE # 投用中
  - SUSPENDED  # 停用
  - RETIRED    # 报废（终态）
```

> 状态机驱动，仅通过领域事件变更。

**状态机**：

```
        NEW ────Retire────────────────────────┐
         │                                    │
    Commission                              │
         │                                    │
         v                                    │
      IN_SERVICE ──Suspend──▶ SUSPENDED ──Retire──▶ RETIRED (终态)
            ▲                   │                              ▲
            └───Reinstate──────┘                              │
                                                               │
                       (任意非 RETIRED 状态均可 Retire)        │
```

**不变式**：
- `NEW → IN_SERVICE`：前置条件为 `AssetSpecificationProfileEstablished`
- `IN_SERVICE → SUSPENDED`：载荷必带 `reason`
- `SUSPENDED → IN_SERVICE`：必须有对应的 SUSPENDED 状态
- `NEW → RETIRED`：允许（资产入库即损坏/丢失，直接报废）
- `SUSPENDED → RETIRED`：允许（故障停用资产直接报废，无需先恢复）
- `IN_SERVICE → RETIRED`：允许（管理决策直接退役）
- `→ RETIRED`：终态，之后任何事件均拒收

### 3.2 AvailabilityRecomputeService（可用性重算服务）

**职责**：消费域外运维事件后，驱动资产状态变更与可用性重算。这是本域最核心的领域服务。

```yaml
AvailabilityRecomputeService:
  方法:
    # ─── 点检保养上下文事件消费 ───
    onInspectionCompleted(asset_id, verdict):
      if verdict == FAIL → asset.suspend(INSPECTION_ANOMALY)  # 若已 SUSPENDED 则 addBlockingReason
                           projection.markUnavailable(INSPECTION_ANOMALY)
      if verdict == PASS → projection.clearBlocking(INSPECTION_ANOMALY)
                           recomputeAndDecideReinstate(asset_id)  # 清除历史点检异常阻塞

    onInspectionOverdue(asset_id):
      projection.markUnavailable(INSPECTION_OVERDUE)  # 条件性阻塞，状态不变

    onMaintenanceOverdue(asset_id):
      projection.markUnavailable(MAINTENANCE_OVERDUE)  # 条件性阻塞，状态不变

    onMaintenanceCompleted(asset_id, metric_reset, reset_metric_kind, reset_baseline):
      projection.clearBlocking(MAINTENANCE_OVERDUE)
      if metric_reset → asset.updateMetricBaseline(reset_baseline, source_task_id)
      recomputeAndDecideReinstate(asset_id)

    # ─── 计量检定上下文事件消费 ───
    onCalibrationCertificateIssued(asset_id, cert_no, valid_until, issued_by):
      asset.updateCertValidity(cert_no, valid_until, issued_by)
      projection.clearBlockingByCategory(CALIBRATION_*)  # 清除 CALIBRATING/CALIBRATION_FAILED/CALIBRATION_EXPIRED
      asset.removeBlockingReason(CALIBRATING/CALIBRATION_FAILED/CALIBRATION_EXPIRED)  # 同步清除资产聚合根硬阻塞
      recomputeAndDecideReinstate(asset_id)  # 综合所有阻塞原因决定是否恢复，不直接 reinstate

    onCalibrationFailed(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(CALIBRATION_FAILED)
      else → asset.addBlockingReason(CALIBRATION_FAILED)  # 已 SUSPENDED 则仅追加
      projection.markUnavailable(CALIBRATION_FAILED)

    onCalibrationExpired(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(CALIBRATION_EXPIRED)
      else → asset.addBlockingReason(CALIBRATION_EXPIRED)
      projection.markUnavailable(CALIBRATION_EXPIRED)

    onCalibrationDispatched(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(CALIBRATING)
      else → asset.addBlockingReason(CALIBRATING)
      projection.markUnavailable(CALIBRATING)

    # ─── 维修上下文事件消费 ───
    onRepairCompleted(asset_id):
      projection.clearBlocking(UNDER_REPAIR / MALFUNCTION_ACTIVE)
      asset.removeBlockingReason(UNDER_REPAIR / MALFUNCTION_ACTIVE)  # 同步清除资产聚合根硬阻塞
      recomputeAndDecideReinstate(asset_id)  # 综合所有阻塞原因判定是否 ReinstateAsset

    onScrapRecommendation(asset_id, source_event_id):
      # 不直接停用资产，而是触发报废审批子流程
      initiateScrapApplication(asset_id, REPAIR, source_event_id)

    # ─── 生产执行服务/维修上下文事件消费 ───
    onMalfunctionReported(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(MALFUNCTION_ACTIVE)
      else → asset.addBlockingReason(MALFUNCTION_ACTIVE)  # 已 SUSPENDED 则仅追加
      projection.markUnavailable(MALFUNCTION_ACTIVE)

    # ─── 接入域事件消费 ───
    onChannelEscalated(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(COMMUNICATION_LOST)
      else → asset.addBlockingReason(COMMUNICATION_LOST)
      projection.markUnavailable(COMMUNICATION_LOST)

    onCriticalAlarmRaised(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(CRITICAL_ALARM)
      else → asset.addBlockingReason(CRITICAL_ALARM)
      projection.markUnavailable(CRITICAL_ALARM)

    # ─── 点检保养上下文寿命事件消费 ───
    onFixtureLifeExceeded(asset_id):
      if asset.state == IN_SERVICE → asset.suspend(LIFE_EXCEEDED)
      else → asset.addBlockingReason(LIFE_EXCEEDED)
      projection.markUnavailable(LIFE_EXCEEDED)

    # ─── 内部辅助 ───
    recomputeAndDecideReinstate(asset_id):
      # 重算可用性，若无任何阻塞维度 → ReinstateAsset（前提：当前为 SUSPENDED）
      projection.recompute(triggering_event)
      if projection.available and asset.state == SUSPENDED → asset.reinstate()

    initiateScrapApplication(asset_id, source, source_event_id):
      # 委托 ScrapApplication 聚合根创建申请
```

> **设计说明**：
> 1. 此服务是本域的**事件消费中枢**，所有域外运维事件都经过此服务翻译为资产状态变更命令。该服务**不包含阈值判定逻辑**——阈值判定归点检保养上下文/计量检定上下文。
> 2. **资产已 SUSPENDED 时的处理**（INV-26）：域外事件到达时若资产已处于 SUSPENDED，不重复调用 `suspend()`（会违反状态机），而是通过 `addBlockingReason()` 追加阻塞原因，并 `markUnavailable()` 更新投影。这是多阻塞维度并存场景（如设备同时通信中断 + 报修 + 检定超期）的关键设计。
> 3. **恢复决策统一走 `recomputeAndDecideReinstate`**：所有"可能恢复"的事件（检定通过、维修完成、保养完成、点检通过）都不直接调用 `asset.reinstate()`，而是先 `clearBlocking` 再 `recompute`，仅当所有阻塞维度均不成立时才恢复。避免在仍有其他阻塞原因时错误恢复。
> 4. **并发与幂等**（INV-27）：多事件并发到达时，`AvailabilityProjection` 通过乐观锁（基于 `asset_id` + `version`）+ 事件去重表保证 `recompute` 幂等。`AssetAvailabilityChanged` 发布前比较新旧状态，无变化则不发布。

```yaml
    approveApplication(application_id, approver):
      # 委托 ScrapApprovalService.approveApplication()
      # 若审批通过 → 执行 RetireAsset
      # 发布事件到 eam.asset.scrap + eam.asset.lifecycle

    rejectApplication(application_id, approver, reason):
      # 委托 ScrapApprovalService.rejectApplication()
      # 发布事件到 eam.asset.scrap（来源域订阅驳回通知）

    withdrawApplication(application_id, applicant):
      # 委托 ScrapApprovalService.withdrawApplication()
      # 发布事件到 eam.asset.scrap（释放资产报废约束）
```

## 6. 不变规则（Invariants）

不变规则是领域模型中**必须始终为真**的业务约束，无论经过何种操作序列。

### 6.1 资产身份不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-01 | 同一资产编码全局唯一 | `AssetIdentifier.code` 在全局范围内不可重复 |
| INV-02 | 资产类型一旦确定不可变更 | `AssetKind` 在 `AssetRegistered` 时确定，之后不可修改 |
| INV-03 | `AssetRetired` 后任何事件均拒收 | 报废是终态，不可逆 |
| INV-04 | 同一资产不可重复提交未决报废申请 | 跨实例不变式，由 `ScrapApprovalService` 查询读模型 + 数据库唯一索引 (asset_id, status=SUBMITTED) 共同保证；非聚合根自身可校验 |

### 6.2 状态流转不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-05 | `AssetCommissioned` 前必须已建立能力规格画像 | `AssetSpecificationProfileEstablished` 是投用的前置条件 |
| INV-06 | `AssetSuspended` 载荷必带 `reason` | 停用必须有明确原因 |
| INV-07 | `AssetReinstated` 必须有对应的 `AssetSuspended` | 只能从 SUSPENDED 状态恢复 |
| INV-08 | 状态流转必须遵循状态机定义 | NEW→IN_SERVICE→SUSPENDED⇄IN_SERVICE→RETIRED；另允许 NEW→RETIRED、SUSPENDED→RETIRED、IN_SERVICE→RETIRED（任意非终态均可直接 Retire） |

### 6.3 累积度量不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-09 | `accumulated_run_hours` 单调递增 | 运行时长绝对累积值永不减小 |
| INV-10 | `accumulated_cycles` 单调递增（翻修清零除外） | 工装使用次数永不减小，除非 `FixtureRefurbished(life_reset=true)` |
| INV-11 | `reset_baseline` 仅由点检保养上下文通过 `MaintenanceCompleted(metric_reset=true)` 或本域 `FixtureRefurbished(life_reset=true)` 驱动更新 | 本域不主动修改基准值；翻修清零时基准同步重置为 0 |
| INV-12 | 偏移量 = 累积值 - 基准值 ≥ 0 | 由单调递增和基准值语义保证；翻修时两者同步清零，维持不变式 |

### 6.4 可用性不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-13 | `AssetAvailabilityChanged.available = true` 当且仅当所有阻塞维度均不成立 | 可用性是所有维度的合取 |
| INV-14 | 任意影响可用性的事件必须触发重算并发布 `AssetAvailabilityChanged` | 可用性变更必须幂等发布 |
| INV-15 | 计量器具可用必须 `cert_valid_until > now()` | 证书过期则不可用；`now()` 取查询时间，由 `CalibrationExpired` 事件 + 定时扫描双重触发 |
| INV-16 | `SuspendAsset` 导致状态变为 SUSPENDED + available=false | 硬阻塞：双重不可用 |
| INV-17 | `MarkAssetUnavailable` 仅导致 available=false，状态保持 IN_SERVICE | 条件阻塞：合规性不满足但资产本身在线 |

### 6.5 报废审批不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-18 | `ScrapApplicationApproved` 后必须执行 `RetireAsset` | 审批通过 → 资产退役，强一致 |
| INV-19 | 审批流步骤必须按顺序推进 | 不可跳步审批；任意步骤驳回即结束整个申请（不退回上一步） |
| INV-20 | 审批驳回后需通知来源域 | 如维修上下文工单退回 DIAGNOSED 重新评估 |

### 6.6 跨域协作不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-21 | `EquipmentOnline ≠ AssetCommissioned` | 通信在线 ≠ 资产投用，两者独立维护 |
| INV-22 | 本域不存储阈值规则 | 保养周期、点检周期、寿命上限、检定周期归点检保养上下文/计量检定上下文 |
| INV-23 | 本域不判定"是否达到阈值" | 阈值判定归点检保养上下文，本域只发布度量值 |
| INV-24 | 资产报废/外转等强生命周期变更只能从本域发起 | 避免下游服务越权 |

### 6.7 一致性保证不变式

| 编号 | 不变式 | 说明 |
|------|--------|------|
| INV-25 | 阻塞原因职责划分明确 | 资产聚合根 `blocking_reasons` 仅记录硬阻塞（与 SUSPENDED 状态联动）；`AvailabilityProjection.blocking_reasons` 记录全部阻塞（含条件性阻塞）。两者通过 `AvailabilityRecomputeService` 同步维护 |
| INV-26 | 资产已 SUSPENDED 时叠加新阻塞不重复状态变更 | 域外事件到达时若资产已 SUSPENDED，调用 `addBlockingReason()` 而非 `suspend()`，仅追加阻塞原因 |
| INV-27 | 可用性重算幂等与并发安全 | `AvailabilityProjection` 通过乐观锁（asset_id + version）+ 事件去重表保证幂等；`AssetAvailabilityChanged` 无状态变化时不发布 |

## 9. 领域服务与应用服务的关系

```
┌──────────────────────────────────────────────────────────────────┐
│                        外部调用入口                               │
│                                                                  │
│  REST API ──▶ AssetCommandService (应用服务)                     │
│               ├─ 单聚合操作：直接委托聚合根方法                    │
│               └─ 跨聚合编排：委托领域服务                         │
│                                                                  │
│  消息消费 ──▶ AssetEventConsumer (应用服务)                      │
│               └─ 委托 AvailabilityRecomputeService (领域服务)     │
│                                                                  │
│  审批操作 ──▶ ScrapApplicationService (应用服务)                 │
│               └─ 委托 ScrapApprovalService (领域服务)             │
│                                                                  │
│  查询请求 ──▶ AssetQueryService (应用服务，读模型)               │
│               └─ 直接查询读模型投影                               │
└──────────────────────────────────────────────────────────────────┘
```

**原则**：
- 应用服务不包含业务逻辑，只做编排和事务边界
- 领域服务封装跨聚合的业务规则
- 聚合根内的业务逻辑由聚合根自身保证
- 不变式由聚合根 + 领域服务共同守护
